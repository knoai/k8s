# 案例研究：电商大促平台工程实践

> 双 11、618、黑五——电商大促是平台工程最极端的压力测试。
> 流量在几分钟内增长 10-100 倍，任何瓶颈都会被无限放大。
> 本案例基于头部电商公司的真实大促实践整理。

## 第二章：弹性架构设计

### 2.1 预热与弹性扩容

```yaml
# 使用 KEDA（Kubernetes Event-driven Autoscaling）
# 基于自定义指标（消息队列长度、订单量）自动扩容

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: order-service-scaler
  namespace: ecommerce
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: order-service
  minReplicaCount: 10
  maxReplicaCount: 500
  triggers:
  # 触发器 1：基于消息队列长度
  - type: kafka
    metadata:
      bootstrapServers: kafka-cluster:9092
      consumerGroup: order-service
      topic: order-requests
      lagThreshold: "100"
  # 触发器 2：基于自定义指标（每秒订单数）
  - type: prometheus
    metadata:
      serverAddress: http://prometheus:9090
      metricName: orders_per_second
      threshold: "1000"
      query: |
        sum(rate(orders_created_total[1m]))
  # 触发器 3：基于时间表（大促期间保持高副本数）
  - type: cron
    metadata:
      timezone: Asia/Shanghai
      start: 0 20 * 11 *    # 11 月每天 20:00 开始
      end: 0 4 * 11 *       # 11 月每天 04:00 结束
      desiredReplicas: "200"
```

### 2.2 热点数据缓存

```
缓存策略：

1. 本地缓存（Caffeine/Guava）
   - 商品基础信息（名称、价格、图片 URL）
   - 缓存时间：5 分钟
   - 命中率：90%+
   - 内存占用：每个 Pod 100MB

2. 分布式缓存（Redis Cluster）
   - 库存信息（实时扣减）
   - 用户购物车
   - 缓存时间：根据业务动态调整
   - 命中率：80%+

3. CDN 缓存
   - 静态资源（图片、CSS、JS）
   - 商品详情页（HTML）
   - 缓存时间：1 小时-24 小时
   - 命中率：95%+

缓存预热：
  - 大促前 1 小时，预加载热点商品数据
  - 使用 Flink 分析历史数据，预测热点商品
  - 预热数据量：Top 10 万商品
```

## 第三章：全链路压测

### 3.1 压测方案

```
压测目标：
  - 验证系统在 10 倍流量下的表现
  - 发现瓶颈点
  - 验证扩容策略

压测工具：
  - 流量生成：自研压测平台 + JMeter
  - 流量镜像：线上真实流量复制到测试环境
  - 数据构造：基于历史订单生成测试数据

压测阶段：
  Phase 1：单系统压测（提前 2 个月）
    - 每个服务独立压测
    - 目标是日常流量的 5 倍
    
  Phase 2：链路压测（提前 1 个月）
    - 核心链路：首页 → 商品 → 购物车 → 下单 → 支付
    - 目标是日常流量的 10 倍
    
  Phase 3：全链路压测（提前 2 周）
    - 所有系统同时压测
    - 目标是日常流量的 20 倍
    - 使用线上影子库（数据写入影子表，不影响真实数据）
```

### 3.2 影子库技术

```
影子库原理：
  - 写操作：同时写入真实库和影子库
  - 读操作：只读真实库
  - 影子库数据不影响业务

实现方式：
  1. SQL 解析（ShardingSphere）
     - 识别压测流量（通过特殊标记）
     - 将压测流量的 SQL 路由到影子库
     
  2. 数据库层面
     - 真实库：order_2024
     - 影子库：order_2024_shadow
     
  3. 消息队列层面
     - 真实 Topic：order-created
     - 影子 Topic：order-created-shadow
     - 消费者过滤影子消息

关键数字：
  - 影子库资源：真实库的 30-50%
  - 压测流量比例：线上流量的 10-20%
  - 压测时间：每天凌晨 1:00-3:00
```

## 第四章：降级与熔断

### 4.1 降级策略

```
降级层级：

Level 1：非核心功能降级
  - 关闭商品推荐（减少 30% 计算量）
  - 关闭用户行为分析（减少 20% 写入量）
  - 关闭日志详情（只记录错误日志）

Level 2：查询功能降级
  - 关闭实时库存查询（显示"有货/无货"而非具体数量）
  - 关闭物流实时追踪（显示"已发货"而非具体位置）
  - 关闭历史订单查询（只显示最近 3 个月）

Level 3：核心功能降级
  - 关闭优惠券计算（显示原价）
  - 关闭组合商品（只卖单品）
  - 关闭部分支付方式（只保留支付宝/微信）

Level 4：限流
  - 新用户注册限流
  - 非核心页面限流
  - 只保证下单和支付可用

降级触发条件：
  - CPU 使用率 > 80% 持续 2 分钟
  - 错误率 > 5% 持续 1 分钟
  - 响应时间 P99 > 2 秒 持续 2 分钟
```

## 第六章：面试核心考点

```
Q: 电商大促时，K8s 集群如何做到分钟级扩容？

A:
   1. 预置镜像：
      - 所有服务的 Docker 镜像提前拉取到节点
      - 使用 ImagePreload DaemonSet
      
   2. 节点池预热：
      - 提前创建 2-5 倍于日常的节点池
      - 节点处于 Ready 状态，等待 Pod 调度
      - 使用 Cluster Autoscaler + 预配置节点组
      
   3. HPA + KEDA：
      - HPA：基于 CPU/内存自动扩容
      - KEDA：基于自定义指标（消息队列长度、QPS）扩容
      - 扩容延迟：30-60 秒
      
   4. 服务预热：
      - Pod 启动后自动执行预热逻辑
      - 加载本地缓存、建立数据库连接池
      - Readiness Probe 通过后才开始接收流量

Q: 什么是影子库？在大促压测中有什么作用？

A:
   影子库是线上数据库的副本，用于压测：
   
   作用：
   1. 在线上环境进行真实流量压测
   2. 压测数据写入影子库，不影响真实业务
   3. 验证系统在真实数据量下的表现
   
   实现：
   - 通过 SQL 解析识别压测流量
   - 压测流量的写操作路由到影子库
   - 读操作仍走真实库（保证数据一致性）
   
   优势：
   - 比测试环境更真实
   - 可以提前发现线上特有的问题
   - 不干扰真实用户

Q: 大促时如何防止缓存击穿和雪崩？

A:
   缓存击穿（单个热点 key 失效）：
   1. 热点 key 永不过期（逻辑过期）
   2. 互斥锁：只有一个线程去加载数据
   3. 多级缓存：本地缓存 + 分布式缓存
   
   缓存雪崩（大量 key 同时失效）：
   1. 过期时间加随机值（避免同时失效）
   2. 缓存预热：大促前预加载所有热点数据
   3. 降级策略：缓存失效时返回默认值
   
   缓存穿透（查询不存在的 key）：
   1. 布隆过滤器：快速判断 key 是否存在
   2. 空值缓存：将不存在的 key 也缓存一段时间
```
