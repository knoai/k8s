## 案例学习方法论

### 如何高效阅读案例

```
阶段 1：背景理解（15 分钟）
  → 了解公司业务模式和用户规模
  → 识别面临的核心技术挑战
  → 理解业务约束（合规、成本、时间）
  
  关键问题：
    - 这家公司是做什么的？赚多少钱？
    - 用户规模多大？增长多快？
    - 技术团队多少人？什么背景？

阶段 2：架构分析（30 分钟）
  → 画出架构图，标注关键组件
  → 理解组件之间的交互关系
  → 识别关键决策点和权衡取舍
  
  关键问题：
    - 为什么选择这个架构？
    - 有哪些替代方案？为什么没选？
    - 架构的优缺点分别是什么？

阶段 3：深入细节（45 分钟）
  → 记录关键技术实现（配置、参数、代码）
  → 记住关键性能数字（P99、QPS、成本）
  → 分析踩坑经验和解决方案
  
  关键问题：
    - 具体的配置参数是什么？
    - 优化前后的数字对比？
    - 如果重来会怎么做不同？

阶段 4：面试准备（30 分钟）
  → 记住 3-5 个关键数字
  → 准备 "如果你是架构师" 的开放性问题
  → 对比不同案例的相似性和差异
  → 准备自己的观点和批判性思考
```

## 面试核心考点

```
Q: "请介绍一个你熟悉的平台工程案例，并分析其关键技术决策。"

A:（回答模板，建议准备 2-3 个案例）
   
   结构化回答（5 分钟）：
   1. 背景（30 秒）：
      "我深入分析过 Netflix 的 Titus 容器平台案例。
      Netflix 是全球最大的流媒体平台，2 亿+ 用户，
      技术栈是 5000+ 微服务，每天部署 1000+ 次。"
   
   2. 挑战（30 秒）：
      "核心挑战是大规模容器调度和高可用交付。
      需要支持每天数千次的无中断部署，
      同时保证 99.99% 的可用性。"
   
   3. 方案（2 分钟）：
      "架构上分为三层：
      - 基础设施层：Titus 容器平台（基于 K8s 定制）
      - 交付层：Spinnaker 持续交付平台
      - 内容层：Open Connect CDN
      
      关键技术决策：
      - 自研 Titus 而非直接用 K8s：因为 Netflix 2015 年开始容器化，当时 K8s 还不成熟
      - 混沌工程：Chaos Monkey 随机终止实例，验证容错能力
      - 红/黑部署：Spinnaker 的双活部署策略，快速回滚"
   
   4. 成果（1 分钟）：
      "量化成果：
      - 部署频率：从周级提升到日级（每天 1000+ 次）
      - 故障恢复时间：从小时级降到分钟级
      - 可用性：99.99%（全年停机 < 1 小时）"
   
   5. 反思（30 秒）：
      "如果重来，可能会在早期引入更多标准化，
      减少后期技术债务。但整体架构方向是正确的。"

Q: "你如何看待不同公司的平台架构差异？"

A:
   1. 业务驱动：
      - 电商关注高并发和弹性（秒杀、大促）
      - 金融关注安全和合规（等保、审计）
      - 游戏关注低延迟（P99 < 50ms）
      - SaaS 关注多租户隔离和数据本地化
   
   2. 规模驱动：
      - 小集群（<50 节点）：简单方案优先，避免过度设计
      - 中集群（50-500 节点）：需要标准化和自动化
      - 大集群（>1000 节点）：需要自研或深度定制
   
   3. 组织驱动：
      - 大厂有专门的平台团队（Platform Engineering）
      - 创业公司需要全栈工程师（DevOps + SRE）
      - 传统行业需要兼顾合规和稳定性
   
   4. 核心原则：
      - 没有最好的架构，只有最适合的架构
      - 架构是演进而非设计的（Evolutionary Architecture）
      - 技术选型要匹配团队能力和业务阶段

Q: "如果让你从零设计一个电商平台的技术架构，你会怎么做？"

A:（结合案例经验）
   1. 基础设施：
      - 多云策略（避免单点故障）
      - K8s + Cilium eBPF（低延迟网络）
      - 混合实例（On-Demand + Spot + Reserved）
   
   2. 平台组件：
      - GitOps：ArgoCD 管理所有 K8s 资源
      - 多租户：Namespace + NetworkPolicy + ResourceQuota
      - 策略：Kyverno 保证安全和合规
      - 可观测性：Prometheus + Loki + Jaeger
   
   3. 业务架构：
      - 微服务 + Service Mesh（Istio）
      - 事件驱动（Kafka）+ 缓存（Redis Cluster）
      - 数据库：分库分表 + 读写分离
   
   4. 大促保障：
      - 弹性伸缩：HPA + Cluster Autoscaler + Karpenter
      - 降级策略：限流、熔断、兜底页面
      - 预热：缓存预热、连接池预热、JVM 预热
      - 演练：压测 + 混沌工程
   
   5. 参考案例：
      - 阿里云双 11 的弹性架构
      - Netflix 的混沌工程实践
      - 电商大促的降级策略
```

## 面试常见问题补充

**Q: 描述一次你处理过的生产故障？**

A: 使用 STAR 法则回答:
- **Situation（背景）**: 什么时间、什么系统、什么症状
- **Task（任务）**: 你的角色和职责
- **Action（行动）**: 具体的排查步骤（按时间线）
- **Result（结果）**: 恢复时间、影响范围、后续改进

示例结构:
```
"去年 3 月，我们的支付服务出现间歇性超时（Situation）。
我是值班 SRE，负责定位和恢复（Task）。
我首先查看了 Prometheus 发现 P99 延迟从 50ms 上升到 2s，
然后查看 Jaeger 追踪发现下游订单服务数据库查询变慢，
进一步查看数据库慢查询日志发现一个新上线的报表查询没有索引，
导致全表扫描锁定订单表（Action）。
我们立即 kill 了慢查询，服务在 5 分钟内恢复，
随后为报表查询添加了合适的索引（Result）。
事后我们建立了慢查询自动告警和索引审查流程。"
```

**Q: 如何从技术故障中提炼平台改进？**

A: 每次故障都是平台优化的机会:
1. **工具改进**: 如果排查花了很长时间 → 增加监控/日志/追踪覆盖
2. **流程改进**: 如果响应慢 → 优化告警分级和值班响应流程
3. **架构改进**: 如果单点故障 → 增加冗余或降级能力
