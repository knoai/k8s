# 案例研究：平台工程失败案例（反面教材）

> 成功案例 instructive，失败案例 transformative。
> 本章分析五个真实发生的平台工程失败案例，每个案例都包含完整的时间线、根因分析、修复过程和量化结果。
> 这些案例来自公开报道、行业访谈和匿名分享。

## 案例 1："K8s 就是平台"——某电商公司

### 背景

- **公司规模**：300 人电商，技术团队 80 人
- **决策**：CTO 在 2023 年决定"全面上 K8s"，要求 3 个月内所有服务迁移
- **执行**：运维团队用 2 周搭建了一个 K8s 集群，直接把 kubeconfig 发给所有开发者

### 失败时间线

```
Month 1: 开发者抱怨 YAML 太复杂
         - 80 个开发者中，只有 20 个能独立写出正确的 Deployment
         - 其余 60 个还在用旧 ECS 部署方式
         - 平台团队收到 "帮我写 YAML" 的 Slack 消息每天 50+
         
Month 2: 生产事故
         - 某开发者在 YAML 里写错了一个标签选择器
         - Service 把流量导到了错误版本的 Pod
         - 支付链路故障 2 小时
         - 直接损失：¥50 万（订单无法完成）
         
         同日：开发者误删了其他团队的 Namespace
         - 没有 RBAC 限制，所有开发者都有 cluster-admin
         - 恢复时间：4 小时（从 etcd 备份恢复）
         
Month 3: 强制迁移截止日期到
         - 只有 30% 的服务完成迁移
         - 已迁移的服务中，40% 的配置是错误的
         
Month 4: 开发者满意度调查 2.1/5
         - "K8s 让部署更麻烦了"
         - Shadow IT 泛滥：开发者自己租云服务器部署
         - 每月非官方云支出 ¥8 万
         
Month 6: 平台团队被解散
         - CTO 离职
         - 新 CTO 决定回退到 ECS + Ansible
         - 前期投入：¥120 万（人力 + 云资源）
         - 全部作废
```

### 根因分析

```
错误 1：直接把 K8s 暴露给开发者
  症状：开发者需要学习 Pod、Deployment、Service、Ingress、ConfigMap
  正确做法：在 K8s 之上构建开发者友好的抽象层
    - 提供 Helm Chart 模板
    - 提供 CI/CD 流水线（GitHub Actions 自动生成 YAML）
    - 提供开发者门户（Backstage）一键部署

错误 2：3 个月强制迁移所有服务
  症状：时间压力下，质量无法保证
  正确做法：
    - 先选 1 个非核心服务（如内部工具）
    - 迁移 + 验证 + 优化，花 3 个月跑通完整流程
    - 再逐步推广到其他服务

错误 3：没有 RBAC 就开放集群
  症状：任何人可以删除任何资源
  正确做法：
    - 每个团队独立的 Namespace
    - RBAC 限制只能操作自己的 Namespace
    - 生产环境修改需要双人复核

错误 4：没有 CI/CD 就手工 kubectl
  症状：配置不版本化，无法回滚
  正确做法：
    - 先建立 GitOps 流水线（ArgoCD）
    - 所有配置存储在 Git
    - 变更通过 PR 审查

错误 5：没有监控就迁移生产
  症状：故障发现时间 30 分钟，定位时间 2 小时
  正确做法：
    - 监控覆盖率 100% 后再切流
    - 包括：Pod 状态、应用指标、业务指标
    - 告警直接发到值班群
```

### 教训

> **"K8s 不是平台，它只是平台的底座。"**
> 
> 平台工程的核心不是引入新技术，而是降低开发者的认知负担。
> 如果开发者需要学习 10 个新概念才能部署一个服务，平台就失败了。

## 案例 2："完美架构综合征"——某金融科技公司

### 背景

- **公司规模**：200 人金融科技，技术团队 60 人
- **平台团队**：5 人（平台负责人是前大厂架构师）
- **决策**：要求"一步到位"，搭建"行业领先的技术平台"

### 架构设计（过度设计）

```
计划中的平台：
├── 3 个 K8s 集群（开发 / 测试 / 生产）
├── 2 个云（阿里云主 + AWS 灾备）
├── Service Mesh（Istio，全链路 mTLS）
├── 自定义调度器（基于 Volcano）
├── 自研 PaaS 层（替代 Backstage）
├── 多租户隔离（vCluster + HNC + Capsule）
├── FinOps 平台（自研成本分摊系统）
├── 自定义 CNI（基于 eBPF）
└── 预计工期：6 个月
```

### 失败时间线

```
Month 3: 自研 PaaS 只完成了 20%
         - 平台负责人坚持"要做得比 Backstage 更好"
         - 开发者在等平台，业务需求被阻塞
         - 业务团队开始投诉："平台团队拖慢了我们"
         
Month 6: 平台延期到 9 个月
         - 只有 Istio 和基础 K8s 完成
         - 业务团队完全失去信心
         
Month 8: Istio 引入后 30% 的服务出现连接问题
         - Envoy sidecar 与某个 Java 框架不兼容
         - 平台团队花了 2 个月排查
         - 发现是 Envoy 1.18 版本的已知 bug
         - 但升级 Envoy 需要重新验证所有服务
         
Month 10: 只有 3 个服务成功上线新平台
         - 其余服务仍在旧 ECS 上运行
         - 运维同时维护两套系统，人力翻倍
         
Month 12: 平台负责人被调岗
         - 新负责人砍掉 70% 功能
         - 保留：单集群 + ArgoCD + Prometheus
         - 3 个月后全部服务迁移完成
         
总成本：
  - 平台建设：¥200 万（5 人 × 12 月）
  - 云资源浪费：¥80 万（多集群空转）
  - 业务延期损失：无法估量
  - 关键员工离职：3 名核心开发者因平台问题离职
```

### 根因分析

```
错误 1：试图一次性建成"完美平台"
  症状：6 个月计划包含 8 个大型子系统
  正确做法：MVP 思维
    - 第一个版本只解决最痛的 1-2 个问题
    - 例："部署需要 2 天" → 先做到 "部署需要 2 小时"
    - 后续版本逐步添加功能

错误 2：自研一切
  症状：自研 PaaS、自研 FinOps、自研 CNI
  正确做法：
    - 先用成熟开源方案（Backstage、Kubecost、Cilium）
    - 验证需求真实存在后，再考虑自研
    - 自研的成本是开源方案的 5-10 倍

错误 3：5 人团队做 20 人的工作量
  症状：平台负责人是架构师，低估了落地难度
  正确做法：
    - 按团队规模裁剪平台范围
    - 200 人技术团队 → 5-8 人平台团队是合理的
    - 但只做最核心的 3-5 个功能

错误 4：引入过多新技术
  症状：Istio + Volcano + 自研 CNI + vCluster
  正确做法：
    - 技术栈收敛
    - 每个领域选 1 个成熟工具
    - 优先选择社区主流方案（流行 = 问题少 + 文档全）

错误 5：不与业务团队沟通
  症状：平台功能由平台团队"猜测"需求
  正确做法：
    - 平台团队 = 产品团队
    - 每月用户访谈
    - 每个功能上线前找 3 个团队试用
```

### 教训

> **"完美是完成的敌人。先让 80% 的服务跑起来，再优化剩下的 20%。"**

## 案例 3："平台团队闭门造车"——某 SaaS 公司

### 背景

- **公司规模**：150 人 SaaS，技术团队 50 人
- **平台团队**：3 人，技术能力很强
- **问题**：没有与开发者沟通，自顶向下设计"标准化"

### 平台设计（脱离实际）

```yaml
# 平台团队设计的"标准化"配置
# 要求所有服务必须遵循

resources:
  limits:
    cpu: "500m"        # 平台团队认为"足够"
    memory: "512Mi"
  requests:
    cpu: "50m"
    memory: "128Mi"

securityContext:
  runAsNonRoot: true
  readOnlyRootFilesystem: true
  allowPrivilegeEscalation: false

# 禁止列表：
# - hostPath
# - NodePort
# - latest 标签
# - 环境变量传 Secret（必须用 Vault）
# - 超过 10 个环境变量
```

### 冲突与后果

```
开发者 A：我的 Java 应用启动就需要 1GB 内存
平台团队：不行，标准限制是 512Mi
→ 结果：应用频繁 OOMKilled
→ 业务影响：每天 3-5 次重启，用户体验差

开发者 B：我需要用 hostPath 挂载本地缓存
平台团队：不行，安全策略禁止
→ 结果：应用性能下降 50%
→ 业务影响：API 响应时间从 100ms 增加到 500ms

开发者 C：我需要快速迭代，用 latest 标签方便
平台团队：不行，必须用具体版本号
→ 结果：开发者每次更新要改 3 个文件的版本号
→ 业务影响：发布频率从每天 5 次降到每周 1 次

开发者 D：我的应用有 50 个环境变量（配置项多）
平台团队：太多了，标准是最多 10 个
→ 结果：开发者把配置写死在代码里
→ 业务影响：配置变更需要重新编译，无法热更新

开发者 E：我需要 Vault 集成来获取数据库密码
平台团队：必须用 HashiCorp Vault
→ 结果：Vault 需要额外学习，开发者绕过平台直接写死密码
→ 安全风险：密码暴露在 Git 中
```

### 量化结果

```
开发者满意度：1.8/5（全公司最低）
Shadow IT 比例：60%（开发者绕过平台自建资源）
平台采用率：35%
生产事故归因于平台限制：40%
平台团队被称为："Blocker Team"（障碍团队）

CEO 介入后的修复：
  第 1 步：平台负责人与每个团队 1-on-1
           → 收集真实痛点
           
  第 2 步：成立"平台委员会"
           → 每个业务团队派 1 名代表
           → 平台决策需要委员会多数同意
           
  第 3 步：放宽非关键限制
           - CPU/Memory 允许团队自定义（只需审批 >4C/8G）
           - hostPath 允许（需审计）
           - latest 标签允许（staging 环境）
           - 环境变量限制取消
           
  第 4 步：引入"例外流程"
           → 不满足标准？提交例外申请
           → 平台团队 24h 内响应
           → 超过 80% 的申请被批准
           
  第 5 步：度量开发者体验
           → 每月 NPS 调研
           → 平台团队 KPI = 开发者满意度
           
  3 个月后：
    - 满意度从 1.8 → 3.8
    - Shadow IT 从 60% → 15%
    - 平台采用率从 35% → 75%
```

### 教训

> **"平台团队的用户是开发者，不是架构图。"**
> 
> 平台工程的最终目标是让开发者更高效，不是展示技术能力。
> 每一个限制都必须有明确的业务价值，否则就是官僚。

## 案例 4："上线后无人维护"——某物流公司

### 背景

- **公司规模**：500 人物流，技术团队 100 人
- **平台**：请了外部咨询公司搭建了一套 K8s 平台
- **问题**：咨询项目结束后，内部无人能维护

### 失败时间线

```
Month 1-3: 咨询公司搭建平台
           → 使用了 15 种开源工具
           → 大量自定义脚本和配置
           → 交付了一份 200 页的运维手册
           
Month 6:  平台出现第一次故障
           → ArgoCD 无法同步应用
           → 内部无人能定位问题
           → 重新请咨询公司，费用 ¥20 万
           → 根因：证书过期，需要更新
           
Month 9:  发现 3 个高危 CVE
           → K8s 版本 1.24，已知漏洞
           → 不知道哪些组件受影响
           → 不敢升级（怕破坏现有配置）
           
Month 12: K8s 版本落后 3 个 minor 版本
           → 社区不再支持 1.24
           → 被迫整体重建
           → 重建成本：¥100 万
           
总成本：
  - 平台建设：¥80 万
  - 维护费用：¥50 万（多次请咨询公司）
  - 重建费用：¥100 万
  - 总计：¥230 万
  - 时间损失：12 个月
```

### 根因分析

```
错误 1：完全外包平台建设
  症状：内部无人理解平台架构
  正确做法：
    - 核心能力必须内化
    - 外包只能做加速（3 个月的项目，外包帮你 2 个月完成）
    - 内部人员必须全程参与建设

错误 2：使用过多小众技术
  症状：15 种开源工具，其中 5 个社区已经不活跃
  正确做法：
    - 技术栈收敛
    - 优先选社区主流方案（GitHub stars > 5000）
    - CNCF 毕业项目优先

错误 3：没有知识转移
  症状：200 页运维手册，但没有人读过
  正确做法：
    - 建设期间必须有内部人员全程参与
    - 交付时进行培训（至少 2 周）
    - 前 3 个月由外包团队 + 内部团队共同运维

错误 4：没有文档
  症状：自定义脚本的用途不清楚
  正确做法：
    - 每一步操作都有 Runbook
    - 架构图、数据流图、故障树
    - 新人 1 周内能独立处理常见问题

错误 5：没有升级计划
  症状：K8s 版本落后 3 个版本
  正确做法：
    - 制定季度升级窗口
    - 测试环境先升级验证
    - 生产环境滚动升级
```

### 教训

> **"买不到平台，只能培养出平台。"**
> 
> 平台工程的核心能力（K8s 运维、故障排查、架构设计）必须内化到团队中。
> 外包可以加速，但不能替代。

## 案例 5："成本失控"——某 AI 创业公司

### 背景

- **公司规模**：80 人 AI 创业公司，技术团队 30 人
- **业务**：大模型推理服务（LLM）
- **问题**：GPU 集群成本 3 个月从 $10k/月涨到 $80k/月

### 成本失控时间线

```
Month 1: 购买 10 台 A100 云服务器
         → 成本 $10k/月
         → GPU 利用率：85%
         
Month 2: 业务发展，增加到 30 台
         → 成本 $30k/月
         → GPU 利用率：60%（部分任务等待中）
         
Month 3: 开发者为了方便，每台机器只跑 1 个模型
         → 实际利用率 15%
         → 但为了"留余量"不愿缩容
         → 成本 $80k/月
         
         详细浪费分析：
         ┌────────────────────┬────────┬────────────────────────────┐
         │ 浪费类型           │ 占比   │ 原因                       │
         ├────────────────────┼────────┼────────────────────────────┤
         │ 开发环境长期运行   │ 30%    │ 晚上/周末不关机            │
         │ 低利用率 GPU       │ 40%    │ 没有 GPU 共享，一卡一模型  │
         │ 过度预留           │ 20%    │ "怕不够"，预留了 3 倍容量  │
         │ 测试任务不清理     │ 10%    │ 测试跑完 Pod 还在          │
         └────────────────────┴────────┴────────────────────────────┘
         
Month 4: 财务总监发现异常
         → 但已签订 1 年预留实例合同
         → 无法退款
         → 每月最低支出 $55k（优化后）
```

### 修复措施

```bash
# 1. 立即措施（Month 4）
# 开发环境定时关机（晚 8 点 → 早 9 点）
# 节省：30% × 13h/24h = 16%

# 使用 K8s CronJob 自动开关机
cat > gpu-shutdown-cronjob.yaml <<EOF
apiVersion: batch/v1
kind: CronJob
metadata:
  name: gpu-dev-shutdown
  namespace: dev
spec:
  schedule: "0 20 * * 1-5"  # 工作日晚上 8 点
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: kubectl
            image: bitnami/kubectl
            command:
            - kubectl
            - scale
            - deployment/gpu-dev-workloads
            - --replicas=0
            - -n
            - dev
          restartPolicy: OnFailure
EOF

# 2. 短期措施（Month 4-5）
# 引入 GPU 共享（MIG - Multi-Instance GPU）
# 1 张 A100 → 7 个 MIG 实例
# 利用率从 15% → 70%

# 3. 中期措施（Month 5-7）
# 训练用 Spot 实例（节省 70%）
# 推理用预留实例（节省 40%）

# 4. 长期措施（Month 7-12）
# 自建 GPU 集群（比云便宜 50%）
# 但需要投入机房和运维人力
```

### 优化结果

```
┌──────────┬──────────┬──────────┬─────────────────────────────┐
│ 月份     │ 优化前   │ 优化后   │ 措施                        │
├──────────┼──────────┼──────────┼─────────────────────────────┤
│ Month 4  │ $80,000  │ $55,000  │ 定时关机 + MIG              │
│ Month 6  │ $80,000  │ $35,000  │ + Spot 实例                 │
│ Month 9  │ $80,000  │ $25,000  │ + 资源配额限制 + 审批流程   │
│ Month 12 │ $80,000  │ $20,000  │ + 自建部分 GPU              │
└──────────┴──────────┴──────────┴─────────────────────────────┘

关键制度：
  1. 所有 GPU 资源申请需要审批（> $500/月）
  2. 自动标签：按 team/project 分摊成本
  3. 每周成本报告发给各团队负责人
  4. GPU 利用率 < 40% 持续 3 天 → 自动告警
```

### 教训

> **"没有 FinOps 的 AI 平台，就是一台碎钞机。"**
> 
> GPU 资源的成本是传统 CPU 的 10-50 倍，必须从一开始就建立成本意识。
> 平台团队的责任不仅是让服务跑起来，还要让服务跑得经济。

## 自检清单：你的平台是否正在走向失败？

```bash
# 评分标准：每个问题 1 分，总分 0-8 分
# 0-2 分：健康
# 3-4 分：警告，需要关注
# 5-6 分：危险，需要立即干预
# 7-8 分：平台正在失败

□ 开发者满意度 < 3.0/5
□ Shadow IT 泛滥（开发者绕过平台自建环境）
□ 平台团队被叫做"Blocker Team"
```
