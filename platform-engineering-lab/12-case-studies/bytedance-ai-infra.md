## Gödel 调度器

### 为什么不用原生 K8s 调度器

```
原生 K8s 调度器的问题：
  1. 单一队列：在线服务和离线训练共享队列，互相阻塞
  2. 无 Gang Scheduling：分布式训练需要同时分配 N 个 Pod
  3. 无拓扑感知：GPU 间的 NVLink 拓扑影响训练效率
  4. 无弹性调度：训练任务弹性缩容会中断训练
  5. 无资源超售：节点利用率低（在线服务峰值利用率 30%）

Gödel 设计目标：
  - 统一调度在线（Service）和离线（Batch/ML）工作负载
  - 支持 Gang Scheduling、弹性调度、抢占调度
  - 拓扑感知（NUMA/GPU/网络）
  - 资源超售（离线任务利用在线空闲资源）
```

### Gödel 架构

```
Gödel 调度器架构：

  ┌─────────────────────────────────────┐
  │  Gödel Scheduler (多调度器)          │
  │                                     │
  │  ┌─────────────┐  ┌─────────────┐  │
  │  │ Pod Scheduler│  │ VM Scheduler│  │  ← 统一调度 Pod 和 VM
  │  │ (K8s Pod)   │  │ (VM + Pod)  │  │
  │  └──────┬──────┘  └──────┬──────┘  │
  │         │                │          │
  │         └───────┬────────┘          │
  │                 ▼                   │
  │  ┌─────────────────────────────┐   │
  │  │      Unified Scheduler Core  │   │
  │  │  ┌─────────┐ ┌─────────┐   │   │
  │  │  │ Cache   │ │ Snapshot│   │   │  ← 节点状态缓存
  │  │  └─────────┘ └─────────┘   │   │
  │  │  ┌─────────┐ ┌─────────┐   │   │
  │  │  │ Queue   │ │ Gang    │   │   │  ← 多级队列
  │  │  │ Manager │ │ Scheduler│   │   │
  │  │  └─────────┘ └─────────┘   │   │
  │  │  ┌─────────┐ ┌─────────┐   │   │
  │  │  │ Plugin  │ │ Permi-  │   │   │  ← 插件框架
  │  │  │ Framework│ │ ssion   │   │   │
  │  │  └─────────┘ └─────────┘   │   │
  │  └─────────────────────────────┘   │
  └─────────────────────────────────────┘

调度队列设计：
  
  Active Queue (高优先级)
    ├── 在线服务 Pod (P0)
    ├── 实时训练任务 (P1)
    └── 紧急任务 (P0)
  
  Backoff Queue (重试)
    ├── 调度失败的任务（指数退避）
    └── 抢占后等待的任务
  
  Unschedulable Queue
    ├── 资源不足的任务
    └── 等待依赖的任务
```

### Gödel vs K8s Scheduler 对比

| 特性 | K8s Scheduler | Gödel |
|------|--------------|-------|
| 队列设计 | 单一 FIFO | 多级优先级队列 |
| Gang Scheduling | 不支持（需 Volcano） | 原生支持 |
| 弹性调度 | HPA（Pod 级别） | 任务级别弹性扩缩容 |
| 资源超售 | 不支持 | 在线空闲资源调度离线任务 |
| 拓扑感知 | Node 级别 | NUMA/GPU/NVLink 级别 |
| 抢占策略 | PriorityClass | 更细粒度（资源维度） |
| 混部支持 | 需 Descheduler | 原生支持 |
| 调度延迟（1万节点） | P99 ~5s | P99 ~500ms |

## MegaScale 训练框架

### 大规模训练挑战

```
GPT-3 规模模型训练（175B 参数）：
  - 训练数据：300B tokens
  - GPU 需求：1024 × A100 (80GB)
  - 训练时间：数周
  
挑战 1：显存不足
  - 175B 参数 FP32 = 700GB
  - 单个 A100 80GB 远远不够
  - 需要模型并行（Model Parallelism）

挑战 2：通信瓶颈
  - 1024 GPU 间梯度同步
  - AllReduce 通信量巨大
  - 网络带宽成为瓶颈

挑战 3：故障频发
  - 1024 GPU 中平均每天故障 1-2 次
  - 故障后需要 checkpoint 恢复
  - 恢复时间从数小时缩短到分钟级

挑战 4：数据加载
  - 300B tokens 的读取
  - 需要从存储系统高速加载
  - CPU 预处理成为瓶颈
```

### MegaScale 解决方案

```
1. 3D 并行策略：

   Data Parallel (DP)
     数据分片 → 每个 GPU 处理不同数据
     通信：AllReduce 梯度
     效率：线性扩展（理想）
   
   Tensor Parallel (TP)
     层内切分 → 每个 GPU 计算部分神经元
     通信：AllReduce 激活值
     限制：通常 2-8 GPU（同一节点 NVLink）
   
   Pipeline Parallel (PP)
     层间切分 → 每个 GPU 负责若干层
     通信：P2P 传递激活值
     限制：需要 bubble 填充
   
   3D 并行组合示例（1024 GPU）：
     DP=16, TP=8, PP=8
     16 × 8 × 8 = 1024
     - 同一 TP 组内 8 GPU 用 NVLink
     - 同一 PP 组内用 RDMA
     - DP AllReduce 跨节点

2. 通信优化：

   NCCL 调优：
     - 拓扑感知集合通信
     - 双二叉树 AllReduce（减少跳数）
     - 梯度压缩（FP16/BF16/1-bit Adam）
   
   网络优化：
     - RDMA RoCE v2（100Gbps+）
     - 多轨网络（Multi-Rail）
     - 自适应路由（Adaptive Routing）
     - 拥塞控制（DCQCN）

3. 容错机制：

   Checkpoint 策略：
     - 同步 checkpoint：每 N 步全量保存
     - 异步 checkpoint：后台写入，不阻塞训练
     - 热备 GPU：预留 1-2% GPU 作为备用
   
   故障检测：
     - 心跳检测：每 30 秒
     - 自动诊断：NCCL 错误码分析
     - 快速恢复：从最近 checkpoint 重启（< 5 分钟）
   
   数据：
     - 故障恢复时间：从 2 小时 → 5 分钟
     - 训练有效时间占比：从 70% → 95%+
```

### 推荐系统推理

```
抖音推荐系统架构：

用户请求 → API Gateway
    │
    ├─ 召回层 (Recall)
    │   │
    │   ├─ 协同过滤：亿级用户-物品矩阵
    │   ├─ 向量检索 (FAISS/Milvus)
    │   │   百万级向量，P99 < 5ms
    │   └─ 规则召回：热门、关注
    │   
    │   输出：千级候选视频
    │
    ├─ 粗排层 (Coarse Ranking)
    │   │
    │   ├─ 轻量模型（DNN，百万参数）
    │   ├─ CPU 推理（成本优化）
    │   └─ 批量推理（32-64 条/批次）
    │   
    │   输出：百级候选视频
    │   延迟：P99 < 10ms
    │
    ├─ 精排层 (Fine Ranking)
    │   │
    │   ├─ 深度模型（DCN/DIN，十亿参数）
    │   ├─ GPU 推理（A10/T4）
    │   ├─ 实时特征拼接
    │   └─ 多目标优化（点击、完播、点赞）
    │   
    │   输出：十级候选视频
    │   延迟：P99 < 30ms
    │
    └─ 重排层 (Re-rank)
        │
        ├─ 多样性控制（MMR 算法）
        ├─ 新鲜度提升（exploration）
        ├─ 业务规则过滤
        └─ 分页组装
        
        输出：最终视频列表
        延迟：P99 < 5ms

端到端延迟：
  - P50: 50ms
  - P99: 100ms
  - P99.9: 200ms
```

### 推理优化技术

```
1. 模型优化：
   - 量化：FP32 → INT8（精度损失 < 1%，速度提升 2-4x）
   - 剪枝：移除不重要权重（参数量减少 30-50%）
   - 蒸馏：大模型 → 小模型（精度损失 < 2%）
   - 算子融合：减少 kernel launch 开销

2. 服务优化：
   - 动态批处理：累积请求批量推理
   - 请求缓存：相同输入直接返回缓存
   - 模型预热：启动时预热，避免冷启动
   - 多实例负载均衡：GPU 利用率均衡

3. 部署架构：
   - Triton Inference Server：多框架支持
   - TensorRT：NVIDIA 专用加速
   - ONNX Runtime：跨平台
   - 自研推理引擎：针对推荐模型优化

实际数据：
  - 推理成本降低 60%（通过量化 + 剪枝）
  - GPU 利用率提升 40%（动态批处理）
  - 延迟降低 50%（算子融合 + TensorRT）
```

## 存储系统

### PFS 并行文件系统

```
PFS 架构：

  客户端 (训练节点)
    │
    ├─ PFS 客户端库
    │   ├─ 数据预取（readahead）
    │   ├─ 写入缓存（writeback）
    │   └─ 一致性保证（close-to-open）
    │
    ▼
  元数据服务器 (MDS)
    │
    ├─ 文件元数据管理
    ├─ 目录操作
    └─ 锁管理
    │
    ▼
  数据服务器 (OSS)
    │
    ├─ 对象存储后端
    ├─ 数据分片（256MB/chunk）
    └─ 冗余编码（EC 4+2）

性能指标：
  - 单客户端读吞吐：10GB/s
  - 聚合读吞吐：1TB/s+
  - 元数据操作：100k+ ops/s
  - 延迟：读 5ms，写 10ms
```

### 训练数据流水线

```
数据加载优化：

  存储层 (PFS/OSS)
    │
    ├─ 数据分片（每个 worker 读取不同分片）
    ├─ 本地缓存（Alluxio/训练节点 SSD）
    │   缓存命中率：90%+
    │
    ▼
  CPU 预处理
    │
    ├─ 数据解码（图片/视频）
    ├─ 数据增强（随机裁剪、翻转）

    ├─ CPU → GPU 异步传输（pin memory）

    │
    ▼
  GPU 训练

优化效果：
  - 数据加载时间占比：从 40% → 5%
```

### 2. 面试要点

```
Q: 为什么需要 Gödel 而不是 Volcano？
A: - Volcano 专注 Batch/AI，不支持在线服务
   - Gödel 统一调度在线+离线，支持混部
   - Gödel 调度延迟更低（P99 500ms vs Volcano 2s）
   - 但 Volcano 开源生态更丰富

Q: 3D 并行（DP+TP+PP）如何选择配置？
A: - TP 优先（NVLink 带宽最高）
   - TP 大小 ≤ 单节点 GPU 数（8-16）
   - PP 大小 = 总 GPU / (DP × TP)
   - 目标：最小化通信量，最大化计算效率

Q: 推理延迟优化有哪些手段？
A: - 模型层：量化、剪枝、蒸馏
   - 服务层：动态批处理、请求缓存、预热
   - 系统层：GPU 亲和性、NUMA 绑定、CPU/GPU 流水线

Q: Checkpoint 频率如何权衡？
A: - 太频繁：I/O 开销大，训练时间增加
   - 太稀疏：故障后重算多，浪费 GPU
   - 字节做法：每 15 分钟异步 checkpoint
   - 热备 GPU 预留 1-2%，故障时秒级切换
```
