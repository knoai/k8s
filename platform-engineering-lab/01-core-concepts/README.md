# 01 - 平台工程核心概念

平台工程（Platform Engineering）是 DevOps 的演进形态，旨在通过构建
内部开发者平台（IDP）来提升工程效率、降低认知负荷、标准化交付流程。
本章建立平台工程的理论基础，为后续技术实践提供上下文。

## 学习目标

1. 理解平台工程与 DevOps、SRE 的区别与联系
2. 掌握 IDP 的核心组件和架构模式
3. 了解平台团队的组织定位和成功指标
4. 建立平台即产品（Platform as a Product）的思维
5. 识别平台建设的常见反模式

## 核心概念

### 平台工程的定义

平台工程是**设计和构建工具链与工作流**的学科，这些工具链和工作流为
云原生时代的软件工程组织提供自助服务能力。平台工程的核心关注点:

- **开发者体验（DX）**: 减少开发者的上下文切换和等待时间
- **自助服务**: 开发者可以自主申请资源、部署服务、查看监控
- **黄金路径**: 提供经过平台团队验证的最佳实践模板
- **抽象复杂性**: 将基础设施、安全、合规的复杂性封装在平台之下
- **规模化**: 让 100 个团队都能以同样的高标准交付

### 平台工程 vs DevOps vs SRE

三者是云原生时代软件交付的不同维度，相互补充而非替代:

| 维度 | DevOps | SRE | Platform Engineering |
|------|--------|-----|---------------------|
| 核心目标 | 打破开发和运维壁垒 | 保障系统可靠性 | 提升开发者效率 |
| 主要实践 | CI/CD、IaC、监控、协作 | SLI/SLO、故障响应、容量规划 | IDP、自助服务、模板化、开发者门户 |
| 组织模式 | 跨职能团队，共担责任 | 嵌入产品团队或独立团队 | 独立平台团队，服务内部客户 |
| 度量指标 | 部署频率、恢复时间、变更失败率 | 可用性、错误预算、MTTR | 开发者满意度、交付速度、平台采用率 |
| 演进关系 | 文化运动（2009-） | 工程实践（2003-） | 组织形态（2020-） |

**关系解读**: DevOps 是"为什么"（文化），SRE 是"怎么做"（实践），
平台工程是"规模化地做"（组织）。

三者互补: DevOps 提供文化基础，SRE 提供工程方法，平台工程提供规模化能力。

### 内部开发者平台（IDP）架构

典型的 IDP 采用分层架构，每层提供不同的抽象级别:

```
┌─────────────────────────────────────────────┐
│         开发者门户层 (Developer Portal)       │
│    Backstage / Port / Cortex / Custom       │
│    服务目录、软件模板、文档、监控仪表盘        │
├─────────────────────────────────────────────┤
│         平台接口层 (Platform API)             │
│    GitOps API / Infra API / Policy API      │
│    将基础设施操作封装为开发者友好的接口         │
├─────────────────────────────────────────────┤
│         能力编排层 (Orchestration)            │
│    ArgoCD / Flux / Crossplane / Terraform   │
│    GitOps、基础设施即代码、策略执行           │
├─────────────────────────────────────────────┤
│         基础设施层 (Infrastructure)           │
│    K8s / Cloud / DB / MQ / Cache / LB       │
│    实际的计算、存储、网络资源                  │
├─────────────────────────────────────────────┤
│         可观测性层 (Observability)            │
│    Prometheus / Grafana / Jaeger / Loki     │
│    日志、指标、追踪、告警                      │
└─────────────────────────────────────────────┘
```

**门户层**: 开发者统一入口。服务目录让开发者发现已有服务；
软件模板让开发者一键创建新项目；文档降低学习成本；
监控仪表盘提供一站式观测。

**接口层**: 平台能力的抽象 API。例如:
- `POST /v1/databases` → 创建 PostgreSQL 实例
- `POST /v1/deployments` → 触发 GitOps 部署
- `GET /v1/costs?team=team-a` → 查询团队成本

**编排层**: 实现自动化的引擎。ArgoCD 监听 Git 变更并同步到 K8s；
Crossplane 将云资源声明转化为实际的云 API 调用；
Terraform 管理基础设施状态。

**基础设施层**: 实际的资源。K8s 集群提供容器编排；
云厂商提供 VM、存储、网络；数据库、消息队列、缓存提供数据服务。

**可观测性层**: 贯穿所有层次。平台团队需要观测平台自身的健康状态，
也需要为应用团队提供观测能力。

### 平台即产品（Platform as a Product）

平台团队应将平台视为一个内部产品，遵循产品管理原则:

1. **用户研究**: 定期访谈开发者（每月至少 2 次），了解痛点和需求
2. **产品路线图**: 按优先级迭代平台能力，每季度发布规划
3. **开发者体验**: 关注 onboarding 时间（目标 <30 分钟）、
   文档质量（NPS > 50）、错误提示的友好性
4. **度量指标**:
   - 平台采用率（% 团队使用平台，目标 >80%）
   - 自助服务成功率（目标 >95%）
   - 开发者满意度（NPS，目标 >50）
   - 工单数量趋势（应下降而非上升）
   - 新服务上线时间（从申请到生产，目标 <1 天）

### 平台团队的反模式

**反模式 1: 平台团队成为新的运维团队**
- 症状: 开发者仍然需要提交工单申请资源，平台团队手动审批
- 根因: 没有建立自助服务能力，平台团队陷入运维操作
- 解决: 所有高频操作都应提供自助服务入口，平台团队聚焦平台建设

**反模式 2: 过度设计**
- 症状: 平台支持 100 个配置选项，开发者只需要 3 个；
  文档 200 页，没人看完
- 根因: 平台团队按自己的理解设计，而非基于用户需求
- 解决: 从简单开始（MVP），按需扩展。80% 场景用默认配置

**反模式 3: 忽视开发者反馈**
- 症状: 平台团队闭门造车 6 个月，发布后发现无人使用
- 根因: 缺乏用户参与的设计过程
- 解决: 建立开发者反馈循环，平台团队与产品团队紧密协作，
  采用敏捷迭代而非瀑布式开发

**反模式 4: 强制迁移**
- 症状: 行政命令"所有团队必须在 3 个月内迁移到平台"，引发抵触
- 根因: 忽视了变更管理的复杂性
- 解决: 用价值吸引而非行政命令。先找早期采用者，
  积累成功案例，逐步推广。允许团队按自己的节奏采用

**反模式 5: 平台团队与业务团队脱节**
- 症状: 平台团队追求技术先进性（如最新 K8s 版本），
  而业务团队需要稳定性
- 根因: 缺乏对齐的业务目标
- 解决: 平台团队的 OKR 应与业务目标挂钩（如"支持业务团队发布频率提升 2 倍"）

- **复杂子系统团队（Complicated Subsystem Team）**: 负责复杂组件
- **流对齐团队（Stream-aligned Team）**: 直接交付业务价值的产品团队

### 平台能力成熟度模型

文件: `maturity-model.md`

从 Level 1（手工）到 Level 5（完全自助）的演进路径:

| 级别 | 名称 | 特征 | 典型工具 |
|------|------|------|---------|
| L1 | 手工 | 人工审批、手动部署、无标准化 | 邮件、JIRA |
| L2 | 脚本化 | 自动化脚本、部分标准化 | Shell、Ansible |
| L3 | 平台化 | 自助服务、黄金路径、基础监控 | K8s、ArgoCD、基础模板 |
| L4 | 产品化 | 开发者门户、完整观测性、成本优化 | Backstage、FinOps、全面可观测性 |
| L5 | 智能化 | AI 辅助、自动优化、预测性维护 | AIOps、智能扩缩容、自动修复 |

大多数企业处于 L2-L3，目标是 L4。

## 面试常见问题

**Q: 平台工程与 DevOps 有什么区别？**

A: DevOps 是一种文化和运动，强调开发和运维的协作与共享责任。
平台工程是 DevOps 的演进，通过构建内部开发者平台来
规模化地实现 DevOps 目标。

- DevOps 关注"怎么做"（CI/CD、IaC、监控）
- 平台工程关注"如何让所有人都能做"（自助服务、黄金路径、开发者门户）

类比: DevOps 像是教每个人开车，平台工程像是构建公交系统。

**Q: 如何衡量平台工程的成功？**

A: 从四个维度建立指标体系:

1. **效率**: 部署频率（目标: 每天多次）、交付周期（目标: <1 天）、
   恢复时间（目标: <1 小时）
2. **质量**: 缺陷率（目标: 下降）、可用性（目标: >99.9%）、
   安全漏洞数（目标: 下降）
3. **体验**: 开发者满意度（NPS，目标: >50）、
   自助服务成功率（目标: >95%）、
   新服务上线时间（目标: <1 天）
4. **成本**: 资源利用率（目标: >60%）、
   云成本优化率（目标: 年降 20%）

**Q: 平台团队应该支持多少配置自由度？**

A: 遵循"黄金路径"（Golden Path）原则:
- **80% 场景**: 使用标准配置（限制选择，降低出错率）
- **15% 场景**: 支持有限自定义（通过参数化模板）
- **5% 场景**: 允许完全自定义（高级用户，需审批和额外支持）

核心思想: 约束带来自由。通过合理的约束，让大多数团队快速、安全地交付。

**Q: 如何处理团队对平台的抵触？**

A: 五个策略:
1. **证明价值**: 对比使用前后效率提升（"团队 A 使用后部署频率提升 3 倍"）
2. **渐进式迁移**: 从自愿采用到逐步推广，允许混合模式
3. **倾听反馈**: 平台团队也是服务团队，定期收集满意度
4. **提供退出路径**: 不强制绑定，允许团队在必要时绕过平台
5. **培养布道师**: 在每个产品团队中找到平台拥护者

**Q: 平台工程的 ROI 如何计算？**

A: 直接收益:
- 开发者时间节省: 每个开发者每天节省 1 小时 × 团队人数 × 时薪
- 故障减少: 减少的 P1 故障数 × 平均损失
- 资源优化: 云成本节省

间接收益:
- 开发者满意度提升 → 留存率提升
- 交付速度提升 → 业务竞争力
- 标准化 → 技术债务减少

通常平台团队的 ROI 在 1 年内即可为正。

**Q: 平台团队和产品团队的协作边界？**

A: 清晰的边界划分:
- **平台团队负责**: 平台稳定性、基础设施升级、安全合规、成本优化
- **产品团队负责**: 应用代码、业务逻辑、应用监控、应用安全

协作方式:
- 平台团队提供 SLA（如 99.9% 可用性）
- 产品团队通过自助服务使用平台能力
- 复杂需求通过工单或需求评审流程
- 定期召开平台用户组会议收集反馈

## 平台工程实施路线图

### 阶段化建设方案

**第一阶段：基础设施即代码（0-3 个月）**
- 目标：消除手动配置，实现环境一致性
- 交付物：
  - Terraform/Opentofu 模块库（VPC、EKS、RDS）
  - GitOps 工作流（ArgoCD 或 Flux）
  - 基础监控告警（Prometheus + Alertmanager）
- 成功标准：`terraform apply` 5 分钟创建完整测试环境

**第二阶段：内部开发者平台雏形（3-6 个月）**
- 目标：降低开发者使用基础设施的门槛
- 交付物：
  - Backstage 门户（服务目录、技术文档）
  - 软件模板（Golden Path）：新建微服务 → 自动创建代码仓库、CI/CD、K8s 资源
  - 自助服务能力：开发者可以自行创建数据库、Redis、Kafka Topic
- 成功标准：新服务从 0 到生产部署从 2 周缩短到 2 天

**第三阶段：平台成熟化（6-12 个月）**
- 目标：平台成为默认工作方式
- 交付物：
  - 多环境管理（dev/staging/prod 统一管理）
  - 成本可见性（OpenCost/Kubecost 集成到门户）
  - 安全合规自动化（策略即代码 Kyverno/OPA）
  - FinOps 流程：自动资源优化、预算告警
- 成功标准：80% 的新服务使用平台自助创建

**第四阶段：智能化与生态（12 个月+）**
- 目标：平台智能化，持续优化
- 交付物：
  - AI 辅助排障（异常检测、根因分析）
  - 自动扩缩容（Karpenter + HPA + VPA 联动）
  - 平台生态：第三方工具集成、插件市场
- 成功标准：平台 NPS（净推荐值）> 50

### 平台工程组织设计

**团队拓扑（Team Topologies）应用**:

| 团队类型 | 职责 | 与平台团队关系 |
|---------|------|--------------|
| 平台团队 | 构建和维护内部开发者平台 | — |
|  stream-aligned 团队 | 交付业务价值 | 平台的主要用户 |
| 赋能团队 (Enabling) | 帮助 stream-aligned 团队掌握新技术 | 短期协作，知识传递后退出 |
| 复杂子系统团队 | 维护高度专业化的组件（如 GPU 调度） | 平台可能包含其子系统 |

**平台团队内部结构**:
```
平台团队 (Platform Team)
├── 平台体验组 (Platform Experience)
│   └── Backstage 门户、CLI 工具、文档
├── 基础设施组 (Infrastructure)
│   └── K8s 集群、网络、存储、计算
├── 可观测性组 (Observability)
│   └── 监控、日志、追踪、告警
├── 安全合规组 (Security & Compliance)
│   └── 策略、审计、镜像扫描、 Secret 管理
└── FinOps 组 (FinOps)
    └── 成本优化、资源配额、预算管理
```

**平台团队规模估算**:
- 每 50 个开发者需要 1 个平台工程师（行业基准）
- 初期：3-5 人（覆盖核心能力）
- 成长期：8-12 人（覆盖全部能力域）
- 成熟期：按开发者数量线性扩展

### 平台成熟度评估

**自评问卷（每个维度 1-5 分）**:

| 维度 | 1 分 | 3 分 | 5 分 |
|------|------|------|------|
| 自助服务 | 完全手动工单 | 部分自助（需审批） | 完全自助，即时生效 |
| 开发者体验 | 文档缺失 | 文档存在但不完整 | 完善的文档 + 工具链 |
| 可观测性 | 基础监控 | 监控 + 日志 | Metrics/Logs/Traces 关联 |
| 安全合规 | 事后审计 | 部分自动化 | 策略即代码，自动执行 |
| 成本管理 | 无可见性 | 月度报告 | 实时可见 + 自动优化 |
| 多租户 | 无隔离 | 命名空间隔离 | 完整的多租户体系 |

**评分解读**:
- 6-12 分: 起步阶段，优先建设基础设施即代码
- 13-18 分: 发展阶段，重点建设自助服务平台
- 19-24 分: 成熟阶段，聚焦智能化和生态建设
- 25-30 分: 领先阶段，探索 AI 和自动化前沿

## 面试常见问题补充

**Q: 平台工程和 DevOps 的区别？**

A: 关键区别:
- **DevOps**: 文化和实践，强调开发和运维的协作
- **平台工程**: 工程学科，通过构建平台产品来赋能开发者

类比:
- DevOps 像"教所有人开车"
- 平台工程像"提供高铁系统，乘客只需买票上车"

关系: 平台工程是 DevOps 的演化和规模化，当团队数量增加时，DevOps 的手工协作模式无法扩展，需要平台来标准化和自动化。

**Q: 如何衡量平台的成功？**

A: 四个维度:
1. **效率指标**: 
   - 部署频率（从每周 1 次到每天 10 次）
   - 前置时间（从代码提交到生产从 1 周缩短到 1 小时）
   - MTTR（平均恢复时间从 2 小时缩短到 15 分钟）

2. **质量指标**:
   - 变更失败率（从 30% 降到 5%）
   - 安全漏洞数量（减少 80%）

3. **满意度指标**:
   - 开发者 NPS（净推荐值）
   - 平台工单响应时间（< 4 小时）

4. **成本指标**:
   - 基础设施成本（降低 30-50%）
   - 平台团队 ROI（通常 1 年内回正）

**Q: 平台工程中的" paved road"（铺好的路）是什么意思？**

A: "Paved Road" 是 Netflix 提出的概念:
- **铺好的路**: 平台提供的标准路径（Golden Path），有完善的支持、文档和工具链
- **土路**: 开发者自行选择的技术栈，平台不承诺支持，但允许使用

核心原则: 
- 让铺好的路比土路更容易走（降低 friction）
- 不禁止走土路，但鼓励走铺好的路
