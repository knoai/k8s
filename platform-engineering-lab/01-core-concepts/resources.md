# 核心概念 - 平台工程权威资源与参考架构

> 平台工程不是"搭个 K8s 集群"，而是一套涵盖组织、流程、技术的完整体系。
> 本节汇总经过生产验证的权威白皮书、参考架构、社区资源和面试必考点。

---

## 一、CNCF 权威白皮书（必读）

### 1.1 CNCF Platforms White Paper（2023）

```
核心定义：
  "平台是将底层能力封装为可复用服务的层，
   让开发者能够自助使用基础设施，而不需要了解底层细节。"

五大原则：
  1. 平台即产品（Platform as a Product）
     - 平台团队必须像产品经理一样对待平台
     - 定期用户访谈、NPS 调研、迭代路线图
  
  2. 自助服务（Self-Service）
     - 开发者可以在 5 分钟内创建新服务
     - 不需要提交工单、不需要等待审批
  
  3. 减少认知负担（Reduce Cognitive Load）
     - 开发者只需要知道 "怎么做"，不需要知道 "为什么"
     - 例：开发者说 "我要一个数据库"，不需要知道 RDS 实例类型
  
  4. 可选的抽象层（Optional Abstraction）
     - 对于高级用户，允许 "escape hatch"（逃生舱）
     - 例：标准配置不满足？可以提交 PR 修改或申请例外
  
  5. 标准化与治理（Standardization）
     - 安全策略、成本标签、监控埋点自动注入
     - 不是约束创造力，而是消除重复劳动
```

### 1.2 Platform Engineering Maturity Model

```
Level 1 - 临时（Ad-hoc）：
  - 每个团队自己管理基础设施
  - 无标准化，重复造轮子
  - 面试信号：公司规模 < 50 人，或技术债务严重

Level 2 - 运营化（Operational）：
  - 有专门的运维/SRE 团队
  - 基础设施通过工单申请
  - 面试信号：平均资源申请时间 > 2 天

Level 3 - 平台化（Platform）：
  - 有平台团队，提供标准化服务
  - 开发者可以自助使用大部分资源
  - 面试信号：公司正在招聘 "Platform Engineer"

Level 4 - 产品化（Product）：
  - 平台被视为内部产品，有产品经理和路线图
  - 度量开发者体验（DXP）和平台采用率
  - 面试信号：有开发者门户（Backstage/Port），NPS > 40

Level 5 - 生态化（Ecosystem）：
  - 平台成为业务竞争力的一部分
  - 对外输出平台能力（如 AWS 的 EKS、阿里云的 ACK）
  - 面试信号：公司有多云/混合云平台团队
```

## 二、生产级参考架构

### 2.1 中小企业平台架构（50-500 人）

```
技术栈（成本优先）：

┌─────────────────────────────────────────────┐
│  开发者门户（Backstage / Port）                │
│  └── 服务模板、文档、API 目录                  │
├─────────────────────────────────────────────┤
│  GitOps（ArgoCD）                             │
│  └── 多环境部署、自动同步、回滚                │
├─────────────────────────────────────────────┤
│  托管 K8s（ACK / EKS / GKE）                  │
│  └── 控制平面免运维                            │
├─────────────────────────────────────────────┤
│  CNI：Terway / AWS VPC CNI（云厂商原生）       │
│  Ingress：云厂商 ALB / NLB（免证书管理）       │
│  存储：云厂商 CSI（免维护）                    │
├─────────────────────────────────────────────┤
│  监控：Prometheus + Grafana + 云厂商托管存储   │
│  日志：云厂商托管日志服务（SLS/CloudWatch）    │
│  告警：PagerDuty / 钉钉 / 企业微信             │
└─────────────────────────────────────────────┘

关键决策：
  - 不自建任何基础设施组件（除非有 5+ 人 SRE 团队）
  - 全部用云厂商托管服务，省下的运维人力投入平台工具开发
  - 月度额外成本：$500-2000（vs 自建节省 3-5 人月）
```

### 2.2 大型企业平台架构（500-5000 人）

```
技术栈（可控性优先）：

┌─────────────────────────────────────────────┐
│  IDP（Backstage + 自研插件）                  │
│  ├── 多集群管理面板                           │
│  ├── 成本分摊看板                             │
│  ├── 服务成熟度评分                           │
│  └── 自动化运维 Runbook                       │
├─────────────────────────────────────────────┤
│  多集群 GitOps（ArgoCD + ApplicationSet）     │
│  ├── 主集群：生产                             │
│  ├── 灾备集群：异地                           │
│  └── 边缘集群：客户现场                       │
├─────────────────────────────────────────────┤
│  多租户隔离（HNC + vCluster + NetworkPolicy） │
│  策略治理（Kyverno / OPA Gatekeeper）         │
│  成本治理（OpenCost + Kubecost）              │
├─────────────────────────────────────────────┤
│  自研/定制调度器（Volcano / Yunikorn）        │
│  服务网格（Istio / Cilium Service Mesh）      │
│  云资源编排（Crossplane / Terraform Controller）│
├─────────────────────────────────────────────┤
│  可观测性（OpenTelemetry + Prometheus + Tempo）│
│  FinOps（按团队/项目/环境的成本分摊）          │
│  混沌工程（Litmus / Chaos Mesh）              │
└─────────────────────────────────────────────┘

关键决策：
  - 核心控制面自研（调度器、配额管理、审计）
  - 外围工具用开源（监控、日志、GitOps）
  - 平台团队规模：技术团队的 5-10%（500 人技术团队 = 25-50 人平台团队）
```

## 三、面试核心考点

### 3.1 平台工程 vs DevOps vs SRE

```
概念对比：

              DevOps           SRE              Platform Engineering
─────────────────────────────────────────────────────────────────────
目标          打破开发与运维墙   用软件工程方法做运维  构建开发者自助服务平台
起源          2009（Velocity）  2003（Google）     2022（Gartner 报告）
核心活动      CI/CD pipeline   可靠性工程          内部开发者平台（IDP）
度量指标      部署频率          SLO/错误预算        开发者体验/上线时间
团队角色      跨功能团队        可靠性工程师        平台产品经理+工程师
─────────────────────────────────────────────────────────────────────

关系（不是替代，是演进）：
  DevOps（文化/运动）
    └── SRE（具体实践）
          └── Platform Engineering（规模化、产品化）

面试回答要点：
  Q: "你们公司做 DevOps，为什么还需要平台工程？"
  A: "DevOps 是文化，SRE 是实践，平台工程是产品化。
      当公司从 50 人增长到 500 人，每个团队自己搭 CI/CD
      的重复劳动不可接受。平台团队把这些重复劳动抽象为
      自助服务，让开发者专注于业务逻辑。"
```

### 3.2 平台团队 KPI 设计

```
不要度量的：
  - K8s 集群数量（多不代表好）
  - 平台功能数量（80% 功能没人用是浪费）
  - 工单处理速度（自助服务应该没有工单）

应该度量的：
  ┌────────────────────┬────────────────────────────────┐
  │ 指标               │ 目标值                         │
  ├────────────────────┼────────────────────────────────┤
  │ 开发者满意度 (NPS) │ > 40                           │
  │ 新服务上线时间     │ < 2 小时                       │
  │ 平台采用率         │ > 80% 的服务在平台上运行        │
  │ MTTR               │ < 15 分钟                      │
  │ 基础设施成本占比   │ < 15% 营收                     │
  │ Shadow IT 比例     │ < 10%（绕过平台自建资源）       │
  │ 安全合规覆盖率     │ 100%                           │
  └────────────────────┴────────────────────────────────┘
```

### 3.3 平台工程反模式

```
反模式 1：平台团队 = 运维团队改个名
  症状：还是手工处理工单，开发者排队申请资源
  解决：平台团队必须有产品经理思维，交付自助服务

反模式 2：完美架构综合征
  症状：要一次性建成"终极平台"，6 个月不上线
  解决：MVP 思维，先解决最痛的 1-2 个问题

反模式 3：平台团队闭门造车
  症状：不跟开发者沟通，强制推行"标准"
  解决：每月用户访谈，平台委员会包含业务团队代表

反模式 4：没有逃生舱（Escape Hatch）
  症状：100% 强制标准化，高级用户无法满足特殊需求
  解决：80% 标准 + 20% 例外流程，允许 PR 修改平台模板

反模式 5：忽视开发者体验
  症状：平台功能强大但文档缺失、CLI 难用、报错信息模糊
  解决：平台团队 KPI 绑定开发者满意度
```

## 四、推荐学习路径

### 4.1 平台工程师技能树

```
Level 1 - 基础设施（3-6 个月）：
  - Linux 系统管理（内核、网络、存储）
  - Docker 容器化与镜像优化
  - Kubernetes 核心概念与日常运维
  - 云厂商基础服务（VPC、ECS、RDS、OSS）

Level 2 - K8s 进阶（6-12 个月）：
  - K8s 网络（CNI、Service Mesh、Ingress）
  - K8s 调度（调度框架、资源管理、调度器插件）
  - K8s 存储（CSI、存储类、快照）
  - K8s 安全（RBAC、NetworkPolicy、PodSecurity）

Level 3 - 平台工程（12-24 个月）：
  - 开发者门户（Backstage 架构与插件开发）
  - GitOps（ArgoCD 原理、多集群管理）
  - 多租户（Namespace、vCluster、HNC、Capsule）
  - 策略即代码（Kyverno、OPA Gatekeeper）
  - 云资源编排（Crossplane、Terraform Controller）

Level 4 - 架构师（24-36 个月）：
  - 大规模集群设计（5000+ 节点）
  - 多集群联邦（Karmada、Clusternet）
  - FinOps 与成本优化
  - 混沌工程与韧性设计
  - 源码级排障（etcd、kube-apiserver、调度器）
```
