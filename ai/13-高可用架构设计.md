## 一、SLA 设计深度

### 1.1 SLA 基础概念与等级体系

**服务等级协议（Service Level Agreement）** 是服务提供方与客户之间就服务质量达成的正式承诺。在 LLM 推理服务中，SLA 不仅仅是"几个 9"的数字游戏，而是涉及可用性、性能、数据一致性等多维度的综合承诺体系。

#### 1.1.1 可用性等级与年停机时间对照

| 可用性等级 | 俗称 | 年停机时间 | 月停机时间 | 周停机时间 | 适用场景 |
|-----------|------|-----------|-----------|-----------|----------|
| 99% | 两个 9 | 3.65 天 | 7.3 小时 | 1.68 小时 | 内部工具、非核心测试环境 |
| 99.9% | 三个 9 | 8.76 小时 | 43.8 分钟 | 10.1 分钟 | 一般企业应用、内部 AI 助手 |
| 99.95% | 三个半 9 | 4.38 小时 | 21.9 分钟 | 5.05 分钟 | 重要业务系统 |
| 99.99% | 四个 9 | 52.6 分钟 | 4.38 分钟 | 1.01 分钟 | 金融交易、核心推荐系统 |
| 99.999% | 五个 9 | 5.26 分钟 | 26.3 秒 | 6.05 秒 | 支付清算、证券核心 |
| 99.9999% | 六个 9 | 31.5 秒 | 2.63 秒 | 0.605 秒 | 航天级、军事级系统 |

### 1.3 可用性计算公式深度解析

#### 1.3.1 核心指标定义

**MTBF（Mean Time Between Failures）平均故障间隔时间**

```
MTBF = 总运行时间 / 故障次数

示例：系统运行 30 天（720 小时），发生 2 次故障
MTBF = 720 / 2 = 360 小时
```

**MTTR（Mean Time To Repair）平均修复时间**

```
MTTR = 总修复时间 / 故障次数

示例：2 次故障分别耗时 10 分钟和 20 分钟修复
MTTR = (10 + 20) / 2 = 15 分钟 = 0.25 小时
```

**MTTF（Mean Time To Failure）平均失效时间**

MTTF 用于不可修复组件（如 GPU 卡），MTBF 用于可修复系统。在云计算环境中，由于实例可快速替换，通常关注 MTBF。

#### 1.3.2 可用性计算公式

```
可用性 A = MTBF / (MTBF + MTTR)

示例：
MTBF = 720 小时
MTTR = 0.5 小时（30 分钟自动恢复）
A = 720 / (720 + 0.5) = 720 / 720.5 ≈ 0.999306 ≈ 99.93%

要达到 99.99%：
0.9999 = MTBF / (MTBF + MTTR)
假设 MTTR = 5 分钟 = 0.0833 小时
MTBF = 0.9999 × MTTR / 0.0001 = 0.9999 × 0.0833 / 0.0001 ≈ 833 小时
即平均约 35 天才能出一次故障，且 5 分钟内恢复
```

#### 1.3.3 串联系统可用性

当多个组件串联时，整体可用性为各组件可用性的乘积：

```
A_total = A1 × A2 × A3 × ... × An

示例：LLM 推理链路
- DNS: 99.999%
- CDN: 99.95%
- 负载均衡: 99.99%
- K8s 集群: 99.95%
- LLM 服务: 99.9%
- 存储: 99.99%

A_total = 0.99999 × 0.9995 × 0.9999 × 0.9995 × 0.999 × 0.9999
        ≈ 0.99778 ≈ 99.78%

结论：即使每个组件都高可用，串联后整体可用性会显著下降
```

#### 1.3.4 并联系统可用性

通过冗余并联可以大幅提升可用性：

```
并联 n 个相同组件：
A_parallel = 1 - (1 - A)^n

示例：两个 99.9% 可用性的 LLM 服务并联
A_parallel = 1 - (1 - 0.999)² = 1 - 0.000001 = 99.9999%

三个 99.9% 组件并联：
A_parallel = 1 - (1 - 0.999)³ = 99.999999%
```

#### 1.3.5 LLM 服务可用性计算模型

```python
# 可用性计算模型
class AvailabilityCalculator:
    def __init__(self):
        self.components = {}
    
    def add_component(self, name, availability, redundancy=1, is_critical=True):
        """
        添加组件及其可用性参数
        redundancy: 冗余副本数
        is_critical: 是否为关键路径（串联）
        """
        self.components[name] = {
            'availability': availability,
            'redundancy': redundancy,
            'is_critical': is_critical
        }
    
    def calculate_series(self, components):
        """计算串联可用性"""
        result = 1.0
        for name in components:
            comp = self.components[name]
            if comp['redundancy'] > 1:
                # 并联冗余
                avail = 1 - (1 - comp['availability']) ** comp['redundancy']
            else:
                avail = comp['availability']
            result *= avail
        return result
    
    def calculate_parallel(self, component_name):
        """计算单组件并联可用性"""
        comp = self.components[component_name]
        return 1 - (1 - comp['availability']) ** comp['redundancy']

# 实际计算示例
calc = AvailabilityCalculator()

# 添加 LLM 推理链路组件
calc.add_component('dns', 0.99999, redundancy=2)        # DNS 双活
calc.add_component('cdn', 0.9995, redundancy=1)
calc.add_component('lb', 0.9999, redundancy=2)          # 负载均衡主备
calc.add_component('k8s_control', 0.9999, redundancy=3) # 3 master
calc.add_component('llm_service', 0.999, redundancy=3)  # 3 副本
calc.add_component('storage', 0.9999, redundancy=2)     # 存储双副本
calc.add_component('gpu', 0.995, redundancy=2)          # GPU 冗余

# 计算关键路径可用性
critical_path = ['dns', 'cdn', 'lb', 'k8s_control', 'llm_service', 'storage', 'gpu']
availability = calc.calculate_series(critical_path)
print(f"整体可用性: {availability:.6f} ({availability*100:.4f}%)")
# 输出: 整体可用性: 0.996215 (99.6215%)
```

### 1.4 SLA 衡量与报告体系

#### 1.4.1 SLA 数据采集与计算

```python
# SLA 指标采集器
import time
from datetime import datetime, timedelta
from dataclasses import dataclass
from typing import List, Dict
import statistics

@dataclass
class SLAMetrics:
    timestamp: datetime
    total_requests: int
    successful_requests: int
    failed_requests: int
    total_latency_ms: List[float]
    downtime_seconds: float
    
    @property
    def availability(self) -> float:
        if self.total_requests == 0:
            return 1.0
        return self.successful_requests / self.total_requests
    
    @property
    def latency_p50(self) -> float:
        return statistics.median(self.total_latency_ms)
    
    @property
    def latency_p99(self) -> float:
        sorted_latencies = sorted(self.total_latency_ms)
        idx = int(len(sorted_latencies) * 0.99)
        return sorted_latencies[min(idx, len(sorted_latencies)-1)]
    
    @property
    def error_rate(self) -> float:
        if self.total_requests == 0:
            return 0.0
        return self.failed_requests / self.total_requests

class SLAMonitor:
    def __init__(self, service_name: str, target_sla: Dict):
        self.service_name = service_name
        self.target = target_sla
        self.metrics_history: List[SLAMetrics] = []
        
    def calculate_monthly_sla(self, year: int, month: int) -> Dict:
        """计算月度 SLA 报告"""
        month_metrics = [
            m for m in self.metrics_history
            if m.timestamp.year == year and m.timestamp.month == month
        ]
        
        total_requests = sum(m.total_requests for m in month_metrics)
        successful = sum(m.successful_requests for m in month_metrics)
        total_downtime = sum(m.downtime_seconds for m in month_metrics)
        
        # 月度可用性
        minutes_in_month = 30 * 24 * 60
        availability = 1 - (total_downtime / (minutes_in_month * 60))
        
        # 所有延迟数据
        all_latencies = []
        for m in month_metrics:
            all_latencies.extend(m.total_latency_ms)
        
        sorted_lat = sorted(all_latencies)
        p50 = sorted_lat[int(len(sorted_lat)*0.5)]
        p95 = sorted_lat[int(len(sorted_lat)*0.95)]
        p99 = sorted_lat[int(len(sorted_lat)*0.99)]
        
        return {
            'period': f"{year}-{month:02d}",
            'availability': availability,
            'availability_target': self.target['availability'],
            'is_meeting_target': availability >= self.target['availability'],
            'total_requests': total_requests,
            'total_downtime_seconds': total_downtime,
            'latency_p50_ms': p50,
            'latency_p95_ms': p95,
            'latency_p99_ms': p99,
            'latency_p99_target_ms': self.target.get('latency_p99_ms', 0),
        }
```

```yaml
apiVersion: monitoring.coreos.com/v1
kind: PrometheusRule
metadata:
  name: sla-calculation-rules
  namespace: monitoring
spec:
  groups:
    - name: sla.availability
      interval: 60s

        # 月度可用性计算（rolling 30d）
        - record: sla:availability:monthly
          expr: |
            1 - (
              sum_over_time(
                (up{job="llm-service"} == 0)[30d:1m]
              ) / count_over_time(up{job="llm-service"}[30d])
            )
        
        # 月度错误率
        - record: sla:error_rate:monthly
          expr: |
            sum(rate(http_requests_total{job="llm-service",status=~"5.."}[30d]))
            /
            sum(rate(http_requests_total{job="llm-service"}[30d]))
        
        # 月度 P99 延迟
        - record: sla:latency_p99:monthly
          expr: |
            histogram_quantile(0.99,
              sum(rate(http_request_duration_seconds_bucket{job="llm-service"}[30d])) by (le)
            )
    
    - name: sla.budget
      interval: 300s
      rules:
        # 错误预算消耗率（月度）

            (
              sum_over_time(
                (up{job="llm-service"} == 0)[30d:1m]

            ) / 43.2

        - alert: ErrorBudgetExhausted
```

## 二、单点故障分析与 FMEA

### 2.1 故障模式与影响分析（FMEA）方法论

FMEA（Failure Mode and Effects Analysis）是一种系统化的风险评估方法，通过识别潜在故障模式、评估其影响，制定预防和应对措施。

#### 2.1.1 FMEA 评估矩阵

每个故障模式从三个维度评分（1-10 分），计算风险优先级数 RPN：

```
RPN = 严重度(S) × 发生度(O) × 探测度(D)

严重度(S) - 故障对客户/业务的影响程度
  1-3: 轻微影响，用户无感知
  4-6: 中等影响，部分功能降级
  7-8: 严重影响，核心功能不可用
  9-10: 灾难性影响，数据丢失/资金损失

发生度(O) - 故障发生的可能性
  1-3: 极低，几乎不可能发生
  4-6: 中等，偶尔发生
  7-8: 较高，经常发生
  9-10: 极高，几乎必然发生

探测度(D) - 故障被检测到的难易程度
  1-3: 极易探测，自动化监控全覆盖
  4-6: 中等，需要一定时间发现
  7-8: 较难探测，依赖用户反馈
  9-10: 极难探测，隐蔽性故障
```

## 三、负载均衡深度

### 3.1 四层 vs 七层负载均衡

#### 3.1.1 层次对比详解

| 特性 | L4 负载均衡（传输层） | L7 负载均衡（应用层） |
|------|---------------------|---------------------|
| **工作层级** | TCP/UDP | HTTP/HTTPS/gRPC |
| **负载依据** | IP + Port + 连接信息 | URL + Header + Cookie + Body |
| **性能** | 极高（内核态/硬件） | 较高（用户态处理） |
| **延迟** | < 1ms | 1-5ms |
| **SSL 卸载** | 有限支持 | 完整支持，证书管理 |
| **内容路由** | 不支持 | 支持基于内容的路由 |
| **健康检查** | TCP 连接检查 | HTTP 深度健康检查 |
| **会话保持** | 基于 IP | 基于 Cookie/Header |
| **适用场景** | 数据库、MQ、LLM 原始 TCP | Web API、微服务网关 |
| **代表产品** | AWS NLB、LVS、IPVS | Nginx、Envoy、Traefik、AWS ALB |
| **资源消耗** | 极低 | 中等 |
| **连接状态** | 无状态，可横向扩展 | 可能需要共享会话状态 |

#### 3.2.3 Envoy 负载均衡算法详解

| 算法 | 名称 | 适用场景 | 优点 | 缺点 |
|------|------|----------|------|------|
| ROUND_ROBIN | 轮询 | 请求处理时间均匀 | 简单公平 | 不考虑负载差异 |
| LEAST_REQUEST | 最少请求 | LLM 推理（请求耗时差异大） | 自动避开慢节点 | 需要跟踪请求数 |
| RING_HASH | 一致性哈希 | 需要会话保持 | 相同请求路由到同一节点 | 热点风险 |
| MAGLEV | Maglev 哈希 | 大规模集群 | 负载均衡好，迁移少 | 内存占用较高 |
| RANDOM | 随机 | 大量短连接 | 简单高效 | 不均匀 |
| CLUSTER_PROVIDED | 集群提供 | 自定义场景 | 灵活性高 | 实现复杂 |

### 3.3 会话保持策略

#### 3.3.1 会话保持 vs 无状态设计

LLM 推理服务通常设计为无状态，但在以下场景需要会话保持：

1. **长对话上下文缓存**：某些实现将对话历史缓存在内存中
2. **Prefix Caching**：vLLM 等框架对相同前缀的请求复用 KV Cache
3. **多轮工具调用**：Function Calling 的多轮交互需要状态
4. **模型热加载**：避免同一用户的请求路由到不同模型版本

```python
# 方案 2: 客户端侧的会话保持（推荐，服务完全无状态）
import hashlib

class SessionAwareRouter:
    def __init__(self, endpoints):
        self.endpoints = endpoints
    
    def get_endpoint(self, session_id: str) -> str:
        hash_value = int(hashlib.md5(session_id.encode()).hexdigest(), 16)
        idx = hash_value % len(self.endpoints)
        return self.endpoints[idx]
    
    def get_endpoint_with_replication(self, session_id: str, replicas: int = 2) -> list:
        hash_value = int(hashlib.md5(session_id.encode()).hexdigest(), 16)
        endpoints = []
        for i in range(replicas):
            idx = (hash_value + i) % len(self.endpoints)
            endpoints.append(self.endpoints[idx])
        return endpoints

# 使用示例
router = SessionAwareRouter([
    "llm-pod-1.llm.svc:8000",
    "llm-pod-2.llm.svc:8000",
    "llm-pod-3.llm.svc:8000"
])

session_id = "user-12345-conversation-678"
endpoint = router.get_endpoint(session_id)
print(f"Session {session_id} routed to {endpoint}")
```

#### 3.3.3 会话保持的最佳实践

| 策略 | 优点 | 缺点 | 推荐场景 |
|------|------|------|----------|
| 无状态 + 外部缓存 | 完全弹性，无单点 | 额外网络开销 | 首选方案 |
| Cookie 哈希 | 简单，客户端无感知 | 首次请求无 Cookie | Web 应用 |
| Header 哈希 | 灵活，API 友好 | 需要客户端配合 | API 服务 |
| Source IP 哈希 | 无侵入 | NAT 场景失效 | 内部服务 |
| Pod 本地缓存 | 零网络开销 | 弹性差 | 特定优化场景 |

### 3.4 健康检查机制

#### 3.4.1 主动健康检查 vs 被动健康检查

```
主动健康检查 (Active Health Check)
  - LB 周期性发送探测请求到后端
  - 检查类型：TCP / HTTP / gRPC / Custom
  - 优点：提前发现问题，精确控制
  - 缺点：额外流量开销，可能干扰服务
  - 适用：所有生产环境必须配置

被动健康检查 (Passive Health Check / Outlier Detection)
  - 分析实际请求的成功/失败/延迟
  - 自动驱逐异常节点
  - 优点：基于真实流量，无额外开销
  - 缺点：故障发现延迟，需要积累错误样本
  - 适用：与主动检查配合使用
```

```python
# vLLM 推理服务健康检查端点实现
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
import torch
import psutil
import time
from typing import Dict, Optional
import asyncio

app = FastAPI()

class HealthStatus:
    def __init__(self):
        self.model_loaded = False
        self.last_inference_time = time.time()
        self.inference_error_count = 0
        self.total_inference_count = 0
        self.gpu_memory_peak = 0.0
        
    def get_gpu_status(self) -> Dict:
        if not torch.cuda.is_available():
            return {"available": False, "error": "CUDA not available"}
        
        gpu_stats = []
        for i in range(torch.cuda.device_count()):
            props = torch.cuda.get_device_properties(i)
            allocated = torch.cuda.memory_allocated(i) / 1024**3
            reserved = torch.cuda.memory_reserved(i) / 1024**3
            total = props.total_memory / 1024**3
            utilization = allocated / total
            
            gpu_stats.append({
                "index": i,
                "name": props.name,
                "total_memory_gb": round(total, 2),
                "allocated_memory_gb": round(allocated, 2),
                "reserved_memory_gb": round(reserved, 2),
                "utilization": round(utilization, 4),
                "temperature": self._get_gpu_temperature(i),
            })
        return {"available": True, "devices": gpu_stats}
    
    def _get_gpu_temperature(self, gpu_index: int) -> Optional[float]:
        try:
            import pynvml
            pynvml.nvmlInit()
            handle = pynvml.nvmlDeviceGetHandleByIndex(gpu_index)
            temp = pynvml.nvmlDeviceGetTemperature(handle, pynvml.NVML_TEMPERATURE_GPU)
            return temp
        except:
            return None
    
    def get_system_status(self) -> Dict:
        return {
            "cpu_percent": psutil.cpu_percent(interval=1),
            "memory_percent": psutil.virtual_memory().percent,
            "disk_usage_percent": psutil.disk_usage('/').percent,
            "load_average": psutil.getloadavg(),
        }

health_status = HealthStatus()

@app.get("/health")
async def health_check():
    return {"status": "alive", "timestamp": time.time()}

@app.get("/health/ready")
async def readiness_check():
    checks = {
        "model_loaded": health_status.model_loaded,
        "gpu_available": torch.cuda.is_available(),
        "gpu_memory_ok": True,
        "recent_inference_ok": True,
    }
    
    gpu_status = health_status.get_gpu_status()
    if gpu_status["available"]:
        for gpu in gpu_status["devices"]:
            if gpu["utilization"] > 0.98:
                checks["gpu_memory_ok"] = False
            if gpu["temperature"] and gpu["temperature"] > 85:
                checks["gpu_temperature_ok"] = False
    
    if health_status.total_inference_count > 10:
        error_rate = health_status.inference_error_count / health_status.total_inference_count
        if error_rate > 0.5:
            checks["recent_inference_ok"] = False
    
    all_healthy = all(checks.values())
    
    if not all_healthy:
        raise HTTPException(
            status_code=503,
            detail={"status": "not_ready", "checks": checks}
        )
    
    return {"status": "ready", "checks": checks}

@app.get("/health/deep")
async def deep_health_check():
    gpu_status = health_status.get_gpu_status()
    system_status = health_status.get_system_status()
    
    score = 100
    issues = []
    
    if not gpu_status["available"]:
        score -= 50
        issues.append("GPU not available")
    else:
        for gpu in gpu_status["devices"]:
            if gpu["utilization"] > 0.95:
                score -= 20
                issues.append(f"GPU {gpu['index']} memory almost full")
            if gpu["temperature"] and gpu["temperature"] > 80:
                score -= 10
                issues.append(f"GPU {gpu['index']} temperature high")
    
    if system_status["cpu_percent"] > 90:
        score -= 10
        issues.append("CPU usage high")
    
    status = "healthy" if score >= 80 else "degraded" if score >= 50 else "unhealthy"
    
    return {
        "status": status,
        "score": score,
        "issues": issues,
        "gpu": gpu_status,
        "system": system_status,
        "inference_stats": {
            "total": health_status.total_inference_count,
            "errors": health_status.inference_error_count,
            "error_rate": round(health_status.inference_error_count / max(health_status.total_inference_count, 1), 4),
            "last_inference_ago_sec": round(time.time() - health_status.last_inference_time, 2),
        }
    }
```

```yaml

        expr: |
          avg(
            rate(http_requests_total{job="llm-service"}[5m])
            and on() (day_of_week() >= 1 and day_of_week() <= 5)
          ) by (hour, minute)
      
      - record: predict:qps:composite
```

### 5.1 RTO/RPO 设计

#### 5.1.1 核心概念

| 指标 | 全称 | 定义 | 说明 |
|------|------|------|------|
| **RTO** | Recovery Time Objective | 恢复时间目标 | 从故障发生到服务恢复的最大允许时间 |
| **RPO** | Recovery Point Objective | 恢复点目标 | 灾难发生后可接受的数据丢失量 |
| **WRT** | Work Recovery Time | 工作恢复时间 | 恢复后验证服务正常的时间 |
| **MTD** | Maximum Tolerable Downtime | 最大可容忍停机时间 | MTD = RTO + WRT |

#### 5.1.2 不同灾备等级的 RTO/RPO

| 灾备等级 | 名称 | RTO | RPO | 架构模式 | 成本 |
|----------|------|-----|-----|----------|------|
| Tier 0 | 无灾备 | 数天 | 数天 | 无备份 | 低 |
| Tier 1 | 冷备 | 数小时~数天 | 24h~数天 | 定期备份 | 低 |
| Tier 2 | 温备 | 1~4 小时 | 1~24h | 定期同步 + 手动切换 | 中 |
| Tier 3 | 热备 | 15~60 分钟 | 15min~1h | 实时同步 + 半自动切换 | 中高 |
| Tier 4 | 同城双活 | < 5 分钟 | 0~秒级 | 双活架构 | 高 |
| Tier 5 | 异地多活 | < 1 分钟 | 0 | 多活架构 | 极高 |
| Tier 6 | 零 RPO | 0 | 0 | 同步多写 | 极高 |

```bash
        "Type": "A",

        "Weight": 0,

      "Action": "UPSERT",
      "ResourceRecordSet": {
        "Name": "$SERVICE_NAME.example.com",
        "Type": "A",

        "TTL": 60,
```

### 6.1 限流算法详解

#### 6.1.1 令牌桶算法（Token Bucket）

```
原理：
- 桶以固定速率生成令牌
- 每个请求消耗一个令牌
- 桶满时多余的令牌被丢弃
- 桶空时请求被拒绝

优点：
- 允许突发流量（桶中有足够令牌时）
- 平滑长期流量
- 实现简单

缺点：
- 突发可能超过预期
- 需要维护令牌状态

适用：API 限流、带宽控制
```

```python
import time
import threading
from collections import deque

class TokenBucket:
    def __init__(self, rate: float, capacity: int):
        """
        rate: 每秒生成的令牌数
        capacity: 桶的最大容量
        """
        self.rate = rate
        self.capacity = capacity
        self.tokens = capacity
        self.last_update = time.time()
        self.lock = threading.Lock()
    
    def _add_tokens(self):
        now = time.time()
        elapsed = now - self.last_update
        self.tokens = min(
            self.capacity,
            self.tokens + elapsed * self.rate
        )
        self.last_update = now
    
    def allow_request(self, tokens: int = 1) -> bool:
        with self.lock:
            self._add_tokens()
            if self.tokens >= tokens:
                self.tokens -= tokens
                return True
            return False
    
    def get_wait_time(self, tokens: int = 1) -> float:
        """计算需要等待多久才能获取令牌"""
        with self.lock:
            self._add_tokens()
            if self.tokens >= tokens:
                return 0.0
            needed = tokens - self.tokens
            return needed / self.rate

# 使用示例：1000 QPS，桶容量 2000（允许 2 秒突发）
bucket = TokenBucket(rate=1000, capacity=2000)

    if bucket.allow_request():
        process_request()
    else:
        return {"error": "Rate limit exceeded", "retry_after": bucket.get_wait_time()}
```

#### 6.1.2 漏桶算法（Leaky Bucket）

```
原理：
- 请求进入漏桶（队列）
- 以固定速率流出处理
- 桶满时新请求被拒绝

优点：
- 严格的速率限制
- 无突发
- 输出绝对平滑

缺点：
- 无法应对突发流量
- 队列可能堆积

适用：需要严格平滑输出的场景
```

```python
import time
import threading
from queue import Queue, Full

class LeakyBucket:
    def __init__(self, rate: float, capacity: int):
        """
        rate: 每秒处理请求数
        capacity: 队列最大长度
        """
        self.rate = rate
        self.capacity = capacity
        self.queue = Queue(maxsize=capacity)
        self.lock = threading.Lock()
        self.running = True
        
        # 启动漏桶处理线程
        self.worker = threading.Thread(target=self._leak)
        self.worker.daemon = True
        self.worker.start()
    
    def _leak(self):
        """以固定速率处理请求"""
        interval = 1.0 / self.rate
        while self.running:
            try:
                request = self.queue.get(timeout=interval)
                self._process(request)
            except:
                continue
    
    def _process(self, request):
        """实际处理请求"""
        # 处理逻辑
        pass
    
    def add_request(self, request) -> bool:
        """添加请求到漏桶"""
        try:
            self.queue.put_nowait(request)
            return True
        except Full:
            return False
    
    def stop(self):
        self.running = False
        self.worker.join()
```

#### 6.1.3 滑动窗口算法（Sliding Window）

```
原理：
- 维护最近时间窗口内的请求计数
- 精确统计单位时间内的请求数
- 无突发，精确控制

优点：
- 最精确的限流
- 无突发问题
- 公平

缺点：
- 内存消耗大（需要存储每个请求时间戳）
- 计算开销较高

适用：需要精确计费的场景
```

```python
import time
import threading
from collections import deque

class SlidingWindowCounter:
    def __init__(self, window_size: int, max_requests: int):
        """
        window_size: 窗口大小（秒）
        max_requests: 窗口内最大请求数
        """
        self.window_size = window_size
        self.max_requests = max_requests
        self.requests = deque()
        self.lock = threading.Lock()
    
    def allow_request(self) -> bool:
        now = time.time()
        
        with self.lock:
            # 移除窗口外的请求
            while self.requests and self.requests[0] < now - self.window_size:
                self.requests.popleft()
            
            if len(self.requests) < self.max_requests:
                self.requests.append(now)
                return True
            return False
    
    def get_current_count(self) -> int:
        now = time.time()
        with self.lock:
            while self.requests and self.requests[0] < now - self.window_size:
                self.requests.popleft()
            return len(self.requests)

# 优化版：滑动窗口日志（更节省内存）
class SlidingWindowLog:
    def __init__(self, window_size: int, max_requests: int):
        self.window_size = window_size
        self.max_requests = max_requests
        self.log = {}
        self.lock = threading.Lock()
    
    def allow_request(self, key: str) -> bool:
        now = int(time.time())
        
        with self.lock:
            if key not in self.log:
                self.log[key] = {}
            
            # 清理旧数据
            window_start = now - self.window_size
            self.log[key] = {t: c for t, c in self.log[key].items() if t >= window_start}
            
            # 计算当前窗口总请求数
            total = sum(self.log[key].values())
            
            if total < self.max_requests:
                self.log[key][now] = self.log[key].get(now, 0) + 1
                return True
            return False
```

### 6.3 自适应限流（根据系统负载）

```python
import time
import threading
import statistics
from typing import Callable

class AdaptiveRateLimiter:
    """
    自适应限流器：根据系统负载动态调整限流阈值
    """
    
    def __init__(
        self,
        initial_limit: int = 1000,
        min_limit: int = 10,
        max_limit: int = 10000,
        adjustment_interval: int = 10,
        latency_threshold_ms: float = 500,
        error_rate_threshold: float = 0.05,
        cpu_threshold: float = 80.0
    ):
        self.current_limit = initial_limit
        self.min_limit = min_limit
        self.max_limit = max_limit
        self.adjustment_interval = adjustment_interval
        
        # 阈值
        self.latency_threshold_ms = latency_threshold_ms
        self.error_rate_threshold = error_rate_threshold
        self.cpu_threshold = cpu_threshold
        
        # 统计
        self.request_count = 0
        self.error_count = 0
        self.latencies = []
        self.lock = threading.Lock()
        
        # 启动调整线程
        self.adjustment_thread = threading.Thread(target=self._adjust_loop, daemon=True)
        self.adjustment_thread.start()
    
    def record_request(self, latency_ms: float, success: bool):
        """记录请求结果"""
        with self.lock:
            self.request_count += 1
            if not success:
                self.error_count += 1
            self.latencies.append(latency_ms)
            # 保持最近 1000 个样本
            if len(self.latencies) > 1000:
                self.latencies = self.latencies[-1000:]
    
    def allow_request(self) -> bool:
        """是否允许请求通过"""
        with self.lock:
            if self.request_count >= self.current_limit:
                return False
            self.request_count += 1
            return True
    
    def _get_metrics(self) -> dict:
        """获取当前指标"""
        with self.lock:
            if not self.latencies:
                return {"p99_latency": 0, "error_rate": 0}
            
            sorted_lat = sorted(self.latencies)
            p99_idx = int(len(sorted_lat) * 0.99)
            p99_latency = sorted_lat[min(p99_idx, len(sorted_lat)-1)]
            
            error_rate = self.error_count / max(self.request_count, 1)
            
            return {
                "p99_latency": p99_latency,
                "error_rate": error_rate,
                "current_limit": self.current_limit
            }
    
    def _adjust_loop(self):
        """定期调整限流阈值"""
        while True:
            time.sleep(self.adjustment_interval)
            self._adjust_limit()
    
    def _adjust_limit(self):
        """根据系统状态调整限流阈值"""
        metrics = self._get_metrics()
        p99 = metrics["p99_latency"]
        error_rate = metrics["error_rate"]
        
        with self.lock:
            # 判断系统状态
            is_overloaded = (
                p99 > self.latency_threshold_ms or
                error_rate > self.error_rate_threshold
            )
            
            if is_overloaded:
                # 系统过载，降低限流
                reduction = max(1, int(self.current_limit * 0.2))
                self.current_limit = max(self.min_limit, self.current_limit - reduction)
                print(f"[Adaptive] System overloaded (p99={p99:.1f}ms, error={error_rate:.2%}). "
                      f"Reducing limit to {self.current_limit}")
            else:
                # 系统健康，尝试提高限流
                increase = max(1, int(self.current_limit * 0.1))
                self.current_limit = min(self.max_limit, self.current_limit + increase)
                print(f"[Adaptive] System healthy (p99={p99:.1f}ms, error={error_rate:.2%}). "
                      f"Increasing limit to {self.current_limit}")
            
            # 重置计数器
            self.request_count = 0
            self.error_count = 0
            self.latencies = []

# 集成到服务中
limiter = AdaptiveRateLimiter(
    initial_limit=100,
    latency_threshold_ms=2000,
    error_rate_threshold=0.01
)

def handle_inference_request(request):

        return {"error": "Service overloaded", "retry_after": 5}
    
    start = time.time()
    try:
        result = do_inference(request)
        latency = (time.time() - start) * 1000
        limiter.record_request(latency, True)
        return result
    except Exception as e:
        latency = (time.time() - start) * 1000
        limiter.record_request(latency, False)
        raise
```

```

│                                                       │     │

│   • CLOSED -> OPEN: 连续失败 >= failure_threshold     │     │
│   • OPEN -> HALF-OPEN: timeout 到达                   │     │
│   • HALF-OPEN -> CLOSED: 连续成功 >= success_threshold│     │

│                                                       │     │
└───────────────────────────────────────────────────────┘     │
```

```python
import time
import threading
from enum import Enum
from typing import Callable, Optional

class CircuitState(Enum):
    CLOSED = "closed"
    OPEN = "open"
    HALF_OPEN = "half_open"

class CircuitBreaker:
    def __init__(
        self,
        failure_threshold: int = 5,
        recovery_timeout: float = 30.0,
        half_open_max_calls: int = 3,
        success_threshold: int = 2,
        expected_exception: type = Exception
    ):
        self.failure_threshold = failure_threshold
        self.recovery_timeout = recovery_timeout
        self.half_open_max_calls = half_open_max_calls
        self.success_threshold = success_threshold
        self.expected_exception = expected_exception
        
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.success_count = 0
        self.half_open_calls = 0
        self.last_failure_time = None
        self.lock = threading.Lock()
    
    def call(self, func: Callable, *args, **kwargs):
        """执行被保护的方法"""
        with self.lock:
            if self.state == CircuitState.OPEN:
                if self._should_attempt_reset():
                    self.state = CircuitState.HALF_OPEN
                    self.half_open_calls = 0
                    self.success_count = 0
                else:
                    raise CircuitBreakerOpen("Circuit breaker is OPEN")
            
            elif self.state == CircuitState.HALF_OPEN:
                if self.half_open_calls >= self.half_open_max_calls:
                    raise CircuitBreakerOpen("Half-open call limit reached")
                self.half_open_calls += 1
        
        # 执行实际调用
        try:
            result = func(*args, **kwargs)
            self._on_success()
            return result
        except self.expected_exception as e:
            self._on_failure()
            raise
    
    def _on_success(self):
        with self.lock:
            if self.state == CircuitState.HALF_OPEN:
                self.success_count += 1
                if self.success_count >= self.success_threshold:
                    self._close_circuit()
            else:
                self.failure_count = 0
    
    def _on_failure(self):
        with self.lock:
            self.failure_count += 1
            self.last_failure_time = time.time()
            
            if self.state == CircuitState.HALF_OPEN:
                self._open_circuit()
            elif self.failure_count >= self.failure_threshold:
                self._open_circuit()
    
    def _open_circuit(self):
        self.state = CircuitState.OPEN
        self.last_failure_time = time.time()
        print(f"[CircuitBreaker] OPENED at {time.time()}")
    
    def _close_circuit(self):
        self.state = CircuitState.CLOSED
        self.failure_count = 0
        self.success_count = 0
        self.half_open_calls = 0
        print(f"[CircuitBreaker] CLOSED at {time.time()}")
    
    def _should_attempt_reset(self) -> bool:
        if self.last_failure_time is None:
            return True
        return (time.time() - self.last_failure_time) >= self.recovery_timeout
    
    @property
    def current_state(self) -> str:
        return self.state.value

class CircuitBreakerOpen(Exception):
    pass

# 使用示例

    failure_threshold=5,
    recovery_timeout=30.0,
    success_threshold=2
)

def call_llm_api(prompt: str):
    return breaker.call(requests.post, url, json={"prompt": prompt}, timeout=30)
```

### 7.2 告警分级（P0/P1/P2/P3）

| 级别 | 名称 | 响应时间 | 升级时间 | 通知方式 | 示例 |
|------|------|----------|----------|----------|------|
| **P0** | 紧急 | 5 分钟 | 15 分钟 | 电话 + 短信 + 钉钉 | GPU Xid 错误、服务完全不可用 |
| **P1** | 严重 | 15 分钟 | 30 分钟 | 短信 + 钉钉 + 邮件 | P99 延迟 > 5s、错误率 > 5% |
| **P2** | 警告 | 1 小时 | 4 小时 | 钉钉 + 邮件 | 队列积压 > 50、节点异常 |
| **P3** | 提示 | 4 小时 | 24 小时 | 邮件 | 资源利用率趋势异常、版本过期 |
| **P4** | 信息 | 无 | 无 | 仅记录 | 常规运维信息、统计报告 |

### 7.4 告警降噪

```yaml
# AlertManager 告警降噪配置
global:
  smtp_smarthost: 'smtp.example.com:587'
  smtp_from: 'alerts@example.com'

# 告警分组
group_by: ['alertname', 'cluster', 'service']
group_wait: 30s
group_interval: 5m
repeat_interval: 4h

# 抑制规则（inhibition）
inhibit_rules:
  # 规则 1: 节点宕机时抑制该节点上的 Pod 告警
  - source_match:
      alertname: 'NodeDown'
      severity: 'critical'
    target_match_re:
      alertname: 'PodNotReady|PodCrashLoopBackOff|GPUNotAvailable'
    equal: ['node']
  
  # 规则 2: 集群不可用时抑制集群内所有服务告警
  - source_match:
      alertname: 'ClusterUnavailable'
      severity: 'critical'
    target_match_re:
      alertname: '.*'
    equal: ['cluster']
  
  # 规则 3: 磁盘满时抑制磁盘相关的次要告警
  - source_match:
      alertname: 'DiskFull'
      severity: 'critical'
    target_match:
      alertname: 'DiskWillFullIn4Hours'
    equal: ['instance', 'device']
  
  # 规则 4: 高错误率时抑制单个 Pod 错误告警
  - source_match:
      alertname: 'HighErrorRate'
      severity: 'critical'
    target_match_re:
      alertname: 'PodHighErrorRate|RequestTimeout'
    equal: ['service']

# 静默规则（silence）
# 通过 AlertManager UI 或 API 创建
# 例如：维护窗口期间静默所有非 P0 告警

# 告警路由
templates:
  - '/etc/alertmanager/templates/*.tmpl'

route:
  receiver: 'default'
  group_by: ['alertname', 'cluster', 'service']
  group_wait: 30s
  group_interval: 5m
  repeat_interval: 4h
  
  routes:
    # P0 告警：立即电话通知
    - match:
        severity: critical
      receiver: 'p0-team'
      group_wait: 0s
      repeat_interval: 5m
      continue: true
    
    # P1 告警：短信 + IM
    - match:
        severity: warning
      receiver: 'p1-team'
      group_wait: 1m
      repeat_interval: 30m
      continue: true
    
    # 业务告警路由到业务团队
    - match_re:
        service: 'llm-.*'
      receiver: 'llm-team'
      routes:
        - match:
            alertname: 'GPUXidError'
          receiver: 'gpu-oncall'
        - match:
            alertname: 'LLMHighLatency'
          receiver: 'llm-perf-team'
    
    # 基础设施告警路由到 SRE
    - match_re:
        service: 'k8s|network|storage'
      receiver: 'sre-team'

receivers:
  - name: 'default'
    email_configs:
      - to: 'ops@example.com'
  
  - name: 'p0-team'
    pagerduty_configs:
      - service_key: '<PAGERDUTY-KEY>'
        severity: critical
    slack_configs:
      - api_url: '<SLACK-WEBHOOK>'
        channel: '#alerts-p0'
        title: 'P0 ALERT: {{ .GroupLabels.alertname }}'
        text: '{{ range .Alerts }}{{ .Annotations.summary }}{{ end }}'
    webhook_configs:
      - url: 'http://phone-gateway/alert'
        send_resolved: true
  
  - name: 'p1-team'
    slack_configs:
      - api_url: '<SLACK-WEBHOOK>'
        channel: '#alerts-p1'
        title: 'P1 ALERT: {{ .GroupLabels.alertname }}'
    email_configs:
      - to: 'oncall@example.com'
  
  - name: 'llm-team'
    slack_configs:
      - api_url: '<SLACK-WEBHOOK>'
        channel: '#llm-alerts'
  
  - name: 'gpu-oncall'
    pagerduty_configs:
      - service_key: '<GPU-PAGERDUTY-KEY>'
        severity: error
  
  - name: 'sre-team'
    slack_configs:
      - api_url: '<SLACK-WEBHOOK>'
        channel: '#sre-alerts'
```

### 8.2 交互流程图

```
用户请求处理流程：

1. 用户 ──HTTP──▶ CDN/WAF ──▶ DNS GSLB
   • DNS 返回最优区域入口 IP
   • TTL: 60s（支持快速故障切换）

2. DNS ──▶ Envoy Gateway (Layer 7)
   • SSL 终止
   • 认证检查 (JWT/API Key)
   • 限流检查 (Rate Limit)
   • 请求路由 (基于 model 参数)
   • 熔断检查 (Circuit Breaker)

3. Envoy ──▶ Kubernetes Service (Layer 4)
   • IPVS 负载均衡
   • 会话保持 (可选)
   • 健康检查过滤

4. Service ──▶ LLM Pod
   • vLLM/TGI 推理引擎
   • Continuous Batching
   • Prefix Caching
   • GPU 调度

5. Pod 内部处理：
   • 检查本地模型缓存
   • 加载模型权重 (如未缓存)
   • 执行推理
   • 返回流式/非流式响应

6. 响应返回路径：
   Pod ──▶ Service ──▶ Envoy ──▶ CDN ──▶ 用户
   • 响应压缩 (Gzip/Brotli)
   • 缓存控制头
   • 链路追踪 ID

故障处理流程：

A. 单 Pod 故障：
   K8s Liveness Probe 失败 ──▶ Pod 重启
   Readiness Probe 失败 ──▶ 从 Service 摘除
   Karpenter 自动调度到新节点

B. 单节点故障：
   Node NotReady ──▶ Pod 驱逐
   HPA 补充新副本
   Karpenter 创建新节点

C. 区域故障：
   健康检查失败 ──▶ DNS 权重调整
   流量路由到健康区域
   跨区域数据同步验证

D. 全链路故障：
   进入降级模式 (Level 3/4)
   返回静态兜底响应
   触发 P0 告警
   启动 War Room
```

## 十、高可用理论基础

### 10.1 CAP 定理在 AI Infra 中的应用

#### 10.1.1 CAP 定理的核心内涵

CAP 定理（Brewer's Theorem）指出，分布式系统无法同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance）这三个特性，最多只能同时满足其中两个。在分布式系统设计中， partition tolerance 是必须保证的（因为网络分区不可避免），因此实际的选择往往在 **CP**（一致性+分区容错）和 **AP**（可用性+分区容错）之间做出权衡。

```
┌─────────────────────────────────────────────────────────────────┐
│                        CAP 定理三角                              │
│                                                                 │
│                           Consistency                           │
│                         （一致性）                               │
│                              ▲                                  │
│                             / \                                 │
│                            /   \                                │
│                           /     \                               │
│                          /   ?   \                              │
│                         /         \                             │
│         Availability   /─────────\   Partition Tolerance       │
│           （可用性）                        （分区容错）         │
│                                                                 │
│   CP 系统: etcd / ZooKeeper / HBase                            │
│   AP 系统: Cassandra / DynamoDB / Eureka                       │
│   CA 系统: 传统单节点数据库（非分布式）                         │
└─────────────────────────────────────────────────────────────────┘
```

#### 10.1.2 LLM 推理基础设施的 CAP 分析

在 LLM 推理基础设施中，不同的组件面临不同的 CAP 权衡：

| 组件 | 优先保证 | 权衡策略 | 原因 |
|------|----------|----------|------|
| **模型权重存储** | CP | 强一致性优先 | 模型文件损坏会导致输出完全错误，不可接受 |
| **推理服务本身** | AP | 可用性优先 | 用户请求不能因网络抖动而失败 |
| **会话/上下文缓存** | AP | 最终一致性 | 短暂的数据不一致可接受，但不能拒绝对话 |
| **向量数据库** | CP/AP 可选 | 查询 AP，写入 CP | 检索可容忍近似结果，索引必须一致 |
| **配置中心** | CP | 强一致性 | 错误的配置会导致全局故障 |
| **监控指标** | AP | 可用性优先 | 允许丢失部分采样点，不能中断数据采集 |
| **日志系统** | AP | 最终一致性 | 日志可延迟聚合，不能阻塞主流程 |
| **计费/审计** | CP | 强一致性 | 资金相关必须精确，可接受短暂延迟 |

#### 10.1.3 实际案例：网络分区下的推理服务

假设两个可用区之间发生网络分区，分析不同组件的行为：

```
场景：AZ-A 与 AZ-B 之间网络断开

CP 策略（配置中心 etcd）：
  - etcd 集群进入只读模式或少数派不可用
  - 保证已提交数据的一致性
  - 可能牺牲部分可用性
  - 适用于：模型版本配置、服务发现

AP 策略（推理服务）：
  - 每个 AZ 的推理服务继续响应请求
  - 使用本地缓存的模型和配置
  - 可能产生短暂的不一致（如新旧模型并存）
  - 适用于：实时推理、对话服务

混合策略（会话缓存）：
  - Redis Cluster 采用主从模式
  - 分区期间各自独立处理写入
  - 网络恢复后通过冲突解决策略合并
  - 牺牲部分一致性换取持续可用
```

#### 10.1.4 PACELC 定理：CAP 的补充

PACELC 定理是 CAP 的扩展，指出即使没有网络分区（Partition），也存在延迟（Latency）和一致性（Consistency）的权衡：

```
PACELC: If there is a Partition (P),
        how does the system trade off Availability (A) and Consistency (C)?
        Else (E),
        how does the system trade off Latency (L) and Consistency (C)?
```

在 LLM 推理场景中：

| 场景 | 策略 | 说明 |
|------|------|------|
| 无分区，低延迟要求 | 牺牲一致性 | 使用本地缓存，异步同步 |
| 无分区，高一致性要求 | 牺牲延迟 | 同步写入多副本，等待确认 |
| 有分区，高可用要求 | 牺牲一致性 | 分区各自服务，最终合并 |
| 有分区，高一致性要求 | 牺牲可用性 | 少数派拒绝服务，等待恢复 |

#### 10.2.3 错误预算（Error Budget）计算方法

错误预算是 SRE 实践中的核心概念，将可用性目标转化为可消耗的"预算"。

```
错误预算 = 1 - SLO（以比例表示）

示例：
SLO = 99.9% (三个 9)
错误预算 = 0.1% = 0.001

月度错误预算（按 30 天计算）：
= 30 天 × 24 小时 × 60 分钟 × 60 秒 × 0.001
= 2592 秒
= 43.2 分钟

即：一个月内，服务总不可用时间不能超过 43.2 分钟
```

**错误预算消耗监控**：

```python
class ErrorBudgetCalculator:
    """错误预算计算器"""
    
    def __init__(self, slo: float, window_days: int = 30):
        self.slo = slo
        self.window_seconds = window_days * 24 * 3600
        self.budget_seconds = self.window_seconds * (1 - slo)
    
    def calculate_consumption(self, downtime_seconds: float) -> dict:
        """计算错误预算消耗情况"""
        consumption_rate = downtime_seconds / self.budget_seconds
        remaining_budget = max(0, self.budget_seconds - downtime_seconds)
        remaining_percentage = (remaining_budget / self.budget_seconds) * 100
        
        return {
            "slo": f"{self.slo*100:.3f}%",
            "total_budget_seconds": self.budget_seconds,
            "total_budget_minutes": self.budget_seconds / 60,
            "consumed_seconds": downtime_seconds,
            "consumed_minutes": downtime_seconds / 60,
            "consumption_rate": consumption_rate,
            "remaining_seconds": remaining_budget,
            "remaining_percentage": remaining_percentage,
            "status": "HEALTHY" if consumption_rate < 0.5 else 
                     "WARNING" if consumption_rate < 1.0 else "EXHAUSTED",
            "projected_monthly": self._project_monthly(downtime_seconds),
        }
    
    def _project_monthly(self, current_downtime: float) -> dict:
        """基于当前消耗速度，预测月度总消耗"""
        from datetime import datetime
        day_of_month = datetime.now().day
        month_progress = day_of_month / 30.0
        
        if month_progress == 0:
            return {"projected_seconds": 0, "projected_rate": 0}
        
        projected_total = current_downtime / month_progress
        projected_rate = projected_total / self.budget_seconds
        
        return {
            "projected_seconds": projected_total,
            "projected_minutes": projected_total / 60,
            "projected_rate": projected_rate,
            "will_exceed_budget": projected_total > self.budget_seconds,
        }

# 使用示例
calc = ErrorBudgetCalculator(slo=0.999, window_days=30)
result = calc.calculate_consumption(downtime_seconds=1200)
print(f"错误预算状态: {result['status']}")
print(f"已消耗: {result['consumed_minutes']:.1f} 分钟 / {result['total_budget_minutes']:.1f} 分钟")
```

#### 10.2.4 SLO 设计原则与反模式

**设计原则：**

1. **可测量性**：SLI 必须能通过监控系统精确采集
2. **代表性**：SLI 必须真实反映用户体验
3. **可实现性**：SLO 必须基于历史数据，目标合理
4. **分层设计**：从用户级 SLO 分解到组件级 SLO
5. **预留缓冲**：内部 SLO 应比对外 SLA 更严格

```
SLA vs SLO 缓冲设计：
对外 SLA: 99.9% (月停机 43.2 分钟)
内部 SLO: 99.95% (月停机 21.6 分钟)
缓冲: 21.6 分钟（用于应对突发事件）
```

**常见反模式：**

| 反模式 | 问题 | 正确做法 |
|--------|------|----------|
| 过度承诺 | SLA 超出技术能力 | 基于历史数据设定 |
| 指标虚荣 | 选择容易达标的指标 | 选择用户感知指标 |
| SLO 过多 | 维护成本过高 | 每个服务 3-5 个核心 SLO |
| 永不调整 | 业务变化后 SLO 过时 | 每季度评审 SLO |
| 忽视长尾 | 只看平均值 | 关注 P99/P99.9 |
| 无错误预算 | 团队不敢发布 | 建立错误预算文化 |

#### 10.3.3 故障域依赖分析

```python
class FailureDomainAnalyzer:
    """故障域依赖分析器"""
    
    def __init__(self):
        self.domains = {}
        self.dependencies = {}
    
    def add_domain(self, name: str, domain_type: str, redundancy: int):
        """添加故障域"""
        self.domains[name] = {
            "type": domain_type,
            "redundancy": redundancy,
            "failure_probability": self._estimate_failure_prob(domain_type),
        }
    
    def add_dependency(self, from_domain: str, to_domain: str, critical: bool = True):
        """添加依赖关系"""
        if from_domain not in self.dependencies:
            self.dependencies[from_domain] = []
        self.dependencies[from_domain].append({
            "to": to_domain,
            "critical": critical,
        })
    
    def _estimate_failure_prob(self, domain_type: str) -> float:
        """估计故障概率（每年）"""
        probabilities = {
            "pod": 0.1,
            "node": 0.05,
            "rack": 0.02,
            "az": 0.01,
            "region": 0.005,
            "cloud": 0.001,
        }
        return probabilities.get(domain_type, 0.01)
    
    def analyze_cascading_failure(self, failed_domain: str) -> dict:
        """分析级联故障影响"""
        affected = set()
        queue = [failed_domain]
        
        while queue:
            current = queue.pop(0)
            affected.add(current)
            
            for domain, deps in self.dependencies.items():
                for dep in deps:
                    if dep["to"] == current and dep["critical"]:
                        if domain not in affected:
                            queue.append(domain)
                            affected.add(domain)
        
        return {
            "failed_domain": failed_domain,
            "cascading_affected": list(affected),
            "total_affected": len(affected),
            "risk_level": "HIGH" if len(affected) > 5 else 
                         "MEDIUM" if len(affected) > 2 else "LOW",
        }
    
    def find_single_points_of_failure(self) -> list:
        """查找单点故障"""
        spofs = []
        
        for domain, info in self.domains.items():
            if info["redundancy"] == 1:
                dependent_count = sum(
                    1 for deps in self.dependencies.values()
                    for dep in deps if dep["to"] == domain and dep["critical"]
                )
                if dependent_count > 1:
                    spofs.append({
                        "domain": domain,
                        "type": info["type"],
                        "dependent_services": dependent_count,
                        "risk": "CRITICAL" if dependent_count > 5 else "HIGH",
                    })
        
        return spofs
```

### 10.4 韧性工程（Resilience Engineering）理论

#### 10.4.1 从可靠性到韧性

传统可靠性工程关注"系统如何不出错"，而韧性工程关注"系统如何在出错后继续工作"。

```
┌─────────────────────────────────────────────────────────────────┐
│               可靠性 vs 韧性 对比                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  可靠性（Reliability）          韧性（Resilience）              │
│  ────────────────────          ─────────────────                │
│                                                                 │
│  目标：避免故障                  目标：吸收故障并恢复            │
│  方法：冗余 + 预防               方法：适应 + 恢复               │
│  假设：故障可避免                假设：故障必然发生              │
│  关注点：平均无故障时间           关注点：平均恢复时间            │
│  应对：已知故障模式              应对：未知故障模式              │
│                                                                 │
│  典型措施：                      典型措施：                      │
│  • 双机热备                      • 自动故障隔离                  │
│  • 定期维护                      • 优雅降级                      │
│  • 严格变更流程                  • 混沌工程                      │
│  • 全面测试                      • 持续学习                      │
│                                                                 │
│  比喻：坚固的桥梁（不垮塌）      比喻：灵活的芦苇（风来低头，风过抬头）│
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 10.4.2 韧性设计的四大能力

```
┌─────────────────────────────────────────────────────────────────┐
│              系统韧性的四大核心能力                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  1. 抵御能力（Robustness）                                      │
│     ├── 定义：在预期故障下保持正常功能                           │
│     ├── 示例：单节点故障时，流量自动切换到其他节点               │
│     ├── 技术：健康检查、自动剔除、重试、熔断                     │
│     └── 度量：故障注入测试通过率                                 │
│                                                                 │
│  2. 恢复能力（Recovery）                                        │
│     ├── 定义：从故障中快速恢复的能力                             │
│     ├── 示例：Pod 崩溃后 30 秒内自动重建并恢复服务               │
│     ├── 技术：自愈、热重启、状态重建、缓存预热                   │
│     └── 度量：MTTR（平均恢复时间）                               │
│                                                                 │
│  3. 适应能力（Adaptability）                                    │
│     ├── 定义：根据环境变化调整自身行为                           │
│     ├── 示例：流量突增时自动扩容，流量下降时自动缩容             │
│     ├── 技术：自动扩缩容、负载均衡调整、自适应限流               │
│     └── 度量：扩容响应时间、资源利用率波动                       │
│                                                                 │
│  4. 学习能力（Learning）                                        │
│     ├── 定义：从故障中学习并改进系统                             │
│     ├── 示例：故障根因分析后更新 Runbook 和自动化修复            │
│     ├── 技术：事后复盘、AIOps、异常检测、预测性维护              │
│     └── 度量：重复故障发生率、Runbook 覆盖率                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 10.4.3 韧性模式目录

**模式 1：舱壁隔离（Bulkhead）**

```
原理：将系统分割为独立的部分，一个部分的故障不会影响其他部分

LLM 推理中的应用：
├── 按模型大小隔离
│   ├── 7B 模型 Pod 独立部署
│   ├── 70B 模型 Pod 独立部署
│   └── 405B 模型 Pod 独立部署
├── 按租户隔离
│   ├── 企业用户独立资源池
│   ├── 免费用户共享资源池
│   └── 内部用户专用资源池
└── 按功能隔离
    ├── 聊天接口独立部署
    ├── Embedding 接口独立部署
    └── 代码补全接口独立部署
```

**模式 2：断路器（Circuit Breaker）**

```
原理：监控故障率，当超过阈值时快速失败，避免级联故障

状态转换：
  CLOSED -> OPEN: 连续失败超过阈值
  OPEN -> HALF_OPEN: 超时后试探
  HALF_OPEN -> CLOSED: 试探成功
  HALF_OPEN -> OPEN: 试探失败

LLM 推理中的应用：
├── 模型服务断路器
│   ├── 连续 5 次推理失败 -> OPEN
│   ├── 返回降级响应（轻量模型/缓存）
│   └── 30 秒后试探恢复
├── 向量数据库断路器
│   ├── RAG 检索失败 -> OPEN
│   ├── 降级为关键词搜索
│   └── 1 分钟后试探恢复
└── 外部 API 断路器
    ├── 计费接口失败 -> OPEN
    ├── 允许继续服务，异步补录计费
    └── 5 分钟后试探恢复
```

**模式 3：重试与退避（Retry with Backoff）**

```
原理：瞬时故障时自动重试，配合退避策略避免雪崩

退避策略：
├── 固定间隔：每次等待固定时间
├── 线性退避：每次等待时间线性增加
├── 指数退避：每次等待时间翻倍（最常用）
├── 抖动（Jitter）：随机化等待时间，避免惊群
└── capped 指数退避：设置最大等待上限

LLM 推理中的重试策略：
  1. 首次请求失败
  2. 等待 100ms（含随机抖动）
  3. 第二次请求失败
  4. 等待 200ms
  5. 第三次请求失败
  6. 等待 400ms
  7. 第四次请求 -> 如果仍失败，触发熔断

注意事项：
  - 仅对幂等操作重试（GET、幂等 POST）
  - 非幂等操作需配合去重机制
  - 设置最大重试次数和总超时时间
```

**模式 4：优雅降级（Graceful Degradation）**

```
原理：当系统部分功能不可用时，保留核心功能，降低非核心功能质量

LLM 推理的降级阶梯：
Level 0 (正常):
  ├── 完整模型推理
  ├── 流式输出
  ├── RAG 增强
  ├── Function Calling
  └── 完整上下文窗口

Level 1 (轻度降级 - 自动):
  ├── 完整模型推理
  ├── 流式输出
  ├── RAG 增强（可能延迟）
  ├── Function Calling
  └── 上下文窗口截断到 4K

Level 2 (中度降级 - 自动):
  ├── 切换到轻量模型（7B）
  ├── 关闭流式输出
  ├── 关闭 RAG
  ├── 关闭 Function Calling
  └── 上下文窗口截断到 2K

Level 3 (重度降级 - 手动):
  ├── 仅返回缓存结果
  ├── 关闭多轮对话
  ├── 仅支持预设模板
  └── 保留核心查询接口

Level 4 (应急模式):
  ├── 静态兜底回复
  └── "服务维护中，请稍后重试"
```

**模式 5：超时与熔断（Timeout & Fallback）**

```
原理：设置合理的超时时间，超时后快速失败并返回预设的兜底响应

超时层级设计：
  客户端超时: 30s (用户可接受的最大等待)
    └── 网关超时: 25s
          └── 服务超时: 20s
                └── 推理超时: 18s
                      └── GPU 内核超时: 15s

Fallback 策略：
├── 推理超时 -> 返回已生成的部分结果
├── 模型加载超时 -> 返回 "模型准备中" + 重试指引
├── RAG 检索超时 -> 直接推理，不带上下文增强
├── 缓存未命中 -> 同步加载，同时返回降级响应
```

## 第16章 高可用理论基础

### 16.1 CAP 定理在 AI Infra 中的应用

CAP 定理指出，分布式系统无法同时满足一致性（Consistency）、可用性（Availability）和分区容错性（Partition Tolerance），最多只能同时满足其中两项。

```
         ┌─────────────┐
         │  Consistency │
         │   (一致性)    │
         └──────┬──────┘
                │
    ┌───────────┼───────────┐
    │           │           │
    ▼           ▼           ▼
┌───────┐   ┌───────┐   ┌───────┐
│  CP   │   │  AP   │   │  CA   │
│系统   │   │系统   │   │（单点）│
└───────┘   └───────┘   └───────┘
                │
         ┌──────┴──────┐
         │ Availability │
         │   (可用性)    │
         └─────────────┘
                │
         ┌──────┴──────┐
         │Partition     │
         │Tolerance     │
         │(分区容错)    │
         └─────────────┘
```

**AI Infra 中的 CAP 权衡：**

| 场景 | 选择 | 原因 |
|------|------|------|
| 模型权重存储 | **CP** | 一致性优先，避免不同节点加载不同版本模型 |
| 推理服务 | **AP** | 可用性优先，部分节点故障仍可服务 |
| KV Cache（多副本）| **AP** | 允许短暂不一致，追求低延迟 |
| 训练 Checkpoint | **CP** | 一致性绝对优先，脏数据会导致训练失败 |
| 配置中心 | **CP** | 配置不一致会导致系统行为异常 |

### 16.2 SLA / SLO / SLI 定义与计算

| 术语 | 全称 | 定义 | 示例 |
|------|------|------|------|
| **SLA** | Service Level Agreement | 服务等级协议，对用户的承诺 | "年度可用性 99.9%" |
| **SLO** | Service Level Objective | 服务等级目标，内部目标 | "月度 P99 延迟 < 500ms" |
| **SLI** | Service Level Indicator | 服务等级指标，可测量的指标 | "HTTP 200 响应比例" |

**可用性计算：**

```
可用性 = (总时间 - 停机时间) / 总时间 × 100%

常见可用性级别：
- 99%（2个9）：年停机 87.6 小时
- 99.9%（3个9）：年停机 8.76 小时
- 99.99%（4个9）：年停机 52.6 分钟
- 99.999%（5个9）：年停机 5.26 分钟

LLM 推理服务 SLA 示例：
- 可用性：99.95%（年停机 < 4.38 小时）
- P99 延迟（TTFT）：< 500ms
- P99 延迟（TPOT）：< 50ms
- 错误率：< 0.1%
- 吞吐：> 1000 tokens/s per GPU
```

**SLO 计算方法：**

```python
# SLO 计算示例
class SLOCalculator:
    def __init__(self):
        self.metrics = []
    
    def record_request(self, latency_ms, success):
        self.metrics.append({
            'latency': latency_ms,
            'success': success,
            'timestamp': time.time()
        })
    
    def calculate_slo(self, window_hours=24):
        """计算过去 window_hours 小时的 SLO"""
        cutoff = time.time() - window_hours * 3600
        recent = [m for m in self.metrics if m['timestamp'] > cutoff]
        
        if not recent:
            return None
        
        total = len(recent)
        successful = sum(1 for m in recent if m['success'])
        latencies = [m['latency'] for m in recent if m['success']]
        
        return {
            'availability': successful / total * 100,
            'p50_latency': sorted(latencies)[len(latencies)//2],
            'p99_latency': sorted(latencies)[int(len(latencies)*0.99)],
            'error_rate': (total - successful) / total * 100,
            'total_requests': total
        }

# 使用
calc = SLOCalculator()
# 记录请求...
result = calc.calculate_slo()
print(f"可用性: {result['availability']:.3f}%")
print(f"P99 延迟: {result['p99_latency']:.1f}ms")
```

### 16.3 故障域设计原则

```
故障域层级（从高到低隔离）：

Region（地域）
  └── 完全独立的地理区域
  └── 网络延迟 > 100ms
  └── 用于灾备
  
  └── Availability Zone（可用区）
    └── 独立的电力、网络、冷却
    └── 网络延迟 < 2ms
    └── 用于高可用
    
    └── Cluster（集群）
      └── 独立的 K8s 控制平面
      └── 用于大规模隔离
      
      └── Node（节点）
        └── 独立的服务器
        └── 用于故障隔离
        
        └── Pod（容器）
          └── 独立的进程
          └── 用于优雅重启
```

**设计原则：**

1. **故障域越小越好**：单 Pod 故障不应影响其他 Pod
2. **无单点故障**：任何组件至少有两个实例
3. **故障隔离**：一个故障域的问题不应扩散到其他域
4. **快速恢复**：自动检测故障并在秒级恢复

### 18.1 RTO / RPO 计算

| 术语 | 定义 | 计算方法 |
|------|------|----------|
| **RTO** | Recovery Time Objective | 从故障发生到恢复服务的目标时间 |
| **RPO** | Recovery Point Objective | 允许丢失的数据量（时间窗口）|

```
时间线：

T0: 正常服务 ──► T1: 故障发生 ──► T2: 发现故障 ──► T3: 恢复完成
                              │                    │
                              │<──── RTO ────────>│
                              │                    │
                    T1-RPO    │                    │
                       │      │                    │
                       ▼      │                    │
                 最后备份点   │                    │
                 （允许丢失）  │                    │
```

**不同级别灾难的 RTO/RPO：**

| 灾难级别 | 场景 | RTO | RPO | 实现成本 |
|----------|------|-----|-----|----------|
| L1 Pod | 容器崩溃 | 30s | 0 | 低 |
| L2 节点 | 服务器故障 | 5min | 0 | 低 |
| L3 AZ | 可用区故障 | 10min | 1min | 中 |
| L4 Region | 地域故障 | 30min | 5min | 高 |
| L5 数据 | 数据损坏 | 1h | 1h | 高 |

```python
        df = historical_data.copy()
        df['timestamp'] = pd.to_datetime(df['timestamp'])
        df['days_since_start'] = (df['timestamp'] - df['timestamp'].min()).dt.days
        
        # 特征工程
        features = df[['days_since_start', 'day_of_week', 'hour', 'is_holiday']]
        target = df['qps']
        
        return features, target
    
    def train(self, historical_data):
        features, target = self.prepare_data(historical_data)
        
        # 多项式特征
        X_poly = self.poly.fit_transform(features)
        
        # 训练模型
        self.model = LinearRegression()
        self.model.fit(X_poly, target)
    
    def predict(self, days_ahead=30):
        """预测未来 days_ahead 天的负载"""
        future_dates = pd.date_range(
            start=pd.Timestamp.now(),
            periods=days_ahead * 24,  # 每小时一个预测
            freq='H'
        )
        
        predictions = []
        for date in future_dates:
            features = pd.DataFrame([{
                'days_since_start': (date - pd.Timestamp.now()).days,
                'day_of_week': date.dayofweek,
                'hour': date.hour,
                'is_holiday': 1 if date in holidays else 0
            }])
            
            X_poly = self.poly.transform(features)
            pred = self.model.predict(X_poly)[0]
            predictions.append({
                'timestamp': date,
                'predicted_qps': max(0, pred)
            })
        
        return pd.DataFrame(predictions)
    
    def recommend_capacity(self, predictions, target_gpu_util=0.8):
        """推荐 GPU 数量"""
        peak_qps = predictions['predicted_qps'].max()
        
        # 假设单 GPU 处理能力
        tokens_per_request = 1000  # 平均
        gpu_throughput = 80  # tokens/s per H100
        
        # 计算所需 GPU
        tokens_per_sec = peak_qps * tokens_per_request
        gpus_needed = tokens_per_sec / (gpu_throughput * target_gpu_util)
        
        return {
            'peak_predicted_qps': peak_qps,
            'gpus_needed': int(np.ceil(gpus_needed)),
            'gpus_with_buffer': int(np.ceil(gpus_needed * 1.3)),
            'confidence': 'medium'  # 基于模型 R²
        }

# 使用示例
historical_data = pd.read_csv('inference_metrics.csv')
predictor = LoadPredictor()
predictor.train(historical_data)

future_load = predictor.predict(days_ahead=30)
recommendation = predictor.recommend_capacity(future_load)

print(f"未来30天峰值QPS预测: {recommendation['peak_predicted_qps']:.0f}")
print(f"推荐GPU数量: {recommendation['gpus_needed']}")
print(f"含缓冲区GPU数量: {recommendation['gpus_with_buffer']}")
```

## 十一、架构模式详解

### 11.1 主备模式（Active-Standby）

#### 11.1.1 模式定义与架构

主备模式是最经典的高可用架构，系统中只有一个主节点处理所有请求，一个或多个备节点处于待机状态。当主节点故障时，备节点接管服务。

```
┌─────────────────────────────────────────────────────────────────┐
│                       主备模式架构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│     流量入口                                                     │
│        │                                                        │
│        ▼                                                        │
│   ┌─────────────────┐                                           │
│   │   健康检查器     │                                           │
│   │   (Watchdog)    │                                           │
│   │                 │                                           │
│   │  每秒探测主节点  │                                           │
│   │  连续 3 次失败   │                                           │
│   │  -> 触发切换     │                                           │
│   └────────┬────────┘                                           │
│            │                                                    │
│     ┌──────┴──────┐                                             │
│     │             │                                             │
│     ▼             ▼                                             │
│  ┌──────┐     ┌──────┐                                         │
│  │ACTIVE│     │STANDBY│                                        │
│  │ 主   │     │ 备   │                                         │
│  │      │◄────│      │  (数据同步)                             │
│  │ 100% │     │ 0%   │                                         │
│  │ 流量 │     │ 待机 │                                         │
│  └──────┘     └──────┘                                         │
│                                                                 │
│  切换前：主 100%，备 0%                                         │
│  切换后：主 0%，备 100%（新主）                                 │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 11.1.2 主备模式的分类

**冷备（Cold Standby）**

```
特点：
  - 备节点完全关闭或不运行服务
  - 数据通过定期备份同步（RPO 较大）
  - 切换时需要启动服务（RTO 较大）
  - 成本最低

适用场景：
  - 非核心内部工具
  - 批处理任务
  - 开发测试环境

RTO: 分钟级 ~ 小时级
RPO: 小时级 ~ 天级
```

**温备（Warm Standby）**

```
特点：
  - 备节点运行但不接受流量
  - 数据通过定期同步或日志复制
  - 切换时快速启动（RTO 中等）
  - 成本中等

适用场景：
  - 一般企业应用
  - 内部 AI 平台
  - 数据分析服务

RTO: 秒级 ~ 分钟级
RPO: 分钟级 ~ 小时级
```

**热备（Hot Standby）**

```
特点：
  - 备节点运行并实时同步数据
  - 可随时接管（RTO 最小）
  - 通常采用共享存储或同步复制
  - 成本最高

适用场景：
  - 金融交易系统
  - 核心推理服务
  - 计费/审计系统

RTO: 秒级
RPO: 0 ~ 秒级
```

#### 11.1.4 主备模式的优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **数据一致性** | 简单，无主从延迟问题 | 切换时可能有短暂不一致 |

- 预算有限但需要基本高可用
- 传统单体应用

### 11.2 双活模式（Active-Active）

#### 11.2.1 模式定义与架构

双活模式（Active-Active）是指两个或多个节点同时处理请求，每个节点都是"活"的，共享负载。当一个节点故障时，其他节点自动接管其流量。

```

│        │                                                        │
│        ▼                                                        │

│   │   负载均衡器     │                                           │
│   │   (50% : 50%)   │                                           │
│   │                 │                                           │
│   │  健康检查失败   │                                           │
│   │  -> 自动调整    │                                           │
│   │     100% : 0%   │                                           │
│   └────────┬────────┘                                           │
│            │                                                    │
│     ┌──────┴──────┐                                             │
│     │             │                                             │
│     ▼             ▼                                             │
│  ┌──────┐     ┌──────┐                                         │
│  │ACTIVE│     │ACTIVE│                                         │
│  │ 节点A│     │ 节点B│                                         │
│  │      │◄───►│      │  (数据同步)                             │
│  │ 50%  │     │ 50%  │                                         │

│  A 恢复时：逐步回切到 50:50                                     │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 11.2.2 双活模式的关键问题

**问题 1：数据同步**

```
双活架构的核心挑战是数据同步。两个节点同时处理写入时，必须保证数据一致性。

同步策略：
├── 共享存储（Shared Storage）
│   ├── 两个节点访问同一份存储
│   ├── 代表：SAN / NAS / 分布式文件系统
│   ├── 优点：无数据复制延迟
│   ├── 缺点：存储成为单点，需要存储级高可用
│   └── 适用：无状态服务、模型权重只读访问
│
├── 同步复制（Synchronous Replication）
│   ├── 写入操作同步到所有节点
│   ├── 代表：etcd Raft / MySQL Group Replication
│   ├── 优点：强一致性
│   ├── 缺点：延迟增加，吞吐量下降
│   └── 适用：配置数据、会话状态
│
└── 异步复制（Asynchronous Replication）
    ├── 写入操作先返回，后台异步复制
    ├── 代表：MySQL 主从 / Redis 主从 / Kafka MirrorMaker
    ├── 优点：低延迟，高吞吐
    ├── 缺点：最终一致性，切换时可能丢数据
    └── 适用：日志、监控、模型权重
```

**问题 2：会话保持**

```
当用户请求需要状态（如多轮对话上下文）时，需要确保同一用户的请求路由到同一节点。

解决方案：
├── 会话亲和性（Session Affinity）
│   ├── 基于客户端 IP 的哈希
│   ├── 基于 Cookie 的粘性
│   └── 缺点：故障时会话丢失
│
├── 共享会话存储
│   ├── Redis Cluster
│   ├── 数据库会话表
│   └── 优点：节点无状态，故障不影响会话
│
└── 客户端状态管理
    ├── JWT Token 携带状态
    ├── 客户端缓存对话历史
    └── 优点：服务器完全无状态
```

**问题 3：脑裂（Split-Brain）**

```
当两个节点之间的网络断开时，每个节点都认为对方故障，各自继续服务，导致数据不一致。

预防措施：
├── 仲裁机制（Quorum）
│   ├── 需要超过半数的节点同意才能服务
│   ├── 示例：etcd 需要 (N/2)+1 个节点
│   └── 3 节点集群：至少 2 个节点存活
│
├── 心跳检测 + 优先级
│   ├── 节点定期交换心跳
│   ├── 心跳超时后，优先级低的节点自动停止服务
│   └── 需要第三方仲裁者（Witness Node）
│
└── fencing（隔离）
    ├── STONITH（Shoot The Other Node In The Head）
    ├── 通过 IPMI/iDRAC 强制关闭疑似故障节点
    └── 确保只有一个节点在写入
```

### 11.3 多活模式（Multi-Active / Geo-Distributed）

#### 11.3.1 模式定义与架构

多活模式是双活模式的扩展，三个或更多节点/区域同时处理请求。每个区域都可以独立服务本地用户，同时与其他区域保持数据同步。

```
┌─────────────────────────────────────────────────────────────────┐
│                       多活模式架构                               │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│                         GSLB 入口                                │
│              （基于地理位置 + 延迟 + 容量路由）                   │
│                              │                                  │
│        ┌─────────────────────┼─────────────────────┐           │
│        │                     │                     │            │
│        ▼                     ▼                     ▼            │
│   ┌──────────┐         ┌──────────┐         ┌──────────┐      │
│   │ Region 1 │         │ Region 2 │         │ Region 3 │      │
│   │  北京    │◄───────►│  上海    │◄───────►│  新加坡  │      │
│   │          │ 同步    │          │ 同步    │          │      │
│   │ LLM x5   │         │ LLM x5   │         │ LLM x3   │      │
│   │ 本地用户 │         │ 本地用户 │         │ 本地用户 │      │
│   │ 流量 40% │         │ 流量 35% │         │ 流量 25% │      │
│   └──────────┘         └──────────┘         └──────────┘      │
│        │                     │                     │            │
│        └─────────────────────┼─────────────────────┘           │
│                              │                                  │
│                              ▼                                  │
│   ┌─────────────────────────────────────────────────────────┐  │
│   │                    跨区域数据同步层                        │  │
│   │  模型权重: S3 Cross-Region Replication (最终一致)       │  │
│   │  会话数据: Redis Active-Active (强一致/冲突解决)        │  │
│   │  配置数据: etcd 跨 Region (Raft 仲裁)                   │  │
│   │  向量索引: 定时同步 + 本地增量                           │  │
│   │  审计日志: Kafka MirrorMaker (异步镜像)                 │  │
│   └─────────────────────────────────────────────────────────┘  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

#### 11.3.2 多活模式的数据一致性策略

```
数据分类与一致性策略：

Class A: 强一致数据（所有区域必须实时一致）
├── 配置数据（模型版本、路由规则）
├── 用户认证信息（Token、权限）
├── 计费/审计日志
└── 同步方式：同步复制 + 仲裁

Class B: 会话一致数据（用户会话内强一致）
├── 对话上下文
├── 用户偏好设置
├── 缓存状态
└── 同步方式：会话亲和 + 区域主从

Class C: 最终一致数据（可容忍分钟级延迟）
├── 模型权重文件
├── 监控指标
├── 日志
└── 同步方式：异步复制 + 定时同步

Class D: 本地数据（无需同步）
├── 本地模型缓存
├── 临时计算结果
├── 节点级指标
└── 同步方式：无需同步
```

#### 11.3.3 多活冲突解决策略

```python
class ConflictResolver:
    """多活架构冲突解决器"""
    
    def __init__(self, strategy="last_write_wins"):
        self.strategy = strategy
    
    def resolve(self, local_value, remote_value, local_timestamp, remote_timestamp):
        if self.strategy == "last_write_wins":
            return local_value if local_timestamp >= remote_timestamp else remote_value
        elif self.strategy == "merge":
            return self._merge_values(local_value, remote_value)
        elif self.strategy == "custom":
            return self._custom_resolve(local_value, remote_value)
        else:
            raise ValueError(f"Unknown strategy: {self.strategy}")
    
    def _merge_values(self, local_value, remote_value):
        if isinstance(local_value, list) and isinstance(remote_value, list):
            return list(set(local_value + remote_value))
        elif isinstance(local_value, dict) and isinstance(remote_value, dict):
            merged = {}
            for key in set(local_value.keys()) | set(remote_value.keys()):
                if key in local_value and key in remote_value:
                    merged[key] = local_value[key]  # 简化处理
                elif key in local_value:
                    merged[key] = local_value[key]
                else:
                    merged[key] = remote_value[key]
            return merged
        else:
            return local_value
    
    def _custom_resolve(self, local_value, remote_value):
        # 对话上下文的合并
        local_msgs = local_value.get("messages", [])
        remote_msgs = remote_value.get("messages", [])
        all_msgs = local_msgs + remote_msgs
        unique_msgs = {m["id"]: m for m in all_msgs}
        sorted_msgs = sorted(unique_msgs.values(), key=lambda x: x["timestamp"])
        return {"messages": sorted_msgs}
```

#### 11.3.4 多活模式的优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **可用性** | 任一区域故障不影响其他区域 | 跨区域同步复杂 |
| **延迟** | 用户访问最近区域，延迟低 | 跨区域写操作延迟高 |
| **扩展性** | 全球水平扩展 | 数据分区策略复杂 |
| **合规** | 数据可留在本地（数据主权） | 需要处理跨区域合规 |
| **成本** | 无单区域容量瓶颈 | 多区域基础设施成本高 |
| **运维** | 区域独立运维 | 需要全局协调 |

**适用场景**：
- 全球化部署
- 大型互联网应用
- 强合规要求（数据不出境）
- 超高可用性要求（99.999%）

### 11.4 分区模式（Sharded / Partitioned）

#### 11.4.1 模式定义与架构

分区模式将数据和流量按照某种规则划分为多个独立的分片（Shard），每个分片独立服务。通过增加分片数量实现水平扩展。

```
│   │   (Shard Router)│                                           │
│   │                 │                                           │
```

#### 11.4.2 分区策略

**策略 1：哈希分区（Hash Partitioning）**

```python
def hash_partition(key: str, num_shards: int) -> int:
    import hashlib
    hash_value = int(hashlib.md5(key.encode()).hexdigest(), 16)
    return hash_value % num_shards

# 优点：数据分布均匀，实现简单，适合随机访问
# 缺点：扩展分片时需要 resharding，范围查询效率低
# 改进：一致性哈希（Consistent Hashing）
```

**策略 2：范围分区（Range Partitioning）**

```python
def range_partition(key: str, ranges: list) -> int:
    for i, (start, end) in enumerate(ranges):
        if start <= key < end:
            return i
    return len(ranges) - 1

# 优点：范围查询效率高，可利用数据局部性
# 缺点：数据分布可能不均（热点问题）
# 适用：按时间分区的日志、按地域分区的用户
```

**策略 3：列表分区（List Partitioning）**

```python
def list_partition(key: str, partition_map: dict) -> int:
    return partition_map.get(key, 0)

# 按租户/企业分区
partition_map = {
    "tenant_a": 0,
    "tenant_b": 1,
    "tenant_c": 2,
}

# 优点：精确控制数据分布，便于数据隔离
# 缺点：需要维护映射表
```

**策略 4：复合分区（Composite Partitioning）**

```
两级分区示例：
第一级：按地域分区（Range）
  ├── 中国区
  ├── 北美区
  └── 欧洲区

第二级：按用户 ID 哈希（Hash）
  └── 每个地域内再分 10 个分片
```

#### 11.4.4 分区模式的优缺点

| 维度 | 优点 | 缺点 |
|------|------|------|
| **扩展性** | 理论上无限水平扩展 | 路由层可能成为瓶颈 |
| **性能** | 分片内无竞争 | 跨分片查询复杂 |
| **可用性** | 单分片故障影响有限 | 需要处理分片故障 |
| **数据管理** | 分片可独立备份/恢复 | 数据迁移复杂 |
| **运维** | 分片独立升级 | 分片数量多，管理复杂 |

**适用场景**：
- 超大规模部署（100+ 节点）
- 多租户 SaaS 平台
- 需要严格数据隔离
- 按业务域自然分片
