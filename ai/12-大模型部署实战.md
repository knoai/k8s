## 一、部署前的核心计算：显存需求公式

### 1.1 显存占用核心原理

大模型推理时的显存占用由四个核心部分组成，理解这四个部分是后续所有部署决策的基础。

#### 1.1.1 模型权重（Model Weights）

模型权重是显存占用的最大头，由参数量和精度决定：

```
模型权重显存 = 参数量 × 每参数占用字节数
```

| 精度格式 | 每参数字节数 | 1B 参数占用 | 说明 |
|---------|------------|-----------|------|
| FP32 (float32) | 4 bytes | ~4 GB | 全精度，训练常用 |
| FP16/BF16 (bfloat16) | 2 bytes | ~2 GB | 半精度，推理主流 |
| FP8 (E4M3/E5M2) | 1 byte | ~1 GB | H100 原生支持，精度损失极小 |
| INT8 | 1 byte | ~1 GB | 8-bit 量化，SmoothQuant/LLM.int8 |
| INT4 / Q4 | 0.5 bytes | ~0.5 GB | 4-bit 量化，AWQ/GPTQ/GGUF |
| Q4_K_M (GGUF) | ~0.6 bytes | ~0.6 GB | llama.cpp 推荐格式，混合精度 |
| Q3_K_M (GGUF) | ~0.4 bytes | ~0.4 GB | 更高压缩，精度损失更大 |

> **重要概念**：BF16 与 FP16 的区别
> - FP16：1 位符号 + 5 位指数 + 10 位尾数，动态范围较小（~5.96e-8 到 65504）
> - BF16：1 位符号 + 8 位指数 + 7 位尾数，动态范围与 FP32 相同，但精度较低
> - 大模型推理推荐 BF16，因大模型对数值范围更敏感，对精度相对不敏感

#### 1.1.2 KV Cache（键值缓存）

KV Cache 是 Transformer 自回归生成时的核心缓存机制。为避免重复计算历史 token 的 Key 和 Value，将其缓存起来。

```
KV Cache 显存 = 2 × num_layers × hidden_size × seq_len × batch_size × bytes_per_param
```

| 变量 | 含义 | 典型值（Llama3-8B） | 典型值（Llama3-70B） |
|-----|------|-------------------|---------------------|
| `2` | K 和 V 两个张量 | — | — |
| `num_layers` | Transformer 层数 | 32 | 80 |
| `hidden_size` | 隐藏层维度 | 4096 | 8192 |
| `seq_len` | 序列长度（上下文窗口） | 8192 | 8192 |
| `batch_size` | 并发请求数 | 1 | 1 |
| `bytes_per_param` | 精度字节数（FP16=2） | 2 | 2 |

**KV Cache 计算示例**：
- Llama3-8B，FP16，seq_len=8192，batch=1：
  - `2 × 32 × 4096 × 8192 × 1 × 2 = 4,294,967,296 bytes ≈ 4.0 GB`
- Llama3-70B，FP16，seq_len=8192，batch=1：
  - `2 × 80 × 8192 × 8192 × 1 × 2 = 21,474,836,480 bytes ≈ 20.0 GB`

> **关键洞察**：KV Cache 随序列长度线性增长！当上下文从 8K 扩展到 128K 时，KV Cache 占用增长 **16 倍**。这是长上下文推理的最大挑战。

#### 1.1.3 激活值（Activations）

激活值是前向传播过程中各层的中间输出：

```
激活值显存 ≈ batch_size × seq_len × hidden_size × num_layers × 4 bytes
```

在 vLLM 等引擎中，由于 PagedAttention 和算子融合，激活值通常占比较小（约权重的 5-10%）。但在大 batch 推理时不可忽视。

#### 1.1.4 系统开销（Overhead）

```
系统开销 = CUDA Context + cuBLAS Workspace + 显存碎片 + 推理引擎缓存
```

| 开销类型 | 估算大小 | 说明 |
|---------|---------|------|
| CUDA Context | ~500 MB - 1 GB | 每个 GPU 进程的基础开销 |
| cuBLAS/cuDNN Workspace | ~200 MB - 2 GB | 库函数的工作区 |
| 显存碎片 | ~5-15% | 内存分配器碎片 |
| vLLM 预留 | ~5-10% | PagedAttention 块表、调度器等 |

### 1.2 完整显存计算公式

综合以上四部分，推理时的**总显存需求**为：

```
总显存 = 模型权重 + KV Cache + 激活值 + 系统开销

其中：
  模型权重 = P × B_w
  KV Cache = 2 × L × H × S × b × B_kv
  激活值 ≈ b × S × H × L × B_a
  系统开销 ≈ (模型权重 + KV Cache) × 10% ~ 20%

变量定义：
  P  = 模型参数量（如 7B = 7×10^9）
  B_w = 权重精度字节数（FP16=2, INT4≈0.5）
  L  = Transformer 层数
  H  = 隐藏层维度（hidden_size）
  S  = 序列长度（max_model_len）
  b  = batch_size（并发序列数）
  B_kv = KV Cache 精度字节数（FP16=2, FP8=1）
  B_a = 激活值精度字节数（通常 4）
```

**简化估算公式**（用于快速心算）：

```
总显存 ≈ P × B_w + 2 × L × H × S × b × B_kv × 1.2
```

#### 1.4.3 长上下文对显存的影响（以 Llama-3-70B FP16 为例）

| 上下文长度 | KV Cache 显存 | 总显存 | 所需显卡 |
|-----------|-------------|--------|---------|
| 2K | 2.5 GB | ~148 GB | 4×A100 40GB |
| 4K | 5.0 GB | ~151 GB | 4×A100 40GB |
| 8K | 10.0 GB | ~156 GB | 4×A100 40GB |
| 16K | 20.0 GB | ~166 GB | 4×A100 40GB |
| 32K | 40.0 GB | ~186 GB | 4×A100 40GB（紧）|
| 64K | 80.0 GB | ~226 GB | 6×A100 40GB |
| 128K | 160.0 GB | ~306 GB | 8×A100 40GB |

> **结论**：长上下文是显存杀手！当从 8K 扩展到 128K 时，KV Cache 从 10GB 暴涨到 160GB。部署长上下文模型时，必须预留充足的 KV Cache 空间，或采用 KV Cache 量化（如 FP8 KV Cache）。

### 1.5 显存优化速查公式

```bash
# 快速估算：模型能否在某张卡上运行
所需显存 ≈ (参数量 × 精度字节数) × 1.3

# 长上下文额外显存
额外 KV Cache = 2 × 层数 × 隐藏维度 × (上下文长度 - 2048) × 精度字节数 × 并发数

# 多卡 Tensor Parallel 时，每卡显存
每卡显存 ≈ 总显存 / TP_SIZE × 1.05  (5% 通信开销)
```

### 2.2 RTX 3060 12GB — 4B-7B 轻量模型

#### 2.2.1 硬件能力分析

- **显存**：12 GB GDDR6
- **带宽**：360 GB/s
- **算力**：FP16 ~24 TFLOPS
- **可运行模型**：
  - ✅ Qwen2.5-4B / Llama-3.2-3B（FP16）
  - ✅ Qwen2.5-7B / Llama-3-8B（INT8/INT4 量化）
  - ❌ Qwen2.5-14B（即使量化后也很难）

#### 2.2.2 部署 4B 模型（原生 FP16）

```bash
# ========== vLLM 部署（推荐）==========
vllm serve Qwen/Qwen2.5-4B-Instruct \
  --dtype half \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.80 \
  --max-num-seqs 64 \
  --port 8000

# 参数说明：
# --dtype half          使用 FP16，节省一半显存
# --max-model-len 8192  限制上下文长度，避免 KV Cache 爆显存
# --gpu-memory-utilization 0.80  只使用 80% 显存，留余量给系统
# --max-num-seqs 64     限制最大并发序列数
```

```bash
# ========== Ollama 部署（最简单）==========
# 安装 Ollama
curl -fsSL https://ollama.com/install.sh | sh

# 运行 4B 模型
ollama run qwen2.5:4b

# 或运行量化版 7B
ollama run qwen2.5:7b

# 自定义 Modelfile（调整上下文）
cat > Modelfile << 'EOF'
FROM qwen2.5:4b
PARAMETER num_ctx 8192
PARAMETER temperature 0.7
EOF
ollama create my-qwen -f Modelfile
ollama run my-qwen
```

```bash
# ========== llama.cpp 部署（CPU+GPU 混合）==========
# 下载 GGUF 量化模型
wget https://huggingface.co/Qwen/Qwen2.5-4B-Instruct-GGUF/resolve/main/qwen2.5-4b-instruct-q4_k_m.gguf

# GPU 卸载 35 层，剩余 CPU 计算
./llama-cli \
  -m qwen2.5-4b-instruct-q4_k_m.gguf \
  -ngl 35 \
  --ctx-size 8192 \
  -n 512 \
  --temp 0.7

# -ngl 35 表示将 36 层中的 35 层卸载到 GPU，最后一层在 CPU 上
# 如果爆显存，将 -ngl 减小到 30 或 25
```

#### 2.2.5 RTX 3060 优化技巧

```bash
# 技巧 1：使用 AWQ 而非 GPTQ，AWQ 在相同压缩率下精度更高
# 技巧 2：限制上下文长度到 4096 以下，节省 KV Cache

  --quantization awq \
  --dtype half \
  --max-model-len 4096 \
  --max-num-seqs 4 \
  --gpu-memory-utilization 0.88 \
  --enforce-eager \
  --kv-cache-dtype fp8  # 如果支持的话
```

```bash
# 步骤 1：检查 MIG 支持
nvidia-smi -i 0 -q | grep "MIG Mode"

sudo nvidia-smi -i 0 -r

# 方案 A：创建 7 个 1g.10gb 实例（适合 4B 模型）
sudo nvidia-smi mig -i 0 -cgi 19,19,19,19,19,19,19 -C

# 方案 B：创建 3 个 2g.20gb + 1 个 1g.10gb
sudo nvidia-smi mig -i 0 -cgi 14,14,14,19 -C

# 方案 C：创建 2 个 3g.40gb 实例（A100 40GB）
sudo nvidia-smi mig -i 0 -cgi 9,9 -C

# 步骤 4：查看创建的实例
nvidia-smi mig -lgi
nvidia-smi mig -lci

# 步骤 5：在 Docker/K8s 中使用 MIG
# MIG 实例会显示为独立的 GPU 设备
nvidia-smi
# 输出中可以看到 mig 设备，如 MIG 1g.10gb [N/A]
```

### 3.7 多机 InfiniBand 部署详解

#### 3.7.1 InfiniBand 网络架构

InfiniBand（IB）是多机 GPU 集群的标配网络，提供低延迟、高带宽的 RDMA 通信能力。

| 网络代际 | 单端口带宽 | 双端口带宽 | 延迟 | 典型用途 |
|---------|----------|----------|------|---------|
| IB HDR | 200 Gb/s | 400 Gb/s | ~600 ns | 上一代集群 |
| IB NDR | 400 Gb/s | 800 Gb/s | ~500 ns | 当前主流 |
| IB NDR400 | 400 Gb/s | — | ~500 ns | H100 集群标配 |
| NVLink 4 + NVSwitch | — | 900 GB/s | ~300 ns | 单机 8 卡 |

#### 3.7.2 多机部署环境准备

```bash
# ========== 每台机器的基础配置 ==========

# 1. 检查 IB 网卡和状态
ibstat
# 应看到 "State: Active" 和 "Physical state: LinkUp"

# 2. 检查 IB 带宽测试
ib_write_bw  # 服务端
ib_write_bw <server_ip>  # 客户端
# 期望达到理论带宽的 90%+

# 3. 配置 RDMA 环境
# /etc/modprobe.d/mlx4.conf
options mlx4_ib enable_4k_uar=1

# 4. 检查 GPUDirect RDMA 是否启用
nvidia-smi topo -p2p r  # 查看 P2P 读状态

# 5. 配置 NCCL 环境变量
export NCCL_DEBUG=INFO
export NCCL_IB_DISABLE=0
export NCCL_SOCKET_IFNAME=ib0  # IB 网卡名
export NCCL_NET_GDR_LEVEL=5    # 启用 GPUDirect RDMA
export NCCL_IB_GID_INDEX=3     # RoCE v2

export NCCL_IB_QPS_PER_CONNECTION=4
export NCCL_IB_TIMEOUT=22
export NCCL_IB_RETRY_CNT=7

# 6. NCCL 多机测试
# 节点 1（rank 0）
mpirun -np 2 -H node1:1,node2:1 \
  -x NCCL_DEBUG=INFO \
  -x NCCL_IB_DISABLE=0 \
  -x NCCL_SOCKET_IFNAME=ib0 \
```

### 4.1 量化技术总览

| 方法 | 量化对象 | 位宽 | 对称/非对称 | 分组大小 | 精度损失 | 速度提升 | 硬件要求 |
|-----|---------|------|-----------|---------|---------|---------|---------|
| **FP8 (E4M3)** | 权重 + 激活 | 8-bit | — | — | <1% | 2× | H100/RTX Ada |
| **AWQ** | 仅权重 | 4-bit | 非对称 | 128 | 2-3% | 1.5× | 通用 |
| **GPTQ** | 仅权重 | 4-bit/3-bit | 非对称 | 128/256 | 3-5% | 1.5× | 通用 |
| **GGUF** | 仅权重 | 4-bit~8-bit | 混合 | 多种 | 5-10% | 1-2× | CPU+GPU |
| **SmoothQuant** | 权重 + 激活 | 8-bit | 对称 | 逐通道 | 2-3% | 1.5× | 通用 |
| **Marlin** | 仅权重 | 4-bit | 非对称 | 128 | 3-5% | 2-3× | Ampere+ |
| **GPTQ + Marlin** | 仅权重 | 4-bit | 非对称 | 128 | 3-5% | 2.5× | Ampere+ |
| **FP8 KV Cache** | KV Cache | 8-bit | — | — | <1% | 1.2× | H100/Ada |

### 4.2 各量化方法原理详解

#### 4.2.1 FP8（H100 原生）

FP8 是 NVIDIA H100 引入的 8 位浮点格式，包含两种变体：
- **E4M3**：4 位指数 + 3 位尾数，用于前向传播权重和激活
- **E5M2**：5 位指数 + 2 位尾数，用于反向传播梯度

**原理**：
- 利用 Transformer 模型对数值范围敏感但对精度相对不敏感的特性
- H100 Tensor Core 原生支持 FP8 矩阵乘法，无需软件模拟
- 配合 Transformer Engine，自动在 FP16 和 FP8 之间动态切换

**适用场景**：
- H100 / H800 / L40S 等支持 FP8 Tensor Core 的显卡
- 对精度要求极高的生产环境
- 70B+ 大模型的单卡/少卡部署

#### 4.2.2 AWQ（Activation-aware Weight Quantization）

**原理**：
- 观察到：保护 1% 的显著权重（salient weights）即可恢复大部分精度损失
- 显著权重的判断标准：该权重对应的激活值较大
- 对显著权重使用更高精度存储，其余权重量化到 4-bit

```
AWQ 量化流程：
1. 计算每个权重通道的激活值范数（L1/L2）
2. 根据激活值大小排序，选出 top-1% 的显著权重通道
3. 对显著权重通道进行缩放保护（scaling）
4. 其余权重进行常规 per-channel/per-group 量化
5. 优化缩放系数 α，最小化量化后的输出差异
```

**适用场景**：
- 通用推荐，精度损失最小的 4-bit 方法
- 消费级显卡部署 7B-70B 模型
- vLLM/SGLang 原生支持

#### 4.2.3 GPTQ（General-purpose Post-Training Quantization）

**原理**：
- 基于 Optimal Brain Surgeon (OBS) 理论，寻找对输出影响最小的权重进行量化
- 逐层量化：量化完一层后，更新后续层以补偿误差
- 使用 Cholesky 分解加速海森矩阵计算

```
GPTQ 量化流程：
1. 为每一层收集校准数据（~128 个样本）
2. 计算该层输出的海森矩阵 H
3. 对权重矩阵按列量化：
   a. 找到当前列中量化误差最小的舍入方案
   b. 更新剩余未量化的列以补偿误差
4. 重复直到所有列量化完成
```

**适用场景**：
- 4-bit 极致压缩
- 3-bit 超低精度（损失较大）
- 开源社区支持最好（TheBloke 等提供大量 GPTQ 模型）

#### 4.2.4 GGUF（llama.cpp 专用格式）

**原理**：
- 专为 llama.cpp 设计的二进制格式
- 支持多种量化策略混合使用：
  - `Q4_0`：4-bit，所有层统一量化
  - `Q4_K_M`：4-bit，attention 层和 FFN 层采用不同量化策略
  - `Q5_K_M`：5-bit，更高精度
  - `Q6_K`：6-bit，接近 FP16 精度
- 支持 CPU 推理（AVX/AVX2/ARM NEON）

**量化策略对比**：

| 格式 | 每参数位宽 | 精度 | 速度 | 推荐场景 |
|-----|----------|------|------|---------|
| Q2_K | 2.5-3 bit | 较低 | 最快 | 极致压缩 |
| Q3_K_M | ~3.5 bit | 中等 | 很快 | 低显存 |
| Q4_K_M | ~4.5 bit | 较好 | 快 | **推荐默认** |
| Q5_K_M | ~5.5 bit | 好 | 较快 | 高精度 |
| Q6_K | ~6 bit | 很好 | 中等 | 接近无损 |
| Q8_0 | 8 bit | 优秀 | 较慢 | 高精度 |

**适用场景**：
- 本地开发、边缘设备
- CPU+GPU 混合推理
- macOS（Apple Silicon）推理

#### 4.2.5 SmoothQuant

**原理**：
- 核心洞察：权重分布较均匀易量化，但激活值分布不均匀（存在异常值 outliers）
- 通过平滑变换将激活值的量化难度"迁移"到权重上

```
SmoothQuant 公式：

原始前向：Y = X · W
平滑变换：Y = (X · diag(s)^-1) · (diag(s) · W)

其中 s_j = max(|X_j|)^α / max(|W_j|)^(1-α)
α = 0.5（默认，迁移难度到权重和激活各一半）

变换后：
  X' = X · diag(s)^-1  （激活值更平滑，易量化）
  W' = diag(s) · W     （权重量化难度增加，但权重本身易量化）
```

**适用场景**：
- INT8 部署（权重 + 激活都量化到 8-bit）
- TensorRT-LLM 原生支持
- 对精度要求较高的 INT8 场景

#### 4.2.6 Marlin

**原理**：
- 专为 Ampere/Ada 架构 GPU 优化的 4-bit 推理 kernel
- 利用 GPU Tensor Core 的 INT8 计算能力模拟 4-bit 乘法
- 重新设计矩阵乘法的数据排布（data layout），提高访存效率

```
Marlin 核心优化：
1. 将 4-bit 权重打包成 8-bit 的 half2 格式
2. 利用 CUDA Warp 级别的协作加载，减少 bank conflict
3. 针对 A100/H100 的共享内存带宽进行优化
4. 支持动态 batch 和变长序列
```

**适用场景**：
- A100 / H100 / RTX 30/40 系列
- vLLM 最新版本已原生支持 Marlin kernel
- 对吞吐要求极高的 4-bit 推理

以 Llama-3-8B 在 WikiText-2 上的 perplexity 为基准：

| 量化方法 | 位宽 | Perplexity ↓ | 相对 FP16 损失 | 吞吐提升 |
|---------|------|-------------|---------------|---------|

### 4.4 量化脚本示例

#### 4.4.1 AutoAWQ 量化脚本

```python
#!/usr/bin/env python3
"""
AutoAWQ 量化脚本
将 FP16 模型量化为 AWQ 4-bit
"""

from awq import AutoAWQForCausalLM
from transformers import AutoTokenizer
import argparse


def quantize_model(model_path: str, quant_path: str, bits: int = 4):
    """
    量化模型到 AWQ 格式
    
    Args:
        model_path: 原始模型路径（HuggingFace 格式）
        quant_path: 量化后模型保存路径
        bits: 量化位宽（默认 4）
    """
    quant_config = {
        "zero_point": True,           # 使用零点量化
        "q_group_size": 128,          # 量化分组大小（越小精度越高，但速度越慢）
        "w_bit": bits,                # 权重量化位宽
        "version": "GEMM"             # GEMM 后端（vLLM 兼容）
    }

    print(f"[*] 加载模型: {model_path}")
    model = AutoAWQForCausalLM.from_pretrained(
        model_path,
        device_map="auto",
        trust_remote_code=True
    )
    tokenizer = AutoTokenizer.from_pretrained(
        model_path,
        trust_remote_code=True
    )

    print(f"[*] 开始 AWQ 量化 (w_bit={bits}, group_size=128)")
    model.quantize(
        tokenizer,
        quant_config=quant_config,
        calib_data="ptb",             # 校准数据集：ptb / c4 / wikitext
        n_samples=128,                # 校准样本数
        seq_len=512                   # 校准序列长度
    )

    print(f"[*] 保存量化模型到: {quant_path}")
    model.save_quantized(quant_path)
    tokenizer.save_pretrained(quant_path)
    print("[*] 量化完成！")


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--model", required=True, help="原始模型路径")
    parser.add_argument("--output", required=True, help="输出路径")
    parser.add_argument("--bits", type=int, default=4, choices=[3, 4, 8])
    args = parser.parse_args()
    
    quantize_model(args.model, args.output, args.bits)
```

#### 4.4.2 AutoGPTQ 量化脚本

```python
#!/usr/bin/env python3
"""
AutoGPTQ 量化脚本
将 FP16 模型量化为 GPTQ 4-bit
"""

from auto_gptq import AutoGPTQForCausalLM, BaseQuantizeConfig
from transformers import AutoTokenizer
import torch
import argparse


def quantize_model_gptq(model_path: str, quant_path: str, bits: int = 4):
    """
    量化模型到 GPTQ 格式
    
    Args:
        model_path: 原始模型路径
        quant_path: 量化后模型保存路径
        bits: 量化位宽（3/4/8）
    """
    quantize_config = BaseQuantizeConfig(
        bits=bits,                    # 量化位宽
        group_size=128,               # 分组大小
        desc_act=True,                # 激活值重排序（descending activation）
        damp_percent=0.1,             # 海森矩阵阻尼系数
        static_groups=False,          # 是否使用静态分组
        sym=True,                     # 对称量化
        true_sequential=True,         # 顺序量化
    )

    print(f"[*] 加载模型: {model_path}")
    model = AutoGPTQForCausalLM.from_pretrained(
        model_path,
        quantize_config,
        device_map="auto",
        trust_remote_code=True,
        torch_dtype=torch.float16
    )
    tokenizer = AutoTokenizer.from_pretrained(
        model_path,
        trust_remote_code=True
    )

    print(f"[*] 准备校准数据")
    examples = []
    from datasets import load_dataset
    dataset = load_dataset("wikitext", "wikitext-2-raw-v1", split="train")
    for i in range(128):
        text = dataset[i]["text"]
        if len(text) > 100:
            examples.append(tokenizer(text, return_tensors="pt"))

    print(f"[*] 开始 GPTQ 量化 (bits={bits})")
    model.quantize(
        examples,
        batch_size=1,
        use_triton=False,             # Triton 加速（Ampere+）
        autotune_warmup_after_quantized=False
    )

    print(f"[*] 保存量化模型到: {quant_path}")
    model.save_quantized(quant_path, use_safetensors=True)
    tokenizer.save_pretrained(quant_path)
    print("[*] 量化完成！")


if __name__ == "__main__":
    parser = argparse.ArgumentParser()
    parser.add_argument("--model", required=True)
    parser.add_argument("--output", required=True)
    parser.add_argument("--bits", type=int, default=4, choices=[2, 3, 4, 8])
    args = parser.parse_args()
    
    quantize_model_gptq(args.model, args.output, args.bits)
```

```bash
#!/bin/bash

MODEL_DIR="/path/to/Qwen2.5-7B-Instruct"
```

### 7.1 OOM（显存溢出）排查

#### 7.1.1 症状识别

```
错误信息：
- CUDA out of memory. Tried to allocate X.XX GiB
- RuntimeError: CUDA error: out of memory
- torch.cuda.OutOfMemoryError
- vLLM: No available block found in 60 second
```

#### 7.1.2 排查步骤

```bash
# 步骤 1：确认当前显存使用
nvidia-smi
nvidia-smi dmon  # 实时监控

# 步骤 2：查看 Python 进程的显存详细分配
python -c "
import torch
print(f'Allocated: {torch.cuda.memory_allocated()/1e9:.2f} GB')
print(f'Reserved:  {torch.cuda.memory_reserved()/1e9:.2f} GB')
print(f'Max allocated: {torch.cuda.max_memory_allocated()/1e9:.2f} GB')
"

# 步骤 3：使用 py-spy 或 memory-profiler 定位
pip install memory_profiler
python -m memory_profiler your_script.py

# 步骤 4：vLLM 专用显存监控
curl http://localhost:8000/metrics | grep vllm_gpu
```

#### 7.1.3 解决方案矩阵

| OOM 原因 | 诊断方法 | 解决方案 |
|---------|---------|---------|
| 模型权重太大 | `nvidia-smi` 显示占用接近 100% | 使用更小模型、更低精度（INT4）、AWQ/GPTQ |
| KV Cache 溢出 | 长上下文时触发 | 降低 `--max-model-len`、减少 `--max-num-seqs`、启用 KV Cache FP8 |
| 并发太高 | batch 增大时触发 | 降低 `--max-num-seqs`、限制请求并发 |
| 激活值溢出 | 大 batch 时触发 | 启用 `--enable-chunked-prefill`、减小 `--max-num-batched-tokens` |
| 显存碎片 | 空闲显存足够但分配失败 | 降低 `--gpu-memory-utilization` 到 0.85 |
| CUDA Context 开销 | 多进程启动 | 确保每个 GPU 只有一个进程 |
| 多卡通信开销 | TP/PP 时触发 | 检查 NVLink 是否启用，`nvidia-smi topo -m` |

#### 7.1.4 OOM 预防脚本

```bash
#!/bin/bash
# OOM 预防性检查脚本

MODEL_SIZE=${1:-7}  # 参数量（B）
GPU_MEM=${2:-24}    # GPU 显存（GB）
PRECISION=${3:-"fp16"}  # fp16/int8/int4
CONTEXT_LEN=${4:-8192}  # 上下文长度

# 计算所需显存
case $PRECISION in
  fp32) BYTES=4 ;;
  fp16|bf16) BYTES=2 ;;
  int8|fp8) BYTES=1 ;;
  int4) BYTES=0.5 ;;
  *) BYTES=2 ;;
esac

WEIGHT_MEM=$(echo "$MODEL_SIZE * $BYTES * 1.3" | bc)
KV_MEM=$(echo "$MODEL_SIZE * $CONTEXT_LEN / 1000000 * 0.5" | bc)
TOTAL_MEM=$(echo "$WEIGHT_MEM + $KV_MEM + 2" | bc)

echo "=== OOM 预估检查 ==="
echo "模型大小: ${MODEL_SIZE}B"
echo "精度: $PRECISION (每参数 ${BYTES} bytes)"
echo "上下文长度: $CONTEXT_LEN"
echo "---"
echo "预估权重显存: ${WEIGHT_MEM} GB"
echo "预估 KV Cache: ${KV_MEM} GB"
echo "预估总显存: ${TOTAL_MEM} GB"
echo "可用显存: ${GPU_MEM} GB"
echo "---"

if (( $(echo "$TOTAL_MEM > $GPU_MEM" | bc -l) )); then
  echo "⚠️  预计 OOM！建议措施："
  echo "   1. 使用更低精度（如 INT8 或 AWQ）"
  echo "   2. 降低上下文长度"
  echo "   3. 使用多卡 Tensor Parallel"
  echo "   4. 降低 --gpu-memory-utilization"
else
  echo "✅ 显存充足，建议 --gpu-memory-utilization 设置为 0.85-0.92"
fi
```

### 7.2 模型加载失败排查

#### 7.2.1 常见错误与解决方案

| 错误信息 | 原因 | 解决方案 |
|---------|------|---------|
| `FileNotFoundError: model.safetensors` | 模型文件缺失 | 检查路径、重新下载 `huggingface-cli download` |
| `OSError: port already in use` | 端口被占用 | `lsof -i :8000` 然后 `kill -9 <pid>` |
| `KeyError: 'rope_scaling'` | 配置文件不兼容 | 更新 transformers 和 vLLM 到最新版 |
| `ValueError: Quantization config not found` | 量化配置缺失 | 确保量化模型目录包含 `quant_config.json` |
| `RuntimeError: Expected all tensors to be on the same device` | 多卡加载错误 | 检查 `tensor-parallel-size` 与 GPU 数量匹配 |
| `ConnectionError: HTTPSConnectionPool` | HuggingFace 连接失败 | 设置 HF 镜像：`export HF_ENDPOINT=https://hf-mirror.com` |
| `PermissionError` | 模型目录权限问题 | `chmod -R 755 /models/` |
| `ModuleNotFoundError: No module named 'vllm'` | vLLM 未安装 | `pip install vllm` |
| `ImportError: cannot import name 'LlamaForCausalLM'` | transformers 版本不兼容 | `pip install --upgrade transformers` |

#### 7.2.2 模型加载诊断脚本

```python
"""模型加载诊断脚本"""

import sys
import torch
from transformers import AutoConfig, AutoTokenizer, AutoModelForCausalLM

def diagnose_model(model_path: str):
    print(f"[*] 诊断模型: {model_path}")
    
    # 检查 1：能否加载配置
```

### 9.1 GPU 架构详解

#### 9.1.1 NVIDIA GPU 架构演进史

NVIDIA GPU 的架构演进直接影响大模型推理的性能天花板。每一代架构在 Tensor Core 设计、显存技术和互连带宽上都有质的飞跃。

| 架构代号 | 代表显卡 | 制造工艺 | Tensor Core 代际 | 关键特性 | 发布时间 |
|---------|---------|---------|-----------------|---------|---------|
| Volta | V100 | 12nm | 1st Gen | 首次引入 Tensor Core，纯 FP16 | 2017 |
| Turing | RTX 20/T4 | 12nm | 2nd Gen | INT8/INT4 支持，RT Core | 2018 |
| Ampere | A100/RTX 30 | 7nm | 3rd Gen | TF32、结构化稀疏、MIG | 2020 |
| Ada Lovelace | RTX 40/L40S | 4nm | 4th Gen | FP8、光流加速器、DLSS 3 | 2022 |
| Hopper | H100/H200 | 4nm | 4th Gen (增强) | Transformer Engine、NVLink 4、HBM3 | 2022 |
| Blackwell | B100/B200 | 4nm | 5th Gen | FP4/FP6、第二代 Transformer Engine | 2024 |

**架构演进对大模型推理的影响**：

1. **Volta → Ampere**：Tensor Core 从仅支持 FP16 扩展到 TF32 和结构化稀疏，A100 的 FP16 算力达到 312 TFLOPS，是 V100 的 2.5 倍。
2. **Ampere → Hopper**：引入 FP8 和 Transformer Engine，H100 的 FP8 算力达到 1979 TFLOPS，是 A100 FP16 的 6.3 倍。HBM3 显存带宽从 2.0 TB/s 提升到 3.35 TB/s。
3. **Hopper → Blackwell**：FP4 精度使 405B 模型可在更少 GPU 上部署，第二代 Transformer Engine 实现动态精度切换。

#### 9.1.2 SM（Streaming Multiprocessor）—— GPU 的计算心脏

SM 是 NVIDIA GPU 的基本计算单元，理解 SM 的结构是理解 GPU 并行计算的关键。

**SM 内部结构（以 A100 为例）**：

```
┌─────────────────────────────────────────────────────────────┐
│                    Streaming Multiprocessor                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │  Warp       │  │  Warp       │  │  Register File      │ │
│  │  Scheduler  │  │  Scheduler  │  │  (256 KB)           │ │
│  │  (×4)       │  │  (×4)       │  │                     │ │
│  └─────────────┘  └─────────────┘  └─────────────────────┘ │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐ │
│  │  Dispatch   │  │  Dispatch   │  │  Shared Memory      │ │
│  │  Unit       │  │  Unit       │  │  (192 KB)           │ │
│  │  (×4)       │  │  (×4)       │  │  / L1 Cache         │ │
│  └─────────────┘  └─────────────┘  └─────────────────────┘ │
│  ┌────────────────────────────────────────────────────────┐│
│  │              Tensor Core (×4)                          ││
│  │     3rd Gen: FP64/FP32/TF32/FP16/BF16/INT8/INT4       ││
│  └────────────────────────────────────────────────────────┘│
│  ┌────────┐ ┌────────┐ ┌────────┐ ┌────────┐              │
│  │ CUDA   │ │ CUDA   │ │ CUDA   │ │ CUDA   │  FP32 Core   │
│  │ Core   │ │ Core   │ │ Core   │ │ Core   │  (×64)       │
│  │ (×64)  │ │ (×64)  │ │ (×64)  │ │ (×64)  │              │
│  └────────┘ └────────┘ └────────┘ └────────┘              │
│  ┌────────────────────────────────────────────────────────┐│
│  │  Load/Store Unit (×16)  │  Special Function Unit     ││
│  └────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────┘
```

**关键组件详解**：

| 组件 | 数量（A100 SM） | 功能 | 对推理的影响 |
|-----|---------------|------|-----------|
| **Tensor Core** | 4 个/SM | 矩阵乘法加速 | 决定 FP16/INT8/FP8 峰值算力 |
| **CUDA Core（FP32）** | 64 个/SM | 通用浮点计算 | 非矩阵运算（LayerNorm、Softmax、激活函数） |
| **Warp Scheduler** | 4 个/SM | 线程束调度 | 调度效率影响 GPU 利用率 |
| **Register File** | 256 KB/SM | 线程私有寄存器 | 大 kernel 可能因寄存器压力降低 occupancy |
| **Shared Memory/L1** | 192 KB/SM | 线程块共享内存 | Flash Attention 利用 shared memory 减少 HBM 访问 |

**Occupancy（占用率）**：

Occupancy 是指 SM 上实际运行的 warp 数与最大可容纳 warp 数的比值。A100 每个 SM 最多容纳 2048 个线程（64 warps）。

```
Occupancy = 活跃 Warp 数 / 64
```

影响 occupancy 的因素：
- **寄存器用量**：每个线程使用的寄存器越多，同时运行的线程越少。A100 每个 SM 有 256 KB 寄存器文件，每个线程最多 255 个寄存器。
- **Shared Memory 用量**：每个线程块使用的 shared memory 越多，同时运行的线程块越少。
- **线程块大小**：Warp 大小固定为 32，线程块大小应为 32 的倍数。

#### 9.1.3 CUDA Core vs Tensor Core：计算双引擎

**CUDA Core**：
- 通用标量/向量计算单元
- 每个 CUDA Core 每周期执行 1 次 FMA（乘加）运算
- 负责 LayerNorm、Softmax、激活函数、embedding lookup 等非矩阵运算
- A100 每 SM 64 个 FP32 CUDA Core，全卡 6912 个

**Tensor Core**：
- 专用矩阵乘法加速器
- 每个 Tensor Core 每周期执行一次 4×4×4 或更大的矩阵乘累加
- 负责 Transformer 中最核心的 `Q @ K^T`、`Attention @ V`、`FFN` 中的矩阵乘法
- 利用 WMMA（Warp Matrix Multiply Accumulate）API 调用

**大模型推理中的计算分布**：

以 Llama-3-8B 的 decoder layer 为例，分析每生成一个 token 的计算量：

```
单次前向传播（per token）计算构成：

1. QKV 投影：3 × [hidden_size × hidden_size] 矩阵乘
   - 计算量：3 × 2 × 4096 × 4096 = 100.7 MFLOP（FP16）
   - 执行单元：Tensor Core

2. Attention 计算：
   - Q @ K^T: 2 × 4096 × 4096 × seq_len = 134.2 MFLOP (seq_len=4096)
   - Softmax: 标量运算
   - Attention @ V: 2 × 4096 × 4096 × seq_len = 134.2 MFLOP
   - 执行单元：Tensor Core（矩阵乘）+ CUDA Core（Softmax）

3. Output 投影：hidden_size × hidden_size
   - 计算量：2 × 4096 × 4096 = 33.6 MFLOP
   - 执行单元：Tensor Core

4. FFN（SwiGLU）：
   - Gate + Up 投影：2 × hidden_size × (8/3 × hidden_size)
   - Down 投影：(8/3 × hidden_size) × hidden_size
   - 计算量：约 3 × 2 × 4096 × 10922 ≈ 268.4 MFLOP
   - 执行单元：Tensor Core

5. LayerNorm/RMSNorm（×2）：
   - 计算量：约 2 × 4096 = 8 KFLOP（可忽略）
   - 执行单元：CUDA Core

总计（32 层）：约 32 × (100.7 + 268.4 + 33.6) × 2 ≈ 25.8 GFLOP/token
```

> **关键洞察**：Transformer 推理中，约 95% 的计算量来自矩阵乘法（Tensor Core），约 5% 来自 element-wise 运算（CUDA Core）。因此，Tensor Core 利用率是决定推理性能的首要因素。

#### 9.1.4 Warp、线程调度与内存合并访问

**Warp**：
- GPU 上 32 个线程组成一个 warp，是调度的基本单位
- 同一 warp 内的线程执行相同的指令（SIMT：Single Instruction, Multiple Threads）
- 当 warp 内线程走不同分支时，会产生分支发散（divergence），降低效率

**Warp Scheduler 工作机制**：

```
每个 SM 有 4 个 Warp Scheduler
每个周期，每个 Scheduler 可以发射 1 条指令到 1 个 warp
→ 每个 SM 每周期最多发射 4 条指令
→ 由于指令延迟（Tensor Core 约 4-16 周期），需要足够多的 warp 来隐藏延迟

理想情况：每个 SM 同时运行 16-32 个 warp
→ 可以在一个 warp 等待内存时，调度其他 warp 执行
→ 这就是 GPU 隐藏内存延迟的核心机制
```

**内存合并访问（Coalesced Memory Access）**：

当 warp 中的 32 个线程访问连续的内存地址时，GPU 可以合并为一次内存事务（128 字节）。如果访问不连续，会产生多次内存事务，严重降低带宽利用率。

```python
# 好的内存访问模式（合并访问）
# 线程 i 访问 data[i]
# warp 0 访问 data[0:32]，一次 128B 事务即可完成

# 坏的内存访问模式（分散访问）
# 线程 i 访问 data[i * 1024]
# warp 需要 32 次独立的内存事务
```

在大模型推理中，权重矩阵的存储布局（row-major vs column-major）直接影响内存访问模式。cuBLAS 和 CUTLASS 会自动处理最优的数据排布。

#### 9.1.5 GPU 内存层次结构

GPU 内存是一个多层次的体系，从快到慢、从小到大的层级设计决定了数据放置策略。

```
┌────────────────────────────────────────────────────────────┐
│                    GPU 内存层次结构                         │
├─────────────────┬──────────────┬─────────────┬─────────────┤
│     层级        │    容量      │   带宽      │    延迟     │
├─────────────────┼──────────────┼─────────────┼─────────────┤
│ Register File   │ 256 KB/SM    │ ~10 TB/s    │ ~1 cycle    │
│ Shared Memory   │ 192 KB/SM    │ ~19 TB/s    │ ~20 cycles  │
│ L1 Cache        │ 192 KB/SM    │ ~10 TB/s    │ ~30 cycles  │
│ L2 Cache        │ 40 MB (A100) │ ~3 TB/s     │ ~100 cycles │
│ HBM/GDDR        │ 40-80 GB     │ 1-3 TB/s    │ ~500 cycles │
│ System Memory   │ Host RAM     │ ~50 GB/s    │ ~1 μs       │
│ SSD/网络        │ 外部存储     │ ~5 GB/s     │ ~10 μs+     │
└─────────────────┴──────────────┴─────────────┴─────────────┘
```

**各层级的访问特性**：

1. **Register File（寄存器文件）**：
   - 每个线程私有的最高速存储
   - 容量极小（A100 每线程最多 255 个 32-bit 寄存器 = 1 KB）
   - 编译器自动分配，开发者通过 kernel 设计间接影响

2. **Shared Memory/L1 Cache（共享内存）**：
   - 同一线程块内的线程可共享
   - Flash Attention 的核心优化：将 K/V 分块加载到 shared memory，减少 HBM 访问
   - 程序员通过 `__shared__` 关键字显式控制

3. **L2 Cache**：
   - 全 GPU 共享的二级缓存
   - A100 40MB，H100 50MB
   - 缓存权重矩阵和 KV Cache 的频繁访问部分
   - 命中率直接影响有效显存带宽

4. **HBM（High Bandwidth Memory）/ GDDR**：
   - 主显存，存储模型权重、KV Cache、激活值
   - A100 使用 HBM2e（1555 GB/s），H100 使用 HBM3（3350 GB/s）
   - 消费级显卡使用 GDDR6/GDDR6X（RTX 4090：1008 GB/s）
   - **带宽是推理瓶颈**：大模型推理通常是 memory-bound（受限于显存带宽而非算力）

5. **System Memory（系统内存）**：
   - CPU RAM，通过 PCIe 与 GPU 通信
   - PCIe 4.0 x16 带宽约 32 GB/s，远低于 HBM 带宽
   - Unified Memory 允许 GPU 透明访问 CPU 内存，但性能极低

**Memory-Bound vs Compute-Bound**：

```
Roofline 模型分析：

峰值算力（Compute Roof）
    │
    │         ╱  Compute-Bound（计算瓶颈）
    │        ╱
    │───────╱
    │      ╱
    │     ╱  Memory-Bound（内存瓶颈）
    │    ╱
    │   ╱
    │  ╱
    └───────────────────→ 算术强度（FLOP/Byte）

大模型推理的算术强度：
- 单次矩阵乘：O(n³) 计算 / O(n²) 访存 = O(n) 算术强度
- Transformer Layer：约 100-200 FLOP/Byte
- A100 HBM 带宽 1555 GB/s × 算术强度 ≈ 155-311 TFLOPS 有效算力
- A100 FP16 峰值 312 TFLOPS

→ 结论：对于 batch=1 的推理，算术强度刚好在 roofline 拐点附近，
  增大 batch 可以提升算术强度，进入 compute-bound 区域。
```

#### 9.1.6 Tensor Core 矩阵乘法微架构

Tensor Core 执行矩阵乘累加（GEMM）的核心操作是：

```
D = A × B + C

其中：
  A 是 M × K 矩阵
  B 是 K × N 矩阵
  C 和 D 是 M × N 矩阵
```

**A100 Tensor Core 支持的精度与形状**：

| 精度 | 矩阵 A | 矩阵 B | 累加器 C/D | 每周期操作 |
|-----|-------|-------|----------|----------|
| FP64 | 8×8×4 | 8×8×4 | FP64 | 512 FMA |
| TF32 | 16×8×8 | 16×8×8 | FP32 | 2048 FMA |
| FP16/BF16 | 16×8×16 | 16×8×16 | FP32 | 4096 FMA |
| INT8 | 16×8×32 | 16×8×32 | INT32 | 8192 ops |
| FP8 (H100) | 16×8×32 | 16×8×32 | FP16/FP32 | 8192 FMA |

**H100 FP8 Tensor Core 工作原理**：

FP8 是 Hopper 架构引入的 8-bit 浮点格式，包含两种变体：

| 格式 | 指数位 | 尾数位 | 最大正值 | 最小正值 | 典型用途 |
|-----|-------|-------|---------|---------|---------|
| E4M3 | 4 | 3 | 448.0 | 1.95×10⁻³ | 权重、激活（前向） |
| E5M2 | 5 | 2 | 57344.0 | 1.53×10⁻⁵ | 梯度（反向） |

FP8 矩阵乘法的硬件实现：
1. 输入 A 和 B 为 FP8（E4M3 或 E5M2）
2. 乘法结果在更高精度的累加器中累加（FP16 或 FP32）
3. 最终结果可选择转回 FP8 或保持高精度
4. Transformer Engine 自动管理缩放因子（scaling factor），防止下溢/上溢

```
FP8 GEMM 流程：

FP8 A[M×K] ──┐
              ├──→ Tensor Core ──→ FP16/FP32 D[M×N]
FP8 B[K×N] ──┘          │
                    累加器 C[M×N]

缩放因子管理：
s_A = max(|A|) / 448.0  (E4M3 的最大值)
s_B = max(|B|) / 448.0
A_fp8 = A / s_A
B_fp8 = B / s_B
D = (A_fp8 × B_fp8) × (s_A × s_B) + C
```

### 9.2 深度学习模型显存占用原理

#### 9.2.1 模型权重存储的精确计算

模型权重的显存占用并非简单的 `参数量 × 精度字节数`。需要深入理解各层的权重构成。

**Transformer Decoder Layer 的权重构成（以 Llama 架构为例）**：

```
每层包含以下权重矩阵：

1. Self-Attention：
   - Q_proj: [hidden_size, hidden_size]
   - K_proj: [hidden_size, hidden_size]  (GQA 时缩小 num_kv_heads/num_heads 倍)
   - V_proj: [hidden_size, hidden_size]  (同上)
   - O_proj: [hidden_size, hidden_size]
   总计：4 × hidden_size²（标准 MHA）
        或 (2 + 2×num_kv_heads/num_heads) × hidden_size²（GQA）

2. Feed-Forward Network（SwiGLU）：
   - gate_proj: [hidden_size, intermediate_size]
   - up_proj:   [hidden_size, intermediate_size]
   - down_proj: [intermediate_size, hidden_size]
   总计：3 × hidden_size × intermediate_size
   Llama 中 intermediate_size ≈ 8/3 × hidden_size

3. Layer Normalization：
   - input_layernorm: [hidden_size]（可忽略）
   - post_attention_layernorm: [hidden_size]（可忽略）

4. Embedding：
   - token_embedding: [vocab_size, hidden_size]
   - output_lm_head:  [vocab_size, hidden_size]（通常与 token_embedding 共享）
```

**精确计算公式**：

```python
def calculate_weight_memory(config, dtype_bytes=2):
    """
    精确计算模型权重显存
    """
    h = config.hidden_size
    i = config.intermediate_size
    l = config.num_hidden_layers
    v = config.vocab_size
    nh = config.num_attention_heads
    nkv = getattr(config, 'num_key_value_heads', nh)
    
    # Attention 权重
    # Q_proj + K_proj + V_proj + O_proj
    # GQA: K/V 的头数减少
    attn_q = h * h
    attn_k = h * h * nkv / nh
    attn_v = h * h * nkv / nh
    attn_o = h * h
    attn_per_layer = attn_q + attn_k + attn_v + attn_o
    
    # FFN 权重 (SwiGLU)
    ffn_per_layer = 3 * h * i
    
    # LayerNorm (每个 2*h 参数：weight + bias，但 Llama 只有 weight)
    ln_per_layer = 2 * h  # 两个 RMSNorm
    
    # 所有层的权重
    layer_weights = l * (attn_per_layer + ffn_per_layer + ln_per_layer)
    
    # Embedding
    embed_weights = v * h  # token embedding
    # lm_head 通常共享，不额外计算
    
    total_params = layer_weights + embed_weights
    
    return total_params * dtype_bytes
```

**Llama-3-8B 权重精确计算**：

```
配置：
  hidden_size = 4096
  intermediate_size = 14336 (≈ 3.5 × 4096，不是严格的 8/3)
  num_layers = 32
  vocab_size = 128256
  num_heads = 32
  num_kv_heads = 8 (GQA)

计算：
  Attention 每层：
    Q: 4096 × 4096 = 16,777,216
    K: 4096 × (4096 × 8/32) = 4,194,304
    V: 4096 × (4096 × 8/32) = 4,194,304
    O: 4096 × 4096 = 16,777,216
    小计：42,943,040

  FFN 每层：
    gate + up + down = 3 × 4096 × 14336 = 176,160,768

  LayerNorm 每层：2 × 4096 = 8,192

  所有层：32 × (42,943,040 + 176,160,768 + 8,192) = 7,011,584,000

  Embedding：128256 × 4096 = 525,336,576

  总参数量：7,011,584,000 + 525,336,576 = 7,536,920,576 ≈ 7.54B

  FP16 显存：7.54B × 2 = 15.07 GB
```

> 注意：官方标称 "8B" 是近似值，实际约 7.54B 参数。这解释了为什么有些 "8B" 模型 FP16 只需要约 15GB 而非 16GB。

### 16.1 NVIDIA GPU 架构演进史

NVIDIA GPU 架构经历了多次重大革新，每一代架构都针对 AI 计算进行了专门优化。理解这些架构的差异，对于选择合适的 GPU 进行大模型部署至关重要。

#### Pascal 架构（2016）

| 特性 | 参数 |
|------|------|
| 代表产品 | GTX 1080 Ti, P100 |
| 制程 | 16nm |
| CUDA 核心 | 3584 |
| 显存 | 11-16GB HBM2/GDDR5X |
| FP16 性能 | 21 TFLOPS |
| 特点 | 首次引入 FP16 支持，但无 Tensor Core |

Pascal 架构是 NVIDIA 首次在消费级产品中支持 FP16 计算，但其 FP16 性能仅为 FP32 的一半，且缺乏专门的 AI 加速单元。今天已经不适合用于大模型推理。

#### Volta 架构（2017）

| 特性 | 参数 |
|------|------|
| 代表产品 | V100 |
| 制程 | 12nm |
| CUDA 核心 | 5120 |
| Tensor Core | 第 1 代 |
| 显存 | 16-32GB HBM2 |
| FP16 性能 | 125 TFLOPS（含 Tensor Core）|
| 特点 | 首次引入 Tensor Core，AI 性能飞跃 |

Volta 架构的革命性创新是 **Tensor Core**。Tensor Core 是专门为矩阵乘加运算（GEMM）设计的硬件单元，可以在一个时钟周期内完成 4×4×4 的 FP16 矩阵乘加运算。这使得 V100 的 AI 训练性能相比 Pascal 提升了约 6 倍。

Tensor Core 的工作原理：

```
传统 CUDA Core 执行矩阵乘法：
C[i][j] = A[i][k] × B[k][j]  →  需要 K 次乘加运算
每次乘加 = 1 个 FMA 指令 → 多个时钟周期

Tensor Core 执行矩阵乘法：
直接计算 4×4 FP16 矩阵乘法 → 1 个 mma 指令 → 1 个时钟周期
加速比：~8×
```

#### Turing 架构（2018）

| 特性 | 参数 |
|------|------|
| 代表产品 | RTX 2080 Ti, T4 |
| 制程 | 12nm |
| CUDA 核心 | 4352 |
| Tensor Core | 第 2 代 |
| 显存 | 11-16GB GDDR6 |
| FP16 性能 | 80 TFLOPS |
| INT8 性能 | 260 TOPS |
| 特点 | 引入 INT8 支持，RT Core 用于光追 |

Turing 架构的 Tensor Core 增加了 **INT8 和 INT4 支持**，这使得推理量化成为可能。T4 是数据中心推理的主力卡，至今仍在大规模部署中使用。

#### Ampere 架构（2020）

| 特性 | 参数 |
|------|------|
| 代表产品 | RTX 3090, A100, A40 |
| 制程 | 7nm |
| CUDA 核心 | 6912-10752 |
| Tensor Core | 第 3 代 |
| 显存 | 24-80GB HBM2e |
| FP16 性能 | 312-624 TFLOPS |
| TF32 性能 | 156-312 TFLOPS |
| 特点 | 结构稀疏性、TF32、MIG |

Ampere 架构带来了三项重大创新：

**1. 第 3 代 Tensor Core**
- 支持 TF32（Tensor Float 32）：使用 FP16 的吞吐量，但保持 FP32 的精度范围
- 支持 BF16（Brain Float 16）：更宽的动态范围，训练更稳定
- 支持稀疏性加速：结构化稀疏矩阵可获得 2 倍加速

**2. MIG（Multi-Instance GPU）**
- A100 可以被物理切分为最多 7 个独立实例
- 每个实例有自己的显存、计算单元和缓存
- 实现了真正的硬件级隔离

**3. 第三代 NVLink**
- 单链路带宽提升至 50GB/s
- A100 SXM 支持 12 条 NVLink，总带宽 600GB/s

#### Hopper 架构（2022）

| 特性 | 参数 |
|------|------|
| 代表产品 | H100, H200 |
| 制程 | 4nm |
| CUDA 核心 | 16896 |
| Tensor Core | 第 4 代（Transformer Engine）|
| 显存 | 80-141GB HBM3 |
| FP16 性能 | 989-1979 TFLOPS |
| FP8 性能 | 1979-3958 TFLOPS |
| 特点 | Transformer Engine、DPX、机密计算 |

Hopper 架构是专为 Transformer 模型设计的架构：

**1. Transformer Engine**
- 硬件自动在 FP8 和 FP16/BF16 之间动态切换
- FP8 精度下可获得 2 倍性能提升
- 自动管理精度，无需手动配置

**2. DPX（Dynamic Programming X）**
- 硬件加速动态规划算法
- 对生物信息学、路径规划等应用有显著加速

**3. 第四代 NVLink**
- 单链路带宽 50GB/s → 900GB/s 总带宽
- 支持 NVLink Switch，实现 256 张 GPU 全互联

**4. HBM3 显存**
- H100: 80GB HBM3，带宽 3.35TB/s
- H200: 141GB HBM3e，带宽 4.8TB/s

**内存访问优化原则：**

1. **尽量使用共享内存**：将频繁访问的数据从显存加载到共享内存，可减少 10-100 倍的延迟
2. **合并访问（Coalesced Access）**：同一 warp 的线程访问连续的内存地址，可以合并为一次传输
3. **避免 bank conflict**：共享内存被分为 32 个 bank，同一 warp 访问同一 bank 的不同地址会导致冲突
4. **利用 L2 缓存**：A100 的 40MB L2 缓存可以缓存大量 KV Cache，对长上下文推理至关重要

### 16.3 显存占用计算原理

#### 模型权重显存

```
FP32 模型：参数量 × 4 bytes
FP16/BF16 模型：参数量 × 2 bytes
INT8 模型：参数量 × 1 byte
INT4/GPTQ/AWQ 模型：参数量 × 0.5 bytes

示例：
- Llama-3-8B FP16: 8B × 2 = 16GB
- Llama-3-70B INT8: 70B × 1 = 70GB
- Qwen-72B AWQ: 72B × 0.5 = 36GB
```

#### KV Cache 显存

KV Cache 是 Transformer 推理中最大的显存开销之一，理解其计算方式是部署优化的核心。

```
KV Cache 计算公式：

kv_cache_size = 2 × num_layers × hidden_size × seq_length × batch_size × sizeof(dtype)

其中：
- 2：K 和 V 两个张量
- num_layers：模型层数（Llama-3-8B 有 32 层）
- hidden_size：隐藏层维度（Llama-3-8B 是 4096）
- seq_length：序列长度
- batch_size：批大小
- sizeof(dtype)：数据类型大小（FP16=2, FP8=1）

示例 1：Llama-3-8B FP16, batch=1, seq=4096
kv_cache = 2 × 32 × 4096 × 4096 × 1 × 2 = 2.15 GB

示例 2：Llama-3-70B FP16, batch=32, seq=8192
kv_cache = 2 × 80 × 8192 × 8192 × 32 × 2 = 655 GB
```

#### 激活值显存

```
激活值显存 ≈ batch_size × seq_length × hidden_size × num_layers × 4 bytes

（×4 是因为反向传播需要保存中间激活值，推理时不需要）

推理时的激活值相对较小：
- 每次只计算一层
- 不需要保存所有层的激活值
- 主要开销来自当前的矩阵乘法中间结果
```

#### 推理总显存估算

```
总显存 = 模型权重 + KV Cache + 激活值 + 系统开销 + 余量

系统开销 ≈ 10-20%（CUDA 上下文、显存碎片等）
余量建议 ≥ 10%（避免 OOM）

精确公式：
total_memory = model_weights × 1.1 + kv_cache × 1.05 + activation_buffer + 2GB

其中 activation_buffer ≈ max(batch_size × seq_length × hidden_size × 4, 1GB)
```

### 16.4 量化技术原理

#### 为什么量化有效？

神经网络权重通常服从高斯分布或近似高斯分布，大部分值集中在 0 附近。量化利用这一特性，用更少的比特表示这些值，同时尽量保持模型的表达能力。

```
FP32: 1位符号 + 8位指数 + 23位尾数 = 32位
FP16: 1位符号 + 5位指数 + 10位尾数 = 16位
BF16: 1位符号 + 8位指数 + 7位尾数 = 16位
INT8: 1位符号 + 7位整数 = 8位
INT4: 1位符号 + 3位整数 = 4位

动态范围：
FP32: ~10^-45 到 10^38
FP16: ~10^-8 到 10^5
BF16: ~10^-45 到 10^38 (与FP32相同)
INT8: -128 到 127
INT4: -8 到 7
```

#### 线性量化（INT8/INT4）

```
量化公式：
q = round((r - zero_point) / scale)
反量化：
r = q × scale + zero_point

其中：
- r 是原始浮点值
- q 是量化后的整数值
- scale = (r_max - r_min) / (q_max - q_min)
- zero_point = round(q_min - r_min / scale)

对称量化（简化版）：
scale = max(|r|) / 127
q = round(r / scale)
```

#### AWQ（Activation-aware Weight Quantization）

AWQ 的核心思想是：**并非所有权重的比特都同等重要**。通过分析激活值，AWQ 发现某些权重通道对输出影响更大，因此对这些通道保留更高的精度。

```python
# AWQ 核心思想示意
import torch

# 原始权重
W = torch.randn(4096, 4096)

# 计算每个输出通道的激活值重要性
# 通过少量校准数据获得激活值
activation = get_activation_from_calib_data()
channel_importance = activation.abs().mean(dim=0)

# 重要通道使用 FP16，不重要通道使用 INT4
# 实际实现是：对所有权重进行混合精度量化
# 重要通道使用更大的 scale（保留更多精度）

# AWQ 还引入了 per-channel scaling
# 通过搜索最优的缩放系数来最小化量化误差
scales = search_optimal_scales(W, activation)
W_scaled = W * scales
W_quantized = quantize(W_scaled, bits=4)
```

#### GPTQ（General-purpose Post-Training Quantization）

GPTQ 是一种逐层量化方法，核心思想是：**量化当前层时，考虑后续层的误差累积**。

```
GPTQ 算法流程：

1. 按层遍历模型
2. 对当前层的权重矩阵 W，按列（输出通道）处理
3. 对每一列 w：
   a. 计算最优量化值 q = round(w / scale)
   b. 计算量化误差：error = w - q × scale
   c. 将误差传播到后续未量化的列：w_next -= error × H[:, next_cols]
   
其中 H 是 Hessian 矩阵的近似，用于衡量参数之间的相关性

4. 重复直到所有列都被量化

关键创新：
- Observe：使用校准数据获得层输出
- Quantize：逐列最优量化
- Update：将量化误差传播到未量化部分
```

#### FP8（8-bit Floating Point）

FP8 是 Hopper（H100）架构引入的新格式，有 E4M3 和 E5M2 两种变体：

| 格式 | 指数位 | 尾数位 | 动态范围 | 精度 | 用途 |
|------|--------|--------|----------|------|------|
| E4M3 | 4 | 3 | 2^-9 到 2^8 | 较低 | 权重、前向激活 |
| E5M2 | 5 | 2 | 2^-17 到 2^16 | 较高 | 梯度、反向传播 |

FP8 的优势：
- 比 INT8 更容易使用（不需要校准数据）
- 比 FP16 快 2 倍，显存省一半
- H100 Tensor Core 原生支持 FP8，无需额外计算开销

### 18.1 vLLM PagedAttention 算法详解

#### 问题：传统 KV Cache 管理的缺陷

```python
# 传统方法的内存分配
# 为每个请求预分配 max_seq_length 的空间
# 例如：max_seq_length=4096, batch_size=16

# 请求1实际长度=100，但分配了4096
# 请求2实际长度=500，但分配了4096
# ...
# 大量内存浪费！

# 传统内存布局（连续分配）：
# [请求1: 4096 tokens][请求2: 4096 tokens]...[请求16: 4096 tokens]
# 实际使用：[100][500][200]...[1000]
# 利用率可能 < 20%
```

#### PagedAttention 的解决方案

PagedAttention 借鉴了操作系统虚拟内存管理的思想：

```
操作系统虚拟内存：                    PagedAttention KV Cache：
┌──────────────────┐                 ┌──────────────────┐
│ 虚拟地址空间      │                 │ 逻辑 KV Cache    │
│ (连续)            │                 │ (连续)           │
│ 0,1,2,3,4,5,6,7  │  ──映射──►      │ Block Table      │
└──────────────────┘                 │ (页表)           │
       │                             └────────┬─────────┘
       │                                      │
       ▼                                      ▼
┌──────────────────┐                 ┌──────────────────┐
│ 物理内存          │                 │ GPU 显存          │
│ (非连续块)        │                 │ (固定大小块)      │
│ [块3][块1][块7]   │                 │ [块2][块5][块1]  │
└──────────────────┘                 └──────────────────┘
```

**核心数据结构：**

```python
# Block Table 示例
block_table = {
    "request_1": [2, 5, 1],      # 请求1的KV分布在块2、5、1
    "request_2": [3, 7],          # 请求2的KV分布在块3、7
    "request_3": [2, 5, 1, 4],   # 请求3复用请求1的前3个块，新增块4
}

# 每个块（Block）大小固定，例如 16 tokens
block_size = 16

# 物理块管理
# 空闲块池：[0, 6, 8, 9, 10, ...]
# 已用块：[1, 2, 3, 4, 5, 7]
```

**内存复制优化（Copy-on-Write）：**

```python
# 场景：两个请求有相同的前缀（如相同的 system prompt）
request_1: [system_prompt, "问题1"]
request_2: [system_prompt, "问题2"]

# 传统方法：复制两份 system_prompt 的 KV Cache
# PagedAttention：共享相同的块，只在 diverge 时复制

# 初始状态：
# request_1: [块0: system_prompt_tokens]
# request_2: [块0: system_prompt_tokens]  ← 共享引用

# 生成新 token 后：
# request_1: [块0][块1: "问题1"的KV]
# request_2: [块0][块2: "问题2"的KV]  ← 块0仍然共享

# 只有在需要修改共享块时，才进行复制（Copy-on-Write）
```

### 18.2 Continuous Batching 详解

#### 静态批处理 vs 连续批处理

```
静态批处理（Static Batching）：
时间 ──────────────────────────────────────────►

Batch 1: [Req1(========)] 
         [Req2(====)    ]  ← 等待 Req1 完成
         [Req3(==========)]
         
         完成时间由最长请求决定，GPU 空闲等待

连续批处理（Continuous Batching / In-flight Batching）：
时间 ──────────────────────────────────────────►

T1: [Req1(==)  ] [Req2(=) ] [Req3(===)]
T2: [Req1(===) ] [Req2(==)] [Req3(====)] [Req4(=)] ← 新请求插入
T3: [Req1(====)] [Req2(done)] [Req3(=====)] [Req4(==)]
T4: [Req1(done)] [Req3(======)] [Req4(===)] [Req5(=)] ← Req2 完成，Req5 插入

特点：
1. 新请求可以在任何时间加入当前 batch
2. 完成的请求可以立即退出
3. GPU 几乎无空闲
```

#### vLLM 中的实现

```python
# vLLM 调度器核心逻辑（简化版）
class Scheduler:
    def __init__(self):
        self.waiting = deque()      # 等待队列
        self.running = []            # 正在运行的请求
        self.swapped = []            # 被换出的请求（显存不足）
        self.block_manager = BlockManager()
    
    def schedule(self):
        # 1. 尝试将 waiting 中的请求加入 running
        for seq_group in self.waiting:
            if self.can_allocate(seq_group):
                self.running.append(seq_group)
                self.waiting.remove(seq_group)
        
        # 2. 尝试将被 swap 的请求换回
        for seq_group in self.swapped:
            if self.can_allocate(seq_group):
                self.running.append(seq_group)
                self.swapped.remove(seq_group)
        
        # 3. 如果显存不足，将低优先级请求 swap 出去
        while not self.has_enough_memory():
            seq_group = self.select_victim()
            self.swapped.append(seq_group)
            self.running.remove(seq_group)
        
        # 4. 生成 batch
        return self.running
    
    def can_allocate(self, seq_group):
        # 检查是否有足够的空闲块
        required_blocks = self.compute_required_blocks(seq_group)
        return self.block_manager.num_free_blocks() >= required_blocks
```

### 18.3 TensorRT-LLM 编译优化原理

TensorRT-LLM 的核心优势在于**编译时优化**：

```
PyTorch/vLLM 执行流程（运行时解释）：
模型定义 → Python 解释器 → PyTorch C++ 后端 → CUDA Kernel 启动 → GPU 执行
        ↑______________________________↓
                每次推理都有 Python 开销

TensorRT-LLM 执行流程（编译时优化）：
模型定义 → 图捕获 → 层融合 → Kernel 自动调优 → 生成优化引擎 → 序列化
                                                            ↓
运行时：反序列化引擎 → 直接执行优化后的 Kernel（无 Python 开销）
```

**主要优化技术：**

1. **算子融合（Layer Fusion）**
```python
# 融合前（多个 Kernel）：
# 1. LayerNorm
# 2. Linear Projection (Q, K, V)
# 3. Bias Add
# 4. Activation

# 融合后（单个 Kernel）：
# LayerNorm + QKV Projection + Bias + Activation
# 减少 Kernel 启动开销和内存读写
```

2. **Kernel 自动调优（Auto-Tuning）**
```python
# TensorRT 会尝试不同的 CUDA Kernel 实现
# 选择当前 GPU 上性能最好的版本

# 例如矩阵乘法 C = A × B
# 尝试： CUTLASS、cuBLAS、自定义 Kernel
# 选择：在 A100 上 cuBLAS 最快，在 H100 上自定义 Kernel 最快
```

3. **精度校准（Calibration）**
```python
# INT8/FP8 量化需要校准
# 使用少量校准数据（~1000 样本）
# 确定每一层的最佳缩放因子

# 校准流程：
# 1. 用 FP16 运行校准数据，记录每层的激活值分布
# 2. 计算每层的最优 scale 和 zero_point
# 3. 生成量化引擎
```

#### 9.2.2 激活值（Activations）的深入分析

激活值是前向传播中各层的中间结果。其显存占用取决于 batch size、序列长度和模型架构。

**激活值来源**：

```
每层 decoder 的激活值构成：

1. Input to Attention (x):
   shape: [batch, seq_len, hidden_size]
   memory: b × s × h × dtype_bytes

2. Q, K, V after projection:
   Q: [batch, num_heads, seq_len, head_dim]
   K: [batch, num_kv_heads, seq_len, head_dim]
   V: [batch, num_kv_heads, seq_len, head_dim]
   memory: b × s × h × dtype_bytes × 3 (标准) 或更少 (GQA)

3. Attention scores (Q @ K^T):
   shape: [batch, num_heads, seq_len, seq_len]
   memory: b × nh × s² × dtype_bytes
   ⚠️ 这是 O(s²) 的内存消耗，是长上下文的主要瓶颈

4. Attention output (after softmax × V):
   shape: [batch, num_heads, seq_len, head_dim]
   memory: b × s × h × dtype_bytes

5. FFN intermediate activations:
   gate_proj output: [batch, seq_len, intermediate_size]
   up_proj output:   [batch, seq_len, intermediate_size]
   memory: 2 × b × s × i × dtype_bytes

6. Residual connections 需要保存输入用于反向传播（训练时）
```

**推理时 vs 训练时的激活值差异**：

| 场景 | 需要保存的激活值 | 原因 |
|-----|----------------|------|
| 训练 | 全部中间激活 | 反向传播需要计算梯度 |
| 推理（生成） | 仅当前层输入 | 无需反向传播，可逐层计算 |
| 推理（prefill）| 部分激活 | 一次处理整个 prompt，需要中间结果 |

**推理激活值精确计算（Prefill 阶段）**：

```python
def calculate_activation_memory_prefill(batch_size, seq_len, config, dtype_bytes=2):
    """
    Prefill 阶段（处理整个 prompt）的激活值显存
    """
    b, s = batch_size, seq_len
    h = config.hidden_size
    i = config.intermediate_size
    nh = config.num_attention_heads
    
    # Attention 部分
    x = b * s * h * dtype_bytes                    # 输入
    qkv = 3 * b * s * h * dtype_bytes              # QKV 投影后
    scores = b * nh * s * s * dtype_bytes          # Attention scores (O(s²)!)
    attn_out = b * s * h * dtype_bytes             # Attention 输出
    
    # FFN 部分
    ffn_mid = 2 * b * s * i * dtype_bytes          # gate + up 输出
    ffn_out = b * s * h * dtype_bytes              # down 输出
    
    # 每层总计（保留最大同时存在的激活）
    # 优化后（算子融合）同时存在的最大激活：
    per_layer = max(x + qkv + scores, x + ffn_mid)
    
    return per_layer

# Llama-3-8B, batch=1, seq=8192, FP16:
# scores = 1 × 32 × 8192 × 8192 × 2 = 4.29 GB
# 这是 prefill 阶段激活值的最大贡献者
```

> **关键洞察**：Prefill 阶段的 Attention scores 占用 `batch × heads × seq² × 2 bytes`。当 seq=8192 时，仅这一项就占 4.29 GB。Flash Attention 通过在线计算 softmax 和分块加载，将这项从 O(s²) 降低到 O(s)，是长上下文推理的核心优化。

#### 9.2.3 梯度与优化器状态（训练场景）

虽然本章主要讲推理，但理解训练时的显存占用有助于理解为什么推理可以大幅节省显存。

**训练时的额外显存占用**：

```
训练显存 = 权重 + 激活值 + 梯度 + 优化器状态

1. 权重（FP16/BF16）：W
2. 权重副本（FP32，用于稳定更新）：W_fp32
3. 梯度（与权重同形状）：∇W
4. 优化器状态：
   - Adam: 动量 m + 二阶矩 v = 2 × W
   - AdamW: 同上
   - SGD: 动量 m = 1 × W
5. 激活值（需要保存用于反向传播）：Activations
```

**各种优化器的显存占用**：

| 优化器 | 权重 | 梯度 | 优化器状态 | 每参数总字节数 |
|-------|------|------|----------|-------------|
| SGD (no momentum) | 2 (FP16) + 4 (FP32) | 4 | 0 | 10 bytes |
| SGD (momentum) | 2 + 4 | 4 | 4 (m) | 14 bytes |
| Adam/AdamW | 2 + 4 | 4 | 8 (m + v) | 18 bytes |
| Adam + ZeRO-3 | 2 | 4 | 8 | 分摊到多卡 |
| 8-bit Adam | 2 + 4 | 4 | 2 (8-bit m + v) | 12 bytes |

**以 Llama-3-8B FP16 + AdamW 训练为例**：

```
权重 (FP16): 7.54B × 2 = 15.1 GB
权重 (FP32 主副本): 7.54B × 4 = 30.2 GB
梯度 (FP32): 7.54B × 4 = 30.2 GB
Adam 状态 (m + v, FP32): 7.54B × 8 = 60.4 GB
激活值 (batch=1, seq=4096): ~10 GB
总计: ~146 GB

→ 单卡 A100 80GB 远远不够，必须使用：
   1. 梯度检查点（Gradient Checkpointing）：激活值从 O(L) 降到 O(1)，但增加重计算开销
   2. ZeRO 优化器状态分片：将优化器状态分到多卡
   3. 混合精度训练 + 8-bit 优化器
```

#### 9.2.4 KV Cache 深入分析

KV Cache 是 Transformer 自回归生成的核心机制，也是长上下文推理的显存瓶颈。

**KV Cache 的生成与复用**：

```python
# 自回归生成的 KV Cache 机制

def generate_with_kv_cache(model, prompt_tokens, max_new_tokens):
    """
    使用 KV Cache 的自回归生成
    """
    # Step 1: Prefill - 一次性处理 prompt
    # 计算所有层的 K 和 V，并缓存起来
    past_key_values = []  # List of (K, V) for each layer
    
    for layer in model.layers:
        # 计算 Q, K, V
        q = layer.q_proj(hidden_states)
        k = layer.k_proj(hidden_states)
        v = layer.v_proj(hidden_states)
        
        # 缓存 K 和 V（这是 KV Cache 的来源）
        past_key_values.append((k, v))
        
        # 计算 attention（使用当前的 K, V）
        attn_output = attention(q, k, v)
        
    # Step 2: Decode - 逐个生成新 token
    for i in range(max_new_tokens):
        # 只处理最新的 1 个 token
        new_token_hidden = embeddings[-1:]  # [1, 1, hidden_size]
        
        for layer_idx, layer in enumerate(model.layers):
            q = layer.q_proj(new_token_hidden)
            k = layer.k_proj(new_token_hidden)
            v = layer.v_proj(new_token_hidden)
            
            # 将新的 K, V 追加到缓存中
            past_k, past_v = past_key_values[layer_idx]
            past_k = torch.cat([past_k, k], dim=2)  # 在 seq_len 维度拼接
            past_v = torch.cat([past_v, v], dim=2)
            past_key_values[layer_idx] = (past_k, past_v)
            
            # Attention 使用完整的 K, V 缓存
            attn_output = attention(q, past_k, past_v)
        
        # 生成下一个 token
        next_token = sample(logits)
```

**KV Cache 显存的精确计算**：

```
KV Cache 每 token 显存 = 2 (K+V) × num_layers × num_kv_heads × head_dim × bytes_per_element
                     = 2 × L × H_kv × dtype_bytes

其中 H_kv = num_kv_heads × head_dim
对于 GQA/MQA，H_kv < hidden_size

总 KV Cache 显存 = 2 × L × H_kv × seq_len × batch_size × dtype_bytes
```

**各模型的 KV Cache 效率比较**：

| 模型 | 注意力机制 | num_kv_heads | head_dim | 每 token KV Cache (FP16) | 相对于 MHA 的节省 |
|-----|----------|-------------|---------|------------------------|-----------------|
| Llama-2-7B | MHA | 32 | 128 | 2 × 32 × 4096 × 2 = 524 KB | 0% |
| Llama-3-8B | GQA | 8 | 128 | 2 × 32 × 1024 × 2 = 131 KB | 75% |
| Llama-3-70B | GQA | 8 | 128 | 2 × 80 × 1024 × 2 = 327 KB | 75% |

### 9.3 推理 vs 训练的显存差异

#### 9.3.1 显存占用对比总览

| 显存组成部分 | 推理（batch=1） | 训练（batch=1, AdamW） | 比例 |
|------------|--------------|---------------------|------|
| 模型权重 | 2 bytes/param (FP16) | 6 bytes/param (FP16+FP32) | 1:3 |
| 激活值 | ~5-10% 权重 | ~50-100% 权重 | 1:10 |
| 梯度 | 0 | 4 bytes/param | — |
| 优化器状态 | 0 | 8 bytes/param (Adam) | — |
| KV Cache | 2×L×H×S | 0（非自回归训练） | — |
| **总计（7B 模型）** | **~16 GB** | **~120 GB** | **1:7.5** |

#### 9.3.2 推理阶段的两个阶段

大模型推理分为两个阶段，显存需求和计算特征截然不同：

**Phase 1: Prefill（预填充/首次前向）**

```
输入：完整的 prompt（长度 = input_len）
输出：第一个新 token

特征：
- 计算密集型（compute-bound）
- 需要一次性计算所有 prompt token 的 KV Cache
- Attention 计算量为 O(input_len²)
- 可利用大 batch 和 Tensor Core 高吞吐

显存需求：
- 权重：W
- 激活值：较大（包含完整的 attention scores）
- KV Cache：从 0 增长到 input_len 长度

时间公式：
TTFT ∝ input_len² × hidden_size² / (batch × GPU_FLOPS)
```

**Phase 2: Decode（自回归生成）**

```
输入：上一个生成的 token
输出：下一个 token

特征：
- 内存密集型（memory-bound）
- 每步只处理 1 个新 token
- Attention 计算量为 O(seq_len)（因为 K,V 已缓存）
- 矩阵乘法的 M 维度为 1，无法充分利用 Tensor Core
- 受限于显存带宽：每步需要读取全部权重

显存需求：
- 权重：W
- KV Cache：持续增长，最终达到 max_seq_len
- 激活值：极小（只处理 1 个 token）

时间公式：
TPOT ∝ hidden_size² / GPU_bandwidth
每 token 需要读取所有权重一次
```

**Prefill vs Decode 的 Roofline 分析**：

```
Prefill:
- 算术强度高（矩阵乘法 M=N=seq_len）
- 位于 Roofline 的 compute-bound 区域
- 提升方向：更大的 batch、更快的 Tensor Core

Decode:
- 算术强度低（矩阵乘法 M=1, N=hidden_size）
- 位于 Roofline 的 memory-bound 区域
- 提升方向：更高的显存带宽、量化降低权重大小、批处理（continuous batching）
```

#### 9.3.3 Continuous Batching 的显存优势

传统静态 batching 等待所有序列完成后才处理下一个 batch，导致 GPU 利用率低下。Continuous Batching（也叫 In-flight Batching 或 Iteration-level Scheduling）在每次迭代中动态组装 batch。

```
静态 Batching：
Batch 1: [seq A (生成中), seq B (已完成), seq C (生成中)]
→ 必须等 A 和 C 都完成才能处理新请求
→ seq B 完成后，GPU 空闲

Continuous Batching：
Iteration 1: [seq A, seq B, seq C] 都 prefill
Iteration 2: [seq A, seq B, seq C] 都 decode 1 个 token
Iteration 3: [seq A, seq C] (seq B 已完成)，加入新请求 seq D
→ GPU 永远满负载
```

Continuous Batching 的显存影响：
- 需要为每个正在处理的序列分配 KV Cache 空间
- 总 KV Cache 显存 = sum(各序列当前长度) × per_token_kv_cache
- 需要预留足够的 KV Cache 槽位给新加入的序列
- vLLM 的 PagedAttention 将 KV Cache 分块管理，支持高效的内存复用和碎片整理


### 9.4 量化原理深度解析

#### 9.4.1 数值表示法：从 FP32 到 INT4

**IEEE 754 浮点数格式**：

```
FP32 (单精度浮点)：
┌────┬─────────────────────┬──────────────────────────────────────┐
│ 符号 │      指数 (8-bit)      │              尾数 (23-bit)             │
│  1  │    偏移 127 (0-255)    │         隐含前导 1.xxxxxx...           │
└────┴─────────────────────┴──────────────────────────────────────┘
值 = (-1)^符号 × 2^(指数-127) × 1.尾数
范围：~1.18×10⁻³⁸ 到 ~3.4×10³⁸
精度：约 7 位十进制有效数字

FP16 (半精度浮点)：
┌────┬───────────────┬────────────────────────────┐
│ 符号 │  指数 (5-bit)  │        尾数 (10-bit)        │
│  1  │  偏移 15      │      隐含前导 1.xxxx...     │
└────┴───────────────┴────────────────────────────┘
范围：~5.96×10⁻⁸ 到 65504
精度：约 3-4 位十进制有效数字
问题：数值范围小，大模型训练/推理容易出现 inf/NaN

BF16 (Brain Floating Point)：
┌────┬─────────────────────┬─────────────────────┐
│ 符号 │      指数 (8-bit)      │    尾数 (7-bit)     │
│  1  │    偏移 127 (同 FP32)   │   精度较低          │
└────┴─────────────────────┴─────────────────────┘
范围：同 FP32（~1.18×10⁻³⁸ 到 ~3.4×10³⁸）
精度：约 2-3 位十进制有效数字
优势：与 FP32 相同的数值范围，不易溢出

FP8 E4M3：
┌────┬──────────────┬────────────────┐
│ 符号 │ 指数 (4-bit)  │   尾数 (3-bit)  │
│  1  │  偏移 7      │  隐含前导 1.xx  │
└────┴──────────────┴────────────────┘
范围：~1.95×10⁻³ 到 448.0
用于：权重和激活（前向传播）

FP8 E5M2：
┌────┬──────────────┬────────────────┐
│ 符号 │ 指数 (5-bit)  │   尾数 (2-bit)  │
│  1  │  偏移 15      │  隐含前导 1.x   │
└────┴──────────────┴────────────────┘
范围：~1.53×10⁻⁵ 到 57344.0
用于：梯度（反向传播）
```

**整数格式**：

```
INT8（有符号 8-bit 整数）：
┌────────────────────────────┐
│      数值 (8-bit)           │
│   -128 到 127               │
└────────────────────────────┘
量化方式：
  量化：   q = round(x / scale) + zero_point
  反量化： x = (q - zero_point) × scale

INT4（4-bit 整数）：
┌────────────────────────────┐
│      数值 (4-bit)           │
│   0 到 15（非对称）          │
│   -8 到 7（对称）            │
└────────────────────────────┘
通常打包存储：2 个 INT4 存入 1 个 INT8
```

#### 9.4.2 量化映射的数学原理

**线性量化（Uniform Quantization）**：

```
给定浮点范围 [x_min, x_max]，映射到 N-bit 整数范围 [q_min, q_max]

scale = (x_max - x_min) / (q_max - q_min)
zero_point = q_min - round(x_min / scale)

量化：   q = clamp(round(x / scale) + zero_point, q_min, q_max)
反量化： x̂ = (q - zero_point) × scale

量化误差：ε = x - x̂
均方误差：MSE = E[ε²]
```

**对称量化 vs 非对称量化**：

| 类型 | zero_point | 适用场景 | 优点 | 缺点 |
|-----|-----------|---------|------|------|
| 对称 | 0 | 权重（通常以 0 为中心） | 硬件实现简单 | 对非 0 中心分布效率低 |
| 非对称 | 可调整 | 激活值（通常有偏置） | 利用整个量化范围 | 需要额外存储 zero_point |

**per-tensor vs per-channel vs per-group 量化**：

```
per-tensor 量化：
  整个张量共享一个 scale 和 zero_point
  存储开销：2 个 float / 张量
  精度：低（不同通道分布差异大）

per-channel 量化：
  每个输出通道有自己的 scale 和 zero_point
  存储开销：2 × num_channels 个 float / 张量
  精度：中（适合权重矩阵）

per-group 量化（分组量化）：
  每 group_size 个元素共享一个 scale
  存储开销：2 × (num_elements / group_size) 个 float / 张量
  精度：高（group_size=128 是常用折中）
  
  示例：权重矩阵 [4096, 4096]，group_size=128
  - 每组 128 个元素共享 scale
  - 共 4096×4096/128 = 131,072 组
  - 额外存储：131,072 × 2 × 4 bytes = 1 MB（相对于 32 MB 权重可忽略）
```

#### 9.4.3 各类量化方法的数学差异

**PTQ（Post-Training Quantization，训练后量化）**：

```
流程：
1. 获取预训练 FP16/BF16 模型
2. 收集校准数据（~128-512 个样本）
3. 计算每层/每通道/每组的 scale 和 zero_point
4. 将权重从 FP16 量化为 INT8/INT4
5.（可选）通过少量校准数据微调 scale

代表方法：
- GPTQ：基于 OBS 理论，逐层量化并补偿误差
- AWQ：保护显著权重通道
- SmoothQuant：将激活的量化难度迁移到权重
```

**QAT（Quantization-Aware Training，量化感知训练）**：

```
流程：
1. 在训练前向传播中模拟量化：
   x_quant = fake_quantize(x_fp32, scale, zero_point)
   = round(clamp(x / scale, q_min, q_max)) × scale
2. 反向传播时，梯度通过 straight-through estimator (STE) 传递：
   ∂L/∂x = ∂L/∂x_quant  (假设量化函数的导数为 1)
3. 模型学习适应量化带来的噪声

优势：精度最高
缺点：需要重新训练，计算成本高
```

#### 9.4.4 精度损失分析

**量化误差的来源**：

1. **裁剪误差（Clipping Error）**：
   - 当数值超出量化范围时，被强制截断到最大/最小值
   - 大模型权重通常服从高斯分布，极端值较少，裁剪误差可控

2. **舍入误差（Rounding Error）**：
   - `round(x / scale)` 引入的误差
   - 误差范围：[-0.5×scale, +0.5×scale]

3. **异常值问题（Outlier Problem）**：
   - 激活值中存在少量极大的异常值（outliers）
   - 这些异常值扩大了量化范围，导致大部分正常值的精度下降
   - SmoothQuant 的核心思想：通过缩放将激活的异常值"迁移"到权重

```
异常值的影响示例：

激活值分布：
  99% 的值在 [-1, 1] 范围内
  1% 的值约为 100（异常值）

如果不处理异常值：
  scale = 200 / 255 ≈ 0.78
  正常值 [-1, 1] 只用到 2/200 = 1% 的量化范围
  → 正常值的精度极差

SmoothQuant 的处理：
  对激活值进行缩放：x' = x / s
  对权重进行反向缩放：w' = w × s
  选择合适的 s，使 x' 和 w' 都易于量化
  由于 w 分布均匀，即使放大后仍易量化
  由于 x 的异常值被缩小，整个分布易于量化
```

**各精度格式的理论信噪比（SNR）**：

```
SNR ≈ 6.02 × N + 1.76 dB（对于均匀量化，满幅度正弦波）

FP32: 理论上无限精度（实际约 150 dB）
FP16: 约 70 dB
BF16: 约 55 dB（范围同 FP32）
FP8 E4M3: 约 35 dB
INT8: 约 49 dB
INT4: 约 25 dB

实际大模型中的有效 SNR 更低，因为：
- 权重分布不是满幅度正弦波
- 存在异常值
- 层间误差会累积
```

**实验数据：量化对模型能力的影响**：

| 量化格式 | WikiText-2 PPL | MMLU (5-shot) | HumanEval | 速度提升 | 显存节省 |
|---------|---------------|--------------|-----------|---------|---------|
| FP16 | 8.12 | 63.4% | 28.7% | 1× | 1× |
| BF16 | 8.12 | 63.4% | 28.7% | 1× | 1× |
| FP8 | 8.14 | 63.2% | 28.0% | 2× | 2× |
| INT8 (SmoothQuant) | 8.21 | 62.8% | 27.5% | 1.6× | 2× |
| AWQ-4bit | 8.35 | 61.5% | 26.0% | 1.8× | 4× |
| GPTQ-4bit | 8.48 | 60.8% | 25.5% | 1.7× | 4× |

> **结论**：FP8 在速度和精度之间取得了最佳平衡，精度损失 <1%。INT8 适合对精度要求较高的生产环境。4-bit 量化适合显存受限的消费级部署，但会损失 2-5% 的模型能力。

#### 9.4.5 AWQ、GPTQ、SmoothQuant 的数学本质

**AWQ（Activation-aware Weight Quantization）**：

```
核心洞察：
  并非所有权重对输出的影响相同。
  权重 w_ij 对输出 y_i 的贡献是 w_ij × x_j。
  如果 x_j（对应的激活值）很大，那么 w_ij 的量化误差会被放大。

数学形式：
  原始输出：y = W × x
  量化输出：ŷ = quantize(W) × x

  目标：最小化 ||y - ŷ||²

  AWQ 发现：只需要保护 1% 的"显著权重通道"（即对应大激活值的通道），
  就能恢复大部分精度。

实现方法：
  1. 计算每个权重通道的激活值范数：s_j = ||X_j||
  2. 对显著通道进行缩放保护：W'_j = W_j × α，同时 x'_j = x_j / α
  3. 量化 W'（此时显著通道的量化误差相对减小）
  4. 反量化时恢复：ŷ = dequantize(quant(W')) × x'
```

**GPTQ（基于 OBS 的量化）**：

```
核心洞察：
  最优脑损伤（Optimal Brain Surgeon, OBS）理论：
  当量化一个权重时，应该同时调整其他未量化的权重来补偿误差。

数学推导：
  目标：最小化量化后的输出误差
  
  对于单层，误差函数：
  E = ||W × X - quant(W) × X||²
    = (W - quant(W))^T × H × (W - quant(W))
  其中 H = X × X^T 是海森矩阵（Hessian）

  OBS 更新规则：
  当量化 w_q 时：
    w_i = w_i - (H^{-1}_{iq} / H^{-1}_{qq}) × (w_q - quant(w_q))
  
  即：用海森矩阵的逆来调整其他权重，补偿被量化权重的误差。

GPTQ 优化：
  - 使用 Cholesky 分解加速 H^{-1} 计算
  - 逐列（而非逐个元素）量化，减少迭代次数
  - 块处理（block processing）减少内存占用
```

**SmoothQuant**：

```
核心洞察：
  权重分布均匀，易于量化；
  激活值分布不均匀（存在异常值），难以量化。

数学变换：
  原始：Y = X × W
  
  引入平滑因子 s（逐通道）：
  Y = (X × diag(s)^{-1}) × (diag(s) × W)
    = X' × W'

  选择 s 使得 X' 和 W' 都易于量化：
  s_j = max(|X_j|)^α / max(|W_j|)^{1-α}
  
  α = 0.5 时，难度均摊到 X 和 W 各一半。
  α → 1 时，更多难度迁移到 W（适合 W 易于量化的情况）。

实际效果：
  - 激活值的异常值被缩小
  - 权重的分布虽然被放大，但由于原本均匀，仍易量化
  - 使得权重和激活都可以使用 INT8 量化，精度损失 <1%
```

#### 10.1.7 常见问题 FAQ

**Q1：RTX 3060 12GB 能跑 13B 模型吗？**
A：不能流畅运行。13B 模型即使 INT4 量化也需要约 7.5GB 权重 + 2GB KV Cache + 系统开销，接近 12GB 上限。且 RTX 3060 的带宽只有 360 GB/s，13B INT4 的理论吞吐仅约 48 t/s，实际可能只有 8-10 t/s，体验较差。建议升级显卡或使用 API 服务。

**Q2：为什么 12GB 显存跑 4B FP16 就占了 10GB？**
A：权重 8GB + KV Cache（seq=8192 时约 2.5GB）+ CUDA Context (~500MB) + 激活值 + 显存碎片。使用 `--max-model-len 4096` 可将 KV Cache 减半。

**Q3：AWQ 和 GPTQ 哪个在 RTX 3060 上更好？**
A：AWQ 更好。AWQ 的显著权重保护机制使其在 4-bit 下精度损失更小（约 2-3% vs GPTQ 的 3-5%）。且 AWQ 在 vLLM 中的 kernel 优化更成熟。

**Q4：RTX 3060 和 RTX 3060 Ti 8GB 哪个更适合大模型？**
A：RTX 3060 12GB 更适合。大模型推理中，显存容量比算力更重要。3060 Ti 虽然算力高 40%，但 8GB 显存连 7B FP16 都跑不了，只能跑量化版。3060 的 12GB 可以原生跑 4B FP16。

**Q5：如何监控 RTX 3060 的推理性能？**
A：
```bash
# 实时 GPU 监控
nvidia-smi dmon -s uct
# u=利用率, c=SM 时钟, t=温度

# vLLM 指标
curl http://localhost:8000/metrics | grep -E "vllm_(gpu_cache_usage|num_requests|time_)"
```

### 10.5 RTX 4090 24GB — 消费级天花板

#### 10.5.1 硬件规格深度解析

| 参数 | 规格 | 分析 |
|-----|------|------|
| 架构 | Ada Lovelace (AD102) | 完整 AD102 核心 |
| CUDA Core | 16384 | 消费级最多 |
| Tensor Core | 512 (4th Gen) | 消费级最强 FP16/FP8 |
| 显存 | 24 GB GDDR6X | 消费级最大 |
| 显存位宽 | 384-bit | 消费级最宽 |
| 显存带宽 | 1008 GB/s | 消费级最高，接近 A100 PCIe |
| TDP | 450W | 需要良好散热 |
| FP16 算力 | ~83 TFLOPS | A100 的 27% |
| NVLink | 不支持 | 多卡只能走 PCIe P2P |

**4090 的独特优势**：
1. 24GB 显存可单卡跑 14B FP16 或 32B AWQ
2. 1008 GB/s 带宽使 decode 速度远超其他消费级卡
3. 双卡 4090（48GB）可跑 70B AWQ，成本远低于 A100 双卡
4. 不支持 NVLink 是多卡部署的最大限制（PCIe P2P 带宽约 64 GB/s）

#### 10.5.2 完整部署方案

**14B INT8（4090 可原生精度运行）**：

```bash
# 注意：14B FP16 ≈ 29GB，单卡 4090 24GB 无法运行！
# 常见认知误区！必须使用 INT8 或双卡

vllm serve Qwen/Qwen2.5-14B-Instruct \
  --quantization fp8 \
  --dtype half \
  --max-model-len 32768 \
  --gpu-memory-utilization 0.92 \
  --max-num-seqs 64

# 或双卡 4090 跑 FP16
vllm serve Qwen/Qwen2.5-14B-Instruct \
  --dtype half \
  --tensor-parallel-size 2 \
  --max-model-len 65536
```

**32B AWQ（4090 杀手锏场景）**：

```bash
vllm serve Qwen/Qwen2.5-32B-Instruct-AWQ \
  --quantization awq \
  --dtype half \
  --max-model-len 8192 \
  --gpu-memory-utilization 0.95 \
  --max-num-seqs 32 \
  --max-num-batched-tokens 4096 \
  --enable-chunked-prefill

# 显存：
# AWQ 权重：32.5B × 0.5 = 16.3 GB
# KV Cache (seq=8192): 2 × 64 × 5120 × 8192 × 2 / (40/8) ≈ 2.1 GB
# 激活值 + 系统开销：~2 GB
# 总计：~20.4 GB，24GB 有充足余量
```

**双卡 4090 部署 70B AWQ**：

#### 10.5.5 FAQ

**Q：4090 能跑 70B FP16 吗？**
A：不能。70B FP16 需要 140GB+ 显存，至少需要 4×A100 40GB 或 2×A100 80GB。双卡 4090 48GB 只能跑 70B AWQ/INT4。

**Q：4090 和 A100 40GB 哪个更适合推理？**
A：取决于场景：
- 14B 以下：4090 更好（带宽 1008 vs 1555 GB/s，但 4090 便宜很多）
- 32B AWQ：4090 单卡即可，A100 40GB 需要量化
- 70B+：A100 多卡更稳定（有 NVLink），4090 多卡依赖 PCIe
- 生产环境：A100（ECC 显存、NVLink、MIG、企业支持）
- 个人/小团队：4090（性价比无敌）

**Q：4090 的 24GB 显存怎么分配最合理？**
A：
```
14B INT8: 权重 14.7GB + KV Cache 可变 + 系统开销 2GB
  → 建议 max-model-len 不超过 16K

32B AWQ: 权重 16.3GB + KV Cache 可变 + 系统开销 2GB
  → 建议 max-model-len 不超过 8K
  → 如果要 32K 上下文，需要进一步限制并发
```
