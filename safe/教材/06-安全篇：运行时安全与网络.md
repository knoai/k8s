# 第6章 运行时安全与网络安全

> **本章目标**：掌握 Kubernetes 运行时的安全监控和网络安全策略。我们将深入理解运行时威胁检测工具（Falco、Tetragon）、容器逃逸检测方法，以及 NetworkPolicy 和 Cilium 的网络安全实践。
>
> 读完本章后，你应该能够部署运行时安全监控系统，配置零信任网络策略，并识别容器逃逸的迹象。

## 6.1 运行时安全概述

### 6.1.1 为什么需要运行时安全

即使镜像扫描和准入控制都通过了，运行时仍可能出现以下安全威胁：

| 威胁类型 | 说明 | 示例 |
|---------|------|------|
| **0-day 漏洞利用** | 未知的漏洞在运行时被利用 | 新发现的容器运行时漏洞 |
| **配置漂移** | 运行时的配置与预期不符 | 特权容器被动态创建 |
| **内部威胁** | 合法用户的恶意操作 | 开发人员 exec 进容器窃取数据 |
| **供应链延迟触发** | 恶意代码在特定条件触发 | 时间炸弹、特定输入触发 |
| **横向移动** | 攻击者在集群内扩展控制 | 从被攻破的 Pod 扫描其他服务 |
| **加密挖矿** | 恶意程序消耗计算资源 | XMRig 挖矿程序 |
| **数据外泄** | 敏感数据被传输到外部 | 数据库转储到远程服务器 |

**安全阶段的对比**：

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│  开发阶段   │ → │  构建阶段   │ → │  部署阶段   │ → │  运行阶段   │
├─────────────┤   ├─────────────┤   ├─────────────┤   ├─────────────┤
│ SAST        │   │ 镜像扫描    │   │ 准入控制    │   │ 运行时检测  │
│ 依赖扫描    │   │ SBOM 生成   │   │ 签名验证    │   │ 行为基线    │
│ 密钥检测    │   │ 签名        │   │ 策略检查    │   │ 异常告警    │
├─────────────┤   ├─────────────┤   ├─────────────┤   ├─────────────┤
│ 已知漏洞    │   │ 已知漏洞    │   │ 配置合规    │   │ 未知威胁    │
│ 静态分析    │   │ 静态分析    │   │ 部署前拦截  │   │ 实时检测    │
└─────────────┘   └─────────────┘   └─────────────┘   └─────────────┘
     预防              预防            预防            检测+响应
```

运行时安全的核心是**检测异常行为**，而不是检测已知漏洞。

### 6.1.2 运行时安全能力矩阵

| 能力 | 说明 | 代表工具 | 实现机制 |
|------|------|---------|---------|
| **系统调用监控** | 跟踪进程的系统调用 | Falco, Tetragon | eBPF / Kernel Module |
| **文件完整性监控** | 检测关键文件变更 | Falco, AIDE | inode 监控 / eBPF |
| **网络流量分析** | 检测异常连接 | Cilium Hubble, Falco | eBPF / Netfilter |
| **进程行为基线** | 检测异常进程启动 | Falco, Tetragon | execve 跟踪 |
| **容器逃逸检测** | 检测突破容器边界 | Falco, Sysdig | 特权操作监控 |
| **K8s 审计集成** | 分析 API Server 审计日志 | Falco | K8s Audit Webhook |
| **自动响应** | 检测到威胁后自动处置 | Tetragon, OPA | kill 进程 / 隔离 Pod |

## 6.2 Falco：云原生威胁检测

### 6.2.1 Falco 架构详解

Falco 由 Sysdig 开源，是 CNCF 孵化项目，专门用于云原生运行时安全检测。

```
┌─────────────────────────────────────────────────────────────────┐
│                        Falco 架构                                │
│                                                                  │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────────┐ │
│  │  eBPF Probe │  │Kernel Module│  │    Modern eBPF Probe    │ │
│  │  (legacy)   │  │  (legacy)   │  │    (推荐，v0.35+)       │ │
│  └──────┬──────┘  └──────┬──────┘  └───────────┬─────────────┘ │
│         │                │                     │               │
│         └────────────────┼─────────────────────┘               │
│                          │                                      │
│                          ▼                                      │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                    Falco 引擎（用户空间）                   │  │
│  │                                                            │  │
│  │  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐   │  │
│  │  │  规则引擎     │  │  字段提取    │  │  输出模块     │   │  │
│  │  │              │  │  (extractor) │  │              │   │  │
│  │  │ • 解析规则   │  │              │  │ • stdout    │   │  │
│  │  │ • 匹配事件   │  │ • 容器元数据 │  │ • file      │   │  │
│  │  │ • 生成告警   │  │ • K8s 元数据 │  │ • webhook   │   │  │
│  │  │              │  │ • 进程信息   │  │ • gRPC      │   │  │
│  │  └──────────────┘  └──────────────┘  └──────────────┘   │  │
│  └──────────────────────────────────────────────────────────┘  │
│                          │                                      │
│                          ▼                                      │
│  ┌──────────────────────────────────────────────────────────┐  │
│  │                      输出目标                               │  │
│  │  stdout / 文件 / Slack / webhook / Falcosidekick / gRPC  │  │
│  └──────────────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
```

**驱动方式对比**：

| 驱动 | 内核要求 | 性能 | 稳定性 | 推荐场景 |
|------|---------|------|--------|---------|
| **Modern eBPF** | 5.8+ | 极高 | 高 | **新部署首选** |
| **eBPF Probe** | 4.14+ | 高 | 高 | 较新内核 |
| **Kernel Module** | 无要求 | 高 | 中 | 旧内核兼容性 |
| **gVisor** | 无要求 | 中 | 高 | gVisor 沙箱 |

Modern eBPF（CO-RE: Compile Once, Run Everywhere）是 Falco 0.35+ 的推荐驱动，它使用 BTF（BPF Type Format）信息实现跨内核版本兼容。

### 6.2.2 部署 Falco

```bash
# 添加 Helm 仓库
helm repo add falcosecurity https://falcosecurity.github.io/charts
helm repo update

# 安装 Falco（Modern eBPF 驱动）
helm install falco falcosecurity/falco \
  --namespace falco \
  --create-namespace \
  --set driver.kind=modern_ebpf \
  --set collectors.enabled=true \
  --set tty=true

# 验证安装
kubectl get pods -n falco
kubectl logs -n falco -l app.kubernetes.io/name=falco

# 查看 Falco 版本和驱动信息
kubectl exec -n falco -it deployment/falco -- falco --version
kubectl exec -n falco -it deployment/falco -- falco --list --markdown | head -50
```

### 6.2.3 Falco 规则详解

Falco 规则由条件、输出和优先级组成：

```yaml
# Falco 规则语法
- rule: <规则名称>
  desc: <描述>
  condition: <过滤条件>
  output: <输出格式>
  priority: <优先级>
  enabled: <是否启用>
  tags: [<标签列表>]

# 条件使用的字段
# 进程相关：proc.name, proc.cmdline, proc.pid, proc.ppid, proc.args
# 容器相关：container.name, container.image, container.id, container.privileged
# 文件相关：fd.name, fd.directory, fd.filename
# 网络相关：fd.sip, fd.sport, fd.cip, fd.cport
# 用户相关：user.name, user.uid
# K8s 相关：k8s.ns.name, k8s.pod.name, ka.user.name
```

**内置关键规则解析**：

```yaml
# 规则 1：容器内运行 shell
- rule: Terminal shell in container
  desc: >
    A shell was used as the entrypoint/exec point into a container with an
    attached terminal. Consider whether this is a legitimate use case.
  condition: >
    spawned_process                      # 新进程创建事件
    and container                        # 在容器内
    and shell_procs                      # 进程是 shell（bash/sh/zsh 等）
    and proc.tty != 0                    # 有 tty（交互式）
    and not container_shell_spawned      # 排除已知例外
  output: >
    A shell was spawned in a container with an attached terminal
    (user=%user.name user_loginuid=%user.loginuid %proc.info
    shell=%proc.name parent=%proc.pname
    cmdline=%proc.cmdline pid=%proc.pid terminal=%proc.tty
    container=%container.info)
  priority: NOTICE
  tags: [container, shell, mitre_execution]

# 规则 2：特权容器启动
- rule: Launch Privileged Container
  desc: >
    Detect the initial process started in a privileged container.
    Exceptions are made for known trusted images.
  condition: >
    spawned_process
    and container
    and container.privileged=true         # 特权容器
    and not user_known_privileged_containers  # 排除已知例外
  output: >
    Privileged container started
    (user=%user.name command=%proc.cmdline
    image=%container.image.repository:%container.image.tag)
  priority: INFO
  tags: [container, cis, mitre_privilege_escalation]

# 规则 3：敏感文件读取
- rule: Read sensitive file untrusted
  desc: >
    An attempt to read any sensitive file (e.g., files containing user/password/authentication
    information). Exceptions are made for known trusted programs.
  condition: >
    sensitive_files                     # 访问敏感文件
    and open_read                       # 读取操作
    and not proc.name in (trusted_binaries)  # 排除可信程序
    and not thread.cap_permitted contains CAP_SYS_ADMIN  # 排除 SYS_ADMIN
  output: >
    Sensitive file opened for reading by non-trusted program
    (user=%user.name command=%proc.cmdline file=%fd.name)
  priority: WARNING

# 规则 4：云元数据服务访问（容器逃逸迹象）
- rule: Contact EC2 Instance Metadata Service From Container
  desc: >
    The EC2 instance metadata service provides a variety of sensitive information
    that can be used for privilege escalation.
  condition: >
    spawned_process
    and container
    and (fd.sip.name=169.254.169.254 or aws_metadata_ip)  # AWS 元数据 IP
  output: >
    EC2 instance metadata service contacted from container
    (cmd=%proc.cmdline container=%container.info)
  priority: CRITICAL
  tags: [container, aws, metadata, mitre_credential_access]

# 规则 5：可疑出站连接（数据外泄）
- rule: Outbound Connection from CI/CD Container
  desc: >
    Detect outbound connections from CI/CD containers which are not expected
    to have network access.
  condition: >
    outbound                            # 出站连接
    and container                       # 在容器内
    and k8s.label.app in (jenkins, gitlab-runner, github-actions)  # CI/CD 容器
    and not trusted_domains             # 排除可信域名
  output: >
    Outbound connection from CI/CD container
    (connection=%fd.name container=%container.info)
  priority: WARNING
```

**Falco 宏（Macro）和列表（List）**：

```yaml
# 宏：可复用的条件片段
- macro: spawned_process
  condition: (evt.type in (execve, execveat) and evt.dir=<)

- macro: container
  condition: (container.id != host)

- macro: shell_procs
  condition: (proc.name in (shell_binaries))

# 列表：值的集合
- list: shell_binaries
  items: [bash, csh, sh, tcsh, zsh, dash, ksh]

- list: sensitive_files
  items: [/etc/shadow, /etc/sudoers, /etc/pam.conf, /etc/security/pwquality.conf]

- list: trusted_binaries
  items: [passwd, sudo, su, systemd, systemd-journal]
```

### 6.2.4 自定义 Falco 规则

```yaml
# custom-rules.yaml
# 自定义规则：检测生产环境命名空间中的特权容器创建
- rule: Privileged Container in Production
  desc: Detect privileged containers in production namespace
  condition: >
    spawned_process
    and container
    and container.privileged=true
    and k8s.ns.name in (production, payment)
  output: >
    Privileged container started in production namespace
    (user=%user.name command=%proc.cmdline
    image=%container.image.repository
    namespace=%k8s.ns.name pod=%k8s.pod.name)
  priority: CRITICAL
  tags: [production, privileged, container]

# 自定义规则：检测数据库容器中的异常进程
- rule: Unexpected Process in Database Container
  desc: Detect non-database processes running in database containers
  condition: >
    spawned_process
    and container
    and container.image.repository contains "postgres"
    and not proc.name in (postgres, pg_ctl, initdb, psql, vacuumdb)
  output: >
    Unexpected process in database container
    (process=%proc.name cmdline=%proc.cmdline
    pod=%k8s.pod.name namespace=%k8s.ns.name)
  priority: WARNING
  tags: [database, unexpected_process]

# K8s 审计规则：检测 RBAC 变更
- rule: K8s RBAC Change
  desc: Detect changes to RBAC resources
  condition: >
    kevt                             # K8s 审计事件
    and kresource in (role, rolebinding, clusterrole, clusterrolebinding)
    and kcreate or kmodify or kdelete
  output: >
    RBAC resource changed
    (user=%ka.user.name verb=%ka.verb resource=%kresource
    name=%ka.target.name namespace=%ka.target.namespace)
  priority: WARNING
  source: k8s_audit
  tags: [k8s, rbac, audit]

# 应用到 Falco
# 通过 Helm values 或 ConfigMap 挂载
```

## 6.3 Tetragon：eBPF 可观测与执行

### 6.3.1 Tetragon 与 Falco 对比

| 特性 | Falco | Tetragon |
|------|-------|----------|
| 检测方式 | 规则匹配（系统调用） | 进程执行跟踪（内核函数级） |
| eBPF 深度 | 系统调用层 | 内核函数级（kprobe/tracepoint） |
| 响应能力 | 仅告警 | **可直接终止进程** |
| 性能开销 | 低 | 极低 |
| 数据丰富度 | 高（K8s 元数据） | 高（进程族谱、网络连接） |
| 使用场景 | 通用威胁检测 | 精细化执行控制、自动响应 |
| 项目归属 | CNCF 孵化（Falcosecurity） | Cilium 生态（Isovalent） |

**选择建议**：
- **Falco**：通用威胁检测、规则丰富的安全监控、告警为主
- **Tetragon**：需要自动响应的场景（kill 进程）、进程级可观测、与 Cilium 集成

### 6.3.2 部署 Tetragon

```bash
# 安装 Tetragon
helm repo add cilium https://helm.cilium.io/
helm install tetragon cilium/tetragon -n kube-system

# 查看事件
kubectl exec -n kube-system -it ds/tetragon -c tetragon -- \
  tetra getevents -o compact

# 查看进程执行事件
kubectl exec -n kube-system -it ds/tetragon -c tetragon -- \
  tetra getevents --processes
```

### 6.3.3 Tetragon 追踪策略

```yaml
# tracing-policy.yaml
# 策略：检测并终止 web 命名空间中的 shell 执行
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: detect-shell-in-web
spec:
  kprobes:
  - call: "__x64_sys_execve"
    syscall: true
    args:
    - index: 0
      type: "string"
    selectors:
    - matchNamespaces:
      - namespace: web
      matchArgs:
      - index: 0
        operator: "Prefix"
        values:
        - "/bin/bash"
        - "/bin/sh"
        - "/usr/bin/bash"
      matchActions:
      - action: Sigkill    # 直接发送 SIGKILL
        message: "Shell execution blocked in web namespace"
```

**Tetragon 动作类型**：

| 动作 | 说明 | 使用场景 |
|------|------|---------|
| `NoAction` | 仅记录事件 | 观察模式 |
| `Sigkill` | 发送 SIGKILL 终止进程 | 高危操作阻断 |
| `Override` | 修改系统调用返回值 | 欺骗攻击者 |
| `GetUrl` | 发送 HTTP 请求 | 集成外部系统 |
| `PostUrl` | 发送 HTTP POST | 集成外部系统 |
| `StackTrace` | 记录堆栈跟踪 | 调试分析 |

## 6.4 容器逃逸检测

### 6.4.1 逃逸手法与检测

| 逃逸方式 | 前提条件 | 检测方法 | Falco 规则 |
|---------|---------|---------|-----------|
| **特权容器** | `--privileged` | 检测特权容器启动 | `Launch Privileged Container` |
| **危险挂载** | 挂载 `/proc`、`/sys` | 检测可疑卷挂载 | `Mount Launched in Privileged Container` |
| **CAP_SYS_ADMIN** | `cap-add SYS_ADMIN` | 检测能力获取 | 自定义规则 |
| **hostPID** | `hostPID: true` | 检测 hostPID 使用 | 自定义规则 |
| **Docker Socket** | 挂载 `/var/run/docker.sock` | 检测 socket 访问 | `Contact K8s API Server From Container` |
| **内核漏洞** | 未修补内核 | 检测异常系统调用 | 异常行为基线 |
| **User Namespace 绕过** | 错误配置 | 检测 UID 映射异常 | 自定义规则 |

### 6.4.2 逃逸检测规则示例

```yaml
# 检测 Docker Socket 挂载后的容器创建
- rule: Docker Socket Access
  desc: Detect access to Docker socket from container
  condition: >
    spawned_process
    and container
    and (fd.name contains "/var/run/docker.sock" or
         fd.name contains "/run/containerd/containerd.sock")
  output: >
    Container runtime socket accessed
    (user=%user.name command=%proc.cmdline container=%container.info)
  priority: CRITICAL

# 检测挂载操作（可能用于逃逸）
- rule: Mount in Container
  desc: Detect mount operations inside containers
  condition: >
    spawned_process
    and container
    and proc.name = "mount"
    and not user_known_mount_in_container
  output: >
    Mount command executed in container
    (user=%user.name command=%proc.cmdline container=%container.info)
  priority: WARNING

# 检测对 /proc/1/environ 的访问（试图读取宿主机 init 进程）
- rule: Access to Host Process Environment
  desc: Detect attempts to read host process environment variables
  condition: >
    open_read
    and container
    and fd.name startswith /proc/
    and fd.name contains "/environ"
    and not proc.name in (ps, top, htop)
  output: >
    Host process environment accessed from container
    (file=%fd.name user=%user.name container=%container.info)
  priority: WARNING
```

## 6.5 网络安全

### 6.5.1 NetworkPolicy 深度解析

NetworkPolicy 是 Kubernetes 的 L3/L4 网络隔离机制。理解其工作原理至关重要：

**默认行为**：
- 没有 NetworkPolicy 的 Pod：**允许所有流量**（入站和出站）
- 有 NetworkPolicy 的 Pod：**只允许策略明确允许的流量**

**这个默认行为意味着**：NetworkPolicy 是一种"白名单"机制，应用了策略的 Pod 默认拒绝所有流量，除非被显式允许。

```yaml
# 零信任网络的起点：默认拒绝所有流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: production
spec:
  podSelector: {}      # 匹配所有 Pod
  policyTypes:
  - Ingress
  - Egress
  # 没有定义 ingress 和 egress 规则 = 拒绝所有流量
```

**精细化策略设计**：

```yaml
# 三层应用的网络策略
---
# 1. Frontend：只允许来自 Ingress 的流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: frontend-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: frontend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: ingress-nginx
    ports:
    - protocol: TCP
      port: 8080
```

### 6.5.2 Cilium 高级网络安全

Cilium 基于 eBPF 提供 L3-L7 的网络安全策略：

```yaml
# CiliumNetworkPolicy：L7 HTTP 层过滤
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: api-l7-policy
  namespace: production
spec:
  endpointSelector:
    matchLabels:
      app: api
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
    toPorts:
    - ports:
      - port: "80"
        protocol: TCP
      rules:
        http:
        - method: "GET"
          path: "/api/v1/users"
        - method: "GET"
          path: "/api/v1/users/.*"
        - method: "POST"
          path: "/api/v1/users"
        - method: "DELETE"
          path: "/api/v1/users/.*"
          # 可以进一步限制 HTTP header
          headers:
          - name: "X-Role"
            presence: true
  egress:
  - toEndpoints:
    - matchLabels:
        k8s:io.kubernetes.pod.namespace: kube-system
        k8s:k8s-app: kube-dns
    toPorts:
    - ports:
      - port: "53"
        protocol: UDP
      rules:
        dns:
        - matchPattern: "*.internal.company.com"
        - matchName: "database.internal.company.com"
  - toFQDNs:
    - matchName: "api.stripe.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP
```

**Cilium 的 DNS 策略**：

```yaml
# 只允许访问特定的外部域名
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: egress-dns
  namespace: production
spec:
  endpointSelector:
    matchLabels:
      app: payment-service
  egress:
  - toFQDNs:
    - matchName: "api.stripe.com"
    - matchName: "api.paypal.com"
    toPorts:
    - ports:
      - port: "443"
        protocol: TCP
```

**Cilium 的 L7 协议支持**：

| 协议 | 可过滤字段 | 使用场景 |
|------|-----------|---------|
| HTTP | Method, Path, Header, Host | API 访问控制 |
| Kafka | Topic, Role (produce/consume) | 消息队列安全 |
| gRPC | Service, Method | 微服务调用控制 |
| DNS | Query, Query Type | 出站 DNS 控制 |

### 6.5.3 零信任网络

**零信任的核心原则**：
1. **永不信任，始终验证**：不假设内部网络是安全的
2. **最小权限**：每个服务只能访问必要的资源
3. **持续验证**：身份和权限持续检查
4. **假设 breach**：假设攻击者已经在网络中

**Kubernetes 中的零信任实现**：

```
传统安全模型                    零信任模型
     │                            │
┌────▼────┐                  ┌────▼────┐
│  防火墙  │                  │ mTLS    │
│ 外部边界 │                  │ 身份认证 │
└────┬────┘                  └────┬────┘
     │                            │
  内部=可信                    每个服务
     │                        互相验证
     ▼                            ▼
  自由通信                    策略控制
     │                            │
     │                       ┌────▼────┐
     │                       │ Network │
     │                       │ Policy  │
     │                       └────┬────┘
     │                            │
     │                       ┌────▼────┐
     │                       │  L7 策略 │
     │                       │ (Cilium) │
     │                       └────┬────┘
     │                            │
     │                       ┌────▼────┐
     │                       │ 运行时  │
     │                       │ 监控    │
     │                       │ (Falco) │
     │                       └─────────┘
```

**技术栈**：
- **身份**：SPIFFE/SPIRE 为每个工作负载发放身份
- **传输安全**：Istio/Linkerd mTLS 自动加密
- **访问控制**：NetworkPolicy + Cilium L7 策略
- **持续验证**：Falco 运行时异常检测

### 实验 6.2：Falco 自定义规则（25 分钟）

```bash
# 步骤 1：创建自定义规则文件
cat <<EOF > custom-rules.yaml
- rule: Production Namespace Shell
  desc: Detect shell execution in production namespace
  condition: >
    spawned_process
    and container
    and shell_procs
    and k8s.ns.name in (production)
  output: >
    Shell executed in production namespace
    (user=%user.name command=%proc.cmdline pod=%k8s.pod.name)
  priority: CRITICAL
EOF

# 步骤 2：安装 Falco 并挂载自定义规则

  --namespace falco \
  --create-namespace \
  --set driver.kind=modern_ebpf \
  --set-file customRules.custom-rules.yaml=./custom-rules.yaml

# 步骤 3：创建 production 命名空间并触发规则
```

### 实验 6.4：Tetragon 进程阻断（25 分钟）

```bash
# 步骤 1：安装 Tetragon
helm repo add cilium https://helm.cilium.io/
helm install tetragon cilium/tetragon -n kube-system

# 步骤 2：创建 TracingPolicy
cat <<EOF | kubectl apply -f -
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: block-shell
spec:
  kprobes:
  - call: "__x64_sys_execve"
    syscall: true
    args:
    - index: 0
      type: "string"
    selectors:
    - matchArgs:
      - index: 0
        operator: "Prefix"
        values:
        - "/bin/bash"
        - "/bin/sh"
      matchActions:
      - action: Sigkill
EOF

# 步骤 3：创建测试 Pod
kubectl run test-shell --image=busybox:1.36 --restart=Never -- sleep 3600

# 步骤 4：尝试执行 shell（应该被终止）
kubectl exec -it test-shell -- /bin/sh -c "echo test" || echo "Process killed by Tetragon"

# 步骤 5：查看 Tetragon 事件
kubectl exec -n kube-system -it ds/tetragon -c tetragon -- \
  tetra getevents -o compact | grep -i "shell\|sigkill"

# 清理
kubectl delete tracingpolicy block-shell
kubectl delete pod test-shell
helm uninstall tetragon -n kube-system
```

## 6.9 本章小结

| 主题 | 工具/机制 | 核心能力 | 使用场景 |
|------|----------|---------|---------|
| **运行时检测** | Falco | 系统调用监控、规则告警 | 通用威胁检测、合规监控 |
| **执行控制** | Tetragon | eBPF 进程跟踪、自动终止 | 高危操作自动阻断 |
| **网络隔离** | NetworkPolicy | L3/L4 访问控制 | 服务间网络隔离 |
| **高级网络** | Cilium | L7 过滤、DNS 控制、身份感知 | API 访问控制、零信任 |
| **逃逸检测** | Falco 规则 | 特权容器、危险挂载、元数据访问 | 容器逃逸防御 |
| **零信任** | Istio/Cilium/SPIRE | mTLS、身份认证、持续验证 | 金融、高安全要求场景 |

**关键安全原则**：
1. **默认拒绝**：没有明确允许的流量都拒绝（NetworkPolicy）
2. **最小权限**：每个服务只能访问必要的资源
3. **持续监控**：运行时行为持续监控，异常立即告警
4. **自动响应**：高危操作自动阻断，减少人工响应时间
5. **多层防御**：NetworkPolicy + Runtime Security + mTLS 叠加
