# 第13章 专题：云厂商托管 Kubernetes 安全（EKS / GKE / AKS）

> **本章目标**：深入理解 AWS EKS、GCP GKE、Azure AKS 三大主流托管 K8s 服务的安全架构。实际工作中，超过 80% 的企业使用云厂商托管的 K8s 而非自建。
>
> 读完本章后，你应该能够：理解云厂商责任共担模型；配置 Pod 级 IAM 身份；实施网络隔离和加密；建立跨云安全基线。

## 13.1 托管 K8s 安全概述

### 13.1.1 责任共担模型深度解析

云安全遵循**责任共担模型**，云厂商和用户各自负责不同层面的安全：

```
┌─────────────────────────────────────────────────────────────────┐
│                        云厂商责任                                │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 控制平面（Control Plane）                                   ││
│  │ ├─ API Server 高可用部署（多 AZ）                           ││
│  │ ├─ etcd 管理（备份、加密、高可用）                          ││
│  │ ├─ Scheduler / Controller Manager                           ││
│  │ ├─ 控制平面网络隔离（不暴露到公网）                         ││
│  │ ├─ 控制平面补丁和版本升级                                   ││
│  │ └─ 托管组件（CNI、CSI、CoreDNS）                            ││
│  └─────────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────────┤
│                        共享责任区                                │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ ├─ 网络配置（VPC、子网、路由、安全组）                      ││
│  │ ├─ 身份认证（IAM 集成、OIDC）                               ││
│  │ ├─ 加密（KMS、数据加密配置）                                ││
│  │ ├─ 审计日志（采集、存储、分析）                             ││
│  │ └─ 节点操作系统（基础镜像、补丁策略）                       ││
│  └─────────────────────────────────────────────────────────────┘│
├─────────────────────────────────────────────────────────────────┤
│                        用户责任                                  │
│  ┌─────────────────────────────────────────────────────────────┐│
│  │ 工作负载安全                                                ││
│  │ ├─ Pod Security Standards / Admission Control               ││
│  │ ├─ RBAC 配置                                                ││
│  │ ├─ NetworkPolicy                                            ││
│  │ ├─ 镜像安全（扫描、签名、来源控制）                         ││
│  │ ├─ Secret 管理                                              ││
│  │ ├─ 应用层安全（WAF、输入验证）                              ││
│  │ └─ 运行时安全（Falco、监控告警）                            ││
│  └─────────────────────────────────────────────────────────────┘│
└─────────────────────────────────────────────────────────────────┘
```

| 安全域 | 自建 K8s | 托管 K8s（EKS/GKE/AKS） |
|--------|---------|----------------------|
| 控制平面可用性 | 自行搭建 HA | 云厂商保证 SLA（99.9%+） |
| etcd 加密 | 自行配置 KMS | 默认启用云 KMS |
| 控制平面网络 | 自行隔离 | 默认私有，可选公网 |
| 节点管理 | 完全负责 | 半托管（自动升级） |
| 身份认证 | 自行集成 LDAP/OIDC | 云 IAM 原生集成 |
| 网络策略 | 自行选择 CNI | 云 CNI + 可选 Calico/Cilium |
| 审计日志 | 自行采集存储 | 云监控集成 |
| 威胁检测 | 自行部署 Falco 等 | 云原生 GuardDuty/SCC/Defender |

### 13.1.2 云原生攻击面

托管 K8s 引入了特有的攻击面：

```
云厂商 K8s 攻击面：

[外部攻击者]
     │
     ├──► API Server（如果暴露公网）
     │
     ├──► 元数据服务（169.254.169.254）
     │    ├─ EC2 IMDS（AWS）
     │    ├─ GCE Metadata（GCP）
     │    └─ Azure IMDS（Azure）
     │
     ├──► 负载均衡器/Ingress
     │
     └──► 节点 SSH/管理端口

[容器内攻击者]
     │
     ├──► 节点 IAM 凭证（Instance Profile）
     │
     ├──► 集群内网横向移动
     │
     └──► 其他 Pod 的 ServiceAccount Token
```

## 13.2 AWS EKS 安全深度指南

### 13.2.1 EKS 身份认证：IRSA（IAM Roles for Service Accounts）

**传统方式的问题**：

```
问题模型（节点级 IAM）：

┌──────────────────────────────────────────────────────┐
│  EC2 节点                                            │
│  ├─ Instance Profile: S3FullAccess                   │
│  ├─ Pod A: 需要访问 S3（合法）                        │
│  ├─ Pod B: 不需要访问 S3                              │
│  └─ Pod C: 恶意 Pod（也能访问 S3！）                  │
└──────────────────────────────────────────────────────┘

所有 Pod 共享节点的 IAM 权限！
```

**IRSA 方案架构**：

```
┌─────────────┐     ┌─────────────────┐     ┌──────────────┐
│    Pod      │────▶│  ServiceAccount │────▶│ OIDC Provider│
│             │     │  (带 Annotation) │     │ (EKS 自带)   │
└─────────────┘     └─────────────────┘     └──────┬───────┘
                                                   │
                              ┌────────────────────┘
                              │ JWT Token 签名验证
                              ▼
                       ┌──────────────┐
                       │   AWS STS    │
                       │ AssumeRole   │
                       │ WithWebIdent │
                       └──────┬───────┘
                              │
                              ▼ 签发临时凭证
                       ┌──────────────┐
                       │   AWS IAM    │
                       │    Role      │
                       │ (最小权限)    │
                       └──────────────┘
```

**IRSA 完整配置流程**：

```bash
# ========== 步骤 1：创建 EKS 集群（带 OIDC）==========
eksctl create cluster \
  --name production \
  --region us-west-2 \
  --node-type t3.medium \
  --nodes 3 \
  --nodes-min 2 \
  --nodes-max 10 \
  --managed \
  --asg-access \
  --external-dns-access

# ========== 步骤 2：启用 OIDC Provider ==========
# 检查是否已启用
aws eks describe-cluster --name production \
  --query "cluster.identity.oidc.issuer" --output text

# 如果没有，启用它
eksctl utils associate-iam-oidc-provider \
  --cluster production \
  --region us-west-2 \
  --approve

# ========== 步骤 3：创建 IAM Policy（最小权限）==========
cat > s3-readonly-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::company-data-bucket",
        "arn:aws:s3:::company-data-bucket/app-data/*"
      ],
      "Condition": {
        "StringEquals": {
          "aws:RequestedRegion": "us-west-2"
        }
      }
    }
  ]
}
EOF

aws iam create-policy \
  --policy-name AppS3ReadOnlyPolicy \
  --policy-document file://s3-readonly-policy.json

# ========== 步骤 4：创建 IAM Role 并绑定 SA ==========
eksctl create iamserviceaccount \
  --name app-sa \
  --namespace production \
  --cluster production \
  --attach-policy-arn arn:aws:iam::123456789012:policy/AppS3ReadOnlyPolicy \
  --approve \
  --override-existing-serviceaccounts

# ========== 步骤 5：Pod 使用 ServiceAccount ==========
# Pod 自动获得 AWS 凭证，无需硬编码
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-irsa
  namespace: production
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: myapp:latest
    env:
    - name: AWS_REGION
      value: "us-west-2"
    # AWS_ROLE_ARN 和 AWS_WEB_IDENTITY_TOKEN_FILE
    # 由 EKS 自动注入，无需手动配置
```

**IRSA 安全最佳实践**：

1. **每个微服务独立的 IAM Role**：不要共享 Role
2. **最小权限**：只授予需要的 Action 和 Resource
3. **不信任节点 IAM**：节点 Instance Profile 只保留 EKS Worker Node 策略
4. **条件约束**：使用 IAM Condition 限制源 IP、VPC、时间等
5. **定期轮换**：OIDC Provider 的证书自动轮换，但应监控

```bash
# 检查节点 IAM 是否过宽
aws iam get-instance-profile \
  --instance-profile-name eks-node-profile | \
  jq '.InstanceProfile.Roles[].AttachedPolicies'

# 不应包含 S3/EC2 等应用级权限
# 只应包含：
# - AmazonEKSWorkerNodePolicy
# - AmazonEC2ContainerRegistryReadOnly
# - AmazonEKS_CNI_Policy
```

### 13.2.2 元数据服务保护（IMDSv2）

AWS EC2 实例元数据服务（IMDS）是容器逃逸后的首要攻击目标：

```bash
# 攻击者在容器内访问元数据服务
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/
# 如果响应了角色名称，说明可以获取临时凭证！

# 获取凭证
curl http://169.254.169.254/latest/meta-data/iam/security-credentials/eks-node-role
# 返回 AccessKeyId, SecretAccessKey, Token
```

**防御：启用 IMDSv2（会话认证）**：

```bash
# 创建节点组时启用 IMDSv2
eksctl create nodegroup \
  --cluster production \
  --name imdsv2-ng \
  --node-type t3.medium \
  --node-private-networking \
  --instance-prefix imdsv2 \
  --instance-name imdsv2-node

# 或者修改现有节点组
eksctl utils update-nodegroup-iam \
  --cluster production \
  --nodegroup-name ng-1 \
  --approve

# 使用 hop limit 限制容器访问
aws ec2 modify-instance-metadata-options \
  --instance-id i-1234567890abcdef0 \
  --http-tokens required \
  --http-put-response-hop-limit 1
# hop-limit=1 意味着容器无法访问 IMDS（NAT 跳数为 2）
```

### 13.2.3 EKS 网络隔离

**VPC CNI + SecurityGroupPolicy**：

```bash
# 启用 Network Policy 支持（EKS VPC CNI 1.14+）
kubectl set env daemonset aws-node -n kube-system \
  ENABLE_NETWORK_POLICY=true

# 验证
kubectl get daemonset aws-node -n kube-system -o yaml | \
  grep ENABLE_NETWORK_POLICY
```

```yaml
# SecurityGroupPolicy：将安全组绑定到 Pod
apiVersion: vpcresources.k8s.aws/v1beta1
kind: SecurityGroupPolicy
metadata:
  name: database-access
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment-service
  securityGroups:
    groupIds:
    - sg-0db8d0a468f3ac5c1  # 只允许访问数据库的安全组
    - sg-0a1b2c3d4e5f6g7h8  # 允许出站到 NAT Gateway

---
# 对应的 NetworkPolicy
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: payment-netpolicy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment-service
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          name: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: postgres
    ports:
    - protocol: TCP
      port: 5432
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
```

## 13.3 GCP GKE 安全深度指南

### 13.3.1 GKE 身份认证：Workload Identity

**Workload Identity 架构**：

```
┌──────────────────┐        ┌──────────────────┐
│  K8s Namespace   │        │  GCP Project     │
│  ┌────────────┐  │        │  ┌────────────┐  │
│  │K8s SA      │──┼──绑定──┼─▶│GCP SA      │  │
│  │my-app-sa   │  │        │  │my-app-gcp  │  │
│  └────────────┘  │        │  └────────────┘  │
│       │          │        │       │          │
│       ▼          │        │       ▼          │
│  ┌────────────┐  │        │  ┌────────────┐  │
│  │Pod         │  │        │  │IAM 权限    │  │
│  │自动获取    │  │        │  │├─ GCS 读写 │  │
│  │GCP Token   │  │        │  │├─ Cloud SQL│  │
│  └────────────┘  │        │  │└─ Secret M │  │
└──────────────────┘        └──────────────────┘
```

**Workload Identity 完整配置**：

```bash
# ========== 步骤 1：创建启用 Workload Identity 的集群 ==========
gcloud container clusters create production \
  --workload-pool=my-project.svc.id.goog \
  --zone us-central1-a \
  --num-nodes 3 \
  --machine-type e2-medium \
  --enable-network-policy \
  --enable-dataplane-v2 \
  --shielded-secure-boot \
  --shielded-integrity-monitoring

# 为现有节点池启用
# gcloud container node-pools update default-pool \
#   --cluster=production \
#   --workload-metadata=GKE_METADATA

# ========== 步骤 2：创建 GCP ServiceAccount ==========
gcloud iam service-accounts create app-gcp-sa \
  --display-name="App GCP Service Account"

# 授予最小权限
gcloud projects add-iam-policy-binding my-project \
  --member="serviceAccount:app-gcp-sa@my-project.iam.gserviceaccount.com" \
  --role="roles/storage.objectViewer"

# ========== 步骤 3：绑定 K8s SA 到 GCP SA ==========
gcloud iam service-accounts add-iam-policy-binding \
  app-gcp-sa@my-project.iam.gserviceaccount.com \
  --role="roles/iam.workloadIdentityUser" \
  --member="serviceAccount:my-project.svc.id.goog[production/app-sa]"

# ========== 步骤 4：为 K8s SA 添加 Annotation ==========
kubectl create namespace production
kubectl create serviceaccount app-sa -n production

kubectl annotate serviceaccount app-sa \
  iam.gke.io/gcp-service-account=app-gcp-sa@my-project.iam.gserviceaccount.com \
  -n production

# ========== 步骤 5：Pod 使用 Workload Identity ==========
```

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: app-with-workload-id
  namespace: production
spec:
  serviceAccountName: app-sa
  containers:
  - name: app
    image: gcr.io/my-project/myapp:latest
    # Pod 自动获取 GCP 访问令牌
    # 可通过 metadata server 访问
```

**验证 Workload Identity**：

```bash
# 进入 Pod
kubectl exec -it app-with-workload-id -- /bin/sh

# 获取 GCP 访问令牌
curl -H "Metadata-Flavor: Google" \
  http://169.254.169.254/computeMetadata/v1/instance/service-accounts/default/token

# 验证可以访问 GCS
gsutil ls gs://my-bucket/

# 验证不能访问其他资源（权限隔离）
gsutil ls gs://other-bucket/  # 应该失败
```

### 13.3.2 GKE 安全特性矩阵

| 特性 | 说明 | 启用方式 | 安全价值 |
|------|------|---------|---------|
| **Shielded GKE Nodes** | 安全启动 + vTPM + 完整性监控 | `--shielded-secure-boot` | 防止启动级攻击 |
| **Workload Identity** | Pod 级 GCP 身份 | `--workload-pool` | 替代节点 SA |
| **Private Cluster** | 控制平面无公网 IP | `--enable-private-nodes` | 减少攻击面 |
| **Authorized Networks** | 限制 API Server 访问 | `--master-authorized-networks` | IP 白名单 |
| **Dataplane V2** | eBPF 网络策略 + 可观测性 | `--enable-dataplane-v2` | 替代 Calico |
| **Autopilot** | 托管节点，锁定安全配置 | `gcloud container clusters create-auto` | 安全默认 |
| **Binary Authorization** | 镜像签名强制验证 | `--enable-binary-authorization` | 供应链安全 |
| **GKE Sandbox (gVisor)** | 沙箱容器运行时 | `runtimeClassName: gvisor` | 内核隔离 |

## 13.4 Azure AKS 安全深度指南

### 13.4.1 AKS 身份认证：Azure AD + Workload Identity

**Azure AD Integration**：

```bash
# ========== 创建启用 AAD 的 AKS 集群 ==========
az aks create \
  --resource-group my-rg \
  --name production \
  --enable-aad \
  --aad-admin-group-object-ids <aad-group-id> \
  --enable-azure-rbac \
  --node-count 3 \
  --node-vm-size Standard_DS2_v2

# 获取凭证（触发 AAD 登录）
az aks get-credentials --resource-group my-rg --name production
kubectl get pods
# 浏览器弹出 Azure AD 登录窗口
```

**Azure Workload Identity**（AKS 1.24+，推荐替代 AAD Pod Identity）：

```bash
# ========== 步骤 1：安装 Workload Identity ==========
helm repo add azure-workload-identity https://azure.github.io/azure-workload-identity/charts
helm install workload-identity-webhook azure-workload-identity/workload-identity-webhook \
  --namespace azure-workload-identity-system \
  --create-namespace

# ========== 步骤 2：创建 Managed Identity ==========
az identity create \
  --name app-identity \
  --resource-group my-rg

# 获取 Identity 信息
export USER_ASSIGNED_CLIENT_ID=$(az identity show \
  --name app-identity \
  --resource-group my-rg \
  --query clientId -o tsv)

# ========== 步骤 3：建立联邦信任 ==========
export AKS_OIDC_ISSUER=$(az aks show \
  --name production \
  --resource-group my-rg \
  --query "oidcIssuerProfile.issuerUrl" -o tsv)

az identity federated-credential create \
  --name app-federated-cred \
  --identity-name app-identity \
  --resource-group my-rg \
```

## 13.7 云原生攻击场景与防御

### 13.7.1 元数据服务攻击（SSRF → 凭证窃取）

```
攻击流程：

[外部用户]
     │
     ▼ 发送恶意请求到存在 SSRF 漏洞的应用
     │    http://169.254.169.254/latest/meta-data/
     │
[Pod 内]
     │
     ▼ 应用发起请求到元数据服务
     │
[元数据服务]
     │
     ▼ 返回 IAM 凭证
     │    AccessKeyId, SecretAccessKey, Token
     │
[攻击者]
     │
     ▼ 使用窃取的凭证访问云资源
```

**防御措施**：

```bash
# AWS：启用 IMDSv2 + hop limit
aws ec2 modify-instance-metadata-options \
  --instance-id <instance-id> \
  --http-tokens required \
  --http-put-response-hop-limit 1

# GCP：Workload Identity 替代节点 SA
# 节点 SA 不授予任何应用权限

# Azure：使用 Managed Identity + Workload Identity
# 不要使用节点上的 Azure Identity
```

### 13.7.2 供应链攻击：公共镜像仓库

```bash
# 攻击者上传 typosquatting 镜像
docker pull nginx_official:latest   # 恶意
docker pull ngix:latest             # 恶意（拼写错误）

# 防御：准入控制限制镜像来源
```

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-image-registries
spec:
  validationFailureAction: Enforce
  rules:
  - name: validate-registries
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "只允许使用批准的镜像仓库"
      pattern:
        spec:
          containers:
          - image: "*.amazonaws.com/* | gcr.io/* | *.azurecr.io/* | registry.company.io/*"
```

## 13.11 多云安全架构设计

### 13.11.1 跨云 K8s 安全挑战

```
多云 K8s 安全挑战：

身份管理       │ 每个云有自己的 IAM，难以统一
网络互联       │ 跨云网络延迟高，安全边界复杂
密钥分发       │ Secret 需要在多个云之间同步
合规审计       │ 不同云有不同合规框架
监控告警       │ 分散的监控数据，难以关联分析
成本控制       │ 安全工具的重复部署

解决方向：
├── 统一身份：SPIFFE/SPIRE 跨云身份
├── 统一网络：Cilium ClusterMesh / Submariner
├── 统一密钥：External Secrets Operator + Vault
├── 统一监控：Thanos / Cortex 聚合多集群指标
└── 统一策略：OPA/Gatekeeper 全局策略
```

## 13.10 本章小结

| 主题 | AWS EKS | GCP GKE | Azure AKS |
|------|---------|---------|-----------|
| **Pod 身份** | IRSA (OIDC + IAM Role) | Workload Identity (GCP SA) | Workload Identity (Federated MI) |
| **元数据保护** | IMDSv2 + hop limit | GKE_METADATA | Managed Identity |
| **网络策略** | VPC CNI / SecurityGroupPolicy | Dataplane V2 (Cilium) | Azure CNI + Calico |
| **日志审计** | CloudWatch + CloudTrail | Cloud Logging | Azure Monitor |
| **威胁检测** | GuardDuty for EKS | Security Command Center | Defender for Containers |
| **Secret 管理** | Secrets Manager + CSI | Secret Manager + CSI | Key Vault + CSI |
| **镜像安全** | ECR + cosign/Notation | Binary Authorization | ACR + Notation |

**核心原则**：
1. **使用 Pod 级 IAM**：绝不要让 Pod 继承节点 IAM 权限
2. **保护元数据服务**：启用 IMDSv2 或等效机制
3. **Private Cluster**：控制平面不暴露公网
4. **云原生工具链**：充分利用云厂商提供的安全工具
5. **持续监控**：启用威胁检测，定期检查配置

**推荐阅读**：
- AWS EKS 安全最佳实践：https://docs.aws.amazon.com/eks/latest/userguide/security.html
- GKE 安全概览：https://cloud.google.com/kubernetes-engine/docs/concepts/security-overview
- AKS 安全基线：https://docs.microsoft.com/azure/aks/security-baseline
