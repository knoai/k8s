## 5.1 镜像安全

### 5.1.1 镜像安全威胁模型

容器镜像是 Kubernetes 工作负载的基础，也是供应链攻击的主要目标之一。

```
开发者编写代码
    │
    ▼
┌─────────────┐    后门代码、密钥泄露、恶意依赖
│  代码仓库    │──────────────────────────────────► 攻击者植入后门
└──────┬──────┘
       │
       ▼
┌─────────────┐    基础镜像漏洞、不安全的构建指令
│  Dockerfile │──────────────────────────────────► 使用有漏洞的基础镜像
└──────┬──────┘
       │
       ▼
┌─────────────┐    构建工具篡改、缓存投毒
│  CI 构建     │──────────────────────────────────► 恶意构建插件
└──────┬──────┘
       │
       ▼
┌─────────────┐    仓库劫持、未授权推送、镜像替换
│  镜像仓库    │──────────────────────────────────► 攻击者替换镜像标签
└──────┬──────┘
       │
       ▼
┌─────────────┐    中间人攻击、镜像篡改
│  镜像拉取    │──────────────────────────────────► 网络劫持替换镜像层
└──────┬──────┘
       │
       ▼
┌─────────────┐    运行时漏洞利用、容器逃逸
│  容器运行    │──────────────────────────────────► 漏洞利用获得宿主机权限
└─────────────┘
```

**镜像供应链攻击案例**：

| 事件 | 时间 | 攻击方式 | 影响 |
|------|------|---------|------|
| **Docker Hub 加密货币挖矿** | 持续 | 上传合法应用的恶意镜像 | 用户下载运行后挖矿 |
| **Codecov Bash Uploader** | 2021 | 修改 CI 脚本泄露环境变量 | 数千个项目的 Secret 泄露 |
| **SolarWinds** | 2020 | 在构建过程中植入后门 | 政府和企业网络被渗透 |
| **PHP Git 服务器入侵** | 2021 | 在源码中植入后门 | 如果发布会影响全球 PHP |

### 5.1.2 基础镜像选择与安全

基础镜像的选择直接影响容器的攻击面：

| 镜像类型 | 大小 | Shell | 包管理器 | 典型漏洞数 | 适用场景 |
|---------|------|-------|---------|-----------|---------|
| `ubuntu:22.04` | ~80MB | bash | apt | 10-50 | 开发、兼容性要求高 |
| `debian:12` | ~120MB | bash | apt | 10-50 | 需要完整 glibc 支持 |
| `alpine:3.19` | ~7MB | ash | apk | 1-5 | 通用生产环境 |
| `distroless` | ~20MB | **无** | **无** | 0-2 | **最佳实践** |
| `scratch` | 0MB | **无** | **无** | 0 | 静态编译语言（Go/Rust） |

**Alpine 的安全考量**：

Alpine 使用 musl libc 替代 glibc，这带来安全优势：
1. **更小的攻击面**：默认安装包更少
2. **内存安全**：musl 的某些实现比 glibc 更安全
3. **但也存在风险**：musl 与 glibc 的行为差异可能导致安全问题（如 DNS 解析差异）

```dockerfile
# ✅ 生产环境推荐：Distroless
# 优点：无 shell、无包管理器、最小攻击面
FROM golang:1.21 AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -o server

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=builder /app/server /server
EXPOSE 8080
USER nonroot:nonroot
ENTRYPOINT ["/server"]

# ⚠️ 注意：Distroless 镜像没有 shell，调试困难
# 解决方案：使用 debug 标签进行调试
# FROM gcr.io/distroless/static-debian12:debug-nonroot
# debug 标签包含 busybox，便于 exec 进入调试
```

**镜像层安全**：

```dockerfile
# ❌ 每层都增加攻击面
FROM ubuntu:22.04
RUN apt-get update && apt-get install -y curl wget vim gcc make
RUN apt-get install -y python3 python3-pip
RUN pip install requests flask
COPY . /app
RUN make build
# 最终镜像包含：curl、wget、vim、gcc、make、python3、pip...

# ✅ 多阶段构建，只保留必要文件
FROM golang:1.21 AS builder
RUN apt-get update && apt-get install -y gcc  # 只在构建阶段需要
COPY . /app
RUN make build

FROM gcr.io/distroless/static-debian12
COPY --from=builder /app/server /server
ENTRYPOINT ["/server"]
# 最终镜像只有一个二进制文件
```

### 5.1.3 镜像漏洞扫描

#### Trivy 深度使用

Trivy 是 Aqua Security 开源的漏洞扫描器，支持镜像、文件系统、Git 仓库和 Kubernetes：

```bash
# 基本扫描
trivy image nginx:alpine

# 只显示 HIGH 和 CRITICAL
trivy image --severity HIGH,CRITICAL nginx:alpine

# 忽略未修复的漏洞
trivy image --ignore-unfixed nginx:alpine

# 输出为 SARIF（用于 GitHub/GitLab Security Dashboard）
trivy image -f sarif -o report.sarif nginx:alpine

# 输出为 JSON（便于程序处理）
trivy image -f json -o report.json nginx:alpine

# 扫描结果包含：
# - CVE ID
# - 严重等级（UNKNOWN/LOW/MEDIUM/HIGH/CRITICAL）
# - 漏洞标题
# - 受影响的包
# - 已修复版本
# - 参考链接

# 扫描文件系统（用于 CI 中扫描源码依赖）
trivy filesystem --scanners vuln,secret,config /path/to/project

# 扫描 Kubernetes 集群
trivy k8s cluster --report summary

# 使用忽略文件（.trivyignore）
# .trivyignore 内容：
# # 忽略特定 CVE
# CVE-2023-1234
# # 忽略某个包的特定 CVE
# CVE-2023-5678 exp:2024-06-01
```

#### Grype 与 Syft

Grype（漏洞扫描）和 Syft（SBOM 生成）来自 Anchore：

```bash
# 安装
curl -sSfL https://raw.githubusercontent.com/anchore/grype/main/install.sh | sh -s -- -b /usr/local/bin
curl -sSfL https://raw.githubusercontent.com/anchore/syft/main/install.sh | sh -s -- -b /usr/local/bin

# 扫描镜像
grype nginx:alpine

# 从 SBOM 扫描（不直接扫描镜像）
syft nginx:alpine -o spdx-json | grype

# 忽略特定漏洞（.grype.yaml）
ignore:
  - vulnerability: CVE-2023-1234
    reason: "Not exploitable in our configuration"
```

| 特性 | Trivy | Grype | Snyk | Clair |
|------|-------|-------|------|-------|
| 开源 | ✅ | ✅ | 部分 | ✅ |
| 镜像扫描 | ✅ | ✅ | ✅ | ✅ |
| 文件系统扫描 | ✅ | ✅ | ✅ | ❌ |
| K8s 扫描 | ✅ | ❌ | ✅ | ❌ |

| 漏洞数据库 | NVD + 厂商 | NVD + 厂商 | 自有 | NVD |
| CI 集成 | 优秀 | 良好 | 优秀 | 一般 |
| 速度 | 快 | 快 | 快 | 中等 |

### 5.1.4 镜像签名与验证

#### Cosign 签名

Cosign 是 Sigstore 项目的一部分，提供无密钥（keyless）签名方案：

```bash
# 安装 cosign
curl -O -L https://github.com/sigstore/cosign/releases/latest/download/cosign-linux-amd64
chmod +x cosign-linux-amd64 && sudo mv cosign-linux-amd64 /usr/local/bin/cosign

# 方式 1：使用密钥对签名
cosign generate-key-pair
# 输入密码保护私钥
# 生成：cosign.key（私钥）和 cosign.pub（公钥）

# 签名镜像
cosign sign --key cosign.key myregistry.example.com/myapp:v1.0.0

# 验证签名
cosign verify --key cosign.pub myregistry.example.com/myapp:v1.0.0

# 方式 2：无密钥签名（Keyless，推荐）
# 使用 Sigstore 的 Fulcio（CA）和 Rekor（透明度日志）
cosign sign myregistry.example.com/myapp:v1.0.0
# 浏览器弹出 OIDC 认证（GitHub/Google/Microsoft）
# 签名后自动上传到 Rekor 日志

# 验证无密钥签名
cosign verify myregistry.example.com/myapp:v1.0.0 \
  --certificate-identity=alice@example.com \
  --certificate-oidc-issuer=https://accounts.google.com
```

**Sigstore 无密钥签名流程**：

```
用户
  │
  │ 1. 发起签名请求
  ▼
cosign
  │
  │ 2. 生成临时密钥对（EPHEMERAL）
  │ 3. 请求 OIDC 认证
  ▼
IdP (GitHub/Google)
  │
  │ 4. 用户完成认证
  │ 5. 返回 ID Token
  ▼
cosign
  │
  │ 6. 用 ID Token 向 Fulcio 申请证书
  ▼
Fulcio (CA)
  │
  │ 7. 签发短期证书（绑定 OIDC 身份）
  │ 8. 证书记录到 Rekor
  ▼
cosign
  │
  │ 9. 使用证书签名镜像
  │ 10. 签名记录到 Rekor
  ▼
镜像仓库 + Rekor 日志
```

**在 Kubernetes 中强制执行镜像签名**：

```yaml
# 使用 Kyverno 验证镜像签名
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  background: false
  rules:
  - name: verify-cosign-signature
    match:
      resources:
        kinds:
        - Pod
    verifyImages:
    - imageReferences:
      - "myregistry.example.com/*"
      attestors:
      - entries:
        - keys:
            publicKeys: |
              -----BEGIN PUBLIC KEY-----
              MFkwEwYHKoZIzj0CAQYIKoZIzj0DAQcDQgAE...
              -----END PUBLIC KEY-----
```

## 5.2 SBOM（软件物料清单）

### 5.2.1 为什么需要 SBOM

SBOM（Software Bill of Materials）是软件组件的完整清单。在供应链攻击日益频繁的背景下，SBOM 成为安全合规的核心要求：

**SBOM 的价值**：
1. **漏洞响应**：当某个组件（如 Log4j）出现漏洞时，快速定位受影响的产品
2. **许可证合规**：确保使用的开源组件符合许可证要求
3. **供应链透明**：了解软件的所有依赖，防止嵌套的恶意组件
4. **监管合规**：美国 FDA、欧盟 Cyber Resilience Act 等要求 SBOM

### 5.2.2 SBOM 标准

| 标准 | 组织 | 格式 | 特点 |
|------|------|------|------|
| **SPDX** | Linux Foundation | JSON/RDF/XML/Tag-Value | 最成熟，ISO 标准 |
| **CycloneDX** | OWASP | JSON/XML | 安全导向，漏洞信息丰富 |
| **SWID** | ISO/IEC 19770-2 | XML | 软件识别标签 |

**SPDX 核心字段**：

```json
{
  "spdxVersion": "SPDX-2.3",
  "SPDXID": "SPDXRef-DOCUMENT",
  "name": "myapp",
  "packages": [
    {
      "SPDXID": "SPDXRef-Package-nginx",
      "name": "nginx",
      "versionInfo": "1.25.3",
      "downloadLocation": "https://nginx.org/download/nginx-1.25.3.tar.gz",
      "filesAnalyzed": false,
      "supplier": "Person: Igor Sysoev",
      "checksums": [
        {
          "algorithm": "SHA256",
          "checksumValue": "..."
        }
      ],
      "licenseConcluded": "BSD-2-Clause"
    }
  ],
  "relationships": [
    {
      "spdxElementId": "SPDXRef-DOCUMENT",
      "relatedSpdxElement": "SPDXRef-Package-nginx",
      "relationshipType": "DESCRIBES"
    }
  ]
}
```

### 5.2.3 生成和管理 SBOM

```bash
# 使用 Syft 生成 SBOM
syft myregistry.example.com/myapp:v1.0.0 -o spdx-json > sbom.spdx.json
syft myregistry.example.com/myapp:v1.0.0 -o cyclonedx-json > sbom.cdx.json

# 生成后附加到镜像仓库（Harbor 2.5+ 支持）
cosign attach sbom --sbom sbom.spdx.json myregistry.example.com/myapp:v1.0.0

# 下载 SBOM
cosign download sbom myregistry.example.com/myapp:v1.0.0

# 从 SBOM 扫描漏洞
grype sbom:sbom.spdx.json

# 在 CI/CD 中集成
# .github/workflows/sbom.yml
# - name: Generate SBOM
#   uses: anchore/sbom-action@v0
#   with:
#     image: myregistry.example.com/myapp:${{ github.sha }}
#     format: spdx-json
#     output-file: sbom.json
```

## 5.3 Secret 安全管理

### 5.3.1 Secret 生命周期与风险

```
Secret 创建
    │
    ├── 存储在 etcd
    │   └── 风险：如果未加密，任何能访问 etcd 的人都能读取
    │
    ├── API Server 处理请求
    │   └── 风险：审计日志可能记录 Secret 内容（取决于审计策略级别）
    │
    ├── 传输到 kubelet
    │   └── 风险：中间人攻击（但 TLS 防止了这一点）
    │
    ├── kubelet 存储在节点
    │   └── 风险：节点被攻破后，Secret 文件可被读取（tmpfs，但 root 可以访问）
    │
    ├── 挂载到容器
    │   └── 风险：任何能 exec 进容器的人都能读取
    │
    └── 应用使用
        └── 风险：应用可能将 Secret 泄露到日志、错误信息、堆栈跟踪
```

### 5.3.2 Secret 注入方式对比

| 方式 | 安全性 | 动态更新 | 使用场景 | 风险 |
|------|--------|---------|---------|------|
| **环境变量** | 低 | 否（需重建 Pod） | 简单配置 | 子进程继承、可能泄露到日志 |
| **Volume 挂载** | 中 | 是（kubelet 同步） | 推荐方式 | 文件权限、exec 访问 |
| **Projected Volume** | 中 | 是 | 组合多个源 | 同上 |
| **External Secrets** | 高 | 是（自动同步） | 生产推荐 | 依赖外部服务可用性 |
| **Secret Store CSI** | 高 | 是 | 不将 Secret 存入 K8s | 复杂度高 |

**环境变量 vs Volume 挂载的安全对比**：

```yaml
# ❌ 环境变量方式：Secret 会暴露给所有子进程
spec:
  containers:
  - name: app
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: db-secret
          key: password
# 风险 1：子进程继承环境变量
# 风险 2：/proc/<pid>/environ 可读
# 风险 3：应用崩溃时堆栈跟踪可能包含环境变量
# 风险 4：某些日志框架默认打印所有环境变量

# ✅ Volume 挂载方式：更安全的文件访问
spec:
  containers:
  - name: app
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secrets
    secret:
      secretName: db-secret
      defaultMode: 0400   # 只读权限
# 优点 1：文件权限可控
# 优点 2：不通过环境变量传播
# 优点 3：应用可以更好地控制读取时机
```

### 5.3.3 Sealed Secrets

Sealed Secrets 允许将加密的 Secret 安全地存储在 Git 中：

```bash
# 1. 安装控制器
kubectl apply -f https://github.com/bitnami-labs/sealed-secrets/releases/download/v0.24.0/controller.yaml

# 2. 安装 kubeseal 客户端
brew install kubeseal  # macOS
# 或下载二进制

# 3. 将普通 Secret 加密为 SealedSecret
cat <<EOF | kubectl create -f - --dry-run=client -o yaml | kubeseal -o yaml > mysealedsecret.yaml
apiVersion: v1
kind: Secret
metadata:
  name: my-secret
  namespace: default
stringData:
  password: supersecret123
EOF

# 4. mysealedsecret.yaml 可以安全地提交到 Git
cat mysealedsecret.yaml
# apiVersion: bitnami.com/v1alpha1
# kind: SealedSecret
# metadata:
#   name: my-secret
# spec:
#   encryptedData:
#     password: AgByL1+...

# 5. 部署时，控制器自动解密为普通 Secret
kubectl apply -f mysealedsecret.yaml
kubectl get secret my-secret -o jsonpath='{.data.password}' | base64 -d
# supersecret123
```

**Sealed Secrets 安全机制**：
- 使用集群中的公钥加密（kubeseal 获取集群公钥）
- 只有集群中的 sealed-secrets-controller 能解密（持有私钥）
- 绑定了命名空间：在 `default` 命名空间加密的 Secret 不能在其他命名空间解密
- 支持 scope：strict（绑定命名空间+名称）、namespace-wide、cluster-wide

### 5.3.4 External Secrets Operator

External Secrets Operator（ESO）将外部 Secret 管理器（Vault、AWS Secrets Manager、Azure Key Vault 等）与 Kubernetes 集成：

```yaml
# 1. 安装 ESO
helm repo add external-secrets https://charts.external-secrets.io
helm install external-secrets external-secrets/external-secrets \
  -n external-secrets --create-namespace

# 2. 创建 SecretStore（连接 Vault）
apiVersion: external-secrets.io/v1beta1
kind: SecretStore
metadata:
  name: vault-backend
  namespace: production
spec:
  provider:
    vault:
      server: "https://vault.internal:8200"
      path: "secret"
      version: "v2"
      auth:
        kubernetes:
          mountPath: "kubernetes"
          role: "external-secrets"
          serviceAccountRef:
            name: external-secrets

# 3. 创建 ExternalSecret（自动同步）
apiVersion: external-secrets.io/v1beta1
kind: ExternalSecret
metadata:
  name: db-credentials
  namespace: production
spec:
  refreshInterval: "1h"    # 每小时同步一次
  secretStoreRef:
    name: vault-backend
    kind: SecretStore
  target:
    name: db-secret        # 创建的 K8s Secret 名称
    creationPolicy: Owner
    deletionPolicy: Retain
  data:
  - secretKey: username    # K8s Secret 中的键
    remoteRef:
      key: secret/data/db   # Vault 中的路径
      property: username    # Vault 中的字段
  - secretKey: password
    remoteRef:
      key: secret/data/db
      property: password
```

**ESO 的安全优势**：
1. Secret 不长期存储在 etcd 中（虽然会创建 K8s Secret，但可以配置短 TTL）
2. 支持自动轮换：外部 Secret 更新后自动同步
3. 集中管理：所有 Secret 在一个地方管理
4. 审计追踪：外部管理器通常有完整的审计日志

### 5.3.5 SOPS（Secrets OPerationS）

SOPS 是 Mozilla 开源的加密文件编辑器，支持 AWS KMS、GCP KMS、Azure Key Vault、PGP 和 Age：

```bash
# 安装 SOPS
brew install sops

# 使用 Age 加密（简单、开源）
age-keygen -o key.txt

# 加密 YAML 文件
sops --encrypt --age $(cat key.txt | grep "public key" | awk '{print $3}') \
  secret.yaml > secret.enc.yaml

# secret.enc.yaml 可以安全地提交到 Git
# 内容示例：
# apiVersion: v1
# kind: Secret
# metadata:
#     name: my-secret
# stringData:
#     password: ENC[AES256_GCM,data:...,iv:...,type:str]

# 解密（CI/CD 中使用）
sops --decrypt secret.enc.yaml | kubectl apply -f -

# 在 Flux CD 中集成
# 配置 SOPS 和 Age 密钥，Flux 会自动解密
```

## 5.4 配置安全管理

### 5.4.1 ConfigMap 安全

ConfigMap 用于存储非敏感配置，但仍需注意：

```bash
# ❌ 常见错误：在 ConfigMap 中存放敏感信息
kubectl create configmap app-config \
  --from-literal=db_password=secret123 \
  --from-literal=api_key=sk-123456

# ✅ 正确做法：敏感信息用 Secret，ConfigMap 只放非敏感配置
kubectl create configmap app-config \
  --from-literal=log_level=info \
  --from-literal=max_connections=100

# 检查现有 ConfigMap 是否泄露敏感信息
kubectl get configmaps -A -o yaml | grep -i -E "password|secret|token|key|credential"
```

```
Git 仓库

    │   └── 可以明文存储
```

### 5.5.2 GitHub Actions 安全流水线

```yaml
# .github/workflows/secure-pipeline.yml
name: Secure Build and Deploy

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ===== 阶段 1：代码安全扫描 =====
  sast:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    
    # SAST 扫描（SonarQube / CodeQL）
    - name: Initialize CodeQL
      uses: github/codeql-action/init@v2
      with:
        languages: go,python
    
    - name: Autobuild
      uses: github/codeql-action/autobuild@v2
    
    - name: Perform CodeQL Analysis
      uses: github/codeql-action/analyze@v2
    
    # 依赖漏洞扫描
    - name: Run Trivy vulnerability scanner in fs mode
      uses: aquasecurity/trivy-action@master
      with:
        scan-type: 'fs'
        scan-ref: '.'
        severity: 'CRITICAL,HIGH'
        exit-code: '1'

  # ===== 阶段 2：构建与镜像扫描 =====
  build:
    needs: sast
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
      id-token: write   # 用于 cosign keyless 签名
    steps:
    - uses: actions/checkout@v4
    
    - name: Set up Docker Buildx
      uses: docker/setup-buildx-action@v3
    
    - name: Log in to Container Registry
      uses: docker/login-action@v3
      with:
        registry: ${{ env.REGISTRY }}
        username: ${{ github.actor }}
        password: ${{ secrets.GITHUB_TOKEN }}
    
    - name: Build and push
      uses: docker/build-push-action@v5
      with:
        context: .
        push: true
        tags: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
        cache-from: type=gha
        cache-to: type=gha,mode=max
    
    # 镜像漏洞扫描（阻断 HIGH/CRITICAL）
    - name: Scan image
      uses: aquasecurity/trivy-action@master
      with:
        image-ref: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
        format: 'sarif'
        output: 'trivy-results.sarif'
        severity: 'CRITICAL,HIGH'
        exit-code: '1'    # 发现高危漏洞则阻断
    
    - name: Upload scan results
      uses: github/codeql-action/upload-sarif@v2
      with:
        sarif_file: 'trivy-results.sarif'
    
    # 生成 SBOM
    - name: Generate SBOM
      uses: anchore/sbom-action@v0
      with:
        image: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
        format: spdx-json
        output-file: sbom.spdx.json
    
    # 镜像签名（keyless）
    - name: Sign image
      uses: sigstore/cosign-installer@v3
    - run: |
        cosign sign --yes \
          ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}
    
    # 验证签名
    - name: Verify signature
      run: |
        cosign verify \
          --certificate-identity=${{ github.server_url }}/${{ github.repository }}/.github/workflows/secure-pipeline.yml@refs/heads/main \
          --certificate-oidc-issuer=https://token.actions.githubusercontent.com \
          ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

  # ===== 阶段 3：部署 =====
  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
    - uses: actions/checkout@v4
    
    - name: Deploy to staging
      run: |
        kubectl apply -f k8s/staging/
        kubectl rollout status deployment/app -n staging
    
    - name: Run smoke tests
      run: |
        curl -f http://app.staging.internal/health
    
    - name: Deploy to production
      if: success()
      run: |
        kubectl apply -f k8s/production/
        kubectl rollout status deployment/app -n production
```

### 实验 5.5：Secret 注入方式对比（20 分钟）

```bash
# 步骤 1：创建 Secret
kubectl create secret generic test-secret \
  --from-literal=password=secret123 \
  --from-literal=apikey=sk-abc

# 步骤 2：创建使用环境变量的 Pod
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: env-pod
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ['sh', '-c', 'env | grep PASSWORD; sleep 3600']
    env:
    - name: DB_PASSWORD
      valueFrom:
        secretKeyRef:
          name: test-secret
          key: password
EOF

kubectl logs env-pod
# DB_PASSWORD=secret123

# 步骤 3：创建使用 Volume 的 Pod
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: vol-pod
spec:
  containers:
  - name: app
    image: busybox:1.36
    command: ['sh', '-c', 'cat /etc/secrets/password; sleep 3600']
    volumeMounts:
    - name: secrets
      mountPath: /etc/secrets
      readOnly: true
  volumes:
  - name: secrets
    secret:
      secretName: test-secret
      defaultMode: 0400
EOF

kubectl logs vol-pod
# secret123

# 步骤 4：对比 /proc 中的暴露
echo "环境变量方式："
kubectl exec env-pod -- cat /proc/self/environ | tr '\0' '\n' | grep PASSWORD

echo "Volume 方式："
kubectl exec vol-pod -- cat /proc/self/environ | tr '\0' '\n' | grep PASSWORD
# 没有密码

# 清理
kubectl delete pod env-pod vol-pod
kubectl delete secret test-secret
```

## 5.10 本章小结

| 主题 | 核心措施 | 推荐工具 |
|------|---------|---------|
| **镜像安全** | 最小化基础镜像、多阶段构建 | distroless, BuildKit |
| **漏洞扫描** | 每次构建扫描、阻断高危漏洞 | Trivy, Grype |
| **镜像签名** | 所有生产镜像必须签名 | Cosign, Notary |
| **SBOM** | 生成并随镜像分发 | Syft, SPDX |
| **Secret 管理** | 不在 Git 中存储明文 Secret | Sealed Secrets, ESO, SOPS |
| **配置安全** | ConfigMap 不存放敏感信息 | 审计扫描 |
| **CI/CD 安全** | 安全左移、自动阻断 | GitHub Actions, GitLab CI |
| **供应链防御** | 多层防御、依赖锁定 | Dependabot, Renovate |

**关键安全原则**：
1. **最小化镜像**：无 shell、无包管理器、只包含必要文件
2. **签名一切**：所有生产镜像必须签名并验证
3. **扫描一切**：代码依赖、镜像、运行时持续扫描
4. **Secret 不落地**：不在 Git 中存储明文 Secret，使用加密或外部管理器
5. **自动化阻断**：高危漏洞自动阻断构建和部署

**安全检查清单**：
- [ ] 使用最小化基础镜像（distroless 或 alpine）
- [ ] 多阶段构建，不遗留构建工具
- [ ] CI/CD 中集成镜像漏洞扫描
- [ ] 所有生产镜像签名
- [ ] 生成并存储 SBOM
- [ ] Secret 使用 Volume 挂载而非环境变量
- [ ] 不在 Git 中存储明文 Secret
- [ ] 部署时验证镜像签名
- [ ] 锁定依赖版本
- [ ] 订阅 CVE 告警
