## 8.1 企业级云原生安全架构


### 8.1.2 安全控制矩阵

| 层级 | 预防（Prevent） | 检测（Detect） | 响应（Respond） |
|------|---------------|---------------|----------------|
| **供应链** | 镜像签名、依赖扫描、SLSA | SBOM 审计、构建日志 | 阻断部署、回滚 |
| **基础设施** | CIS 加固、加密、最小化 | kube-bench、节点监控 | 自动修复、隔离节点 |
| **工作负载** | PSS、最小权限、不可变 | 运行时扫描、漂移检测 | 隔离 Pod、重启 |
| **网络** | NetworkPolicy、Cilium L7 | Hubble 流量分析 | 阻断连接、限速 |
| **运行时** | Seccomp、Capabilities | Falco、Tetragon | 终止进程、告警 |
| **集群** | RBAC、准入控制 | 审计日志、异常 API 调用 | 撤销权限、Token 轮换 |
| **边界** | WAF、API 网关 | 流量分析、Bot 检测 | IP 阻断、Rate Limit |

### 8.2.3 Cilium ClusterMesh 实现多集群安全

ClusterMesh 允许跨集群的服务发现和网络策略：

```bash
# 集群 1：启用 ClusterMesh
helm upgrade cilium cilium/cilium -n kube-system \
  --set cluster.name=cluster1 \
  --set cluster.id=1 \
  --set clustermesh.useAPIServer=true

# 集群 2：启用 ClusterMesh
helm upgrade cilium cilium/cilium -n kube-system \
  --set cluster.name=cluster2 \
  --set cluster.id=2 \
  --set clustermesh.useAPIServer=true

# 连接集群
cilium clustermesh connect \
  --context cluster1 \
  --destination-context cluster2

# 创建全局网络策略
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: global-backend-policy
  namespace: production
spec:
  endpointSelector:
    matchLabels:
      app: backend
  ingress:
  - fromEndpoints:
    - matchLabels:
        app: frontend
        # 只允许来自 cluster1 的 frontend
        io.cilium.k8s.policy.cluster: cluster1
    toPorts:
    - ports:
      - port: "8080"
        protocol: TCP
```

**多集群 NetworkPolicy 注意事项**：
1. 跨集群 Pod IP 需要在 CIDR 范围内可达
2. 使用标签标识集群来源（`io.cilium.k8s.policy.cluster`）
3. 全局策略需要所有集群的 Cilium 版本一致

## 8.3 零信任架构实现

### 8.3.1 零信任核心原则

1. **永不信任，始终验证**：不假设任何网络位置是可信的
2. **假设网络已被攻破**：防御措施不依赖网络边界
3. **最小权限访问**：每个请求都需要明确授权
4. **持续验证和监控**：身份和权限持续检查

### 8.3.2 SPIFFE/SPIRE 工作负载身份

SPIFFE（Secure Production Identity Framework For Everyone）为工作负载提供统一的身份标准：

```
SPIFFE ID 格式：spiffe://信任域/工作负载标识

示例：
- spiffe://company.com/k8s/production/payment-service
- spiffe://company.com/k8s/staging/frontend-service

SPIRE 组件：
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│ SPIRE Server │────►│ SPIRE Agent  │────►│ 工作负载      │
│ (签发 SVID)  │     │ (节点级代理)  │     │ (使用 SVID)  │
└──────────────┘     └──────────────┘     └──────────────┘
       │                     │                    │
       │ 签发 X.509-SVID     │ 提供 workload API  │ 获取身份凭证
       │                     │                    │
       └─────────────────────┴────────────────────┘
```

**SVID（SPIFFE Verifiable Identity Document）**：
- X.509 证书，包含 SPIFFE ID
- 短期有效（默认 1 小时），自动轮换
- 工作负载通过 Unix Domain Socket 获取

```bash
# 工作负载获取 SVID
curl --unix-socket /tmp/spire-agent/public/api.sock \
  "http://localhost/v1/workload/jwt?audience=api-backend"

# 返回 JWT-SVID，包含 SPIFFE ID
# 用于服务间认证
```

### 8.3.3 Istio 服务网格安全

Istio 提供自动化的 mTLS 和授权策略：

```yaml
# 强制 mTLS（全局）
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: istio-system
spec:
  mtls:
    mode: STRICT        # STRICT / PERMISSIVE / DISABLE
```

```yaml
# 细粒度授权策略
apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: backend-policy
  namespace: production
spec:
  selector:
    matchLabels:
      app: backend
  action: ALLOW
  rules:
  # 只允许来自 frontend ServiceAccount 的请求
  - from:
    - source:
        principals: ["cluster.local/ns/production/sa/frontend-sa"]
    to:
    - operation:
        methods: ["GET", "POST"]
        paths: ["/api/v1/*"]
  # 只允许带有特定 header 的请求
    when:
    - key: request.headers[x-request-id]
      values: ["*"]
  # 拒绝所有其他请求（默认拒绝）
```

**Istio 安全特性**：

| 特性 | 说明 | 配置 |
|------|------|------|
| **自动 mTLS** | 自动加密服务间通信 | PeerAuthentication STRICT |
| **授权策略** | 基于身份和属性的访问控制 | AuthorizationPolicy |
| **JWT 验证** | 验证请求中的 JWT Token | RequestAuthentication |
| **出口网关** | 控制对外部服务的访问 | Egress Gateway |
| **入口网关** | 统一的入口安全策略 | Ingress Gateway |

### 8.3.4 零信任网络实现路径

```
阶段 1：身份基础
├── 部署 SPIRE/SPIFFE
├── 为所有工作负载发放身份
└── 更新应用支持 SVID

阶段 2：传输加密
├── 部署 Istio/Linkerd
├── 启用自动 mTLS
└── 验证加密状态

阶段 3：访问控制
├── 定义服务间授权策略
├── 实施最小权限
└── 拒绝默认访问

阶段 4：持续验证
├── 部署 Falco 运行时监控
├── 部署 Cilium 网络策略
└── 集成 SIEM 告警

阶段 5：自动化响应
├── 自动隔离异常工作负载
├── 自动轮换泄露凭证
└── 自适应策略调整
```

## 8.5 安全治理与合规

### 8.5.1 策略即代码（Policy as Code）

策略即代码将安全策略以代码形式管理，实现版本控制、自动化测试和持续部署：

**OPA Gatekeeper 深度使用**：

```yaml
# 约束模板：禁止特定镜像仓库
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8sallowedrepos
spec:
  crd:
    spec:
      names:
        kind: K8sAllowedRepos
      validation:
        openAPIV3Schema:
          type: object
          properties:
            repos:
              type: array
              items:
                type: string
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8sallowedrepos
      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        satisfied := [good | repo = input.parameters.repos[_] ; good = startswith(container.image, repo)]
        not any(satisfied)
        msg := sprintf("Container %v uses forbidden image %v", [container.name, container.image])
      }
      violation[{"msg": msg}] {
        container := input.review.object.spec.initContainers[_]
        satisfied := [good | repo = input.parameters.repos[_] ; good = startswith(container.image, repo)]
        not any(satisfied)
        msg := sprintf("Init container %v uses forbidden image %v", [container.name, container.image])
      }

# 应用约束
---
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sAllowedRepos
metadata:
  name: allowed-repos
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
    namespaces: ["production"]
  parameters:
    repos:
    - "myregistry.io/"
    - "gcr.io/company/"
    - "ghcr.io/company/"
```

**Kyverno vs OPA Gatekeeper 对比**：

| 特性 | Kyverno | OPA Gatekeeper |
|------|---------|---------------|
| 学习曲线 | 低（YAML） | 高（Rego） |
| 策略语言 | YAML | Rego |
| 验证 | 内置 | 需额外工具 |
| 变异策略 | 强大 | 有限 |
| 生成策略 | 支持 | 不支持 |
| 社区 | 活跃 | 活跃 |
| 性能 | 高 | 高 |
| 适用场景 | K8s 原生策略 | 复杂逻辑、跨平台 |

## 8.6 密钥管理与证书架构

### 8.6.1 HashiCorp Vault 集成

```
┌─────────────────────────────────────────────────────────┐
│                    Vault 架构                            │
│                                                         │
```

### 实验 8.1：Istio mTLS 配置与验证（30 分钟）

```bash
# 步骤 1：安装 Istio
istioctl install --set profile=demo -y

# 步骤 2：启用自动注入
kubectl label namespace default istio-injection=enabled

# 步骤 3：部署示例应用
kubectl apply -f https://raw.githubusercontent.com/istio/istio/release-1.20/samples/bookinfo/platform/kube/bookinfo.yaml

# 步骤 5：创建 PeerAuthentication 强制 mTLS
kubectl apply -f - <<EOF
apiVersion: security.istio.io/v1beta1
kind: PeerAuthentication
metadata:
  name: default
  namespace: default
spec:
  mtls:
    mode: STRICT
EOF

# 步骤 6：验证 mTLS 已启用
istioctl authn tls-check $(kubectl get pod -l app=productpage -o jsonpath='{.items[0].metadata.name}').default

apiVersion: security.istio.io/v1beta1
kind: AuthorizationPolicy
metadata:
  name: productpage-policy
  namespace: default
spec:
  selector:
    matchLabels:
      app: productpage
  action: ALLOW
  rules:
  - from:
    - source:
        principals: ["cluster.local/ns/default/sa/bookinfo-gateway-istio"]
EOF

# 清理

kubectl delete namespace default --wait=false
kubectl create namespace default
```

### 实验 8.3：OPA Gatekeeper 策略部署（25 分钟）

```bash
# 步骤 1：安装 OPA Gatekeeper
helm repo add gatekeeper https://open-policy-agent.github.io/gatekeeper/charts
helm install gatekeeper gatekeeper/gatekeeper \
  -n gatekeeper-system --create-namespace

# 步骤 2：创建约束模板（禁止 latest 标签）
cat <<EOF | kubectl apply -f -
apiVersion: templates.gatekeeper.sh/v1
kind: ConstraintTemplate
metadata:
  name: k8srequiredtags
spec:
  crd:
    spec:
      names:
        kind: K8sRequiredTags
  targets:
  - target: admission.k8s.gatekeeper.sh
    rego: |
      package k8srequiredtags
      violation[{"msg": msg}] {
        container := input.review.object.spec.containers[_]
        endswith(container.image, ":latest")
        msg := sprintf("Container %v uses forbidden 'latest' tag", [container.name])
      }
EOF

# 步骤 3：应用约束
kubectl apply -f - <<EOF
apiVersion: constraints.gatekeeper.sh/v1beta1
kind: K8sRequiredTags
metadata:
  name: no-latest-tag
spec:
  match:
    kinds:
    - apiGroups: [""]
      kinds: ["Pod"]
EOF

# 步骤 4：测试 - 使用 latest 标签应该失败
kubectl run test-latest --image=nginx:latest 2>&1 | head -5

# 步骤 5：测试 - 使用具体版本应该成功
kubectl run test-version --image=nginx:1.25-alpine
kubectl get pod test-version

# 清理
kubectl delete pod test-latest test-version 2>/dev/null
kubectl delete constraints.gatekeeper.sh no-latest-tag
kubectl delete constrainttemplates k8srequiredtags
helm uninstall gatekeeper -n gatekeeper-system
kubectl delete namespace gatekeeper-system
```

### 8.10.3 生产环境安全基线检查脚本

```bash
#!/bin/bash
# production-security-baseline.sh
# 生产环境 K8s 安全基线检查

PASS=0
FAIL=0

function check() {
  if [ $1 -eq 0 ]; then
    echo "  [PASS] $2"
    ((PASS++))
  else
    echo "  [FAIL] $2"
    ((FAIL++))
  fi
}

echo "=== K8s 生产安全基线检查 ==="

# API Server
echo "--- API Server ---"
check $(kubectl get pod -n kube-system -l component=kube-apiserver -o yaml | grep -q "anonymous-auth=false"; echo $?) "API Server 匿名认证已禁用"

# etcd
echo "--- etcd ---"
check $(kubectl get pod -n kube-system -l component=etcd -o yaml | grep -q "client-cert-auth=true"; echo $?) "etcd 客户端证书认证已启用"

# kubelet
echo "--- kubelet ---"
check $(kubectl get nodes -o json | jq -e '[.items[].status.conditions[] | select(.reason=="KubeletReady")] | length > 0' > /dev/null; echo $?) "所有节点 kubelet 健康"

# Pod Security
echo "--- Pod Security ---"
PRIV_COUNT=$(kubectl get pods -A -o json | jq '[.items[] | select(.spec.containers[]?.securityContext?.privileged==true)] | length')
check $([ "$PRIV_COUNT" -eq 0 ]; echo $?) "无特权容器 (发现 $PRIV_COUNT 个)"

# RBAC
echo "--- RBAC ---"
check $(kubectl get clusterrolebinding -o json | jq -e '[.items[] | select(.roleRef.name=="cluster-admin") | .subjects[]? | select(.kind=="ServiceAccount" and .namespace=="default")] | length == 0' > /dev/null; echo $?) "default ServiceAccount 未绑定 cluster-admin"

# NetworkPolicy
echo "--- NetworkPolicy ---"
NS_WITHOUT_NP=$(kubectl get ns -o json | jq -r '.items[] | select(.metadata.name!="kube-system" and .metadata.name!="kube-public" and .metadata.name!="kube-node-lease") | .metadata.name' | while read ns; do
  count=$(kubectl get networkpolicy -n "$ns" --no-headers 2>/dev/null | wc -l)
  [ "$count" -eq 0 ] && echo "$ns"
done | wc -l)
check $([ "$NS_WITHOUT_NP" -eq 0 ]; echo $?) "所有命名空间都有 NetworkPolicy"

# 容器镜像
echo "--- 容器镜像 ---"
LATEST_COUNT=$(kubectl get pods -A -o json | jq -r '[.items[].spec.containers[].image | select(contains(":latest"))] | length')
check $([ "$LATEST_COUNT" -eq 0 ]; echo $?) "无 latest 标签镜像"

echo ""
echo "=== 检查结果: $PASS 通过, $FAIL 失败 ==="
[ "$FAIL" -eq 0 ] && echo "基线检查全部通过!" || echo "请修复上述失败项"
```

## 8.11 本章小结

| 主题 | 核心设计 | 关键工具 |
|------|---------|---------|
| **分层架构** | 七层纵深防御，每层独立 | 多层工具栈 |
| **多集群安全** | ClusterMesh、统一身份、全局策略 | Cilium、Rancher |
| **零信任** | SPIFFE 身份、mTLS、持续验证 | Istio、SPIRE、Falco |
| **可观测性** | 日志+指标+追踪，安全专用告警 | Thanos、Loki、Jaeger |
| **治理合规** | Policy as Code、持续扫描 | OPA、Kubescape |
| **密钥管理** | Vault 集中管理、动态凭证 | Vault、cert-manager |

**关键架构原则**：
1. **纵深防御**：多层安全控制叠加，单点失效不导致整体失守
2. **零信任**：永不信任，始终验证，最小权限
3. **统一治理**：策略即代码，跨集群一致执行
4. **可观测性**：全面监控，快速检测和响应
5. **自动化**：减少人工干预，降低人为错误
