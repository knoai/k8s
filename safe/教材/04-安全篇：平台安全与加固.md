## 4.1 Kubernetes 威胁模型

### 4.1.1 攻击面分析

```
┌─────────────────────────────────────────────────────────────────┐
│                      Kubernetes 攻击面                           │
│                                                                  │
│  外部攻击者                                                       │
│       │                                                          │
│       ▼                                                          │
│  ┌─────────────┐    ┌─────────────┐    ┌─────────────┐          │
│  │   Ingress    │    │  API Server  │    │  NodePort   │          │
│  │  Controller  │    │   (6443)     │    │  Services   │          │
│  └──────┬──────┘    └──────┬──────┘    └──────┬──────┘          │
│         │                   │                   │                │
│         └───────────────────┼───────────────────┘                │
│                             ▼                                    │
│  ┌────────────────────────────────────────────────────────────┐ │
│  │                    集群内部                                   │ │
│  │                                                              │ │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │ │
│  │  │  Pod → Pod   │  │Pod → API    │  │Pod → Node   │        │ │
│  │  │  横向移动    │  │Server 提权  │  │  逃逸        │        │ │
│  │  └─────────────┘  └─────────────┘  └─────────────┘        │ │
│  │                                                              │ │
│  │  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │ │
│  │  │ 容器逃逸     │  │ 凭证窃取    │  │ 供应链攻击   │        │ │
│  │  │ → 节点控制   │  │ → 权限滥用  │  │ → 恶意镜像   │        │ │
│  │  └─────────────┘  └─────────────┘  └─────────────┘        │ │
│  │                                                              │ │

│  └────────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

### 4.1.2 安全分层模型

Kubernetes 安全遵循**纵深防御（Defense in Depth）**原则，分为四个层次：

| 层次 | 防护对象 | 核心机制 |
|------|---------|---------|
| **云/基础设施层** | 节点、网络、存储 | IAM、VPC、防火墙、磁盘加密 |
| **集群层** | 控制平面、节点 | 认证、RBAC、准入控制、NetworkPolicy |
| **容器层** | 容器运行时 | Seccomp、AppArmor、Capabilities、User Namespace |
| **应用层** | 应用代码、配置 | 镜像扫描、Secret 管理、代码审计 |

## 4.2 认证（Authentication）

### 4.2.1 认证机制总览

API Server 支持多种认证方式，可以同时启用多个：

```
客户端请求 → API Server
                  │
                  ├── X.509 客户端证书认证
                  ├── Bearer Token 认证
                  │   ├── ServiceAccount Token
                  │   ├── 静态 Token 文件（已废弃）
                  │   ├── Bootstrap Token
                  │   └── OIDC Token
                  ├── Webhook Token 认证
                  ├── 匿名认证（可禁用）
                  └── 静态密码文件（已废弃，禁用！）
```

| 认证方式 | 适用场景 | 安全性 | 生产建议 |
|---------|---------|--------|---------|
| **X.509 客户端证书** | 管理员、组件通信 | 高 | 推荐，妥善保管私钥 |
| **ServiceAccount Token** | Pod 访问 API | 中 | 1.24+使用短期Token，限制权限 |
| **OIDC / OAuth2** | 企业人员认证 | 高 | **人员认证首选** |
| **Webhook Token** | 自定义认证 | 高 | 对接企业 IAM/SSO |
| **Bootstrap Token** | 节点加入集群 | 高 | 短期有效，自动清理 |
| **静态密码/Token文件** | 测试 | **极低** | **生产绝对禁用！** |

### 4.2.2 X.509 客户端证书认证

X.509 证书是 Kubernetes 组件间和管理员认证的主要方式。

**证书认证流程**：

```
客户端                        API Server
    │                            │
    │  TLS 握手                  │
    │ ─────────────────────────► │
    │  发送客户端证书             │
    │                            │
    │                            │  验证证书：
    │                            │  1. 证书是否由集群 CA 签发
    │                            │  2. 证书是否在有效期内
    │                            │  3. 证书是否被撤销（K8s不支持CRL）
    │                            │
    │  从证书的 CN 字段提取用户名 │
    │  从证书的 O 字段提取组信息  │
    │                            │
    │ ◄───────────────────────── │
    │  TLS 连接建立              │
    │                            │
    │  HTTP 请求                 │
    │ ─────────────────────────► │
    │  Authorization: 无需       │  使用证书中的身份信息
    │                            │  进行 RBAC 鉴权
```

**为用户创建证书**：

```bash
# 1. 创建私钥
openssl genrsa -out alice.key 2048

# 2. 创建证书签名请求 (CSR)
# CN (Common Name) = 用户名
# O (Organization) = 组名（可多个）
openssl req -new -key alice.key -out alice.csr \
  -subj "/CN=alice/O=developers/O=devops"

# 3. 使用 Kubernetes CA 签名
sudo openssl x509 -req -in alice.csr \
  -CA /etc/kubernetes/pki/ca.crt \
  -CAkey /etc/kubernetes/pki/ca.key \
  -CAcreateserial \
  -out alice.crt -days 365

# 4. 配置 kubeconfig
kubectl config set-cluster kubernetes \
  --server=https://k8s-api.example.com:6443 \
  --certificate-authority=/etc/kubernetes/pki/ca.crt \
  --embed-certs=true

kubectl config set-credentials alice \
  --client-certificate=alice.crt \
  --client-key=alice.key \
  --embed-certs=true

kubectl config set-context alice-context \
  --cluster=kubernetes --user=alice

kubectl config use-context alice-context
```

**证书中的身份信息**：

| 证书字段 | K8s 身份映射 | 示例 |
|---------|-------------|------|
| `CN` (Common Name) | 用户名 | `alice` |
| `O` (Organization) | 用户组 | `developers`, `system:masters` |
| `OU` | 不直接映射 | - |

**⚠️ 警告**：`system:masters` 是一个特殊的内置组，**自动绑定到 cluster-admin ClusterRole**，拥有集群的完全控制权。不要将普通用户放入此组！

### 4.2.3 ServiceAccount 与 Token 安全演变

ServiceAccount 是 Pod 访问 API Server 的身份凭证。Kubernetes 的 Token 机制经历了多次安全升级：

**历史演变**：

| 版本 | Token 机制 | 安全性 | 说明 |
|------|-----------|--------|------|
| < 1.20 | 长期 Secret Token（无过期） | 低 | Token 永久有效，泄露风险大 |
| 1.20-1.23 | Token 投射（Projected Volume）| 中 | 支持 TTL，但仍挂载到 Pod |
| 1.24+ | **不再自动生成 Secret Token** | 高 | 需手动创建或使用 TokenRequest API |

**Token 投射（推荐方式）**：

```yaml
apiVersion: v1
kind: ServiceAccount
metadata:
  name: app-sa
  namespace: default
automountServiceAccountToken: false   # 禁止自动挂载

---
apiVersion: v1
kind: Pod
metadata:
  name: app
spec:
  serviceAccountName: app-sa
  automountServiceAccountToken: true   # 只在需要时启用
  containers:
  - name: app
    image: myapp:1.0
    volumeMounts:
    - name: token
      mountPath: /var/run/secrets/kubernetes.io/serviceaccount
  volumes:
  - name: token
    projected:
      sources:
      - serviceAccountToken:
          path: token
          expirationSeconds: 3600    # Token 1 小时过期
          audience: api
```

**Token 投射的安全优势**：
1. **短期有效**：Token 有过期时间（如 1 小时）
2. **自动刷新**：kubelet 在 Token 过期前自动轮换
3. **受众绑定**：Token 只能用于特定的受众（audience）
4. **不存储在 etcd**：Token 是动态生成的，不持久化

**传统 Secret Token（已不推荐）**：

```bash
# 查看 ServiceAccount 的 Secret（旧方式）
kubectl get sa default -o yaml
# secrets:
# - name: default-token-xxxxx

# 这个 Secret 包含长期有效的 Token
kubectl get secret default-token-xxxxx -o jsonpath='{.data.token}' | base64 -d
# 此 Token 永不过期，泄露后风险极大
```

**手动创建短期 Token（1.24+）**：

```bash
# 创建 ServiceAccount（不自动生成 Secret）
kubectl create serviceaccount my-sa

# 手动创建 Token（使用 TokenRequest API）
kubectl create token my-sa --duration=1h

# 或创建长期 Token（不推荐，除非必要）
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Secret
metadata:
  name: my-sa-token
  annotations:
    kubernetes.io/service-account.name: my-sa
type: kubernetes.io/service-account-token
EOF
```

**ServiceAccount 安全最佳实践**：

```yaml
# 1. 不需要访问 API 的 Pod，禁用 ServiceAccount Token
apiVersion: v1
kind: Pod
spec:
  automountServiceAccountToken: false
  containers:
  - name: app
    image: nginx:alpine

# 2. 为每个应用创建独立的 ServiceAccount
apiVersion: v1
kind: ServiceAccount
metadata:
  name: payment-service
  namespace: production
automountServiceAccountToken: false

# 3. 最小权限 RBAC（只给需要的权限）
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  name: payment-reader
  namespace: production
rules:
- apiGroups: [""]
  resources: ["configmaps"]
  resourceNames: ["payment-config"]
  verbs: ["get"]
```

### 4.2.4 OIDC 认证集成

OIDC（OpenID Connect）是 Kubernetes 人员认证的**推荐方式**：

```
用户                          Kubernetes                    IdP (Keycloak/OKTA/Azure AD)
  │                              │                                │
  │  1. 访问 kubectl             │                                │
  │ ───────────────────────────► │                                │
  │                              │                                │
  │  2. 重定向到 IdP 登录         │                                │
  │ ◄─────────────────────────── │                                │
  │                              │                                │
  │  3. 用户在 IdP 登录           │                                │
  │ ─────────────────────────────────────────────────────────────► │
  │                              │                                │
  │  4. IdP 返回 ID Token        │                                │
  │ ◄───────────────────────────────────────────────────────────── │
  │                              │                                │
  │  5. 携带 ID Token 访问 API   │                                │
  │ ───────────────────────────► │                                │
  │                              │  6. 验证 Token 签名            │
  │                              │     (使用 IdP 的公钥)          │
  │                              │                                │
  │                              │  7. 从 Token 中提取用户信息    │
  │                              │     (sub, groups, email)       │
  │                              │                                │
  │  8. 返回资源                 │                                │
  │ ◄─────────────────────────── │                                │
```

**API Server OIDC 配置**：

```bash
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    - --oidc-issuer-url=https://auth.example.com/realms/kubernetes
    - --oidc-client-id=kubernetes
    - --oidc-username-claim=email       # 使用 email 作为用户名
    - --oidc-groups-claim=groups        # 使用 groups 作为用户组
    - --oidc-ca-file=/etc/kubernetes/pki/oidc-ca.crt
```

**kubectl 配置 OIDC**：

```bash
# 使用 kubelogin 插件
kubectl oidc-login setup --oidc-issuer-url=https://auth.example.com/realms/kubernetes \
  --oidc-client-id=kubernetes \
  --oidc-client-secret=xxx

# 或使用 Exec 插件
kubectl config set-credentials oidc-user \
  --exec-api-version=client.authentication.k8s.io/v1beta1 \
  --exec-command=kubectl \
  --exec-arg=oidc-login \
  --exec-arg=get-token \
  --exec-arg=--oidc-issuer-url=https://auth.example.com/realms/kubernetes \
  --exec-arg=--oidc-client-id=kubernetes
```

**OIDC 相比 X.509 的优势**：
- Token 有过期时间，可以自动刷新
- 支持 MFA（多因素认证）
- 用户离职后，在 IdP 中禁用即可（无需回收证书）
- 支持审计和用户活动追踪

### 4.2.5 Webhook Token 认证

Webhook Token 认证允许将认证请求转发到外部服务：

```bash
# API Server 配置
--authentication-token-webhook-config-file=/etc/kubernetes/webhook-auth.yaml
```

```yaml
# webhook-auth.yaml
clusters:
- name: webhook-auth
  cluster:
    certificate-authority: /etc/kubernetes/pki/webhook-ca.crt
    server: https://auth-service.internal/verify
current-context: webhook-auth
contexts:
- context:
    cluster: webhook-auth
  name: webhook-auth
```

外部认证服务接收 TokenReview 对象，返回认证结果：

```json
{
  "apiVersion": "authentication.k8s.io/v1",
  "kind": "TokenReview",
  "status": {
    "authenticated": true,
    "user": {
      "username": "alice@example.com",
      "groups": ["developers", "devops"],
      "extra": {
        "issuer": ["https://auth.example.com"]
      }
    }
  }
}
```

## 4.3 授权（Authorization）：RBAC 深度解析

### 4.3.1 RBAC 核心概念

RBAC（Role-Based Access Control）是 Kubernetes 的授权机制，基于角色的访问控制：

```
Subject（主体：User/Group/ServiceAccount）
         │
         │ RoleBinding / ClusterRoleBinding
         ▼
    Role / ClusterRole（权限规则集合）
         │
         │ 定义对哪些资源的哪些操作
         ▼
    Resource + Verb（pods, get/list/watch/create/update/delete）
```

**四种 RBAC 资源对比**：

| 资源 | 作用范围 | 用途 | 示例 |
|------|---------|------|------|
| **Role** | 命名空间内 | 命名空间级权限 | 在 `dev` 命名空间中管理 Pod |
| **ClusterRole** | 集群范围 | 集群级权限 | 管理节点、命名空间、集群角色 |
| **RoleBinding** | 命名空间内 | 将 Role/ClusterRole 绑定到主体 | 将 `dev-role` 绑定到 `alice` |
| **ClusterRoleBinding** | 集群范围 | 将 ClusterRole 绑定到主体 | 将 `cluster-admin` 绑定到 `admin-group` |

**重要规则**：RoleBinding 可以引用 Role 或 ClusterRole。如果引用 ClusterRole，权限只在 RoleBinding 所在的命名空间中生效。

### 4.3.2 权限规则详解

```yaml
rules:
# 规则 1：对核心资源的权限
- apiGroups: [""]           # "" 表示核心 API 组（v1）
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list", "watch"]

# 规则 2：对 apps 组的权限
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets", "statefulsets"]
  verbs: ["*"]              # "*" 表示所有操作

# 规则 3：对特定资源的权限（resourceNames）
- apiGroups: [""]
  resources: ["secrets"]
  resourceNames: ["app-config", "app-credentials"]
  verbs: ["get"]

# 规则 4：对非资源 URL 的权限
- nonResourceURLs: ["/healthz", "/metrics"]
  verbs: ["get"]

# 规则 5：对子资源的权限
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get"]
  resourceNames: ["my-pod"]
```

**Verb 详解**：

| Verb | 说明 | HTTP 方法 |
|------|------|----------|
| `create` | 创建资源 | POST |
| `get` | 获取单个资源 | GET |
| `list` | 列出资源集合 | GET（集合） |
| `watch` | 监听资源变化 | GET + watch=true |
| `update` | 完整更新 | PUT |
| `patch` | 部分更新 | PATCH |
| `delete` | 删除单个资源 | DELETE |
| `deletecollection` | 删除集合 | DELETE（集合） |
| `*` | 所有操作 | - |

**子资源（Subresource）**：

```bash
# Pod 的子资源
kubectl get pod <pod> --subresource=log        # logs
kubectl get pod <pod> --subresource=status     # status
kubectl get pod <pod> --subresource=exec       # exec
kubectl get pod <pod> --subresource=portforward # portforward

# 子资源的 RBAC 规则
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get"]

- apiGroups: [""]
  resources: ["pods/exec"]
  verbs: ["create"]
```

### 4.3.3 聚合 ClusterRole（Aggregated ClusterRole）

Kubernetes 1.9+ 支持通过标签选择器聚合多个 ClusterRole：

```yaml
# 基础权限提供者
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: aggregate-pod-reader
  labels:
    rbac.example.com/aggregate-to-reader: "true"
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]

---
# 聚合角色（自动合并所有带标签的角色的规则）
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: reader
aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
      rbac.example.com/aggregate-to-reader: "true"
rules: []  # 自动填充
```

**内置的聚合角色**：`admin`、`edit`、`view` 都是聚合角色。

### 4.3.4 内置默认角色

Kubernetes 提供了一组预定义的 ClusterRole：

| 角色 | 权限范围 | 适用场景 |
|------|---------|---------|
| **cluster-admin** | 所有资源的完全控制 | 集群管理员（极度危险！） |
| **admin** | 命名空间内所有资源（除 ResourceQuota） | 命名空间管理员 |
| **edit** | 命名空间内大部分资源的读写 | 开发人员 |
| **view** | 命名空间内资源的只读 | 只读用户 |
| **system:masters** | 绕过所有 RBAC 检查 | 紧急恢复（特殊组） |

```bash
# 查看 cluster-admin 的权限
kubectl describe clusterrole cluster-admin
# Name:         cluster-admin
# Labels:       kubernetes.io/bootstrapping=rbac-defaults
# Annotations:  rbac.authorization.kubernetes.io/autoupdate: true
# PolicyRule:
#   Resources  Non-Resource URLs  Resource Names  Verbs
#   ---------  -----------------  --------------  -----
#   *.*        []                 []              [*]
#              [*]                []              [*]

# cluster-admin 的权限 = 对所有资源的所有操作 + 所有非资源 URL
```

### 4.3.5 RBAC 安全最佳实践

**✅ 推荐做法**：

```yaml
# 1. 按团队/应用创建独立命名空间
# 2. 使用 Role + RoleBinding 限制在命名空间内
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: payment
  name: payment-developer
rules:
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/exec"]
  verbs: []  # 明确禁止 exec

---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: payment
  name: payment-dev-binding
subjects:
- kind: Group
  name: payment-team
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: payment-developer
  apiGroup: rbac.authorization.k8s.io
```

**❌ 禁止做法**：

```yaml
# 绝对禁止：给普通用户 cluster-admin
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: dangerous-admin-binding
subjects:
- kind: User
  name: alice@example.com
roleRef:
  kind: ClusterRole
  name: cluster-admin       # ❌ 极度危险！
  apiGroup: rbac.authorization.k8s.io

# 绝对禁止：允许创建特权 Pod
rules:
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["*"]
# 这允许创建 privileged: true 的 Pod = 容器逃逸

# 绝对禁止：允许修改 RBAC
rules:
- apiGroups: ["rbac.authorization.k8s.io"]
  resources: ["roles", "rolebindings", "clusterroles", "clusterrolebindings"]
  verbs: ["*"]
# 这允许权限提升 = 获得 cluster-admin
```

### 4.3.6 RBAC 权限审计

```bash
# 查看当前用户的所有权限
kubectl auth can-i --list

# 模拟特定用户检查权限
kubectl auth can-i create pods --as=alice
kubectl auth can-i delete secrets --as=system:serviceaccount:default:default
kubectl auth can-i '*' '*' --as=alice  # 检查是否有完全权限

# 查看谁拥有 cluster-admin
kubectl get clusterrolebinding -o json | \

kubectl get clusterrole,role --all-namespaces -o json | \

    select(.rules[]?.verbs[]? == "*" or .rules[]?.verbs[]? == "create") | 
    .metadata.name'

# 使用 rback 工具可视化 RBAC（第三方工具）
# https://github.com/team-soteria/rback
kubectl get rolebindings,clusterrolebindings,roles,clusterroles --all-namespaces -o yaml | \
  rback > rbac-graph.dot
```

## 4.4 准入控制（Admission Control）

### 4.4.1 准入控制器完整链

```
客户端请求
    │
    ▼
┌─────────────────┐
│ Authentication  │  ← 认证（你是谁）
└────────┬────────┘
         ▼
┌─────────────────┐
│ Authorization   │  ← 授权（你能做什么）
└────────┬────────┘
         ▼
┌─────────────────────────────────────────────────────────┐
│                  Mutating Admission                      │
│  （按顺序执行，每个都可以修改对象）                        │
│                                                          │
│  1. MutatingAdmissionWebhook   ← 自定义 Webhook         │
│  2. PodPreset (已废弃)                                   │
│  3. LimitRanger               ← 设置默认资源限制        │
│  4. ServiceAccount            ← 自动挂载 SA             │
│  5. DefaultStorageClass       ← 设置默认 StorageClass   │
│  6. NodeRestriction           ← 限制 kubelet 节点操作   │
│  ...                                                     │
└─────────────────────────────────────────────────────────┘
         │
         ▼
┌─────────────────┐
│ Object Schema   │  ← 验证对象格式（OpenAPI schema）
│ Validation      │
└────────┬────────┘
         ▼
┌─────────────────────────────────────────────────────────┐
│                  Validating Admission                    │
│  （按顺序执行，只能验证不能修改）                          │
│                                                          │
│  1. ValidatingAdmissionWebhook  ← 自定义 Webhook        │
│  2. PodSecurity                 ← Pod 安全标准          │
│  3. ResourceQuota               ← 资源配额检查          │
│  4. NamespaceLifecycle          ← 命名空间生命周期       │
│  5. LimitRanger                 ← 资源限制验证          │
│  ...                                                     │
└─────────────────────────────────────────────────────────┘
         │
         ▼
       etcd
```

### 4.4.2 内置关键准入控制器详解

| 控制器 | 类型 | 作用 | 安全意义 |
|--------|------|------|---------|
| **NamespaceLifecycle** | Validating | 阻止在终止中的命名空间创建资源 | 防止资源泄漏 |
| **LimitRanger** | Mutating+Validating | 设置默认资源限制，验证不超限 | 防止资源耗尽 |
| **ServiceAccount** | Mutating | 自动挂载默认 ServiceAccount | 注意自动挂载风险 |
| **ResourceQuota** | Validating | 限制命名空间资源总量 | 防止资源耗尽 |
| **NodeRestriction** | Validating | 限制 kubelet 只能修改自己的节点 | 防止节点伪装 |
| **PodSecurity** | Validating | 执行 Pod Security Standards | 强制安全基线 |
| **AlwaysPullImages** | Mutating | 强制总是拉取镜像 | 防止使用本地缓存的恶意镜像 |
| **SecurityContextDeny** | Validating | 拒绝某些 SecurityContext（已废弃） | 被 PSS 替代 |

**启用方式**：

```bash

kubectl get pod -n kube-system kube-apiserver-<node> -o yaml | grep enable-admission-plugins

# 推荐的准入控制器列表（v1.29+）
--enable-admission-plugins=
  NamespaceLifecycle,
  LimitRanger,
  ServiceAccount,
  TaintNodesByCondition,
  PodSecurity,
  Priority,
  DefaultTolerationSeconds,
  DefaultStorageClass,
  StorageObjectInUseProtection,
  PersistentVolumeClaimResize,
  RuntimeClass,
  CertificateApproval,
  CertificateSigning,
  CertificateSubjectRestriction,
  ClusterTrustBundleAttestation,
  DefaultIngressClass,
  MutatingAdmissionWebhook,
  ValidatingAdmissionWebhook,
  ResourceQuota,
  NodeRestriction
```

### 4.4.3 Pod Security Admission（PSA）

PSA 是 Kubernetes 内置的 Pod 安全策略执行机制，替代了已废弃的 PodSecurityPolicy（PSP）。

**三个安全级别**：

| 级别 | 风险容忍度 | 适用场景 |
|------|-----------|---------|
| **privileged** | 无限制 | 系统基础设施、需要完全控制的组件 |
| **baseline** | 阻止已知高危配置 | 大多数非特权应用 |
| **restricted** | 遵循 Pod 强化最佳实践 | 安全敏感应用、多租户环境 |

**baseline 级别禁止的配置**：

| 禁止项 | 说明 |
|--------|------|
| `privileged: true` | 特权容器 |
| `hostPID: true` | 共享主机 PID 命名空间 |
| `hostIPC: true` | 共享主机 IPC 命名空间 |
| `hostNetwork: true` | 共享主机网络命名空间 |
| `hostPorts` | 绑定主机端口（特定范围除外） |
| `volumes` | 某些卷类型（hostPath 等） |
| `allowedCapabilities` | 超出默认的能力 |
| `seLinuxOptions` | 自定义 SELinux 选项 |
| `procMount` | 非默认的 proc 挂载类型 |
| `seccompProfile` | 未设置或使用 Unconfined |
| `sysctls` | 不安全的 sysctl |

**restricted 级别额外禁止**（在 baseline 基础上）：

| 额外禁止项 | 说明 |
|-----------|------|
| 必须以非 root 运行 | `runAsNonRoot: true` |
| 禁止特权提升 | `allowPrivilegeEscalation: false` |
| 必须丢弃所有 capabilities | `drop: ["ALL"]` |
| 必须设置 seccomp | `type: RuntimeDefault` 或 `Localhost` |
| 必须设置 runAsUser | 明确的 UID（非 0） |
| 限制卷类型 | 只允许特定的安全卷类型 |

**PSA 实施方式**：

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    # enforce: 违反时拒绝创建
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: v1.29
    
    # audit: 违反时记录审计日志，但不拒绝
    pod-security.kubernetes.io/audit: restricted
    pod-security.kubernetes.io/audit-version: v1.29
    
    # warn: 违反时向用户返回警告信息
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: v1.29
```

**豁免（Exemptions）**：

```yaml
# /etc/kubernetes/pki/admission-configuration.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: AdmissionConfiguration
plugins:
- name: PodSecurity
  configuration:
    apiVersion: pod-security.admission.config.k8s.io/v1
    kind: PodSecurityConfiguration
    defaults:
      enforce: "baseline"
      enforce-version: "latest"
      audit: "restricted"
      audit-version: "latest"
      warn: "restricted"
      warn-version: "latest"
    exemptions:
      # 按用户名豁免
      usernames: ["admin"]
      # 按 RuntimeClass 豁免
      runtimeClasses: ["gvisor"]
      # 按命名空间豁免
      namespaces: ["kube-system", "ingress-nginx"]
```

### 4.4.4 自定义准入控制器（Webhook）

当内置准入控制器无法满足需求时，可以开发自定义 Webhook：

```
API Server
    │
    │ AdmissionReview 请求
    ▼
┌─────────────────┐
│  Webhook Server │  ← 部署在集群内或外部
│  (HTTPS)        │
│                 │
│  1. 接收请求     │
│  2. 执行业务逻辑 │
│  3. 返回允许/拒绝 │
└─────────────────┘
    │
    │ AdmissionReview 响应
    ▼
API Server 根据响应决定允许或拒绝
```

**Webhook 配置**：

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: pod-security-webhook
webhooks:
- name: validate-pod.example.com
  rules:
  - apiGroups: [""]
    apiVersions: ["v1"]
    operations: ["CREATE", "UPDATE"]
    resources: ["pods"]
  clientConfig:
    service:
      namespace: security
      name: webhook-service
      path: /validate-pod
    caBundle: <base64-encoded-ca>
  admissionReviewVersions: ["v1"]
  sideEffects: None
  timeoutSeconds: 5
  failurePolicy: Fail  # Fail 或 Ignore
```

**常见 Webhook 实现**：

| 工具 | 类型 | 功能 | 学习曲线 |
|------|------|------|---------|
| **OPA Gatekeeper** | Validating | 通用策略引擎 | 中 |
| **Kyverno** | Mutating+Validating | K8s 原生策略 | 低 |
| **Kubewarden** | Mutating+Validating | WebAssembly 策略 | 中 |
| **jsPolicy** | Mutating+Validating | JavaScript 策略 | 低 |

## 4.5 API Server 安全加固

### 4.5.1 关键安全参数

```yaml
# /etc/kubernetes/manifests/kube-apiserver.yaml
spec:
  containers:
  - command:
    - kube-apiserver
    
    # === 认证 ===
    # 禁用匿名访问（默认 true，建议显式禁用）
    - --anonymous-auth=false
    
    # 禁用基本认证（已废弃，确保禁用）
    # - --basic-auth-file  ← 绝对不要设置！

    # - --token-auth-file  ← 绝对不要设置！

    # 授权模式：Node + RBAC
    # Node: 允许 kubelet 访问自己的节点信息

    - --authorization-mode=Node,RBAC

    - --enable-priority-and-fairness=true
```

### 4.5.2 审计日志策略

审计日志记录所有 API 请求，是安全分析和合规的关键：

```yaml
# /etc/kubernetes/audit-policy.yaml
apiVersion: audit.k8s.io/v1
kind: Policy
# 不记录 RequestReceived 阶段（减少日志量）
omitStages:
- "RequestReceived"

rules:
  # 规则 1：记录所有认证失败
  - level: Metadata
    users: ["system:anonymous"]
    verbs: ["*"]
    resources:
    - group: ""
      resources: ["*"]

  # 规则 2：记录 RBAC 变更（RequestResponse 级别）
  - level: RequestResponse
    resources:
    - group: "rbac.authorization.k8s.io"
      resources: ["roles", "rolebindings", "clusterroles", "clusterrolebindings"]
    verbs: ["create", "update", "patch", "delete"]
    omitStages:
    - "RequestReceived"

  # 规则 3：记录 Secret 的访问和变更
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["secrets", "configmaps"]
    verbs: ["create", "update", "patch", "delete"]
    omitStages:
    - "RequestReceived"

  # 规则 4：记录 Pod 创建（特别是特权 Pod）
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["pods"]
    verbs: ["create", "update"]
    omitStages:
    - "RequestReceived"

  # 规则 5：记录 exec/attach/portforward（危险操作）
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["pods/exec", "pods/attach", "pods/portforward"]
    verbs: ["create"]
    omitStages:
    - "RequestReceived"

  # 规则 6：记录节点变更
  - level: RequestResponse
    resources:
    - group: ""
      resources: ["nodes"]
    verbs: ["create", "update", "patch", "delete"]
    omitStages:
    - "RequestReceived"

  # 规则 7：记录生产命名空间的所有变更
  - level: RequestResponse
    namespaces: ["production", "payment"]
    verbs: ["create", "update", "patch", "delete"]
    omitStages:
    - "RequestReceived"

  # 规则 8：默认只记录元数据（所有其他请求）
  - level: Metadata
    omitStages:
    - "RequestReceived"
```

**审计级别**：

| 级别 | 记录内容 | 日志量 | 适用场景 |
|------|---------|--------|---------|
| `None` | 不记录 | 无 | 健康检查等高频请求 |
| `Metadata` | 请求元数据（用户、时间、资源、动作） | 小 | 大多数请求 |
| `Request` | Metadata + 请求体 | 中 | 需要查看请求内容的场景 |
| `RequestResponse` | Metadata + 请求体 + 响应体 | 大 | 关键操作（RBAC、Secret 变更） |

### 4.5.3 API 优先级和公平性（APF）

Kubernetes 1.20+ 引入 APF（API Priority and Fairness），防止某个用户或控制器耗尽 API Server 资源：

```yaml
# 默认的 FlowSchema 和 PriorityLevelConfiguration
apiVersion: flowcontrol.apiserver.k8s.io/v1
kind: FlowSchema
metadata:
  name: health-for-strangers
spec:
  priorityLevelConfiguration:
    name: exempt            # 健康检查请求不限制
  rules:
  - nonResourceRules:
    - nonResourceURLs:
      - /healthz
      - /livez
      - /readyz
      verbs: ["get"]
    subjects:
    - kind: Group
      group:
        name: system:unauthenticated
```

## 4.6 kubelet 安全加固

### 4.6.1 kubelet 攻击面

kubelet 是 Kubernetes 攻击的重要目标，因为它：
1. 运行在每个节点上，拥有该节点的最高权限
2. 暴露 10250 端口（HTTPS，用于 API Server 调用）
3. 如果配置不当，可能泄露敏感信息或允许未授权操作

```
攻击路径：
    │
    ├── 10250 端口未认证
    │   └── 获取节点上所有 Pod 的日志、在容器内执行命令
    │
    ├── 10255 只读端口启用
    │   └── 泄露 Pod 列表、资源使用、环境变量
    │
    └── kubelet 证书泄露
        └── 伪装 kubelet，修改节点状态、读取 Secret
```

### 4.6.2 kubelet 安全配置

```yaml
# /var/lib/kubelet/config.yaml
# 认证配置
authentication:
  anonymous:
    enabled: false          # ❗ 禁用匿名认证
  webhook:
    enabled: true           # 启用 Webhook 认证（使用 API Server 验证 Token）
  x509:
    clientCAFile: /etc/kubernetes/pki/ca.crt  # 只接受 CA 签名的客户端证书

# 授权配置
authorization:
  mode: Webhook             # 使用 Webhook 授权（向 API Server 查询权限）

# 禁用只读端口（10255）
readOnlyPort: 0             # 0 表示禁用

# 保护内核参数
protectKernelDefaults: true

# 限制不安全的 sysctl
allowedUnsafeSysctls: []    # 空列表表示禁止所有不安全的 sysctl

# 禁用性能分析端点（减少攻击面）
debugging:
  enableContentionProfiling: false
  enableProfiling: false

# 镜像拉取策略
serializeImagePulls: false  # 并行拉取镜像（性能优化）

# 驱逐配置（防止节点资源耗尽）
evictionHard:
  memory.available: "500Mi"
  nodefs.available: "10%"
  imagefs.available: "15%"
evictionSoft:
  memory.available: "1Gi"
  nodefs.available: "15%"
  imagefs.available: "20%"
evictionSoftGracePeriod:
  memory.available: "1m"
  nodefs.available: "1m30s"
  imagefs.available: "1m30s"
```

**kubelet 命令行参数**（在 systemd service 或 kubeadm 配置中）：

```bash
# /etc/systemd/system/kubelet.service.d/10-kubeadm.conf
ExecStart=/usr/bin/kubelet \
  --config=/var/lib/kubelet/config.yaml \
  --tls-cert-file=/var/lib/kubelet/pki/kubelet-client-current.pem \
  --tls-private-key-file=/var/lib/kubelet/pki/kubelet-client-current.pem \
  --rotate-certificates=true          # 自动轮换证书
  --rotate-server-certificates=true   # 自动轮换服务器证书
```

### 4.6.3 NodeRestriction 准入控制器

NodeRestriction 限制 kubelet 只能修改属于自己的节点和 Pod：

```bash
# 启用 NodeRestriction
--enable-admission-plugins=...,NodeRestriction,...

# NodeRestriction 的限制：
# 1. kubelet 只能修改自己的 Node 对象（与 kubelet 节点名匹配）
# 2. kubelet 只能修改绑定到自己节点的 Pod 对象
# 3. kubelet 不能修改其他 kubelet 的节点或 Pod
# 4. kubelet 不能删除自己的 Node 对象
# 5. kubelet 不能创建 Mirror Pod（静态 Pod）以外的 Pod

# 验证 NodeRestriction 生效
curl -k https://<node-ip>:10250/pods
# 无认证 → 401 Unauthorized

# 使用正确证书访问
curl -k --cert /etc/kubernetes/pki/apiserver-kubelet-client.crt \
  --key /etc/kubernetes/pki/apiserver-kubelet-client.key \
  https://<node-ip>:10250/pods
```

## 4.7 etcd 安全

### 4.7.1 etcd 安全要点

etcd 存储了集群的所有数据，包括 Secret 的内容。etcd 的安全至关重要：

| 安全层面 | 措施 | 配置 |
|---------|------|------|
| **静态加密** | 加密 etcd 中的 Secret 数据 | `--encryption-provider-config` |
| **传输加密** | etcd peer/client 通信使用 TLS | `--peer-client-cert-auth`, `--client-cert-auth` |
| **访问控制** | 使用独立的 CA，限制证书 | 独立的 etcd CA |
| **备份加密** | 备份文件加密存储 | 加密后存储到安全位置 |
| **网络隔离** | etcd 只在控制平面节点间通信 | 防火墙限制 2379/2380 端口 |

### 4.7.2 etcd 访问控制

```bash
# etcd 使用证书进行身份验证和授权
# 三类证书：
# 1. peer 证书：etcd 节点间通信
# 2. server 证书：etcd 客户端连接（API Server）
# 3. client 证书：etcdctl 等工具连接

# etcd 启动参数中的安全相关配置
--client-cert-auth=true
--trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
--cert-file=/etc/kubernetes/pki/etcd/server.crt
--key-file=/etc/kubernetes/pki/etcd/server.key

--peer-client-cert-auth=true
--peer-trusted-ca-file=/etc/kubernetes/pki/etcd/ca.crt
--peer-cert-file=/etc/kubernetes/pki/etcd/peer.crt
--peer-key-file=/etc/kubernetes/pki/etcd/peer.key

# 验证 etcd 是否启用客户端证书认证
ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  get /registry/namespaces/default
# 无证书 → Error: etcdserver: user name is empty

ETCDCTL_API=3 etcdctl \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
```

## 4.9 NetworkPolicy 基础

### 4.9.1 NetworkPolicy 原理

NetworkPolicy 是 Kubernetes 的 L3/L4 网络隔离机制，但**默认允许所有流量**：

```
没有 NetworkPolicy 的命名空间：
    所有 Pod ←──────► 所有 Pod（允许）
    所有 Pod ←──────► 外部（允许）
    外部   ←──────► 所有 Pod（允许）

应用了 NetworkPolicy 的 Pod：
    只有策略允许的流量可以通过
    其他流量默认被拒绝
```

**重要**：NetworkPolicy 的实现依赖于 CNI 插件（Calico、Cilium 等）。如果 CNI 不支持 NetworkPolicy，策略不会生效。

### 4.9.2 默认拒绝策略

```yaml
# 默认拒绝所有入站流量（零信任网络）
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-ingress
  namespace: production
spec:
  podSelector: {}      # 匹配命名空间中的所有 Pod
  policyTypes:
  - Ingress            # 只限制入站，不限制出站
---
# 默认拒绝所有出站流量
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-egress
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Egress
```

### 4.9.3 精细化网络策略

```yaml
# 允许 frontend → backend 的通信
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-frontend
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
---
# 允许 backend → database 的通信
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-allow-db
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306
  # 允许访问 DNS（CoreDNS）
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
```

## 4.11 本章实验

### 实验 4.1：X.509 证书认证（25 分钟）

```bash
# 步骤 1：创建用户证书
openssl genrsa -out developer.key 2048
openssl req -new -key developer.key -out developer.csr \
  -subj "/CN=developer/O=dev-team"

sudo openssl x509 -req -in developer.csr \
  -CA /etc/kubernetes/pki/ca.crt \
  -CAkey /etc/kubernetes/pki/ca.key \
  -CAcreateserial -out developer.crt -days 30

# 步骤 2：配置 kubeconfig
kubectl config set-credentials developer \
  --client-certificate=developer.crt \
  --client-key=developer.key
kubectl config set-context developer-context \
  --cluster=kubernetes --user=developer

# 步骤 3：测试（此时没有权限）
kubectl --context=developer-context get pods
# Error from server (Forbidden): pods is forbidden

# 步骤 4：授予只读权限
cat <<EOF | kubectl apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: developer-reader
rules:
- apiGroups: [""]
  resources: ["pods", "services", "configmaps"]
  verbs: ["get", "list", "watch"]
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: developer-binding
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: ClusterRole
  name: developer-reader
  apiGroup: rbac.authorization.k8s.io
EOF

# 步骤 5：验证权限
kubectl --context=developer-context get pods
kubectl --context=developer-context get secrets   # 应该失败

# 清理
kubectl delete clusterrole developer-reader
kubectl delete clusterrolebinding developer-binding
rm developer.key developer.csr developer.crt
```

## 4.13 本章小结

| 安全机制 | 作用 | 关键配置 | 风险等级 |
|---------|------|---------|---------|
| **认证** | 确认"你是谁" | X.509、OIDC、禁用匿名 | 高（身份冒用） |
| **RBAC** | 确认"你能做什么" | 最小权限、避免 cluster-admin | 高（权限提升） |
| **准入控制** | 请求拦截和验证 | PodSecurity、LimitRanger、ResourceQuota | 中 |
| **PSS** | Pod 安全基线 | restricted 级别、命名空间标签 | 中 |
| **API Server** | 入口安全 | 审计日志、TLS、禁用不安全端口 | 高 |
| **kubelet** | 节点代理安全 | 禁用只读端口、NodeRestriction | 高 |
| **etcd** | 数据安全 | 静态加密、TLS、独立 CA | **极高** |
| **NetworkPolicy** | 网络隔离 | 默认拒绝、显式允许 | 中 |
| **CIS Benchmark** | 合规检查 | kube-bench 定期扫描 | 中 |

**关键安全原则**：
1. **最小权限**：每个主体只拥有完成工作所需的最小权限
2. **纵深防御**：多层安全机制叠加，单点失效不导致整体失守
3. **默认拒绝**：没有明确允许的操作都禁止（RBAC、NetworkPolicy）
4. **审计一切**：所有敏感操作都有日志记录，可追溯
5. **定期审查**：RBAC 权限、证书有效期、安全基线定期审查
6. **加密数据**：etcd 静态加密、网络传输加密、备份加密

**安全检查清单**：
- [ ] 禁用匿名认证
- [ ] 禁用不安全端口（--insecure-port=0）
- [ ] 启用审计日志
- [ ] etcd 启用静态加密
- [ ] 禁用 kubelet 只读端口
- [ ] 启用 NodeRestriction
- [ ] 实施 Pod Security Standards
- [ ] 配置 NetworkPolicy
