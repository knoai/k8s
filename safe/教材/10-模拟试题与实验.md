### 题目 2：etcd 备份与恢复

**场景**：需要对 etcd 进行备份，并在灾难后恢复。

**任务**：
1. 备份 etcd 到 `/opt/backup/etcd.db`
2. 模拟删除 deployment `web-app`
3. 从备份恢复 etcd
4. 验证 deployment 已恢复

<details>
<summary>参考答案</summary>

```bash
# 1. 备份
ETCDCTL_API=3 etcdctl snapshot save /opt/backup/etcd.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# 2. 删除 deployment
kubectl delete deployment web-app

# 3. 恢复
sudo systemctl stop kubelet
sudo ETCDCTL_API=3 etcdctl snapshot restore /opt/backup/etcd.db \
  --data-dir=/var/lib/etcd-restored
# 修改 /etc/kubernetes/manifests/etcd.yaml 中的 hostPath 为 /var/lib/etcd-restored
sudo systemctl start kubelet

# 4. 验证
kubectl get deployment web-app
```

</details>

### 题目 3：配置 RBAC

**场景**：用户 `developer` 需要在 `dev` 命名空间中管理 Deployment，但不能访问其他命名空间。

**任务**：创建必要的 Role 和 RoleBinding。

<details>
<summary>参考答案</summary>

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: deployment-manager
rules:
- apiGroups: ["apps"]
  resources: ["deployments", "replicasets"]
  verbs: ["get", "list", "watch", "create", "update", "patch", "delete"]
- apiGroups: [""]
  resources: ["pods"]
  verbs: ["get", "list", "watch"]
- apiGroups: [""]
  resources: ["pods/log"]
  verbs: ["get", "list"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: dev
  name: developer-binding
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: deployment-manager
  apiGroup: rbac.authorization.k8s.io
```

</details>

### 题目 5：存储动态供给

**场景**：需要为应用提供动态持久化存储。

**任务**：
1. 创建一个 StorageClass `fast-ssd`
2. 创建一个 PVC `data-pvc` 申请 5Gi 存储
3. 在 Pod 中使用该 PVC

<details>
<summary>参考答案</summary>

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-ssd
provisioner: kubernetes.io/gce-pd
parameters:
  type: pd-ssd
volumeBindingMode: WaitForFirstConsumer
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: data-pvc
spec:
  accessModes:
  - ReadWriteOnce
  storageClassName: fast-ssd
  resources:
    requests:
      storage: 5Gi
---
apiVersion: v1
kind: Pod
metadata:
  name: data-pod
spec:
  containers:
  - name: app
    image: nginx:alpine
    volumeMounts:
    - name: data
      mountPath: /data
  volumes:
  - name: data
    persistentVolumeClaim:
      claimName: data-pvc
```

</details>

### 题目 6：网络故障排查

**场景**：`frontend` Pod 无法访问 `backend` Service。

**任务**：
1. 诊断问题原因
2. 修复网络连通性

<details>
<summary>参考答案</summary>

```bash
# 1. 检查 Service 是否存在
kubectl get svc backend
kubectl get endpoints backend

# 2. 检查 Pod 标签是否匹配 Service selector
kubectl get pod -l app=backend --show-labels

# 3. 检查 Pod 是否 Running 且 Ready

# 5. 检查 DNS 解析
kubectl exec frontend -- nslookup backend

# 6. 检查 iptables/ipvs 规则（在节点上）
sudo iptables -t nat -L KUBE-SERVICES -n | grep <service-ip>
sudo ipvsadm -Ln | grep <service-ip>

# 常见原因和修复：
# - Pod 标签不匹配 → 修改标签或 selector
# - Pod 未 Ready → 检查 readinessProbe
# - DNS 问题 → 检查 CoreDNS
# - NetworkPolicy 阻断 → 添加允许规则
```

</details>

<details>
<summary>参考答案</summary>

```bash
# 1. 检查证书过期时间

# 2. 备份现有证书

# 3. 轮换所有证书

# 4. 更新 kubeconfig 文件
sudo kubeadm init phase kubeconfig all

# 5. 重启 kubelet
sudo systemctl restart kubelet

# 6. 验证
kubeadm certs check-expiration
kubectl get nodes
kubectl get pods -n kube-system
```

### 题目 3：NetworkPolicy 配置

**场景**：`frontend` Pod 需要访问 `backend` Pod 的 8080 端口，`backend` 需要访问 `database` Pod 的 3306 端口。其他流量全部拒绝。

**任务**：在 `production` 命名空间中配置 NetworkPolicy。

<details>
<summary>参考答案</summary>

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny
  namespace: production
spec:
  podSelector: {}
  policyTypes:
  - Ingress
  - Egress
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: backend-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: backend
  policyTypes:
  - Ingress
  - Egress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend
    ports:
    - protocol: TCP
      port: 8080
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database
    ports:
    - protocol: TCP
      port: 3306
  - to:
    - namespaceSelector:
        matchLabels:
          kubernetes.io/metadata.name: kube-system
      podSelector:
        matchLabels:
          k8s-app: kube-dns
    ports:
    - protocol: UDP
      port: 53
---
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: database-policy
  namespace: production
spec:
  podSelector:
    matchLabels:
      app: database
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: backend
    ports:
    - protocol: TCP
      port: 3306
```

</details>

### 题目 4：启用 etcd 静态加密

**任务**：配置 API Server 对 Secret 进行 etcd 静态加密。

<details>
<summary>参考答案</summary>

```bash
# 1. 生成加密密钥
ENCRYPTION_KEY=$(head -c 32 /dev/urandom | base64)

# 2. 创建配置文件
sudo mkdir -p /etc/kubernetes/enc
cat <<EOF | sudo tee /etc/kubernetes/enc/encryption-config.yaml
apiVersion: apiserver.config.k8s.io/v1
kind: EncryptionConfiguration
resources:
  - resources:
    - secrets
    providers:
    - aescbc:
        keys:
        - name: key1
          secret: ${ENCRYPTION_KEY}
    - identity: {}
EOF

# 3. 修改 kube-apiserver 静态 Pod 清单
# 编辑 /etc/kubernetes/manifests/kube-apiserver.yaml
# 添加：--encryption-provider-config=/etc/kubernetes/enc/encryption-config.yaml
# 添加 volume 和 volumeMount

# 4. 等待 API Server 重启
kubectl wait --for=condition=Ready pod -n kube-system -l component=kube-apiserver

# 5. 重写所有 Secret 触发加密
kubectl get secrets --all-namespaces -o json | kubectl replace -f -

# 6. 验证
sudo ETCDCTL_API=3 etcdctl --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  get /registry/secrets/default/my-secret | strings
```

</details>

### 题目 5：AppArmor Profile

**任务**：为容器配置 AppArmor 配置文件 `localhost/k8s-apparmor-deny-write`，禁止写入 `/etc` 目录。

```bash
# 2. 在 Pod 中引用
```

### 题目 7：创建安全上下文 Pod

**任务**：创建一个符合 `restricted` Pod Security Standards 的 Pod，要求：
- 非 root 用户运行（UID 1000）
- 丢弃所有 capabilities
- 只读根文件系统
- 禁止特权提升
- 使用 RuntimeDefault seccomp

<details>
<summary>参考答案</summary>

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:
    runAsNonRoot: true
    runAsUser: 1000
    seccompProfile:
      type: RuntimeDefault
  containers:
  - name: app
    image: nginx:alpine
    securityContext:
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop:
        - ALL
    volumeMounts:
    - name: tmp
      mountPath: /tmp
    - name: cache
      mountPath: /var/cache/nginx
    - name: run
      mountPath: /var/run
  volumes:
  - name: tmp
    emptyDir: {}
  - name: cache
    emptyDir: {}
  - name: run
    emptyDir: {}
```

</details>

### 题目 8：RBAC 审计与修复

**场景**：发现多个用户绑定了 `cluster-admin` ClusterRole。

**任务**：
1. 列出所有绑定到 `cluster-admin` 的主体
2. 删除不必要的绑定
3. 为 `developer` 用户创建最小权限 Role

<details>
<summary>参考答案</summary>

```bash
# 1. 列出 cluster-admin 绑定
kubectl get clusterrolebinding -o json | \
  jq -r '.items[] | select(.roleRef.name=="cluster-admin") | 
    "Binding: \(.metadata.name), Subjects: \(.subjects | map(.name) | join(", "))"'

# 2. 删除不必要的绑定（根据审计结果）
kubectl delete clusterrolebinding <unnecessary-binding>

# 3. 创建最小权限 Role
cat <<EOF | kubectl apply -f -
apiVersion: rbac.authorization.k8s.io/v1
kind: Role
metadata:
  namespace: dev
  name: developer
rules:
- apiGroups: ["apps"]
  resources: ["deployments"]
  verbs: ["get", "list", "watch", "create", "update", "patch"]
- apiGroups: [""]
  resources: ["pods", "pods/log"]
  verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  namespace: dev
  name: developer
subjects:
- kind: User
  name: developer
  apiGroup: rbac.authorization.k8s.io
roleRef:
  kind: Role
  name: developer
  apiGroup: rbac.authorization.k8s.io
EOF
```

## 10.3 故障排查场景题

### 场景 1：API Server 无法访问

**现象**：`kubectl` 命令全部超时，显示 `Unable to connect to the server: dial tcp <apiserver-ip>:6443: i/o timeout`

**排查步骤**：

```bash
# 1. 检查本地 kubeconfig
kubectl config view --raw | grep server

# 2. 检查 API Server Pod 状态
kubectl get pod -n kube-system -l component=kube-apiserver
# 如果 kubectl 完全不可用，在控制平面节点上：
sudo crictl ps | grep kube-apiserver

# 3. 检查 API Server 日志
sudo crictl logs <apiserver-container-id>
# 或
sudo journalctl -u kubelet -f

# 4. 检查证书是否过期
openssl x509 -in /etc/kubernetes/pki/apiserver.crt -noout -dates

# 5. 检查 etcd 是否健康
sudo crictl ps | grep etcd
sudo crictl logs <etcd-container-id>
ETCDCTL_API=3 etcdctl endpoint health

# 6. 检查节点资源
sudo df -h
sudo free -h

# 常见原因：
# - 证书过期 → 轮换证书
# - etcd 故障 → 恢复 etcd
# - 节点磁盘满 → 清理空间
# - 网络问题 → 检查防火墙/路由
```

---

### 场景 2：Pod 反复重启（CrashLoopBackOff）

**现象**：Pod 状态为 `CrashLoopBackOff`，RESTARTS 计数不断增加

**排查步骤**：

```bash
# 1. 查看 Pod 状态和事件
kubectl describe pod <pod-name>

# 2. 查看当前日志
kubectl logs <pod-name>

# 3. 查看上一次启动的日志
kubectl logs <pod-name> --previous

# 4. 进入容器调试（如果可能）
kubectl exec -it <pod-name> -- /bin/sh

# 5. 检查资源限制
kubectl describe pod <pod-name> | grep -A 5 "Limits"

# 常见原因：
# - 应用启动失败 → 检查日志，修复代码/配置
# - 内存不足 OOMKilled → 增加 memory limit
# - Liveness Probe 配置错误 → 调整探针
# - 缺少依赖/文件 → 检查镜像和挂载
```

---

### 场景 3：DNS 解析失败

**现象**：Pod 内无法解析 Service 名称，`nslookup` 超时

**排查步骤**：

```bash
# 1. 检查 CoreDNS Pod
kubectl get pods -n kube-system -l k8s-app=kube-dns

# 2. 检查 CoreDNS 日志
kubectl logs -n kube-system -l k8s-app=kube-dns

# 3. 从 Pod 测试 DNS
kubectl exec <pod> -- nslookup kubernetes.default
kubectl exec <pod> -- cat /etc/resolv.conf

# 4. 检查 Service DNS 记录
kubectl get svc kube-dns -n kube-system

# 5. 检查网络策略是否阻止 DNS
kubectl get networkpolicy -n <namespace>

# 6. 检查 CNI 插件状态
# 在节点上：
systemctl status kubelet
ls /etc/cni/net.d/

# 常见原因：
# - CoreDNS 未运行 → 检查资源/调度
# - NetworkPolicy 阻止 UDP 53 → 添加允许规则
# - CNI 故障 → 检查 CNI 插件
# - ndots 问题 → 使用 FQDN
```

## 10.4 综合实验：构建安全集群

### 实验目标

从零开始搭建一个符合以下安全标准的 Kubernetes 集群：

1. ✅ 禁用匿名认证
2. ✅ etcd 启用静态加密
3. ✅ 启用审计日志
4. ✅ Pod Security Standards: restricted（所有命名空间）
5. ✅ NetworkPolicy：默认拒绝所有入站流量
6. ✅ Falco 监控运行时
7. ✅ 镜像漏洞扫描（Trivy）
8. ✅ 准入策略：禁止特权容器、禁止 latest 标签、强制资源限制
9. ✅ RBAC：最小权限原则
10. ✅ kube-bench CIS 合规分数 > 90%

### 实验步骤大纲

```bash
# ===== 阶段 1：部署集群 =====
# 使用 kubeadm 部署 HA 集群（参考第 3 章）

# ===== 阶段 2：加固 API Server =====
# 编辑 /etc/kubernetes/manifests/kube-apiserver.yaml
# - 禁用匿名认证：--anonymous-auth=false
# - 启用审计日志：--audit-log-path=/var/log/audit.log
# - 配置 TLS：--tls-cipher-suites=...
# - 启用准入控制：--enable-admission-plugins=...

# ===== 阶段 3：加固 etcd =====
# 1. 创建 encryption-config.yaml
# 2. 配置 API Server 使用加密
# 3. 配置防火墙只允许 API Server 访问 2379
```

### 题目 11：Symlink 攻击防御

**场景**：防止容器通过符号链接攻击访问宿主机文件。

**任务**：
1. 解释 symlink 攻击的原理
2. 配置 Pod 阻止这类攻击（使用 `readOnlyRootFilesystem` + 限制 volume 类型）
3. 使用 Kyverno 策略限制 Volume 类型

<details>
<summary>参考答案</summary>

```yaml
# Kyverno 策略：限制允许的 Volume 类型
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: restrict-volumes
spec:
  validationFailureAction: Enforce
  rules:
  - name: allowed-volumes
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "只允许使用 emptyDir、configMap、secret、persistentVolumeClaim 和 projected 卷。"
      pattern:
        spec:
          volumes:
          - =(emptyDir): {}
          - =(configMap): {}
          - =(secret): {}
          - =(persistentVolumeClaim): {}
          - =(projected): {}
```

**symlink 攻击防御要点**：
1. `readOnlyRootFilesystem: true` 阻止向根文件系统写入恶意 symlink
2. 限制 Volume 类型，禁止 hostPath、emptyDir 的滥用路径
3. 使用 `allowPrivilegeEscalation: false` 阻止提权后创建 symlink
4. 运行非 root 用户，降低 symlink 攻击的影响面

## 10.8 企业安全巡检手册

### 每日巡检清单

```bash
#!/bin/bash
# daily-security-check.sh

echo "=== K8s 每日安全巡检 ==="
echo "日期: $(date)"

# 1. 检查异常 Pod
echo "--- 异常状态的 Pod ---"
kubectl get pods -A | grep -v Running | grep -v Completed

# 2. 检查高权限 RBAC
echo "--- cluster-admin 绑定 ---"
kubectl get clusterrolebinding -o json | \
  jq -r '.items[] | select(.roleRef.name=="cluster-admin") | 
    "\(.metadata.name): \(.subjects | map(.name) | join(", "))"'

# 3. 检查特权容器
echo "--- 特权容器 ---"
kubectl get pods -A -o json | \
  jq -r '.items[] | select(.spec.containers[]?.securityContext?.privileged==true) |
    "\(.metadata.namespace)/\(.metadata.name)"' | sort | uniq

# 4. 检查无 NetworkPolicy 的命名空间
echo "--- 无 NetworkPolicy 的命名空间 ---"
for ns in $(kubectl get ns -o name | cut -d/ -f2); do
  count=$(kubectl get networkpolicy -n "$ns" --no-headers 2>/dev/null | wc -l)
  if [ "$count" -eq 0 ] && [ "$ns" != "kube-system" ]; then
    echo "  $ns"
  fi
done

# 5. 检查镜像漏洞（Trivy 快速扫描）
echo "--- 高危镜像漏洞 ---"
kubectl get pods -A -o json | \
  jq -r '[.items[].spec.containers[].image] | unique | .[]' | \
  while read img; do
    trivy image --severity HIGH,CRITICAL --exit-code 0 --quiet "$img" 2>/dev/null | \
      grep -E "^(High|Critical)" | head -5
  done

# 6. 检查证书过期（30天内）
echo "--- 即将过期的证书 ---"
kubeadm certs check-expiration | grep -E "(EXPIRES|less than)"

echo "=== 巡检完成 ==="
```

## 10.9 红蓝对抗演练场景

### 场景 A：红队 - 容器逃逸

**目标**：从一个普通 Pod 逃逸到宿主机

**步骤**：
1. 扫描 Pod 中的敏感信息（Secret、环境变量）
2. 检查挂载的宿主机路径
3. 利用特权容器或 CAP_SYS_ADMIN 逃逸
4. 在宿主机上建立持久化后门

**蓝队防御**：
- Falco 告警："Privileged container started"
- 审计日志：追踪 exec 和 create 操作
- NetworkPolicy：阻止异常出站连接

### 场景 B：红队 - 横向移动

**目标**：从一个命名空间渗透到另一个命名空间

**步骤**：
1. 获取 ServiceAccount Token
2. 检查 RBAC 权限：`kubectl auth can-i --list`
3. 利用 overly permissive 的 RoleBinding
4. 访问其他命名空间的 Secret

**蓝队防御**：
- 审计日志：检测跨命名空间访问
- Kyverno：限制 Token 投影范围
- NetworkPolicy：命名空间间隔离

### 场景 C：蓝队 - 入侵检测与响应

**目标**：检测并响应模拟的容器逃逸攻击

**响应流程**：
```
1. 检测阶段
   └─ Falco 告警 → Slack 通知
   └─ 审计日志异常 → 自动标记

2. 分析阶段
   └─ 查看 Pod 安全上下文
   └─ 检查 RBAC 绑定
   └─ 分析网络流量（Cilium Hubble）

3. 遏制阶段
   └─ 隔离受影响 Pod（NetworkPolicy）
   └─ 撤销可疑 ServiceAccount Token
   └─ 封锁源 IP（Cilium 网络策略）

4. 根除阶段
   └─ 终止逃逸容器
   └─ 修复漏洞配置
   └─ 轮换凭证

5. 恢复阶段
   └─ 验证修复有效性
   └─ 重新部署受影响的 workload

6. 总结阶段
   └─ 编写事件报告
   └─ 更新防御策略
   └─ 开展团队复盘
```
